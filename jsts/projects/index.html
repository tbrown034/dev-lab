Project 01

Type-safe CLI Framework

A Commander-inspired CLI framework where command definitions, arguments, and flags are fully type-inferred. No runtime type checking needed -- TypeScript catches invalid flag combinations at compile time.

Template Literal Types Conditional Types Process.argv Parsing Plugin System Vitest
Architecture
process.argv → Tokenizer → Parser → Validator → Handler
CLI Definition (DSL) → Inferred Types → Type-safe Handlers

How I Built This

  • Template literal types parse flag strings like "--port <number>" into typed option records at compile time
  • Conditional types infer argument types from the definition DSL -- no manual type annotations needed
  • Plugin system lets users add custom commands, middleware, and output formatters via a Plugin interface
  • Tokenizer handles quoted strings, escaped characters, and double-dash separators correctly
  • Help text is auto-generated from the type definitions, including descriptions and default values
  • Full test suite with Vitest covering edge cases: missing args, unknown flags, conflicting options

What This Demonstrates

Advanced Generics Template Literal Types Type Inference Plugin Architecture Process Management Testing Strategy
Project 02

Real-time Data Pipeline

An event-driven data processing pipeline that ingests, transforms, and routes streaming data. Uses async generators for backpressure, typed channels for message passing, and a circuit breaker for fault tolerance.

AsyncGenerator Backpressure Circuit Breaker Typed Channels Worker Threads
Data Flow
Sources → Ingest Queue → Transform Pipeline
Transform Pipeline → Router → Sinks (DB / API / File)
Monitor ← Metrics Collector ← Health Checks

How I Built This

  • Async generators provide natural backpressure -- downstream consumers pull data at their own pace
  • Typed channels use discriminated unions so each message type has a known shape at compile time
  • Circuit breaker wraps external service calls, opening after N failures and retrying after a cooldown
  • Worker threads handle CPU-intensive transforms (JSON parsing, schema validation) off the main thread
  • Router uses a Map<MessageType, Sink[]> to fan-out messages to multiple typed destinations
  • Metrics collector tracks throughput, latency, and error rates per pipeline stage

What This Demonstrates

Async Generators Backpressure Error Recovery Worker Threads Observability Stream Processing
Project 03

Testing Library from Scratch

A minimal but complete testing framework with describe/it blocks, assertion matchers, async support, mocking, and parallel test execution. Proves deep understanding of how tools like Jest and Vitest actually work.

Test Runner Assertion Library Mock System Async Support Parallel Execution
Test Runner Architecture
Test Files → File Discovery → Parser (describe/it)
Test Suite Tree → Runner (Serial/Parallel) → Reporter
expect() → Matchers mock() → Spy Tracking

How I Built This

  • describe/it blocks build a tree structure at registration time, then the runner traverses it with proper beforeEach/afterEach lifecycle hooks
  • expect() returns a proxy-based matcher chain: expect(x).toBe(y), .toThrow(), .toHaveBeenCalledWith()
  • Mock system uses Proxy to record calls, arguments, and return values with full TypeScript inference
  • Parallel execution uses worker_threads, distributing test files across CPU cores for fast feedback
  • Reporter interface supports multiple output formats: dot, spec (verbose), and JSON for CI integration
  • Async tests have configurable timeouts with proper cleanup and unhandled rejection detection

What This Demonstrates

Proxy Tree Data Structures Worker Threads Error Handling API Design Metaprogramming
Project 04

Browser Extension with TypeScript

A productivity browser extension that tracks reading time, highlights and saves article excerpts, and syncs across devices via Chrome Storage API. Built with Manifest V3, service workers, and content scripts -- all fully typed.

Manifest V3 Service Workers Content Scripts Chrome Storage API Message Passing
Extension Architecture
Content Script ↔ Service Worker (Background) ↔ Popup UI
DOM Observer Chrome Storage Alarms API
Typed Messages → Discriminated Union Router → Handlers

How I Built This

  • Message passing between content script, service worker, and popup uses discriminated unions for type-safe routing
  • Content script uses MutationObserver to detect article content and calculate reading time from word count
  • Highlight system injects CSS classes and stores selection ranges in Chrome Storage for persistence
  • Service worker handles background sync, periodic cleanup via Alarms API, and cross-tab coordination
  • Popup UI is vanilla TypeScript with a custom reactive state system (no framework dependency)
  • Build pipeline uses esbuild for fast bundling, with separate entry points for each extension context

What This Demonstrates

Browser APIs Service Workers Message Passing DOM Manipulation State Management Build Tooling
