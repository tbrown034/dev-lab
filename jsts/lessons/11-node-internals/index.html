← Back to lessons

Node.js is V8 + libuv + a standard library. You've used the browser's event loop. Node has its own, with different phases and different capabilities. Understanding it is essential for server-side TypeScript.

The Mental Model: The Printing Press

If the browser is a newsroom (lesson 01), Node.js is the printing press. It handles the heavy machinery: reading files from disk, opening network connections, compressing data, managing databases. The newsroom (V8) writes the stories. The press (libuv) handles all the physical operations that involve waiting for real-world I/O.

Node's event loop has phases, unlike the browser's simpler model. Understanding these phases tells you exactly when your callbacks will fire.

The Node.js Event Loop (libuv)

Node's event loop runs in 6 phases, in order:

// The 6 phases of Node's event loop (each tick):

// 1. TIMERS
//    Execute callbacks from setTimeout() and setInterval()

// 2. PENDING CALLBACKS
//    Execute I/O callbacks deferred to the next iteration

// 3. IDLE / PREPARE
//    Internal use only

// 4. POLL
//    Retrieve new I/O events. Execute I/O callbacks.
//    Node will BLOCK here if nothing else is scheduled.

// 5. CHECK
//    Execute setImmediate() callbacks

// 6. CLOSE CALLBACKS
//    Execute close event callbacks (socket.on('close', ...))

// Between EVERY phase: drain the microtask queue
//   (process.nextTick first, then Promises)

setTimeout vs setImmediate

// In the main module, order is non-deterministic:
setTimeout(() => console.log("timeout"), 0);
setImmediate(() => console.log("immediate"));
// Could be either order! Depends on how long setup takes.

// Inside an I/O callback, setImmediate always fires first:
const fs = require('fs');
fs.readFile('/etc/hosts', () => {
  setTimeout(() => console.log("timeout"), 0);
  setImmediate(() => console.log("immediate"));
});
// Always: "immediate" then "timeout"
// Because after the poll phase, the check phase (setImmediate) runs
// before looping back to the timers phase (setTimeout)

process.nextTick vs queueMicrotask

// process.nextTick runs BEFORE other microtasks (Promise.then)
Promise.resolve().then(() => console.log("promise"));
process.nextTick(() => console.log("nextTick"));

// Output: "nextTick", "promise"
// nextTick has its own separate queue that drains first

Avoid recursive process.nextTick. Since nextTick runs between every event loop phase and drains completely, recursive nextTick calls will starve the event loop. I/O callbacks, timers, and everything else will wait indefinitely. Use setImmediate for recursive scheduling instead.

Streams

Streams are Node's most powerful abstraction. Instead of loading an entire file into memory, streams process data in chunks:

Without streams (bad for large files)
import { readFile, writeFile } from 'fs/promises';

// Loads ENTIRE file into memory
const data = await readFile('huge.csv', 'utf-8');
const processed = data.toUpperCase();
await writeFile('output.csv', processed);

// 2GB file = 2GB in memory
With streams (constant memory)
import { createReadStream, createWriteStream }
  from 'fs';
import { Transform } from 'stream';

const upper = new Transform({
  transform(chunk, enc, cb) {
    cb(null, chunk.toString().toUpperCase());
  }
});

createReadStream('huge.csv')
  .pipe(upper)
  .pipe(createWriteStream('output.csv'));
// Constant ~64KB memory usage

The 4 Stream Types

import { Readable, Writable, Transform, Duplex } from 'stream';

// 1. Readable — source of data
//    Examples: fs.createReadStream, http.IncomingMessage, process.stdin

// 2. Writable — destination for data
//    Examples: fs.createWriteStream, http.ServerResponse, process.stdout

// 3. Transform — reads AND writes, modifying data in transit
//    Examples: zlib.createGzip(), crypto.createCipher()

// 4. Duplex — reads AND writes independently (like a phone call)
//    Examples: net.Socket, WebSocket

Modern Stream API (async iterators)

import { createReadStream } from 'fs';
import { createInterface } from 'readline';

// Streams are async iterables!
const fileStream = createReadStream('articles.jsonl', 'utf-8');
const lines = createInterface({ input: fileStream });

for await (const line of lines) {
  const article = JSON.parse(line);
  console.log(article.title);
}

// Or use stream/consumers (Node 16+)
import { text, json, arrayBuffer } from 'stream/consumers';

const body = await text(response.body);   // Consume a ReadableStream as text
const data = await json(response.body);   // Consume as parsed JSON

Streams and async generators are the same pattern. In lesson 06, you learned that async generators yield values over time. Node streams do the same thing — that's why streams implement the async iterator protocol. for await (const chunk of stream) is the modern way to consume streams.

Worker Threads

JavaScript is single-threaded, but Node can spawn additional threads for CPU-intensive work:

// main.ts — the main thread
import { Worker, isMainThread, parentPort, workerData } from 'worker_threads';

if (isMainThread) {
  // Spawn a worker
  const worker = new Worker('./worker.ts', {
    workerData: { articles: largeDataset }
  });

  worker.on('message', (result) => {
    console.log('Worker finished:', result);
  });

  worker.on('error', (err) => {
    console.error('Worker error:', err);
  });
}

// worker.ts — runs on a separate thread
import { parentPort, workerData } from 'worker_threads';

const { articles } = workerData;

// CPU-intensive work that won't block the main event loop
const processed = articles.map(article => ({
  ...article,
  wordCount: article.body.split(/\s+/).length,
  readingTime: Math.ceil(article.body.split(/\s+/).length / 200),
}));

parentPort?.postMessage(processed);

When to Use Workers

// USE workers for:
// - CPU-intensive computation (parsing, compression, image processing)
// - Heavy data transformation
// - Cryptographic operations

// DON'T use workers for:
// - I/O operations (the event loop handles these efficiently)
// - Simple async tasks (network requests, file reads)
// - Low-overhead operations (the cost of spawning a worker outweighs the benefit)

SharedArrayBuffer allows workers to share memory directly (no copying). Combined with Atomics for synchronization, this enables high-performance parallel processing. But it's advanced territory — message passing with postMessage is simpler and sufficient for most use cases.

Knowledge Check

1. Why does setImmediate always run before setTimeout(fn, 0) inside an I/O callback?

Inside an I/O callback, you're in the poll phase. After the poll phase completes, the event loop moves to the check phase (where setImmediate callbacks run). Only after completing the full loop does it reach the timers phase again (where setTimeout callbacks run). So setImmediate fires in the current tick; setTimeout fires in the next tick.

2. Why use streams instead of readFile for large files?

readFile loads the entire file into memory as a single buffer. For a 2GB file, that's 2GB of memory. Streams process data in chunks (typically 64KB), so memory usage stays constant regardless of file size. Streams also start processing immediately — you don't wait for the entire file to load before working on the first chunk.

3. When should you use worker threads vs the event loop?

Use worker threads for CPU-bound work (computation, parsing, compression) that would block the event loop. Use the event loop for I/O-bound work (network, filesystem, database). The event loop handles I/O extremely efficiently via non-blocking callbacks. Workers add overhead (thread creation, message serialization) that only pays off for heavy computation.
