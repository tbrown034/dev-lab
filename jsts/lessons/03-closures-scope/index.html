03 Closures & Scope — The Backpack Every Function Carries

← Back to lessons

Closures are the single most important concept in JavaScript. They power React hooks, the module pattern, event handlers, callbacks, and currying. If you understand closures, you understand why JavaScript works the way it does.

The Mental Model: The Reporter's Backpack

When a journalist leaves a beat, they take their contacts with them. The beat might be over, the newsroom might have moved on, but the reporter still has the phone numbers, the notes, the context from that assignment in their backpack.

A closure works the same way. When a function is created inside another function, it packs a backpack with all the variables from its birth environment. Even after the outer function returns and its scope "disappears," the inner function still has access to everything it packed.

A closure is not a special syntax. It's an automatic behavior. Every function in JavaScript is a closure. The question is: does it reference variables from an outer scope? If yes, those variables survive as long as the function does.

Lexical Scope: Where You Were Born Matters

Lexical means "relating to the source code text." Lexical scope means: a function's scope is determined by where it was written, not where it's called.

const newsroom = "CNN";

function getReporter() {
  const beat = "politics";

  function report() {
    // report() can access both `beat` and `newsroom`
    // because of where it was WRITTEN, not where it's called
    console.log(`${beat} reporter at ${newsroom}`);
  }

  return report;
}

const myReporter = getReporter();
myReporter();  // "politics reporter at CNN"

// getReporter() has already returned
// its local scope should be "gone"
// but report() still has `beat` in its backpack

The Scope Chain

When JavaScript looks up a variable, it follows a chain:

  1. Check the current function's local scope
  2. Check the enclosing function's scope
  3. Check the next enclosing scope... and so on
  4. Check the global scope
  5. If not found anywhere: ReferenceError
const global = "I'm global";

function outer() {
  const outerVar = "I'm in outer";

  function middle() {
    const middleVar = "I'm in middle";

    function inner() {
      // inner can see: middleVar, outerVar, global
      console.log(global, outerVar, middleVar);
    }

    inner();
  }

  middle();
  // outer CANNOT see middleVar (scope only goes UP, not DOWN)
}

Closures in Practice

1. Data Privacy (The Module Pattern)

Before ES modules, closures were the ONLY way to create private variables in JavaScript:

function createCounter() {
  let count = 0;  // private — no one can access this directly

  return {
    increment() { count++; },
    decrement() { count--; },
    getCount()  { return count; }
  };
}

const counter = createCounter();
counter.increment();
counter.increment();
counter.getCount();  // 2
counter.count;       // undefined — can't access directly

React hooks are closures. When you call useState(0), React returns a getter (the value) and a setter. The actual state value is stored in a closure inside React's fiber tree. You can't access it directly — only through the functions React gives you. Same pattern as createCounter() above.

2. Function Factories

function createMultiplier(factor) {
  return function(number) {
    return number * factor;  // factor is in the backpack
  };
}

const double = createMultiplier(2);
const triple = createMultiplier(3);

double(5);  // 10
triple(5);  // 15

// Each returned function has its OWN backpack
// with its OWN value of `factor`

3. Event Handlers

function setupButton(buttonId, message) {
  const button = document.getElementById(buttonId);

  button.addEventListener('click', function() {
    // This handler closes over `message`
    // Even though setupButton has already returned
    alert(message);
  });
}

setupButton('btn-1', 'Hello!');
setupButton('btn-2', 'Goodbye!');

Hoisting: The Pre-Publication Edit

Before your code runs, JavaScript makes a first pass and "hoists" declarations to the top of their scope. But different declarations hoist differently:

var — hoisted and initialized to undefined
console.log(x); // undefined
var x = 5;
console.log(x); // 5

// JavaScript sees this as:
// var x = undefined;
// console.log(x);
// x = 5;
let/const — hoisted but NOT initialized (TDZ)
console.log(y); // ReferenceError!
let y = 5;

// The "Temporal Dead Zone" (TDZ):
// y exists in scope but can't
// be accessed until declaration

Function Hoisting

// Function DECLARATIONS are fully hoisted
sayHello();  // Works!
function sayHello() { console.log("Hello"); }

// Function EXPRESSIONS are NOT
sayGoodbye();  // TypeError: sayGoodbye is not a function
const sayGoodbye = function() { console.log("Goodbye"); };

// Arrow functions behave like function expressions
greet();  // TypeError
const greet = () => console.log("Hi");

Rule of thumb: Use const for everything. Use let only when you need to reassign. Never use var. This eliminates hoisting confusion entirely.

The Classic Loop Closure Problem

This is the most common closure bug, and understanding it proves you understand closures:

// THE BUG
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// Output: 3, 3, 3 (not 0, 1, 2!)

// WHY: var is function-scoped. All three callbacks
// close over the SAME `i`. By the time they run,
// the loop is done and i === 3.
Fix 1: use let (block scope)
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// Output: 0, 1, 2
// Each iteration gets its
// own block-scoped `i`
Fix 2: IIFE (the old way)
for (var i = 0; i < 3; i++) {
  (function(j) {
    setTimeout(() => console.log(j), 100);
  })(i);
}
// Output: 0, 1, 2
// Each IIFE creates a new
// scope with its own `j`

This is why let was invented. The loop closure problem was so common and confusing that TC39 (the JavaScript standards committee) introduced block scoping specifically to solve it. Understanding the why behind let vs var is more valuable than just knowing "use let."

Stale Closures (The React Hook Trap)

Closures capture values at creation time. If the value changes later, the closure still has the old value:

// In React, this is the "stale closure" problem
function Timer() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      // This closure captured `count` as 0
      // It will ALWAYS log 0, never the updated value
      console.log(count);
    }, 1000);
    return () => clearInterval(id);
  }, []); // Empty deps: effect runs once, closure captures once

  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}

// FIX: Use the functional updater or add count to deps
useEffect(() => {
  const id = setInterval(() => {
    setCount(c => c + 1); // Functional updater — no closure needed
  }, 1000);
  return () => clearInterval(id);
}, []);

IIFE: Immediately Invoked Function Expressions

// Classic IIFE pattern
(function() {
  const secret = "hidden";
  console.log(secret);  // "hidden"
})();

// secret is not accessible here
console.log(secret);  // ReferenceError

// Modern alternative: block scope
{
  const secret = "hidden";
  console.log(secret);  // "hidden"
}
// secret is not accessible here either

IIFEs are mostly historical. Before ES modules and let/const, IIFEs were the only way to create private scope. You'll see them in legacy code and library source. In modern code, use modules and block scope instead.

Knowledge Check

1. What does this print and why?

function outer() {
  let x = 10;
  function inner() { console.log(x); }
  x = 20;
  return inner;
}
outer()();
20. The closure captures a reference to x, not a snapshot of its value. By the time inner() runs, x has been reassigned to 20.

2. Why does let fix the classic loop closure problem but var doesn't?

var is function-scoped — one variable shared across all iterations. let is block-scoped — each loop iteration creates a new binding. Each closure captures a different i.

3. In React, why does an empty dependency array in useEffect sometimes cause stale data?

The effect callback is a closure that captures state values at the time the effect runs. With [], the effect runs once (on mount), so the closure always has the initial state values. When state updates, the closure still holds the old values because it was never re-created.

Exercise: Build a Closure-Based State Manager

Create a createStore(initialState) function that returns an object with:

  • getState() — returns the current state
  • setState(newState) — merges new state with current state
  • subscribe(listener) — registers a callback; returns an unsubscribe function

This is a simplified Redux. All state should be private (only accessible through closures).

const store = createStore({ count: 0, name: "Trevor" });
const unsub = store.subscribe(state => console.log(state));
store.setState({ count: 1 }); // logs: { count: 1, name: "Trevor" }
unsub(); // unsubscribe
store.setState({ count: 2 }); // no log (unsubscribed)
store.getState(); // { count: 2, name: "Trevor" }
