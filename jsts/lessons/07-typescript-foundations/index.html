07 TypeScript Foundations — The Fact-Checker

← Back to lessons

TypeScript is a static analysis tool, not a runtime. It checks your code before it runs and then disappears. Understanding this boundary — what types can and cannot do — is the foundation for everything in lessons 08 and 09.

The Mental Model: The Fact-Checking Desk

Every newsroom has a fact-checking desk. Reporters write stories (JavaScript code). The fact-checkers (TypeScript compiler) review every claim before publication. They catch errors like "you said this source is a doctor, but they're actually a dentist" — or in code terms, "you said this is a string, but it's actually number | undefined."

But here's the critical thing: fact-checkers don't change the story. They flag problems and suggest fixes. The published article (compiled JavaScript) has no trace of the fact-checking process. Types are erased at compile time.

Types are erased. This is the single most important thing to understand about TypeScript. interface User { name: string } does not exist at runtime. You cannot check instanceof User. You cannot log the type. All type information is stripped during compilation. TypeScript is a development-time tool only.

Type Annotations

// Primitives
const name: string = "Trevor";
const age: number = 30;
const active: boolean = true;

// Arrays
const tags: string[] = ["JS", "TS"];
const scores: Array<number> = [98, 87, 92];  // generic syntax

// Objects
const user: { name: string; age: number } = { name: "Trevor", age: 30 };

// Functions
function greet(name: string): string {
  return `Hello, ${name}`;
}

// Arrow functions
const add = (a: number, b: number): number => a + b;

// Optional parameters
function search(query: string, limit?: number): void {
  // limit is number | undefined
}

Type Inference: Let TypeScript Work

// TypeScript infers the type — no annotation needed
const name = "Trevor";       // type: "Trevor" (literal type!)
let name2 = "Trevor";        // type: string (wider because let can be reassigned)
const numbers = [1, 2, 3];   // type: number[]
const mixed = [1, "two"];    // type: (string | number)[]

// Rule: annotate function parameters and return types.
// Let inference handle local variables.
function getUser(id: number): User {
  // TypeScript infers types of variables inside the function
  const name = fetchName(id);  // inferred from fetchName's return type
  return { id, name };
}

const vs let inference: const x = "hello" gives type "hello" (literal). let x = "hello" gives type string (wide). This is because const can never be reassigned, so TypeScript narrows to the exact value. This distinction matters when passing values to functions that expect specific literals.

interface vs type

The most common TypeScript question. Here are the real differences:

interface
interface User {
  name: string;
  age: number;
}

// Extends via extends
interface Admin extends User {
  role: "admin";
}

// Declaration merging (UNIQUE)
interface User {
  email: string;
}
// User now has name, age, email

// Only for object shapes
type
type User = {
  name: string;
  age: number;
};

// Extends via intersection
type Admin = User & {
  role: "admin";
};

// NO declaration merging
// type User = { email: string }
// Error: Duplicate identifier

// Can represent ANY type
type ID = string | number;
type Fn = () => void;

The practical rule: Use interface for object shapes that might be extended (API responses, component props). Use type for unions, intersections, mapped types, or anything that isn't purely an object shape. In a React/Next.js codebase, interface for props, type for everything else works well.

Unions and Intersections

Union Types (OR)

type Status = "draft" | "published" | "archived";
type ID = string | number;

function printId(id: string | number) {
  // TypeScript only allows methods common to BOTH types
  console.log(id.toString());  // OK (both have toString)
  console.log(id.toUpperCase());  // Error! number doesn't have toUpperCase
}

Intersection Types (AND)

type HasName = { name: string };
type HasAge = { age: number };
type Person = HasName & HasAge;

// Person must have BOTH name AND age
const trevor: Person = { name: "Trevor", age: 30 };  // OK
const bad: Person = { name: "Trevor" };  // Error: missing age

Type Narrowing

Narrowing is how you go from a wide type to a specific type. TypeScript understands control flow:

function process(input: string | number | null) {
  // typeof guard
  if (typeof input === "string") {
    // TypeScript KNOWS input is string here
    return input.toUpperCase();
  }

  // Truthiness guard
  if (!input) {
    // TypeScript KNOWS input is null or 0 here
    return "empty";
  }

  // TypeScript KNOWS input is number here (eliminated string and null)
  return input.toFixed(2);
}

Discriminated Unions (The Most Useful Pattern)

type Article = {
  type: "article";
  title: string;
  body: string;
};

type Video = {
  type: "video";
  title: string;
  duration: number;
};

type Content = Article | Video;

function render(content: Content) {
  switch (content.type) {
    case "article":
      // TypeScript knows: content is Article
      return `<p>${content.body}</p>`;
    case "video":
      // TypeScript knows: content is Video
      return `<video duration="${content.duration}" />`;
  }
}

You use this in Next.js already. Server actions return { success: true, data: T } | { success: false, error: string }. The success field is the discriminant. Narrowing on it gives you type-safe access to either data or error. Same pattern, different scale.

Custom Type Guards

// The `is` keyword tells TypeScript what a boolean check means for types
function isArticle(content: Content): content is Article {
  return content.type === "article";
}

function process(content: Content) {
  if (isArticle(content)) {
    // TypeScript knows: content is Article
    console.log(content.body);
  }
}

Essential Utility Types

interface User {
  id: number;
  name: string;
  email: string;
  role: "admin" | "user";
}

// Partial — all properties optional
type UpdateUser = Partial<User>;
// { id?: number; name?: string; email?: string; role?: "admin" | "user" }

// Required — all properties required
type FullUser = Required<User>;

// Pick — select specific properties
type UserPreview = Pick<User, "id" | "name">;
// { id: number; name: string }

// Omit — exclude specific properties
type CreateUser = Omit<User, "id">;
// { name: string; email: string; role: "admin" | "user" }

// Record — object with specific key and value types
type UserMap = Record<string, User>;

// ReturnType — extract a function's return type
type ApiResponse = ReturnType<typeof fetchUser>;

// Parameters — extract function parameter types as a tuple
type FetchParams = Parameters<typeof fetch>;

Next.js pattern: Omit<ComponentProps, 'className'> is how you create wrapper components that forward all props except the ones you override. Partial<FormData> is how you type form state that fills in incrementally.

Strict Mode: The Non-Negotiable

// tsconfig.json — always enable strict
{
  "compilerOptions": {
    "strict": true
    // This enables ALL of these:
    // strictNullChecks — null/undefined are not assignable to other types
    // noImplicitAny — error on implicit `any` types
    // strictFunctionTypes — stricter function type checking
    // strictBindCallApply — strict bind/call/apply typing
    // noImplicitThis — error when `this` has an implicit `any` type
    // alwaysStrict — emit "use strict" in JS output
  }
}

Never disable strict. Non-strict TypeScript is barely better than JavaScript. The null checks alone prevent entire categories of runtime errors. If you inherit a non-strict codebase, migrating to strict mode is worth the effort.

Knowledge Check

1. Why does const x = "hello" have type "hello" but let x = "hello" has type string?

const can never be reassigned, so TypeScript narrows to the literal type "hello". let can be reassigned to any string, so TypeScript widens to string. This is called "literal widening."

2. When would you use interface over type?

Use interface when: (1) defining object shapes that might be extended by other interfaces, (2) you need declaration merging (rare, but used when augmenting third-party types), (3) defining component props in React. Use type for unions, intersections, mapped types, and non-object types.

3. What happens to TypeScript types at runtime?

They are completely erased. All type annotations, interfaces, and type aliases are removed during compilation. The output is plain JavaScript with no type information. This means you cannot use instanceof with interfaces, and you cannot inspect types at runtime. This is called "type erasure."

Exercise: Type a REST API Layer

Create types for a blog API. The goal is to make invalid states unrepresentable:

// 1. Define a discriminated union for API responses
type ApiResponse<T> =
  | { status: "success"; data: T; }
  | { status: "error"; error: string; code: number; };

// 2. Define the data models
interface Article { id: number; title: string; body: string; status: "draft" | "published"; }
interface Author  { id: number; name: string; articles: Article[]; }

// 3. Type these functions so TypeScript catches all errors:
async function fetchArticle(id: number): Promise<ApiResponse<Article>> { ... }
async function updateArticle(id: number, data: Partial<Omit<Article, "id">>): Promise<ApiResponse<Article>> { ... }

// 4. Write a consumer that narrows the response correctly
const result = await fetchArticle(1);
if (result.status === "success") {
  console.log(result.data.title);  // TypeScript knows data exists
} else {
  console.error(result.error);     // TypeScript knows error exists
}
