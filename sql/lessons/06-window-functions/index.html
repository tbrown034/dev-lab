06 Data Modification — INSERT, UPDATE, DELETE

← Back to lessons

You've been reading data for five lessons. Now it's time to write, change, and remove it. In Prisma, you use .create(), .update(), and .delete(). The SQL behind those methods is straightforward once you know the patterns.

The Mental Model — CRUD in SQL

Every web app does four things with data: Create, Read, Update, Delete. You've mastered Read (SELECT). The other three are simpler than SELECT — they have fewer options and fewer clauses.

OperationSQLPrismaRisk Level
CreateINSERT INTO.create()Low — adds data
ReadSELECT.findMany()None — just reads
UpdateUPDATE ... SET.update()Medium — changes data
DeleteDELETE FROM.delete()High — removes data

The golden rule of data modification: Always use a WHERE clause with UPDATE and DELETE. Without WHERE, you affect EVERY row in the table. In production, run a SELECT with the same WHERE clause first to preview which rows you'll change.

INSERT — Adding Data

Single Row Insert

Prisma
const article = await prisma.article.create({
  data: {
    title: 'Breaking: City Council Vote',
    body: 'The city council voted...',
    category: 'politics',
    author_id: 1,
    published: true,
  },
});
// Returns the created record with id
SQL
INSERT INTO articles
  (title, body, category, author_id, published)
VALUES
  ('Breaking: City Council Vote',
   'The city council voted...',
   'politics',
   1,
   true);

-- To get the created record back:
INSERT INTO articles (title, body, category)
VALUES ('...', '...', 'politics')
RETURNING *;

RETURNING * is a PostgreSQL feature that returns the inserted row, including auto-generated columns like id and created_at. This is what Prisma uses behind the scenes when .create() returns the full object.

-- Return only specific columns
INSERT INTO articles (title, body, category)
VALUES ('New Story', 'Content here...', 'culture')
RETURNING id, title, created_at;

Multiple Row Insert

Prisma
await prisma.article.createMany({
  data: [
    { title: 'Story A', body: '...',
      category: 'politics' },
    { title: 'Story B', body: '...',
      category: 'sports' },
    { title: 'Story C', body: '...',
      category: 'culture' },
  ],
});
SQL
INSERT INTO articles
  (title, body, category)
VALUES
  ('Story A', '...', 'politics'),
  ('Story B', '...', 'sports'),
  ('Story C', '...', 'culture')
RETURNING id, title;

Multiple VALUES rows in one INSERT is much faster than separate INSERT statements. The database handles them as a single operation.

INSERT...SELECT — Insert from a Query

This is a powerful pattern Prisma can't do directly. You can insert rows based on the result of a SELECT query — no VALUES clause needed:

-- Copy all published articles into an archive table
INSERT INTO archived_articles (title, body, category, original_id)
SELECT title, body, category, id
FROM articles
WHERE published = true
  AND published_at < '2023-01-01';

-- Create a summary row from aggregated data
INSERT INTO daily_stats (date, article_count, total_views)
SELECT
  CURRENT_DATE,
  COUNT(*),
  SUM(views)
FROM articles
WHERE published_at::DATE = CURRENT_DATE;

INSERT...SELECT is a data migration superpower. Need to copy data between tables, transform data, or populate a reporting table? This does it in one query, entirely on the database server. No need to pull data into your app and push it back.

UPDATE — Changing Existing Data

Basic UPDATE

Prisma
// Update one record
await prisma.article.update({
  where: { id: 5 },
  data: {
    title: 'Updated Headline',
    published: true,
    views: { increment: 1 },
  },
});
SQL
-- Update one record
UPDATE articles
SET
  title = 'Updated Headline',
  published = true,
  views = views + 1
WHERE id = 5;

The pattern: UPDATE table SET column = value WHERE condition. You can SET multiple columns, separated by commas.

Update Many Rows

Prisma
// Update many
await prisma.article.updateMany({
  where: {
    published: false,
    created_at: { lt: cutoffDate },
  },
  data: {
    status: 'archived',
  },
});
SQL
-- Update many
UPDATE articles
SET status = 'archived'
WHERE published = false
  AND created_at < '2023-06-01';

-- See how many rows were affected:
-- PostgreSQL shows "UPDATE 42" meaning
-- 42 rows were changed
-- Preview what will be updated FIRST
SELECT id, title, status
FROM articles
WHERE published = false
  AND created_at < '2023-06-01';
-- Check the results, then run the UPDATE

Always preview with SELECT first. Before running an UPDATE, write a SELECT with the same WHERE clause to see which rows will be affected. This is especially important in production databases where mistakes are expensive.

UPDATE with Expressions

SET can use expressions, not just literal values:

-- Increment views
UPDATE articles SET views = views + 1 WHERE id = 5;

-- Apply a 10% raise
UPDATE employees SET salary = salary * 1.10 WHERE department = 'engineering';

-- Set based on another column
UPDATE articles SET slug = LOWER(REPLACE(title, ' ', '-')) WHERE slug IS NULL;

-- Set using CASE
UPDATE articles
SET priority = CASE
  WHEN views > 10000 THEN 'high'
  WHEN views > 1000  THEN 'medium'
  ELSE 'low'
END;

UPDATE with RETURNING

-- Get the updated rows back
UPDATE articles
SET published = true,
    published_at = NOW()
WHERE id = 5
RETURNING id, title, published, published_at;

This is what Prisma's .update() uses to return the updated record.

UPDATE with JOIN

Sometimes you need to update rows based on data from another table. PostgreSQL uses a FROM clause for this:

-- Update articles: set the beat column based on the author's beat
UPDATE articles a
SET beat = au.beat
FROM authors au
WHERE a.author_id = au.id
  AND a.beat IS NULL;

-- Give a views boost to all articles by top-performing authors
UPDATE articles
SET views = views + 500
FROM (
  SELECT author_id
  FROM articles
  GROUP BY author_id
  HAVING SUM(views) > 10000
) top_authors
WHERE articles.author_id = top_authors.author_id;

PostgreSQL UPDATE...FROM is non-standard. Standard SQL uses UPDATE ... SET ... FROM ... WHERE ... syntax, but other databases use different patterns (MySQL uses UPDATE table1 JOIN table2). This is one area where SQL differs across databases.

DELETE — Removing Data

Basic DELETE

Prisma
// Delete one
await prisma.article.delete({
  where: { id: 5 },
});

// Delete many
await prisma.article.deleteMany({
  where: {
    published: false,
    created_at: { lt: cutoffDate },
  },
});
SQL
-- Delete one
DELETE FROM articles
WHERE id = 5;

-- Delete many
DELETE FROM articles
WHERE published = false
  AND created_at < '2023-01-01';
-- Delete with RETURNING (see what was deleted)
DELETE FROM articles
WHERE id = 5
RETURNING id, title;

-- Delete ALL rows (use with extreme caution)
DELETE FROM articles;
-- This is different from DROP TABLE:
-- DELETE removes rows but keeps the table structure
-- DROP TABLE removes the entire table

DELETE is permanent. There's no undo, no recycle bin. Once a row is deleted, it's gone (unless you have backups or are inside a transaction you can ROLLBACK). In production, prefer "soft deletes" — set a deleted_at timestamp instead of actually deleting the row.

DELETE with Subqueries

-- Delete articles by authors who are no longer active
DELETE FROM articles
WHERE author_id IN (
  SELECT id FROM authors WHERE active = false
);

-- Delete articles that have no comments
DELETE FROM articles a
WHERE NOT EXISTS (
  SELECT 1 FROM comments c WHERE c.article_id = a.id
);

Cascading Deletes

When you delete a row that other rows reference (via foreign key), what happens to those dependent rows? This is defined when creating the table:

-- When creating the table, define cascade behavior
CREATE TABLE articles (
  id SERIAL PRIMARY KEY,
  title TEXT NOT NULL,
  author_id INTEGER REFERENCES authors(id) ON DELETE CASCADE
);

-- ON DELETE CASCADE: deleting an author automatically deletes their articles
-- ON DELETE SET NULL: deleting an author sets author_id to NULL on their articles
-- ON DELETE RESTRICT: prevents deleting an author who has articles (default)
Prisma schema
model Article {
  id       Int    @id @default(autoincrement())
  title    String
  author   Author @relation(fields: [authorId],
    references: [id],
    onDelete: Cascade)
  authorId Int
}
SQL equivalent
CREATE TABLE articles (
  id SERIAL PRIMARY KEY,
  title TEXT NOT NULL,
  author_id INTEGER
    REFERENCES authors(id)
    ON DELETE CASCADE
);
ON DELETE ActionBehaviorPrisma Equivalent
CASCADEDelete related rows automaticallyonDelete: Cascade
SET NULLSet foreign key to NULLonDelete: SetNull
RESTRICTPrevent deletion if references existonDelete: Restrict
SET DEFAULTSet foreign key to its default valueonDelete: SetDefault
NO ACTIONSame as RESTRICT (check deferred)Default behavior

UPSERT — INSERT or UPDATE

An "upsert" inserts a row if it doesn't exist, or updates it if it does. This is common when syncing data from external sources — you don't know if the record already exists.

Prisma
await prisma.author.upsert({
  where: { email: 'sarah@news.com' },
  create: {
    email: 'sarah@news.com',
    name: 'Sarah Chen',
    beat: 'politics',
  },
  update: {
    name: 'Sarah Chen',
    beat: 'politics',
  },
});
SQL
INSERT INTO authors (email, name, beat)
VALUES ('sarah@news.com', 'Sarah Chen', 'politics')
ON CONFLICT (email)
DO UPDATE SET
  name = EXCLUDED.name,
  beat = EXCLUDED.beat;

Let's break down the SQL:

  • INSERT INTO ... VALUES ... — try to insert the row
  • ON CONFLICT (email) — if the email already exists (conflict on the unique constraint)
  • DO UPDATE SET ... — update those columns instead
  • EXCLUDED — refers to the values that were about to be inserted
-- If you want to do nothing on conflict (skip the row):
INSERT INTO authors (email, name, beat)
VALUES ('sarah@news.com', 'Sarah Chen', 'politics')
ON CONFLICT (email) DO NOTHING;

-- Upsert with RETURNING
INSERT INTO authors (email, name, beat)
VALUES ('sarah@news.com', 'Sarah Chen', 'politics')
ON CONFLICT (email)
DO UPDATE SET
  name = EXCLUDED.name,
  beat = EXCLUDED.beat
RETURNING *;

EXCLUDED is a special table. Inside ON CONFLICT DO UPDATE, EXCLUDED refers to the row that was attempted to be inserted. You can mix EXCLUDED values with existing values: SET views = authors.views + EXCLUDED.views to increment instead of replace.

Bulk Upsert

-- Upsert multiple rows at once
INSERT INTO daily_stats (date, category, article_count, total_views)
VALUES
  ('2024-03-15', 'politics', 12, 45000),
  ('2024-03-15', 'sports', 8, 28000),
  ('2024-03-15', 'tech', 5, 62000)
ON CONFLICT (date, category)
DO UPDATE SET
  article_count = EXCLUDED.article_count,
  total_views = EXCLUDED.total_views;

Transactions — All or Nothing

When you need multiple operations to succeed or fail together, use a transaction. Classic example: transferring money between accounts — you need both the debit and credit to happen, or neither.

Prisma
await prisma.$transaction([
  prisma.article.update({
    where: { id: 5 },
    data: { published: true },
  }),
  prisma.notification.create({
    data: {
      message: 'Article published!',
      article_id: 5,
    },
  }),
]);
SQL
BEGIN;

UPDATE articles
SET published = true
WHERE id = 5;

INSERT INTO notifications
  (message, article_id)
VALUES
  ('Article published!', 5);

COMMIT;
-- If anything fails, use ROLLBACK
-- instead of COMMIT

BEGIN starts the transaction. COMMIT saves all changes. ROLLBACK undoes everything since BEGIN. If the database crashes between BEGIN and COMMIT, all changes are automatically rolled back.

-- Practical example: publishing an article
BEGIN;

-- 1. Mark article as published
UPDATE articles
SET published = true, published_at = NOW()
WHERE id = 5;

-- 2. Update author's article count cache
UPDATE authors
SET published_count = published_count + 1
WHERE id = (SELECT author_id FROM articles WHERE id = 5);

-- 3. Create audit log entry
INSERT INTO audit_log (action, table_name, record_id, timestamp)
VALUES ('publish', 'articles', 5, NOW());

-- All good? Save everything
COMMIT;

Practical — Managing Newsroom Data

1. Onboarding a New Author

-- Insert the author and get their ID back
INSERT INTO authors (name, email, beat, hire_date)
VALUES ('Alex Kim', 'alex@news.com', 'technology', CURRENT_DATE)
RETURNING id;
-- Returns: id = 5

-- Assign them to their first story
UPDATE articles
SET author_id = 5, status = 'in_progress'
WHERE id = 42 AND author_id IS NULL
RETURNING title;

2. End-of-Day Content Report Update

-- Upsert daily stats for each category
INSERT INTO daily_stats (date, category, article_count, total_views)
SELECT
  CURRENT_DATE,
  category,
  COUNT(*),
  SUM(views)
FROM articles
WHERE published = true
  AND published_at::DATE = CURRENT_DATE
GROUP BY category
ON CONFLICT (date, category)
DO UPDATE SET
  article_count = EXCLUDED.article_count,
  total_views = EXCLUDED.total_views;

3. Archiving Old Drafts

BEGIN;

-- Move old drafts to archive table
INSERT INTO archived_articles (title, body, category, author_id, created_at)
SELECT title, body, category, author_id, created_at
FROM articles
WHERE published = false
  AND created_at < CURRENT_DATE - INTERVAL '90 days';

-- Delete them from the main table
DELETE FROM articles
WHERE published = false
  AND created_at < CURRENT_DATE - INTERVAL '90 days';

COMMIT;

4. Reassigning an Author's Articles

-- When an author leaves, reassign their articles
UPDATE articles
SET author_id = 2  -- reassign to Sarah
WHERE author_id = 5  -- from Alex
  AND published = false;  -- only unpublished ones

-- Soft-delete the departing author
UPDATE authors
SET active = false, deactivated_at = NOW()
WHERE id = 5;

Quick Check

1. What does RETURNING * do?

Returns the affected rows (inserted, updated, or deleted) as a result set. PostgreSQL-specific. It's how Prisma's .create() and .update() return the full record including auto-generated fields like id.

2. What does ON CONFLICT DO NOTHING do?

If the INSERT would violate a unique constraint, it silently skips the row instead of throwing an error. Useful for idempotent imports where you want to insert only new records.

3. What does EXCLUDED refer to in an ON CONFLICT clause?

EXCLUDED is a special table that refers to the row that was proposed for insertion but caused the conflict. You use it to access the new values: SET name = EXCLUDED.name updates the column with the value that was attempted to be inserted.

4. What's the difference between ON DELETE CASCADE and ON DELETE SET NULL?

CASCADE automatically deletes the dependent rows (e.g., deleting an author deletes all their articles). SET NULL keeps the dependent rows but sets their foreign key to NULL (articles remain but with no author).

5. Why should you use transactions for multi-step operations?

Transactions guarantee atomicity: either all operations succeed or none of them do. Without a transaction, if step 2 of 3 fails, step 1 has already been committed and you're left with inconsistent data.

Cheat Sheet

-- INSERT single row
INSERT INTO table (col1, col2) VALUES ('val1', 'val2');
INSERT INTO table (col1, col2) VALUES ('val1', 'val2') RETURNING *;

-- INSERT multiple rows
INSERT INTO table (col1, col2)
VALUES ('a', 'b'), ('c', 'd'), ('e', 'f');

-- INSERT from SELECT
INSERT INTO target (col1, col2)
SELECT col1, col2 FROM source WHERE condition;

-- UPDATE
UPDATE table SET col = 'value' WHERE id = 1;
UPDATE table SET col = col + 1 WHERE condition;
UPDATE table SET col = 'value' WHERE id = 1 RETURNING *;

-- UPDATE with JOIN (PostgreSQL)
UPDATE table1 SET col = t2.col
FROM table2 t2 WHERE table1.id = t2.ref_id;

-- DELETE
DELETE FROM table WHERE id = 1;
DELETE FROM table WHERE condition RETURNING *;

-- UPSERT (INSERT ON CONFLICT)
INSERT INTO table (col1, col2) VALUES ('a', 'b')
ON CONFLICT (col1) DO UPDATE SET col2 = EXCLUDED.col2;
-- or
ON CONFLICT (col1) DO NOTHING;

-- CASCADE options (on CREATE TABLE)
REFERENCES other_table(id) ON DELETE CASCADE
REFERENCES other_table(id) ON DELETE SET NULL
REFERENCES other_table(id) ON DELETE RESTRICT

-- TRANSACTIONS
BEGIN;
  -- multiple statements
COMMIT;   -- save all changes
ROLLBACK; -- undo all changes
