04 Aggregation & GROUP BY — Crunching Numbers

← Back to lessons

This is where SQL becomes a data analysis superpower. Aggregation turns thousands of rows into meaningful summaries: totals, averages, counts. It's the difference between "here are 50,000 rows" and "here's what they mean."

The Mental Model — Reduce, Not Map

You know Array.map() — it transforms each item. Aggregation is more like Array.reduce() — it takes many items and boils them down to one value. COUNT takes 1000 rows and gives you the number 1000. AVG takes all the salaries and gives you one average.

JavaScript reduce
// Count all articles
articles.length;

// Sum all views
articles.reduce((sum, a) => sum + a.views, 0);

// Average views
const total = articles.reduce(
  (sum, a) => sum + a.views, 0
);
const avg = total / articles.length;
SQL aggregates
-- Count all articles
SELECT COUNT(*) FROM articles;

-- Sum all views
SELECT SUM(views) FROM articles;

-- Average views
SELECT AVG(views) FROM articles;

SQL does in one line what takes multiple lines of JavaScript. And it does it on the database server, where the data already lives — no need to pull millions of rows into your app's memory.

SELECT
  COUNT(*) AS total_articles,
  SUM(views) AS total_views,
  AVG(views) AS avg_views,
  MIN(views) AS min_views,
  MAX(views) AS max_views
FROM articles

COUNT(*) vs COUNT(column). COUNT(*) counts all rows, including those with NULLs. COUNT(column) counts only rows where that column is not NULL. COUNT(DISTINCT column) counts unique non-NULL values. These are three different things.

Prisma Equivalents

Prisma
const result = await prisma.article.aggregate({
  _count: { _all: true },
  _sum: { views: true },
  _avg: { views: true },

SQL
SELECT
  COUNT(*) AS total,
  SUM(views) AS total_views,
  AVG(views) AS avg_views,
  MIN(views) AS min_views,
  MAX(views) AS max_views
FROM articles
WHERE published = true;

AVG ignores NULLs. If you have values [100, 200, NULL], AVG returns 150 (averages 100 and 200, ignores the NULL). If you want NULLs treated as 0, use AVG(COALESCE(column, 0)).

GROUP BY — Aggregating by Category

Without GROUP BY, aggregates collapse the entire table into one row. GROUP BY splits the table into buckets first, then runs the aggregate on each bucket separately.

Think of it like sorting your laundry into piles by color, then counting each pile.

JavaScript (lodash-style)
// Group by category, count each
const grouped = {};
articles.forEach(a => {
  if (!grouped[a.category]) {
    grouped[a.category] = [];
  }
  grouped[a.category].push(a);
});

Object.entries(grouped).map(
  ([cat, items]) => ({
    category: cat,
    count: items.length,
  })
);
SQL
SELECT
  category,
  COUNT(*) AS article_count
FROM articles
GROUP BY category;

Result:

| category   | article_count |
|------------|---------------|
| politics   |             2 |
| sports     |             2 |
| technology |             2 |
| culture    |             1 |

SQL did in 3 lines what took 12 lines of JavaScript. That's the power of GROUP BY.

Multiple Aggregates per Group

SELECT
  category,
  COUNT(*) AS article_count,
  SUM(views) AS total_views,
  ROUND(AVG(word_count)) AS avg_word_count,
  MAX(views) AS top_article_views
FROM articles
WHERE published = true
GROUP BY category
ORDER BY total_views DESC;

Result:

| category   | article_count | total_views | avg_word_count | top_article_views |
|------------|---------------|-------------|----------------|-------------------|
| politics   |             2 |        9300 |           1350 |              5200 |
| technology |             1 |        8900 |           2100 |              8900 |
| sports     |             1 |        3100 |            800 |              3100 |
| culture    |             1 |        1200 |            600 |              1200 |

The GROUP BY rule: Every column in SELECT must either be (1) in the GROUP BY clause, or (2) inside an aggregate function. You can't SELECT title when grouping by category because there are multiple titles per category — SQL wouldn't know which one to show.

Prisma groupBy

Prisma
const result = await prisma.article.groupBy({
  by: ['category'],
  _count: { _all: true },
  _avg: { word_count: true },
  _sum: { views: true },
  where: { published: true },
  orderBy: { _sum: { views: 'desc' } },
});
SQL
SELECT
  category,
  COUNT(*) AS count,
  AVG(word_count) AS avg_words,
  SUM(views) AS total_views
FROM articles
WHERE published = true
GROUP BY category
ORDER BY total_views DESC;

HAVING — Filtering Groups

WHERE filters individual rows before grouping. HAVING filters groups after aggregation. Think of it as: WHERE is a bouncer at the door (checks each person), HAVING is a bouncer for the VIP section (checks the whole group).

-- WHERE: filter rows before grouping
-- "Only count published articles"
SELECT category, COUNT(*) AS article_count
FROM articles
WHERE published = true
GROUP BY category;

-- HAVING: filter groups after aggregation
-- "Only show categories with more than 1 article"
SELECT category, COUNT(*) AS article_count
FROM articles
WHERE published = true
GROUP BY category
HAVING COUNT(*) > 1;

Result with HAVING:

| category | article_count |
|----------|---------------|
| politics |             2 |

Only politics has more than 1 published article, so it's the only group that passes the HAVING filter.

JavaScript equivalent
// WHERE = filter before grouping
const published = articles
  .filter(a => a.published);

// GROUP BY = group the remaining
const groups = groupBy(published, 'category');

// HAVING = filter after grouping
const result = Object.entries(groups)
  .filter(([cat, items]) =>
    items.length > 1
  );
SQL
SELECT category, COUNT(*) AS cnt
FROM articles
WHERE published = true
GROUP BY category
HAVING COUNT(*) > 1;

WHERE vs HAVING: Use WHERE for conditions on individual columns (WHERE views > 100). Use HAVING for conditions on aggregated results (HAVING COUNT(*) > 5). You can't use column aliases in HAVING — repeat the aggregate function.

HAVING with Multiple Conditions

-- Categories with more than 1 article AND average views above 3000
SELECT
  category,
  COUNT(*) AS article_count,
  ROUND(AVG(views)) AS avg_views
FROM articles
WHERE published = true
GROUP BY category
HAVING COUNT(*) > 1
  AND AVG(views) > 3000;

Aggregation with JOINs

This is where things get really powerful. Combine JOINs from Lesson 03 with aggregates to answer questions across tables: "How many articles has each author written?"

-- Articles per author
SELECT
  au.name AS author,
  COUNT(a.id) AS article_count,
  COALESCE(SUM(a.views), 0) AS total_views
FROM authors au
LEFT JOIN articles a ON au.id = a.author_id
GROUP BY au.id, au.name
ORDER BY article_count DESC;

Result:

| author        | article_count | total_views |
|---------------|---------------|-------------|
| Sarah Chen    |             2 |        9300 |
| Marcus Rivera |             2 |        3100 |
| Priya Patel   |             2 |        8900 |
| James Wilson  |             1 |        1200 |

Notice the LEFT JOIN. We start from authors and LEFT JOIN articles, so authors with zero articles still show up (with 0 for count and views). If we used INNER JOIN, those authors would vanish. Always think about which JOIN type matches your question.

Filtering Aggregated JOINs

-- Authors who have written more than 1 published article
SELECT
  au.name AS author,
  COUNT(a.id) AS published_count
FROM authors au
INNER JOIN articles a
  ON au.id = a.author_id
WHERE a.published = true
GROUP BY au.id, au.name
HAVING COUNT(a.id) > 1;

The query pipeline: FROM (authors) → JOIN (articles) → WHERE (published only) → GROUP BY (per author) → HAVING (more than 1) → SELECT (name, count).

Conditional Aggregation — CASE Inside Aggregates

From Lesson 02, you learned CASE expressions. Combined with aggregates, they let you count or sum subsets in a single query — no need for multiple queries.

SELECT
  au.name AS author,
  COUNT(*) AS total_articles,
  COUNT(CASE WHEN a.published = true THEN 1 END) AS published,
  COUNT(CASE WHEN a.published = false THEN 1 END) AS drafts,
  COALESCE(SUM(CASE WHEN a.published = true THEN a.views END), 0) AS published_views
FROM authors au
LEFT JOIN articles a ON au.id = a.author_id
GROUP BY au.id, au.name
ORDER BY published_views DESC;

Result:

| author        | total_articles | published | drafts | published_views |
|---------------|----------------|-----------|--------|-----------------|
| Priya Patel   |              2 |         1 |      1 |            8900 |
| Sarah Chen    |              2 |         2 |      0 |            9300 |
| Marcus Rivera |              2 |         1 |      1 |            3100 |
| James Wilson  |              1 |         1 |      0 |            1200 |

One query, four different metrics per author. This pattern is essential for building dashboards and generating reports.

PostgreSQL shortcut: FILTER. PostgreSQL has a cleaner syntax for conditional aggregation: COUNT(*) FILTER (WHERE published = true). It's equivalent to COUNT(CASE WHEN published = true THEN 1 END) but more readable.

-- Same query using FILTER (PostgreSQL only)
SELECT
  au.name AS author,
  COUNT(*) AS total_articles,
  COUNT(*) FILTER (WHERE a.published = true) AS published,
  COUNT(*) FILTER (WHERE a.published = false) AS drafts
FROM authors au
LEFT JOIN articles a ON au.id = a.author_id
GROUP BY au.id, au.name;

SQL Execution Order

SQL doesn't execute in the order you write it. Understanding the actual execution order makes GROUP BY and HAVING click:

-- Writing order:       Execution order:
SELECT   (5)            FROM     (1)  -- pick the table
FROM     (1)            WHERE    (2)  -- filter individual rows
WHERE    (2)            GROUP BY (3)  -- bucket into groups
GROUP BY (3)            HAVING   (4)  -- filter groups
HAVING   (4)            SELECT   (5)  -- pick columns & compute
ORDER BY (6)            ORDER BY (6)  -- sort results
LIMIT    (7)            LIMIT    (7)  -- cap the count

This is why you can't use aliases in WHERE or HAVING. SELECT runs after WHERE and GROUP BY. So the alias doesn't exist yet. You can use aliases in ORDER BY because it runs after SELECT. (PostgreSQL is actually lenient about this, but standard SQL is strict.)

Practical — Newsroom Analytics Dashboard

Let's build the queries a newsroom editor would want on their dashboard:

1. Articles per Author (Top Performers)

SELECT
  au.name,
  COUNT(a.id) AS articles,
  ROUND(AVG(a.views)) AS avg_views,
  SUM(a.views) AS total_views
FROM authors au
INNER JOIN articles a ON au.id = a.author_id
WHERE a.published = true
GROUP BY au.id, au.name
ORDER BY total_views DESC;

2. Average Word Count by Category

SELECT
  category,
  ROUND(AVG(word_count)) AS avg_word_count,
  MIN(word_count) AS shortest,
  MAX(word_count) AS longest,
  COUNT(*) AS article_count
FROM articles
WHERE published = true
GROUP BY category
ORDER BY avg_word_count DESC;

3. Publishing Activity by Month

SELECT
  DATE_TRUNC('month', published_at) AS month,
  COUNT(*) AS articles_published,
  SUM(views) AS monthly_views,
  ROUND(AVG(word_count)) AS avg_length
FROM articles
WHERE published = true
GROUP BY DATE_TRUNC('month', published_at)
ORDER BY month DESC;

DATE_TRUNC rounds a timestamp to the specified precision ('month', 'week', 'year', etc.). It's a PostgreSQL function that's incredibly useful for time-series grouping. It turns 2024-03-15 14:30:00 into 2024-03-01 00:00:00.

4. Category Performance with Engagement Tiers

SELECT
  category,
  COUNT(*) AS articles,
  SUM(views) AS total_views,
  CASE
    WHEN AVG(views) > 5000 THEN 'High'
    WHEN AVG(views) > 2000 THEN 'Medium'
    ELSE 'Low'
  END AS engagement_tier
FROM articles
WHERE published = true
GROUP BY category
ORDER BY total_views DESC;

Quick Check

1. What's the difference between COUNT(*) and COUNT(column)?

COUNT(*) counts all rows regardless of NULL values. COUNT(column) counts only rows where that column is not NULL. Use COUNT(DISTINCT column) for unique non-NULL values.

2. What's the difference between WHERE and HAVING?

WHERE filters individual rows before grouping. HAVING filters groups after aggregation. Use WHERE for column values (WHERE views > 100), HAVING for aggregate results (HAVING COUNT(*) > 5).

3. Why can't you SELECT title when GROUP BY category?

Because there are multiple titles per category. SQL doesn't know which one to show. Every column in SELECT must be in GROUP BY or inside an aggregate function (like MAX(title)).

4. What Prisma method is equivalent to SQL GROUP BY?

prisma.model.groupBy({ by: ['column'], _count: true }). Prisma also has .aggregate() for simple aggregations without grouping.

5. How does AVG handle NULL values?

AVG ignores NULLs. If values are [100, 200, NULL], AVG returns 150 (not 100). If you want NULLs treated as 0, use AVG(COALESCE(column, 0)).

Practice Exercise

Using the tables from this lesson: articles (id, title, author_id, category, word_count, views, published) and authors (id, name, beat).

A: Count the total number of published articles.

SELECT COUNT(*) AS published_count
FROM articles
WHERE published = true;

B: Find the average word count for each category. Order by average descending.

SELECT
  category,
  ROUND(AVG(word_count)) AS avg_word_count

GROUP BY category
ORDER BY avg_word_count DESC;

C: Find authors who have written more than 1 article. Show their name and article count.

SELECT au.name, COUNT(a.id) AS article_count
FROM authors au
INNER JOIN articles a ON au.id = a.author_id
GROUP BY au.id, au.name
HAVING COUNT(a.id) > 1;

D: For each author, show total articles, total views, and what percentage of their articles are published.

SELECT
  au.name,
  COUNT(a.id) AS total_articles,
  COALESCE(SUM(a.views), 0) AS total_views,
  ROUND(
    100.0 * COUNT(CASE WHEN a.published THEN 1 END)
    / COUNT(a.id)
  ) AS published_pct
FROM authors au
LEFT JOIN articles a ON au.id = a.author_id
GROUP BY au.id, au.name
ORDER BY total_views DESC;

E: Which categories have a total view count above 5000? Show category, article count, and total views.

SELECT
  category,
  COUNT(*) AS article_count,
  SUM(views) AS total_views
FROM articles
WHERE published = true
GROUP BY category
HAVING SUM(views) > 5000
ORDER BY total_views DESC;

Cheat Sheet

-- Aggregate functions
SELECT COUNT(*) FROM table;              -- count all rows
SELECT COUNT(column) FROM table;         -- count non-NULL values
SELECT COUNT(DISTINCT column) FROM table;-- count unique values
SELECT SUM(column) FROM table;           -- total
SELECT AVG(column) FROM table;           -- average (ignores NULLs)
SELECT MIN(column) FROM table;           -- smallest
SELECT MAX(column) FROM table;           -- largest

-- GROUP BY: aggregate per group
SELECT category, COUNT(*)
FROM articles
GROUP BY category;

-- HAVING: filter groups
SELECT category, COUNT(*) AS cnt
FROM articles
GROUP BY category
HAVING COUNT(*) > 3;

-- Aggregate + JOIN
SELECT au.name, COUNT(a.id) AS articles
FROM authors au
LEFT JOIN articles a ON au.id = a.author_id
GROUP BY au.id, au.name;

-- Conditional aggregation
COUNT(CASE WHEN condition THEN 1 END)
SUM(CASE WHEN condition THEN column END)

-- PostgreSQL FILTER syntax
COUNT(*) FILTER (WHERE condition)

-- Execution order
-- FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
