Lesson 01 gave you the vocabulary. Now we're going to get fluent. SELECT is the query you'll write 100 times a day. Let's master every option it gives you.

SELECT Expressions & Aliases

SELECT doesn't just pick columns — it can compute things. Think of it like .map(item => ({ ...item, fullName: item.first + ' ' + item.last })). You're transforming data on the way out.

Aliases with AS

AS renames a column in the output. The data doesn't change — just the label.

-- Rename columns in the result
SELECT
  first_name AS first,
  last_name AS last,
  salary / 12 AS monthly_salary
FROM employees;

Without AS, that calculated column would show up as ?column? in the results. Aliases give computed columns a proper name.

JavaScript equivalent
employees.map(emp => ({
  first: emp.first_name,
  last: emp.last_name,
  monthly_salary: emp.salary / 12,
}));
SQL
SELECT
  first_name AS first,
  last_name AS last,
  salary / 12 AS monthly_salary
FROM employees;

String Concatenation

In PostgreSQL, use || to concatenate strings (like + in JavaScript):

-- Combine first and last name
SELECT first_name || ' ' || last_name AS full_name
FROM employees;

-- Output: "Sarah Chen", "Marcus Rivera", etc.

PostgreSQL-specific: The || operator is the standard SQL concatenation operator. MySQL uses CONCAT() instead. PostgreSQL supports both, but || is more idiomatic.

Math in SELECT

SELECT
  name,
  salary,
  salary * 0.30 AS tax,
  salary * 0.70 AS take_home,
  ROUND(salary / 2080.0, 2) AS hourly_rate
FROM employees;

Standard math operators work: +, -, *, /, % (modulo). ROUND(value, decimals) rounds to the specified decimal places.

Comparison Operators

You learned the basics in Lesson 01. Let's go deeper with the operators you'll use every day.

BETWEEN

BETWEEN is inclusive on both ends. It's cleaner than writing >= AND <=:

-- These are identical:
WHERE salary BETWEEN 50000 AND 100000
WHERE salary >= 50000 AND salary <= 100000

-- Works with dates too
WHERE hire_date BETWEEN '2023-01-01' AND '2023-12-31'

IN

IN checks if a value matches any item in a list. Like .includes() in JavaScript:

JavaScript
['NY', 'CA', 'TX'].includes(emp.state)
SQL
WHERE state IN ('NY', 'CA', 'TX')
-- NOT IN = exclude specific values
WHERE department NOT IN ('HR', 'Legal')

-- IN with numbers
WHERE id IN (1, 5, 12, 27)

Pattern Matching — LIKE, ILIKE, and Regex

Pattern matching is essential for investigative work. Searching names, addresses, document text — you need to match partial strings.

LIKE and Wildcards

LIKE uses two wildcards:

  • % = any number of characters (like .* in regex)
  • _ = exactly one character (like . in regex)
-- Starts with 'John'
WHERE name LIKE 'John%'
-- Matches: 'John', 'Johnny', 'John Smith'

-- Ends with 'son'
WHERE name LIKE '%son'
-- Matches: 'Johnson', 'Ericson', 'son'

-- Contains 'smith'
WHERE name LIKE '%smith%'
-- Matches: 'Smith', 'Blacksmith', 'Smithson'
-- But NOT 'SMITH' (LIKE is case-sensitive!)

-- Exactly 5 characters
WHERE code LIKE '_____'

-- Starts with 'A', any char, then 'C'
WHERE code LIKE 'A_C%'
-- Matches: 'ABC', 'AXC123', 'A1C'

ILIKE (PostgreSQL-Specific)

ILIKE is the case-insensitive version of LIKE. This is a PostgreSQL extension — standard SQL doesn't have it.

-- Case-sensitive (misses 'SMITH', 'Smith')
WHERE name LIKE '%smith%'

-- Case-insensitive (catches all variations)
WHERE name ILIKE '%smith%'
-- Matches: 'smith', 'Smith', 'SMITH', 'Blacksmith'

For journalism: always use ILIKE. Public records have inconsistent capitalization. ILIKE saves you from missing records because someone typed "JOHNSON" instead of "Johnson".

PostgreSQL Regex Matching

For more complex patterns, PostgreSQL supports regex with ~ and ~*:

-- ~ = regex match (case-sensitive)
WHERE name ~ '^John(son|ston)$'

-- ~* = regex match (case-insensitive)
WHERE email ~* '^[a-z]+@gmail\.com$'

-- !~ = does NOT match regex
WHERE phone !~ '^\d{3}-\d{3}-\d{4}$'
JavaScript regex
// Case-insensitive match
if (/^john/i.test(name))

// Test email format
if (/^[a-z]+@gmail\.com$/i.test(email))
PostgreSQL regex
-- Case-insensitive match
WHERE name ~* '^john'

-- Test email format
WHERE email ~* '^[a-z]+@gmail\.com$'
OperatorMeaningCase
~Matches regexSensitive
~*Matches regexInsensitive
!~Does NOT match regexSensitive
!~*Does NOT match regexInsensitive

NULL Handling

This is the concept that trips up every developer. NULL in SQL is not null in JavaScript. It means "unknown" or "not applicable" — not "empty" or "zero".

The critical rule: NULL is not equal to anything — not even itself. NULL = NULL is NOT true. It evaluates to NULL (unknown). You MUST use IS NULL or IS NOT NULL instead of =.

-- WRONG: this finds nothing (even if phone is null)
SELECT * FROM contacts WHERE phone = NULL;

-- CORRECT: use IS NULL
SELECT * FROM contacts WHERE phone IS NULL;

-- Find contacts that DO have a phone
SELECT * FROM contacts WHERE phone IS NOT NULL;
JavaScript null checks
// Nullish coalescing
const display = name ?? 'Anonymous';

// Ternary with null check
const label = name !== null
  ? name
  : 'Unknown';
SQL null checks
-- COALESCE (like ?? in JS)
SELECT COALESCE(name, 'Anonymous') AS display;

-- COALESCE with multiple fallbacks
SELECT COALESCE(nickname, first_name, 'Unknown')
AS display_name;

COALESCE — SQL's Nullish Coalescing

COALESCE returns the first non-NULL value from a list. It's exactly like ?? in JavaScript.

-- If phone is NULL, show 'No phone on file'
SELECT
  name,
  COALESCE(phone, 'No phone on file') AS phone_display
FROM contacts;

-- Chain multiple fallbacks (like a ?? b ?? c)
SELECT
  COALESCE(mobile_phone, work_phone, home_phone, 'No phone') AS best_phone
FROM contacts;

NULLIF

NULLIF(a, b) returns NULL if a = b, otherwise returns a. Useful for preventing division by zero:

-- Without NULLIF: crashes if total_votes = 0
SELECT votes_for / total_votes AS percentage FROM results;

-- With NULLIF: returns NULL instead of crashing
SELECT votes_for / NULLIF(total_votes, 0) AS percentage FROM results;

Boolean Logic — AND, OR, NOT

Same as JavaScript's &&, ||, !, but with one important addition: operator precedence matters more in SQL because you can't rely on short-circuit evaluation the same way.

Precedence

AND binds tighter than OR (just like && binds tighter than || in JS). Use parentheses to be explicit:

-- DANGEROUS: AND binds first, so this means:
-- "politics AND trust_level > 7" OR "beat = 'crime'"
-- (Returns ALL crime beat sources regardless of trust!)
WHERE beat = 'politics' AND trust_level > 7 OR beat = 'crime'

-- SAFE: use parentheses to be explicit
WHERE (beat = 'politics' OR beat = 'crime') AND trust_level > 7

Rule of thumb: If you have both AND and OR in a WHERE clause, always use parentheses. Don't rely on precedence rules. Make your intent crystal clear.

NOT

-- NOT reverses a condition
WHERE NOT published
WHERE NOT (status = 'draft' OR status = 'archived')
WHERE name NOT LIKE '%test%'
WHERE department NOT IN ('HR', 'Legal')
WHERE phone IS NOT NULL

DISTINCT — Unique Values

DISTINCT removes duplicate rows. It's like [...new Set(array)] in JavaScript.

JavaScript
// Unique departments
const departments = [
  ...new Set(
    employees.map(e => e.department)
  )
];
SQL
-- Unique departments
SELECT DISTINCT department
FROM employees;
-- Count unique values
SELECT COUNT(DISTINCT department) AS dept_count
FROM employees;

-- Distinct combinations
SELECT DISTINCT department, title
FROM employees;
-- Returns unique (department, title) pairs

DISTINCT applies to the entire row. SELECT DISTINCT department, title gives unique combinations of department + title, not unique departments and unique titles separately.

CASE Expressions — SQL's Ternary / Switch

CASE is SQL's version of if/else or switch. You use it to create computed columns with conditional logic.

JavaScript ternary / switch
// Ternary
const label = salary > 100000
  ? 'Senior' : 'Standard';

// Switch-like
const tier = salary > 150000
  ? 'Executive'
  : salary > 100000
    ? 'Senior'
    : salary > 60000
      ? 'Mid'
      : 'Entry';
SQL CASE
-- Simple CASE (like ternary)
SELECT name,
  CASE WHEN salary > 100000
    THEN 'Senior' ELSE 'Standard'
  END AS label;

-- Multi-condition CASE (like switch)
SELECT name,
  CASE
    WHEN salary > 150000 THEN 'Executive'
    WHEN salary > 100000 THEN 'Senior'
    WHEN salary > 60000  THEN 'Mid'
    ELSE 'Entry'
  END AS tier
FROM employees;

The CASE structure is: CASE WHEN condition THEN result ... ELSE default END. The END keyword closes the expression. Conditions are evaluated top to bottom — first match wins.

CASE in WHERE

You can use CASE anywhere an expression is valid — SELECT, WHERE, ORDER BY, even inside functions:

-- Sort by custom priority
SELECT * FROM tickets
ORDER BY
  CASE priority
    WHEN 'critical' THEN 1
    WHEN 'high'     THEN 2
    WHEN 'medium'   THEN 3
    WHEN 'low'      THEN 4
    ELSE 5
  END;

CASE for Conditional Counting

This is a power pattern for data analysis — count rows that match different conditions:

SELECT
  COUNT(*) AS total_articles,
  COUNT(CASE WHEN published = true THEN 1 END) AS published_count,
  COUNT(CASE WHEN published = false THEN 1 END) AS draft_count,
  COUNT(CASE WHEN views > 1000 THEN 1 END) AS viral_count
FROM articles;

This pattern is incredibly useful for journalism. You can count subgroups in a single query: how many complaints per category, how many donors per bracket, how many violations per district. We'll build on this in Lesson 04 with GROUP BY.

Casting & Type Conversion

Sometimes you need to convert between types. PostgreSQL has two syntaxes for this:

CAST() — Standard SQL

-- Convert string to integer
SELECT CAST('42' AS INTEGER);

-- Convert integer to text
SELECT CAST(salary AS TEXT);

-- Convert to date
SELECT CAST('2024-01-15' AS DATE);

:: Operator — PostgreSQL Shorthand

PostgreSQL has a shortcut ::type that does the same thing as CAST. It's more concise:

-- Same conversions, shorter syntax
SELECT '42'::INTEGER;
SELECT salary::TEXT;
SELECT '2024-01-15'::DATE;

-- Practical use: round a calculation
SELECT ROUND(salary / 2080.0, 2)::NUMERIC(10,2) AS hourly_rate
FROM employees;
JavaScript type conversion
Number('42')        // 42
String(42)          // '42'
parseInt('3.14')    // 3
parseFloat('3.14')  // 3.14
PostgreSQL type casting
'42'::INTEGER       -- 42
42::TEXT             -- '42'
'3.14'::INTEGER     -- 3
'3.14'::NUMERIC     -- 3.14

When do you need casting? Most often when: (1) comparing columns of different types, (2) formatting output, (3) doing math with mixed types (integer division gives integers — use ::NUMERIC to get decimals).

Putting It All Together

Here's a realistic query that combines everything from this lesson. Imagine you're investigating campaign donations:

SELECT
  donor_name,
  UPPER(donor_state) AS state,
  amount,
  CASE
    WHEN amount >= 10000 THEN 'Major Donor'
    WHEN amount >= 1000  THEN 'Mid-Level'
    ELSE 'Small Donor'
  END AS donor_tier,
  COALESCE(employer, 'Not disclosed') AS employer_display,
  donation_date::TEXT AS date_string
FROM donations
WHERE donor_state IN ('NY', 'CA', 'TX', 'FL')
  AND amount > 500
  AND donor_name ILIKE '%smith%'
  AND donation_date BETWEEN '2024-01-01' AND '2024-06-30'
ORDER BY amount DESC
LIMIT 50;

Read it out loud: "Select the donor name, state (uppercased), amount, a tier label based on the amount, the employer (or 'Not disclosed' if null), and the date as text — from donations — where the state is one of four states, amount is over 500, the name contains 'smith' (case-insensitive), and the date is in the first half of 2024 — sorted by amount descending, top 50."

Quick Check

1. What's the difference between LIKE and ILIKE?

LIKE is case-sensitive. ILIKE is case-insensitive. ILIKE is a PostgreSQL extension. Always prefer ILIKE when searching user-entered data.

2. Why does WHERE phone = NULL not work?

NULL is "unknown" in SQL. You can't compare something to unknown with =. The result is always NULL (not true or false). Use WHERE phone IS NULL instead.

3. What does COALESCE(a, b, c) return?

The first non-NULL value among a, b, c. Like a ?? b ?? c in JavaScript. If all are NULL, returns NULL.

4. SELECT DISTINCT department, title FROM employees — what does DISTINCT apply to?

DISTINCT applies to the combination of all selected columns. It returns unique (department, title) pairs, not unique departments and unique titles separately.

5. In WHERE a OR b AND c, which binds first?

AND binds first (higher precedence). So it's evaluated as a OR (b AND c). Always use parentheses to be explicit: (a OR b) AND c.

Practice Exercise

You have a table campaign_donations with columns: id, donor_name, amount, candidate, party, state, employer (nullable), donation_date.

A: Get all donations over $5,000, showing donor name, amount, and candidate. Sort by amount descending.

SELECT donor_name, amount, candidate
FROM campaign_donations
WHERE amount > 5000
ORDER BY amount DESC;

B: Find all donors whose name contains "johnson" (case-insensitive) from Texas or Florida.

SELECT *
FROM campaign_donations
WHERE donor_name ILIKE '%johnson%'
  AND state IN ('TX', 'FL');

C: Show each donation with the employer (or "Self-funded" if employer is NULL) and a label: "Large" if over $10,000, "Medium" if over $1,000, else "Small".

SELECT
  donor_name,
  amount,
  COALESCE(employer, 'Self-funded') AS employer_display,
  CASE
    WHEN amount > 10000 THEN 'Large'
    WHEN amount > 1000  THEN 'Medium'
    ELSE 'Small'
  END AS size_label
FROM campaign_donations;

D: Count how many unique states appear in the donations table.

SELECT COUNT(DISTINCT state) AS state_count
FROM campaign_donations;

E: Get all donors where the employer field is missing (NULL) and the amount is between $1,000 and $50,000.

SELECT *
FROM campaign_donations
WHERE employer IS NULL
  AND amount BETWEEN 1000 AND 50000;
