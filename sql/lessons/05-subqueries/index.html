05 Subqueries & CTEs — Queries Inside Queries

← Back to lessons

Sometimes one query isn't enough. You need the answer from one query to feed into another. That's what subqueries and CTEs do — they let you build complex analysis step by step, like chaining .then() in a Promise chain.

The Mental Model — Queries as Building Blocks

Think of subqueries like function calls in JavaScript. A function returns a value, and you use that value in an expression. A subquery returns data (a single value, a list, or a table), and you use that data inside another query.

JavaScript (nested function calls)
// Step 1: get the average
const avg = getAverage(articles.views);

// Step 2: filter using that average
const aboveAvg = articles.filter(
  a => a.views > avg
);
SQL (subquery)
-- Steps 1 and 2 in one query
SELECT *
FROM articles
WHERE views > (
  SELECT AVG(views) FROM articles
);

The inner query SELECT AVG(views) FROM articles runs first and returns a number. The outer query then uses that number in its WHERE clause. Nesting. Just like functions.

There are three places you can put subqueries:

  • In WHERE — filter based on another query's result
  • In FROM — treat a query result as a temporary table
  • In SELECT — compute a value for each row

Subqueries in WHERE

The most common use. The subquery returns a value (or a list), and you use it to filter rows.

Scalar Subqueries — Return One Value

A scalar subquery returns exactly one value (one row, one column). Use it anywhere you'd use a number or string.

-- Articles with above-average views
SELECT title, views
FROM articles
WHERE views > (SELECT AVG(views) FROM articles);

The subquery (SELECT AVG(views) FROM articles) returns a single number, say 3214. The outer query becomes effectively WHERE views > 3214.

-- Find the most recent article
SELECT title, published_at
FROM articles
WHERE published_at = (
  SELECT MAX(published_at) FROM articles
);

-- Articles by the author with the most views
SELECT title, views
FROM articles
WHERE author_id = (
  SELECT author_id
  FROM articles
  GROUP BY author_id
  ORDER BY SUM(views) DESC
  LIMIT 1
);

IN Subqueries — Return a List

When a subquery returns multiple values (one column, multiple rows), use IN:

Prisma (nested where)
// Articles by politics-beat authors
await prisma.article.findMany({
  where: {
    author: {
      beat: 'politics',
    },
  },
});
SQL (IN subquery)
-- Articles by politics-beat authors
SELECT title
FROM articles
WHERE author_id IN (
  SELECT id
  FROM authors
  WHERE beat = 'politics'
);

The inner query returns a list of author IDs: (1, 5, 12). The outer query becomes WHERE author_id IN (1, 5, 12).

-- NOT IN: articles by authors NOT on the politics beat
SELECT title
FROM articles
WHERE author_id NOT IN (
  SELECT id FROM authors WHERE beat = 'politics'
);

Watch out for NULLs with NOT IN. If the subquery returns any NULL values, NOT IN returns no rows at all. This is a notorious SQL gotcha. Use NOT EXISTS instead if NULLs are possible.

EXISTS Subqueries — Check If Rows Exist

EXISTS returns true if the subquery finds at least one row. It doesn't care about the values — just whether anything matches.

-- Authors who have written at least one article
SELECT name
FROM authors au
WHERE EXISTS (
  SELECT 1
  FROM articles a
  WHERE a.author_id = au.id
);

-- Authors who have NEVER written an article
SELECT name
FROM authors au
WHERE NOT EXISTS (
  SELECT 1
  FROM articles a
  WHERE a.author_id = au.id
);

EXISTS vs IN — when to use which? EXISTS is better when: (1) you just need to know if rows exist, not what they are, (2) the subquery might return NULLs (EXISTS handles them safely), (3) the subquery would return a huge list (EXISTS stops as soon as it finds one match). Use IN when the list is small and readable.

Correlated vs Uncorrelated Subqueries

This distinction matters for understanding performance and behavior.

Uncorrelated — Independent Inner Query

The inner query runs once, produces a result, and the outer query uses it. Like calling a function that has no reference to the outer scope:

-- Uncorrelated: inner query runs ONCE
SELECT title, views
FROM articles
WHERE views > (SELECT AVG(views) FROM articles);
-- The AVG is computed once, then reused for every row

Correlated — Inner Query References Outer Query

The inner query references a column from the outer query, so it runs once per row. Like a closure that captures a variable from the enclosing scope:

JavaScript (closure analogy)
// For each author, find their
// article with the most views
authors.map(author => {
  const topArticle = articles
    .filter(a => a.author_id === author.id)
    .sort((a, b) => b.views - a.views)
    [0];
  return { author, topArticle };
});
SQL (correlated subquery)
-- For each article, check if it's
-- the top article by its author
SELECT title, views
FROM articles a
WHERE views = (
  SELECT MAX(views)
  FROM articles a2
  WHERE a2.author_id = a.author_id
);

The inner query references a.author_id from the outer query. For each article, SQL runs the subquery with that specific author's ID to find the MAX views. This is powerful but can be slow on large datasets because the subquery runs once per row.

-- Another correlated example:
-- Articles with views above their category's average
SELECT title, category, views
FROM articles a
WHERE views > (
  SELECT AVG(views)
  FROM articles a2
  WHERE a2.category = a.category
);

Performance note: Correlated subqueries can be slow because they run once per row. Often you can rewrite them as JOINs or CTEs for better performance. The database optimizer sometimes does this automatically, but not always.

Subqueries in FROM — Derived Tables

A subquery in the FROM clause creates a temporary table (called a "derived table") that you can query against. Think of it as creating a temporary variable that holds a dataset:

JavaScript
// Step 1: compute author stats
const authorStats = authors.map(au => ({
  name: au.name,
  articleCount: articles.filter(
    a => a.author_id === au.id
  ).length,
}));

// Step 2: filter the computed stats
const prolific = authorStats
  .filter(s => s.articleCount > 3);
SQL (derived table)
-- Step 1 is the subquery
-- Step 2 is the outer query
SELECT name, article_count
FROM (
  SELECT
    au.name,
    COUNT(a.id) AS article_count
  FROM authors au
  LEFT JOIN articles a
    ON au.id = a.author_id
  GROUP BY au.id, au.name
) AS author_stats
WHERE article_count > 3;

The subquery in FROM must have an alias (AS author_stats). You can then reference its columns like a regular table.

-- More practical: find the top category by average views
SELECT category, avg_views
FROM (
  SELECT
    category,
    ROUND(AVG(views)) AS avg_views,
    COUNT(*) AS article_count
  FROM articles
  WHERE published = true
  GROUP BY category
) AS category_stats
WHERE article_count >= 2
ORDER BY avg_views DESC
LIMIT 1;

Derived tables solve the "can't filter on aggregates in WHERE" problem. You can't write WHERE COUNT(*) > 3 because WHERE runs before GROUP BY. But you can compute the count in a subquery, then filter on it in the outer query. (HAVING also solves this, but derived tables are more flexible.)

Subqueries in SELECT — Computed Columns

A scalar subquery in SELECT adds a computed column to each row. It runs once per row (it's correlated):

SELECT
  title,
  views,
  (SELECT AVG(views) FROM articles) AS overall_avg,
  views - (SELECT AVG(views) FROM articles) AS diff_from_avg
FROM articles
ORDER BY diff_from_avg DESC;

Result:

| title                  | views | overall_avg | diff_from_avg |
|------------------------|-------|-------------|---------------|
| AI in the Newsroom     |  8900 |        3214 |          5686 |
| Election Results 2024  |  5200 |        3214 |          1986 |
| Budget Crisis Deepens  |  4100 |        3214 |           886 |
| City Marathon Recap    |  3100 |        3214 |          -114 |
| New Art Gallery Opens  |  1200 |        3214 |         -2014 |
-- Per-author article count alongside each article
SELECT
  a.title,
  au.name,
  (SELECT COUNT(*)
   FROM articles a2
   WHERE a2.author_id = a.author_id
  ) AS author_total_articles
FROM articles a
LEFT JOIN authors au ON a.author_id = au.id;

CTEs — Common Table Expressions (WITH)

CTEs are the clean, readable alternative to nested subqueries. Instead of nesting queries inside queries (which gets messy fast), you define named temporary results at the top using WITH, then use them below.

Think of CTEs as const variables for queries:

JavaScript (named steps)
// Step 1: define intermediate result
const authorStats = getAuthorStats();

// Step 2: define another result
const avgCount = average(
  authorStats.map(a => a.count)
);

// Step 3: use both
const aboveAvg = authorStats
  .filter(a => a.count > avgCount);
SQL (CTE with WITH)
-- Step 1
WITH author_stats AS (
  SELECT author_id, COUNT(*) AS cnt
  FROM articles
  GROUP BY author_id
),
-- Step 2
avg_count AS (
  SELECT AVG(cnt) AS val
  FROM author_stats
)
-- Step 3
SELECT au.name, s.cnt
FROM author_stats s
JOIN authors au ON s.author_id = au.id
WHERE s.cnt > (SELECT val FROM avg_count);

The WITH clause defines one or more named queries. Each CTE can reference the ones defined before it. The final SELECT at the bottom uses them.

CTE Syntax

WITH cte_name AS (
  -- any SELECT query
  SELECT ...
),
another_cte AS (
  -- can reference cte_name
  SELECT ... FROM cte_name ...
)
-- final query uses the CTEs
SELECT ...
FROM cte_name
JOIN another_cte ON ...;

CTEs make complex queries readable. Compare a 3-level nested subquery (hard to read) to a CTE with 3 named steps (reads top to bottom like a story). They're the same logic, but CTEs are dramatically easier to understand and debug.

When to Use CTEs vs Subqueries

Use CTEs WhenUse Subqueries When
Query has multiple logical stepsSimple one-off filter or value lookup
You need to reference a result multiple timesSubquery result is used only once
Readability matters (always)Query is short and self-explanatory
You're building up a complex analysisYou need a quick inline check

Recursive CTEs — Hierarchical Data

Recursive CTEs handle tree-structured data: org charts, category hierarchies, threaded comments. The CTE calls itself to walk down the tree level by level.

-- Org chart: employees table
| id | name           | manager_id |
|----|----------------|------------|
|  1 | Diana (CEO)    |       NULL |
|  2 | Sarah (Editor) |          1 |
|  3 | Marcus (Writer)|          2 |
|  4 | Priya (Writer) |          2 |
|  5 | James (Intern) |          3 |
-- Get the full reporting chain for every employee
WITH RECURSIVE org_tree AS (
  -- Base case: start with the CEO (no manager)
  SELECT id, name, manager_id, 0 AS depth,
         name AS chain
  FROM employees
  WHERE manager_id IS NULL

  UNION ALL

  -- Recursive step: find each person's reports
  SELECT e.id, e.name, e.manager_id, t.depth + 1,
         t.chain || ' > ' || e.name
  FROM employees e
  INNER JOIN org_tree t ON e.manager_id = t.id
)
SELECT name, depth, chain
FROM org_tree
ORDER BY chain;

Result:

| name           | depth | chain                                    |
|----------------|-------|------------------------------------------|
| Diana (CEO)    |     0 | Diana (CEO)                              |
| Sarah (Editor) |     1 | Diana (CEO) > Sarah (Editor)             |
| Marcus (Writer)|     2 | Diana (CEO) > Sarah (Editor) > Marcus    |
| James (Intern) |     3 | Diana (CEO) > Sarah (Editor) > Marcus > James |
| Priya (Writer) |     2 | Diana (CEO) > Sarah (Editor) > Priya     |

The pattern: a base case (the top of the tree), UNION ALL, and a recursive step that joins back to the CTE itself. SQL runs the base case first, then repeats the recursive step until no new rows are found.

Be careful with recursive CTEs. If your data has cycles (A reports to B, B reports to A), the recursion will loop forever. PostgreSQL will eventually stop with an error. Add a WHERE depth < 10 safeguard in the recursive step to prevent this.

Prisma and Complex Queries

Prisma can't express subqueries, CTEs, or recursive queries in its query builder. When you need these, you have two options:

Prisma $queryRaw
// Use raw SQL inside Prisma
const result = await prisma.$queryRaw`
  WITH author_stats AS (
    SELECT
      author_id,
      COUNT(*) AS article_count,
      AVG(views) AS avg_views
    FROM articles
    GROUP BY author_id
  )
  SELECT
    au.name,
    s.article_count,
    s.avg_views
  FROM author_stats s
  JOIN authors au ON s.author_id = au.id
  WHERE s.article_count > ${minCount}
`;
When to use $queryRaw
// Use raw SQL when Prisma can't:
//
// 1. Subqueries
//    WHERE col > (SELECT AVG...)
//
// 2. CTEs (WITH ... AS)
//    Complex multi-step analysis
//
// 3. Recursive queries
//    Hierarchical data traversal
//
// 4. Window functions
//    ROW_NUMBER, LAG, LEAD
//
// 5. Complex aggregations
//    FILTER, conditional sums
//
// Prisma's query builder handles
// 80% of cases. Raw SQL handles
// the other 20%.

The template literal is important. prisma.$queryRaw with a tagged template literal (backticks) automatically parameterizes your variables (${minCount}) to prevent SQL injection. Never use string concatenation with raw SQL.

Practical — Finding Above-Average Authors

Here's a realistic analysis: find authors who produce more articles than the average author, along with their performance stats.

Approach 1: Nested Subqueries

SELECT au.name, sub.article_count, sub.total_views
FROM authors au
INNER JOIN (
  SELECT author_id,
         COUNT(*) AS article_count,
         SUM(views) AS total_views
  FROM articles
  WHERE published = true
  GROUP BY author_id
) sub ON au.id = sub.author_id
WHERE sub.article_count > (
  SELECT AVG(author_count)
  FROM (
    SELECT COUNT(*) AS author_count
    FROM articles
    WHERE published = true
    GROUP BY author_id
  ) counts
)
ORDER BY sub.total_views DESC;

That works but it's hard to read. The same logic with CTEs:

Approach 2: CTEs (Much Cleaner)

WITH author_stats AS (
  -- Step 1: count articles per author
  SELECT
    author_id,
    COUNT(*) AS article_count,
    SUM(views) AS total_views,
    ROUND(AVG(views)) AS avg_views
  FROM articles
  WHERE published = true
  GROUP BY author_id
),
avg_article_count AS (
  -- Step 2: get the average article count across authors
  SELECT AVG(article_count) AS avg_count
  FROM author_stats
)
-- Step 3: filter to above-average authors
SELECT
  au.name,
  s.article_count,
  s.total_views,
  s.avg_views,
  a.avg_count AS newsroom_avg
FROM author_stats s
JOIN authors au ON s.author_id = au.id
CROSS JOIN avg_article_count a
WHERE s.article_count > a.avg_count
ORDER BY s.total_views DESC;

Same result, but you can read it top-to-bottom like a story. Each CTE has a clear name and purpose.

Bonus: Author Performance Report

WITH article_stats AS (
  SELECT
    author_id,
    COUNT(*) AS total_articles,
    COUNT(*) FILTER (WHERE published) AS published,
    COUNT(*) FILTER (WHERE NOT published) AS drafts,
    COALESCE(SUM(views), 0) AS total_views,
    COALESCE(ROUND(AVG(views)), 0) AS avg_views,
    MAX(published_at) AS last_published
  FROM articles
  GROUP BY author_id
)
SELECT
  au.name,
  au.beat,
  COALESCE(s.total_articles, 0) AS total_articles,
  COALESCE(s.published, 0) AS published,
  COALESCE(s.drafts, 0) AS drafts,
  COALESCE(s.total_views, 0) AS total_views,
  s.avg_views,
  s.last_published
FROM authors au
LEFT JOIN article_stats s ON au.id = s.author_id
ORDER BY total_views DESC;

Quick Check

1. What's the difference between a correlated and uncorrelated subquery?

An uncorrelated subquery runs once independently. A correlated subquery references the outer query, so it runs once per row of the outer query. Correlated subqueries are more flexible but slower.

2. When would you use EXISTS instead of IN?

Use EXISTS when: (1) you only care whether rows exist, not their values, (2) the subquery might return NULLs (NOT IN fails with NULLs, NOT EXISTS doesn't), (3) the subquery returns a large result set (EXISTS stops at the first match).

3. Why must subqueries in FROM have an alias?

Because SQL needs a name to reference the derived table's columns. Without an alias, you'd have no way to refer to it in SELECT, WHERE, or JOIN clauses. It's like a variable name.

4. What is a CTE and why is it better than nested subqueries?

A CTE (Common Table Expression) is a named temporary result defined with WITH ... AS (...). It's better because: (1) reads top-to-bottom instead of inside-out, (2) you can reference a CTE multiple times, (3) each step has a descriptive name.

5. How do you use raw SQL in Prisma?

prisma.$queryRaw`SELECT ... WHERE col > ${variable}`. Use the tagged template literal (backticks) to automatically parameterize variables and prevent SQL injection.

Practice Exercise

Using tables: articles (id, title, author_id, category, views, word_count, published), authors (id, name, beat).

A: Find all articles with above-average views (use a subquery).

SELECT title, views
FROM articles
WHERE views > (SELECT AVG(views) FROM articles);

B: Find authors who have written at least one article in the 'politics' category (use EXISTS).

SELECT name
FROM authors au
WHERE EXISTS (
  SELECT 1 FROM articles a
  WHERE a.author_id = au.id
    AND a.category = 'politics'
);

C: Use a CTE to find each author's article count, then return only authors with more articles than the average author.

WITH author_counts AS (
  SELECT author_id, COUNT(*) AS cnt
  FROM articles
  GROUP BY author_id
)
SELECT au.name, ac.cnt
FROM author_counts ac
JOIN authors au ON ac.author_id = au.id
WHERE ac.cnt > (SELECT AVG(cnt) FROM author_counts);

D: For each article, show the title, views, and the overall average views as a separate column (use a subquery in SELECT).

SELECT
  title,
  views,
  (SELECT ROUND(AVG(views)) FROM articles) AS avg_views,
  views - (SELECT ROUND(AVG(views)) FROM articles) AS diff
FROM articles
ORDER BY diff DESC;

E: Write a CTE that finds the top category by total views, then returns all articles in that category.

WITH top_category AS (
  SELECT category
  FROM articles
  WHERE published = true
  GROUP BY category
  ORDER BY SUM(views) DESC
  LIMIT 1
)
SELECT a.title, a.views, a.category
FROM articles a
WHERE a.category = (SELECT category FROM top_category);

Cheat Sheet

-- Scalar subquery in WHERE (returns one value)
SELECT * FROM articles
WHERE views > (SELECT AVG(views) FROM articles);

-- IN subquery (returns a list)
SELECT * FROM articles
WHERE author_id IN (SELECT id FROM authors WHERE beat = 'politics');

-- EXISTS (check if rows exist)
SELECT * FROM authors au
WHERE EXISTS (SELECT 1 FROM articles WHERE author_id = au.id);

-- NOT EXISTS (check if no rows exist)
WHERE NOT EXISTS (SELECT 1 FROM articles WHERE author_id = au.id);

-- Subquery in FROM (derived table)
SELECT * FROM (
  SELECT category, COUNT(*) AS cnt
  FROM articles GROUP BY category
) AS stats WHERE cnt > 3;

-- Subquery in SELECT (computed column)
SELECT title, (SELECT AVG(views) FROM articles) AS avg
FROM articles;

-- CTE (Common Table Expression)
WITH my_cte AS (
  SELECT ... FROM ...
)
SELECT * FROM my_cte WHERE ...;

-- Multiple CTEs
WITH step1 AS (...),
     step2 AS (SELECT ... FROM step1 ...)
SELECT ... FROM step2;

-- Recursive CTE
WITH RECURSIVE tree AS (
  SELECT ... WHERE parent IS NULL   -- base case
  UNION ALL
  SELECT ... FROM table             -- recursive step
  JOIN tree ON ...
)
SELECT * FROM tree;

-- Prisma raw SQL
await prisma.$queryRaw`SELECT ... WHERE col > ${value}`;
