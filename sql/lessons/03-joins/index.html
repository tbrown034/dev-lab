03 JOINs & Relationships — Connecting Tables

← Back to lessons

You already understand relationships. Every time you use Prisma's include to fetch an author with their articles, Prisma is writing a JOIN behind the scenes. Now you'll write JOINs yourself — and understand exactly what's happening.

The Mental Model — Why JOINs Exist

In a well-designed database, data is split across multiple tables. Articles live in one table, authors in another, categories in a third. This is called normalization — and it means you don't repeat data. An author's name is stored once, not copied into every article row.

But that creates a problem: when you want "articles with their author names," the data lives in two places. A JOIN is the bridge. It stitches rows from two tables together based on a matching column.

Here's our newsroom schema for this lesson:

-- authors table
| id | name           | beat       |
|----|----------------|------------|
|  1 | Sarah Chen      | politics   |
|  2 | Marcus Rivera   | sports     |
|  3 | Priya Patel     | technology |
|  4 | James Wilson    | culture    |

-- articles table
| id | title                    | author_id | category   | views |
|----|--------------------------|-----------|------------|-------|
|  1 | Election Results 2024    |         1 | politics   |  5200 |
|  2 | City Marathon Recap      |         2 | sports     |  3100 |
|  3 | AI in the Newsroom       |         3 | technology |  8900 |
|  4 | Budget Crisis Deepens    |         1 | politics   |  4100 |
|  5 | New Art Gallery Opens    |      NULL | culture    |  1200 |

Notice that articles.author_id points to authors.id. This is a foreign key — the same concept as a relation in your Prisma schema. Article #5 has NULL for author_id (maybe it was submitted anonymously). That NULL will matter when we compare JOIN types.

The Venn diagram mental model. Think of the two tables as two circles that overlap. The overlap is rows that match on the join condition. Different JOIN types control which parts of the circles you keep: just the overlap, the whole left circle, the whole right circle, or everything.

INNER JOIN — Only Matching Rows

INNER JOIN returns only the rows where both tables have a match. If an article has no author (NULL author_id), it's excluded. If an author has no articles, they're excluded too. It's the overlap in the Venn diagram.

Prisma
// Only returns articles that HAVE an author
await prisma.article.findMany({
  where: { author: { isNot: null } },
  include: { author: true },
});
SQL
SELECT
  articles.title,
  authors.name AS author_name
FROM articles
INNER JOIN authors
  ON articles.author_id = authors.id;

The result:

| title                    | author_name     |
|--------------------------|-----------------|
| Election Results 2024    | Sarah Chen      |
| City Marathon Recap      | Marcus Rivera   |
| AI in the Newsroom       | Priya Patel     |
| Budget Crisis Deepens    | Sarah Chen      |

Notice: "New Art Gallery Opens" is missing because its author_id is NULL. James Wilson is missing because he has no articles. INNER JOIN is strict — both sides must match.

Let's break down the syntax:

  • FROM articles — start with the articles table (the "left" table)
  • INNER JOIN authors — combine with the authors table (the "right" table)
  • ON articles.author_id = authors.id — the matching condition. "Where the article's author_id equals the author's id."

Table prefixes matter. Since both tables might have a column called id, you prefix with the table name: articles.id vs authors.id. Without the prefix, SQL wouldn't know which id you mean.

Table Aliases — Less Typing

Typing full table names gets tedious. Use aliases (just like AS for columns):

SELECT a.title, au.name AS author_name
FROM articles AS a
INNER JOIN authors AS au
  ON a.author_id = au.id;

The AS keyword is optional with table aliases — FROM articles a works too. Most SQL developers use short aliases: a for articles, au for authors.

LEFT JOIN — Keep All Left Rows

LEFT JOIN keeps every row from the left table, even if there's no match in the right table. Where there's no match, the right table's columns are filled with NULL.

Prisma
// Returns ALL articles, author may be null
await prisma.article.findMany({
  include: { author: true },
});
// article.author could be null
SQL
SELECT a.title, au.name AS author_name
FROM articles a
LEFT JOIN authors au
  ON a.author_id = au.id;

The result:

| title                    | author_name     |
|--------------------------|-----------------|
| Election Results 2024    | Sarah Chen      |
| City Marathon Recap      | Marcus Rivera   |
| AI in the Newsroom       | Priya Patel     |
| Budget Crisis Deepens    | Sarah Chen      |
| New Art Gallery Opens    | NULL            |

Now "New Art Gallery Opens" is included with a NULL author. The left table (articles) always keeps all its rows. This is the most common JOIN type you'll use — it matches Prisma's default include behavior.

LEFT JOIN is your default. Use it when you want all rows from your "main" table and optionally attach related data. Think of it as: "Give me all articles, and their author info if they have one."

RIGHT JOIN & FULL OUTER JOIN

RIGHT JOIN

RIGHT JOIN is the mirror of LEFT JOIN — it keeps all rows from the right table. In practice, you rarely see RIGHT JOIN because you can always rewrite it as a LEFT JOIN by swapping the table order.

-- RIGHT JOIN: keep all authors, even those with no articles
SELECT a.title, au.name AS author_name
FROM articles a
RIGHT JOIN authors au
  ON a.author_id = au.id;

The result:

| title                    | author_name     |
|--------------------------|-----------------|
| Election Results 2024    | Sarah Chen      |
| Budget Crisis Deepens    | Sarah Chen      |
| City Marathon Recap      | Marcus Rivera   |
| AI in the Newsroom       | Priya Patel     |
| NULL                     | James Wilson    |

James Wilson appears even though he has no articles. His title column is NULL. But the anonymous article (#5) is gone — no author to match.

Pro tip: Most SQL developers just swap the table order and use LEFT JOIN instead of RIGHT JOIN. A RIGHT JOIN B is identical to B LEFT JOIN A. Stick with LEFT JOIN for consistency.

FULL OUTER JOIN

FULL OUTER JOIN keeps everything from both tables. Unmatched rows on either side get NULLs. It's the complete Venn diagram — both circles, including the parts that don't overlap.

SELECT a.title, au.name AS author_name
FROM articles a
FULL OUTER JOIN authors au
  ON a.author_id = au.id;

The result:

| title                    | author_name     |
|--------------------------|-----------------|
| Election Results 2024    | Sarah Chen      |
| City Marathon Recap      | Marcus Rivera   |
| AI in the Newsroom       | Priya Patel     |
| Budget Crisis Deepens    | Sarah Chen      |
| New Art Gallery Opens    | NULL            |
| NULL                     | James Wilson    |

Now everything shows up: the anonymous article AND the author with no articles. Use FULL OUTER JOIN when you need a complete picture of both sides.

JOIN Type Summary

JOIN TypeLeft Table RowsRight Table RowsWhen to Use

LEFT JOINAll rowsOnly matchedWant all main records + optional related data

Multi-Table JOINs

Real queries often join three or more tables. Each JOIN adds another table to the mix. Let's add a categories table and an article_tags table:

-- categories table
| id | name       | section    |
|----|------------|------------|
|  1 | politics   | news       |
|  2 | sports     | lifestyle  |
|  3 | technology | news       |

-- article_tags table (many-to-many bridge)
| article_id | tag        |
|------------|------------|
|          1 | election   |
|          1 | breaking   |
|          3 | AI         |
|          3 | opinion    |

Now query articles with their author AND category section:

Prisma
await prisma.article.findMany({
  include: {
    author: true,
    category: true,
  },
});
SQL
SELECT
  a.title,
  au.name AS author_name,
  c.section
FROM articles a
LEFT JOIN authors au
  ON a.author_id = au.id
LEFT JOIN categories c
  ON a.category = c.name;

Each JOIN clause adds one table. The pattern is always the same: JOIN table ON condition. Chain as many as you need.

-- Three-table join: articles + authors + tags
SELECT
  a.title,
  au.name AS author_name,
  at.tag
FROM articles a
LEFT JOIN authors au ON a.author_id = au.id
LEFT JOIN article_tags at ON a.id = at.article_id
ORDER BY a.title;

Result:

| title                    | author_name   | tag      |
|--------------------------|---------------|----------|
| AI in the Newsroom       | Priya Patel   | AI       |
| AI in the Newsroom       | Priya Patel   | opinion  |
| Budget Crisis Deepens    | Sarah Chen    | NULL     |
| City Marathon Recap      | Marcus Rivera | NULL     |
| Election Results 2024    | Sarah Chen    | election |
| Election Results 2024    | Sarah Chen    | breaking |
| New Art Gallery Opens    | NULL          | NULL     |

JOINs can multiply rows. Notice "AI in the Newsroom" appears twice — once for each tag. And "Election Results 2024" appears twice too. When one article has multiple tags, the JOIN creates one row per combination. This is important to understand when counting — you can't just use COUNT(*) after a JOIN without accounting for this.

Self Joins — A Table Joining Itself

Sometimes a table has a relationship with itself. Think of employees where each one has a manager who is also an employee. Or articles that reference other articles.

-- employees table with manager relationship
| id | name           | manager_id |
|----|----------------|------------|
|  1 | Diana (Editor) |       NULL |
|  2 | Sarah          |          1 |
|  3 | Marcus         |          1 |
|  4 | Priya          |          2 |

To get each employee with their manager's name, you join the table to itself using different aliases:

SELECT
  e.name AS employee,
  m.name AS manager
FROM employees e
LEFT JOIN employees m
  ON e.manager_id = m.id;

Result:

| employee       | manager        |
|----------------|----------------|
| Diana (Editor) | NULL           |
| Sarah          | Diana (Editor) |
| Marcus         | Diana (Editor) |
| Priya          | Sarah          |

The trick is aliases: e for the employee, m for the manager. They're the same table, but SQL treats them as two separate copies. Diana has no manager (NULL), so LEFT JOIN keeps her with a NULL manager.

Prisma include() vs SQL JOINs

Here's the key difference: Prisma's include gives you nested objects. SQL JOINs give you flat rows. Understanding both helps you choose the right tool.

Prisma result (nested)
{
  id: 1,
  title: "Election Results 2024",
  author: {
    id: 1,
    name: "Sarah Chen",
    beat: "politics"
  }
}
SQL result (flat row)
| id | title                 | author_id | author_name | beat     |
|----|-----------------------|-----------|-------------|----------|
|  1 | Election Results 2024 |         1 | Sarah Chen  | politics |

Prisma actually runs multiple queries under the hood (one for articles, one for authors) and stitches them together in JavaScript. Raw SQL does it in one query with a JOIN. For simple includes, Prisma's approach is fine. For complex reporting with many joins, raw SQL is often faster and more flexible.

Prisma (nested include)
await prisma.article.findMany({
  include: {
    author: {
      select: { name: true, beat: true }
    },
  },
  where: { author: { beat: 'politics' } },
});
SQL (JOIN with filter)
SELECT a.title, au.name, au.beat
FROM articles a
INNER JOIN authors au
  ON a.author_id = au.id
WHERE au.beat = 'politics';

When to use raw SQL over Prisma: (1) Complex reports with multiple JOINs and aggregations, (2) When you need to control exactly which JOIN type is used, (3) Performance-critical queries where Prisma's multiple-query approach is too slow, (4) Data analysis and journalism investigations.

Advanced JOIN Conditions

Joining on Multiple Columns

Sometimes you need to match on more than one column:

-- Match on both department AND role
SELECT e.name, t.team_name
FROM employees e
INNER JOIN teams t
  ON e.department = t.department
  AND e.role = t.required_role;

Joining with Inequalities

The ON clause doesn't have to use =. You can use any comparison:

-- Find articles published after an author joined
SELECT a.title, au.name, a.published_at, au.hire_date
FROM articles a
INNER JOIN authors au
  ON a.author_id = au.id
  AND a.published_at > au.hire_date;

WHERE vs ON in JOINs

For INNER JOIN, putting conditions in ON vs WHERE gives the same result. But for LEFT JOIN, they're different:

-- ON: filter happens DURING the join (keeps all left rows)
SELECT a.title, au.name
FROM articles a
LEFT JOIN authors au
  ON a.author_id = au.id
  AND au.beat = 'politics';
-- Returns ALL articles; author_name is NULL for non-politics authors

-- WHERE: filter happens AFTER the join (removes rows)
SELECT a.title, au.name
FROM articles a
LEFT JOIN authors au
  ON a.author_id = au.id
WHERE au.beat = 'politics';
-- Returns ONLY articles by politics authors (defeats the LEFT JOIN)

This is a common trap. If you LEFT JOIN and then filter on the right table in WHERE, you effectively turn it into an INNER JOIN. Put right-table filters in the ON clause to preserve the LEFT JOIN behavior.

Practical — Newsroom Article Report

Let's build a realistic query: a report showing every article with its author, category section, and tags, sorted by views:

SELECT
  a.title,
  COALESCE(au.name, 'Anonymous') AS author,
  au.beat,
  c.section,
  a.views,
  STRING_AGG(at.tag, ', ') AS tags
FROM articles a
LEFT JOIN authors au ON a.author_id = au.id
LEFT JOIN categories c ON a.category = c.name
LEFT JOIN article_tags at ON a.id = at.article_id
GROUP BY a.id, a.title, au.name, au.beat, c.section, a.views
ORDER BY a.views DESC;

Result:

| title                  | author        | beat       | section   | views | tags             |
|------------------------|---------------|------------|-----------|-------|------------------|
| AI in the Newsroom     | Priya Patel   | technology | news      |  8900 | AI, opinion      |
| Election Results 2024  | Sarah Chen    | politics   | news      |  5200 | election, breaking|
| Budget Crisis Deepens  | Sarah Chen    | politics   | news      |  4100 | NULL             |
| City Marathon Recap    | Marcus Rivera | sports     | lifestyle |  3100 | NULL             |
| New Art Gallery Opens  | Anonymous     | NULL       | NULL      |  1200 | NULL             |

Notice: COALESCE replaces the NULL author with "Anonymous." STRING_AGG combines multiple tags into a comma-separated string. GROUP BY collapses the duplicate rows from the tag join (we'll cover GROUP BY properly in Lesson 04).

Quick Check

1. What's the difference between INNER JOIN and LEFT JOIN?

INNER JOIN only returns rows that match in both tables. LEFT JOIN returns ALL rows from the left table, filling in NULLs where there's no match in the right table.

2. If you have 3 articles and 2 tags per article, how many rows does a JOIN between articles and tags produce?

6 rows. Each article appears once per matching tag. JOINs can multiply rows when there's a one-to-many or many-to-many relationship.

3. What's the Prisma equivalent of a LEFT JOIN?

include: { author: true } — Prisma's include returns all records and sets the relation to null when there's no match, which is the same behavior as LEFT JOIN.

4. Why should you avoid putting right-table filters in WHERE with a LEFT JOIN?

Because WHERE filters happen after the JOIN. If the right table's value is NULL (no match), the WHERE condition fails and removes the row — effectively turning your LEFT JOIN into an INNER JOIN. Put right-table filters in the ON clause instead.

5. What is a self join?

A join where a table is joined to itself, using different aliases. Common for hierarchical data like employees/managers or categories/subcategories.

Practice Exercise

You have three tables: articles (id, title, author_id, category, views), authors (id, name, beat), and comments (id, article_id, commenter_name, body).

A: Get all articles with their author's name. Only include articles that have an author.

SELECT a.title, au.name AS author_name
FROM articles a
INNER JOIN authors au
  ON a.author_id = au.id;

B: Get ALL articles with their author's name (include articles with no author).

SELECT a.title, au.name AS author_name
FROM articles a
LEFT JOIN authors au
  ON a.author_id = au.id;

C: Get all authors and their articles. Include authors who haven't written any articles yet.

SELECT au.name, a.title
FROM authors au
LEFT JOIN articles a
  ON au.id = a.author_id;

D: Get articles with both author name and comment count.

SELECT
  a.title,
  au.name AS author_name,
  COUNT(c.id) AS comment_count
FROM articles a
LEFT JOIN authors au ON a.author_id = au.id
LEFT JOIN comments c ON a.id = c.article_id
GROUP BY a.id, a.title, au.name;

E: Find all articles by "politics" beat authors with more than 3000 views.

SELECT a.title, au.name, a.views
FROM articles a
INNER JOIN authors au
  ON a.author_id = au.id
WHERE au.beat = 'politics'
  AND a.views > 3000;

Cheat Sheet

-- INNER JOIN: only matching rows
SELECT a.col, b.col
FROM table_a a
INNER JOIN table_b b ON a.id = b.a_id;

-- LEFT JOIN: all left rows + matching right (or NULL)
SELECT a.col, b.col
FROM table_a a
LEFT JOIN table_b b ON a.id = b.a_id;

-- RIGHT JOIN: all right rows + matching left (or NULL)
-- (Prefer swapping tables and using LEFT JOIN instead)
SELECT a.col, b.col
FROM table_a a
RIGHT JOIN table_b b ON a.id = b.a_id;

-- FULL OUTER JOIN: all rows from both tables
SELECT a.col, b.col
FROM table_a a
FULL OUTER JOIN table_b b ON a.id = b.a_id;

-- Multi-table JOIN
SELECT a.col, b.col, c.col
FROM table_a a
LEFT JOIN table_b b ON a.id = b.a_id
LEFT JOIN table_c c ON a.id = c.a_id;

-- Self JOIN
SELECT e.name AS employee, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;

-- Table aliases
FROM articles AS a   -- or just: FROM articles a
