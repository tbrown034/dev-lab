01 SQL Fundamentals — From Prisma to Raw SQL

← Back to lessons

You already know this stuff. You've been writing Prisma queries for months. Now you're going to see the SQL that Prisma generates under the hood. It reads like English.

What is SQL?

SQL stands for Structured Query Language. It's the language you use to talk directly to a database. Every database understands it: PostgreSQL, MySQL, SQLite, SQL Server.

Here's the key insight: Prisma is a translator. When you write prisma.user.findMany(), Prisma turns that into SQL and sends it to your database. The database only speaks SQL. Prisma just makes it more convenient.

What you write (Prisma)
const users = await prisma.user.findMany({
  where: { active: true },
  orderBy: { name: 'asc' },
});
What your database actually receives (SQL)
SELECT *
FROM users
WHERE active = true
ORDER BY name ASC;

Why learn raw SQL? Three reasons: (1) You'll debug Prisma issues faster when you understand what it generates. (2) Some queries are too complex for ORMs. (3) Job interviews and data journalism almost always require SQL.

Databases & Tables

Think of it this way:

  • A database is like your project folder — it holds everything.
  • A table is like a TypeScript interface that also stores data. It defines the shape AND holds the rows.
  • A row is like a JavaScript object — one record with specific values.
  • A column is like a property on that object — name, email, age.
TypeScript interface
interface User {
  id: number;
  name: string;
  email: string;
  age: number;
  active: boolean;
}
SQL table (same idea)
-- This is a table called "users"
-- Each column has a name and a type

| id | name      | email           | age | active |
|----|-----------|-----------------|-----|--------|
|  1 | Sarah     | sarah@mail.com  |  28 | true   |
|  2 | Marcus    | marcus@mail.com |  34 | true   |
|  3 | Priya     | priya@mail.com  |  31 | false  |

Naming convention: In PostgreSQL, tables and columns use snake_case (lowercase with underscores): first_name, created_at, user_id. This is different from Prisma/TypeScript which uses camelCase. Prisma maps between them automatically with @map.

CREATE TABLE

In Prisma, you define your data shape in schema.prisma. In SQL, you use CREATE TABLE. Same concept, different syntax.

Prisma schema
model Article {
  id        Int      @id @default(autoincrement())
  title     String
  body      String
  published Boolean  @default(false)
  views     Int      @default(0)
  createdAt DateTime @default(now())
}
SQL equivalent
CREATE TABLE articles (
  id         SERIAL PRIMARY KEY,
  title      VARCHAR(255) NOT NULL,
  body       TEXT NOT NULL,
  published  BOOLEAN DEFAULT false,
  views      INTEGER DEFAULT 0,
  created_at TIMESTAMP DEFAULT NOW()
);

Let's break down the SQL version:

  • SERIAL = auto-incrementing integer (Prisma's @default(autoincrement()))
  • PRIMARY KEY = unique identifier for each row (Prisma's @id)
  • VARCHAR(255) = string up to 255 chars. TEXT = unlimited length string
  • NOT NULL = required field (Prisma fields are required by default, optional with ?)
  • DEFAULT = default value if none is provided (same as Prisma's @default())

Prisma runs CREATE TABLE for you. When you run npx prisma migrate dev, Prisma generates and executes the CREATE TABLE SQL. Now you know what it's actually doing.

INSERT — Adding Data

INSERT is like prisma.user.create(). You specify which table, which columns, and what values.

Prisma
await prisma.article.create({
  data: {
    title: 'Breaking Story',
    body: 'The investigation...',
    published: true,
  },
});
SQL
INSERT INTO articles (title, body, published)
VALUES ('Breaking Story', 'The investigation...', true);

The pattern is: INSERT INTO table (columns) VALUES (values). Notice you don't need to specify id or created_at — those have defaults.

Inserting Multiple Rows

Prisma (createMany)
await prisma.article.createMany({
  data: [
    { title: 'Story 1', body: 'Content 1' },
    { title: 'Story 2', body: 'Content 2' },
    { title: 'Story 3', body: 'Content 3' },
  ],
});
SQL (multiple VALUES)
INSERT INTO articles (title, body)
VALUES
  ('Story 1', 'Content 1'),
  ('Story 2', 'Content 2'),
  ('Story 3', 'Content 3');

Strings in SQL use single quotes, not double quotes. 'hello' is correct. "hello" means something different in SQL (it refers to a column or table name). This trips up every JavaScript developer at first.

SELECT — Reading Data

SELECT is the most important SQL command. You'll use it 90% of the time. Think of it like array.map() — it picks which columns you want from the result.

Prisma
// Get all articles (all columns)
await prisma.article.findMany();

// Get only title and body
await prisma.article.findMany({
  select: { title: true, body: true },
});
SQL
-- Get all articles (all columns)
SELECT * FROM articles;

-- Get only title and body
SELECT title, body
FROM articles;

The * in SELECT * means "all columns." It's like findMany() with no select option. When you list specific columns, you're choosing exactly which fields to return.

Read it like English: SELECT title, body FROM articles literally says "Select the title and body from the articles table." SQL is designed to be readable.

Get One Row

Prisma
await prisma.article.findUnique({
  where: { id: 5 },
});
SQL
SELECT *
FROM articles
WHERE id = 5;

WHERE — Filtering Rows

WHERE is like .filter() in JavaScript or the where clause in Prisma. It keeps only the rows that match your condition.

JavaScript
// Array.filter()
articles.filter(a => a.published === true);
articles.filter(a => a.views > 1000);
articles.filter(a =>
  a.published && a.views > 1000
);
SQL WHERE
SELECT * FROM articles
WHERE published = true;

SELECT * FROM articles WHERE views > 1000;

SELECT * FROM articles
WHERE published = true AND views > 1000;

LIKE — Pattern Matching

LIKE is for partial string matching. The % wildcard means "any characters." Think of it like a simplified regex.

Prisma
// Contains
where: { title: { contains: 'election' } }

// Starts with
where: { title: { startsWith: 'Breaking' } }

// Ends with
where: { email: { endsWith: '@gmail.com' } }
SQL
-- Contains
WHERE title LIKE '%election%'

-- Starts with
WHERE title LIKE 'Breaking%'

-- Ends with
WHERE email LIKE '%@gmail.com'

IN — Multiple Values

Prisma
where: {
  status: { in: ['draft', 'review'] }
}
SQL
WHERE status IN ('draft', 'review')

BETWEEN — Range

Prisma
where: {
  views: { gte: 100, lte: 1000 }
}
SQL
WHERE views BETWEEN 100 AND 1000

AND, OR — Combining Conditions

JavaScript
// AND (both must be true)
if (published && views > 1000)

// OR (either can be true)
if (status === 'draft' || status === 'review')
SQL
-- AND (both must be true)
WHERE published = true AND views > 1000

-- OR (either can be true)
WHERE status = 'draft' OR status = 'review'

ORDER BY — Sorting Results

ORDER BY is like .sort() in JavaScript or orderBy in Prisma. It controls the order of your results.

Prisma
await prisma.article.findMany({
  orderBy: { created_at: 'desc' },
});

// Multiple sort fields
await prisma.article.findMany({
  orderBy: [
    { published: 'desc' },
    { title: 'asc' },
  ],
});
SQL
SELECT * FROM articles
ORDER BY created_at DESC;


-- Multiple sort fields
SELECT * FROM articles
ORDER BY published DESC, title ASC;
  • ASC = ascending (A-Z, 1-100, oldest first). This is the default.
  • DESC = descending (Z-A, 100-1, newest first).

LIMIT & OFFSET — Pagination

If you've used pagination in Prisma with take and skip, you already know LIMIT and OFFSET.

Prisma
// First 10 results
await prisma.article.findMany({
  take: 10,
});

// Page 3 (skip first 20, take 10)
await prisma.article.findMany({
  skip: 20,
  take: 10,
});
SQL
-- First 10 results
SELECT * FROM articles
LIMIT 10;

-- Page 3 (skip first 20, take 10)
SELECT * FROM articles
LIMIT 10 OFFSET 20;

LIMIT goes at the very end. The order of SQL clauses matters: SELECT ... FROM ... WHERE ... ORDER BY ... LIMIT ... OFFSET ...

UPDATE & DELETE — Modifying Data

UPDATE

UPDATE changes existing rows. Like prisma.user.update(), you specify which rows to change (WHERE) and what to change (SET).

Prisma
await prisma.article.update({
  where: { id: 5 },
  data: {
    title: 'Updated Title',
    published: true,
  },
});

// Update many
await prisma.article.updateMany({
  where: { published: false },
  data: { views: 0 },
});
SQL
UPDATE articles
SET title = 'Updated Title',
    published = true
WHERE id = 5;


-- Update many
UPDATE articles
SET views = 0
WHERE published = false;

Always include WHERE with UPDATE and DELETE! Without it, you'll update or delete every row in the table. UPDATE articles SET views = 0 without a WHERE resets views for ALL articles. This is the SQL equivalent of rm -rf /.

DELETE

Prisma
// Delete one
await prisma.article.delete({
  where: { id: 5 },
});

// Delete many
await prisma.article.deleteMany({
  where: { published: false },
});
SQL
-- Delete one
DELETE FROM articles
WHERE id = 5;

-- Delete many
DELETE FROM articles
WHERE published = false;

Key Mental Model — SQL Reads Like English

Here's the thing that makes SQL click. Read this query out loud:

SELECT title, published, views
FROM articles
WHERE published = true
  AND views > 1000
ORDER BY views DESC
LIMIT 5;

In English: "Select the title, published status, and view count from the articles table, where published is true and views are more than 1000, ordered by views descending, and give me the top 5."

That's it. SQL is a declarative language — you tell it what you want, not how to get it. Compare that to JavaScript where you'd write loops and conditionals to get the same result:

JavaScript (imperative)
const result = articles
  .filter(a => a.published && a.views > 1000)
  .sort((a, b) => b.views - a.views)
  .slice(0, 5)
  .map(a => ({
    title: a.title,
    published: a.published,
    views: a.views,
  }));
SQL (declarative)
SELECT title, published, views
FROM articles
WHERE published = true
  AND views > 1000
ORDER BY views DESC
LIMIT 5;

SQL's clause order maps to a logical pipeline. Think of each clause as a step: (1) FROM = which table, (2) WHERE = filter rows, (3) SELECT = pick columns, (4) ORDER BY = sort, (5) LIMIT = cap the count. Like chaining array methods.

The Full Query Template

SELECT columns        -- What columns to return (like .map())
FROM table_name       -- Which table to query
WHERE conditions      -- Filter rows (like .filter())
ORDER BY column       -- Sort results (like .sort())
LIMIT n               -- Cap the results (like .slice())
OFFSET n;             -- Skip results (for pagination)

You don't need all of these every time. SELECT ... FROM ... is the minimum. Add the rest as needed.

Quick Check

1. What's the Prisma equivalent of SELECT * FROM users WHERE age > 25?

prisma.user.findMany({ where: { age: { gt: 25 } } })

2. What does SELECT title FROM articles LIMIT 3 return?

The title column of the first 3 rows from the articles table (in whatever order the database chooses, since there's no ORDER BY).

3. What's wrong with DELETE FROM articles (no WHERE)?

It deletes EVERY row in the articles table. Always include a WHERE clause unless you truly want to wipe the entire table.

4. Strings in SQL use single quotes or double quotes?

Single quotes: 'hello'. Double quotes are for identifiers (column/table names): "users"."firstName".

Practice Exercise

Imagine you have a table called sources with columns: id, name, outlet, beat, trust_level (1-10), last_contact.

Write SQL queries for each of these (check yourself by clicking "Show answer"):

A: Get all sources.

SELECT * FROM sources;

B: Get names and outlets of sources with trust_level >= 8.

SELECT name, outlet
FROM sources
WHERE trust_level >= 8;

C: Get all sources on the "politics" beat, sorted by trust_level (highest first), limit 5.

SELECT *
FROM sources
WHERE beat = 'politics'
ORDER BY trust_level DESC
LIMIT 5;

D: Update trust_level to 10 for the source with id = 42.

UPDATE sources
SET trust_level = 10
WHERE id = 42;

E: Delete sources with trust_level less than 3.

DELETE FROM sources
WHERE trust_level < 3;

Cheat Sheet

-- Create a table
CREATE TABLE table_name (
  id     SERIAL PRIMARY KEY,
  name   VARCHAR(255) NOT NULL,
  active BOOLEAN DEFAULT true
);

-- Insert a row
INSERT INTO table_name (name, active)
VALUES ('Trevor', true);

-- Insert multiple rows
INSERT INTO table_name (name, active)
VALUES ('Alice', true), ('Bob', false);

-- Select all
SELECT * FROM table_name;

-- Select with filter, sort, pagination
SELECT name, active
FROM table_name
WHERE active = true
ORDER BY name ASC
LIMIT 10 OFFSET 0;

-- Update
UPDATE table_name
SET active = false
WHERE id = 1;

-- Delete
DELETE FROM table_name
WHERE id = 1;

-- Drop (delete) the entire table
DROP TABLE table_name;
