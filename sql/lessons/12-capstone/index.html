Pagination — OFFSET/LIMIT vs Cursor-Based

You already know LIMIT and OFFSET from Lesson 01. But there's a catch: OFFSET gets slower as the page number grows. Page 1000 means the database has to scan and skip 10,000 rows before giving you 10. Let's look at both approaches.

OFFSET/LIMIT (Simple but Slow)

Prisma
// Page 1
const page1 = await prisma.article
  .findMany({
    take: 10,
    skip: 0,
    orderBy: { createdAt: 'desc' },
  });

// Page 50
const page50 = await prisma.article
  .findMany({
    take: 10,
    skip: 490, // Skips 490 rows!
    orderBy: { createdAt: 'desc' },
  });
SQL
-- Page 1
SELECT * FROM articles
ORDER BY created_at DESC
LIMIT 10 OFFSET 0;

-- Page 50
SELECT * FROM articles
ORDER BY created_at DESC
LIMIT 10 OFFSET 490;
-- Database scans 500 rows,
-- throws away 490, returns 10.
-- Imagine OFFSET 10000...

OFFSET pagination works fine when:

  • The table is small (under ~100K rows)
  • Users rarely go past page 10
  • You need "jump to page 47" functionality

Cursor-Based Pagination (Fast at Any Depth)

Instead of saying "skip N rows," cursor pagination says "give me rows after this specific row." It uses an indexed column (usually id or created_at) as a bookmark.

Prisma cursor
// First page
const page1 = await prisma.article
  .findMany({
    take: 10,
    orderBy: { id: 'desc' },
  });

// Next page: start after last item
const page2 = await prisma.article
  .findMany({
    take: 10,
    cursor: { id: lastId },
    skip: 1, // skip the cursor itself
    orderBy: { id: 'desc' },
  });
SQL cursor
-- First page
SELECT * FROM articles
ORDER BY id DESC
LIMIT 10;

-- Next page (last item had id = 95)
SELECT * FROM articles
WHERE id < 95
ORDER BY id DESC
LIMIT 10;
-- Uses the index! Instant, no matter
-- how deep you paginate.

Cursor pagination is what Twitter, Instagram, and infinite scroll use. The API returns a "next cursor" token with each response. Your frontend sends it back to get the next batch. It's always fast because WHERE id < 95 uses the primary key index directly — no scanning required.

Full-Text Search with tsvector & tsquery

You know LIKE '%word%' and ILIKE. But for real search — the kind where users type "election results 2024" and get relevant articles ranked by relevance — you need PostgreSQL's full-text search.

The mental model: PostgreSQL breaks text into tokens (words), stems them (so "running" matches "run"), removes stop words ("the", "a"), and stores them in a special data type called tsvector. You search with tsquery.

Basic Full-Text Search

-- Convert text to tsvector (tokenized, stemmed)
SELECT to_tsvector('english', 'The election results are in for 2024');
-- Result: '2024':7 'elect':2 'result':3

-- Create a search query
SELECT to_tsquery('english', 'election & results');
-- Result: 'elect' & 'result'

-- Search!
SELECT title, body
FROM articles
WHERE to_tsvector('english', title || ' ' || body)
      @@ to_tsquery('english', 'election & results');

The @@ operator means "matches." It checks if the tsvector (document) matches the tsquery (search terms).

Adding a Search Column (Performance)

Calling to_tsvector() on every query is slow. Instead, store the vector in a dedicated column and index it:

-- Add a search column
ALTER TABLE articles
ADD COLUMN search_vector tsvector;

-- Populate it
UPDATE articles
SET search_vector = to_tsvector('english',
  COALESCE(title, '') || ' ' || COALESCE(body, '')
);

-- Create a GIN index for fast search
CREATE INDEX articles_search_idx
ON articles USING GIN(search_vector);

-- Keep it updated with a trigger
CREATE FUNCTION update_search_vector()
RETURNS TRIGGER AS $$
BEGIN
  NEW.search_vector := to_tsvector('english',
    COALESCE(NEW.title, '') || ' ' || COALESCE(NEW.body, '')
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER articles_search_update
BEFORE INSERT OR UPDATE ON articles
FOR EACH ROW
EXECUTE FUNCTION update_search_vector();

Ranked Search Results

-- Search with relevance ranking
SELECT
  title,
  ts_rank(search_vector,
    to_tsquery('english', 'election & results')) AS relevance
FROM articles
WHERE search_vector @@ to_tsquery('english', 'election & results')
ORDER BY relevance DESC
LIMIT 20;

Search Query Syntax

OperatorMeaningExample
&AND (both words)'election & fraud'
|OR (either word)'election | primary'

<->Followed by (phrase)'election <-> results'
Prisma (limited)
// Prisma has no built-in full-text search
// Use raw SQL:
const results = await prisma.$queryRaw`
  SELECT id, title,
    ts_rank(search_vector,
      to_tsquery('english', ${query})
    ) AS relevance
  FROM articles
  WHERE search_vector @@
    to_tsquery('english', ${query})
  ORDER BY relevance DESC
  LIMIT 20
`;
SQL (full power)
-- Fuzzy user input with websearch_to_tsquery
-- (handles plain language input)
SELECT title, ts_rank(search_vector, q) AS rank
FROM articles,
  websearch_to_tsquery('english',
    'election fraud 2024') AS q
WHERE search_vector @@ q
ORDER BY rank DESC
LIMIT 20;

-- websearch_to_tsquery handles:
-- "election fraud" -> phrase search
-- election OR primary -> OR logic
-- -spam -> exclude word

PostgreSQL full-text search is surprisingly good. For most apps, you don't need Elasticsearch or Algolia. PostgreSQL handles stemming, ranking, phrase matching, and fuzzy search right in your database. Only reach for a dedicated search engine if you need typo tolerance or need to search millions of documents with sub-100ms response times.

JSONB Columns — Structured Flexibility

Sometimes you need to store data that doesn't fit neatly into columns. Metadata, settings, API responses, tags. PostgreSQL's JSONB type lets you store JSON documents inside a column and query into them with SQL.

Prisma Json type
model Article {
  id       Int  @id
  title    String
  metadata Json  // stores any JSON
}

// Write
await prisma.article.create({
  data: {
    title: 'Story',
    metadata: {
      source: 'AP',
      tags: ['politics', 'breaking'],
      wordCount: 850,
    },
  },
});
SQL JSONB
CREATE TABLE articles (
  id       SERIAL PRIMARY KEY,
  title    TEXT NOT NULL,
  metadata JSONB DEFAULT '{}'
);

-- Write
INSERT INTO articles (title, metadata)
VALUES ('Story', '{
  "source": "AP",
  "tags": ["politics", "breaking"],
  "wordCount": 850
}');

Querying JSONB

-- Extract a value (returns JSON)
SELECT metadata->'source' FROM articles;
-- Result: "AP" (with quotes, it's JSON)

-- Extract as text (returns text)
SELECT metadata->>'source' FROM articles;
-- Result: AP (no quotes, plain text)

-- Nested access
SELECT metadata->'stats'->>'readTime' FROM articles;

-- Filter by JSONB value
SELECT * FROM articles
WHERE metadata->>'source' = 'AP';

-- Check if JSONB contains a key-value pair
SELECT * FROM articles
WHERE metadata @> '{"source": "AP"}';

-- Check if a key exists
SELECT * FROM articles
WHERE metadata ? 'source';

-- Query into arrays
SELECT * FROM articles
WHERE metadata->'tags' ? 'politics';
-- Finds articles where tags array contains 'politics'

JSONB Operators Quick Reference

OperatorReturnsExample
->JSON elementmetadata->'source' returns "AP"
->>Text valuemetadata->>'source' returns AP
@>Containsmetadata @> '{"source":"AP"}'
?Key existsmetadata ? 'source'
?|Any key existsmetadata ?| array['a','b']
?&All keys existmetadata ?& array['a','b']

Indexing JSONB

-- GIN index for @>, ?, ?|, ?& operators
CREATE INDEX articles_metadata_idx
ON articles USING GIN(metadata);

-- B-tree index on a specific key (for = and range queries)
CREATE INDEX articles_source_idx
ON articles ((metadata->>'source'));

When to use JSONB vs columns: Use columns for data you filter, sort, or join on frequently (author_id, published, created_at). Use JSONB for variable/optional metadata that differs between rows (SEO settings, API response data, user preferences). If you're querying a JSONB field in every request, it should probably be a column instead.

Soft Deletes

In production, you almost never want DELETE FROM to actually remove data. What if an editor accidentally deletes an article? What if you need to comply with data retention laws? The solution is soft deletes: mark rows as deleted without removing them.

Hard delete (permanent)
-- Data is GONE
DELETE FROM articles WHERE id = 42;

-- Prisma
await prisma.article.delete({
  where: { id: 42 },
});
Soft delete (recoverable)
-- Data stays, marked as deleted
UPDATE articles
SET deleted_at = NOW()
WHERE id = 42;

-- Prisma
await prisma.article.update({
  where: { id: 42 },
  data: { deletedAt: new Date() },
});

Setting Up Soft Deletes

-- Add the column
ALTER TABLE articles
ADD COLUMN deleted_at TIMESTAMP DEFAULT NULL;

-- A row is "deleted" when deleted_at is NOT NULL
-- A row is "active" when deleted_at IS NULL

The Key: Filter Everywhere

The catch is you need to remember to filter out deleted rows in every query. This is where views help:

-- Create a view that hides deleted rows
CREATE VIEW active_articles AS
SELECT * FROM articles
WHERE deleted_at IS NULL;

-- Now use the view instead of the table
SELECT * FROM active_articles;
-- Only shows non-deleted articles

-- When you need to see deleted items (admin panel, recovery)
SELECT * FROM articles
WHERE deleted_at IS NOT NULL
ORDER BY deleted_at DESC;

-- Restore a soft-deleted article
UPDATE articles
SET deleted_at = NULL
WHERE id = 42;
Prisma middleware approach
// Prisma middleware to auto-filter
prisma.$use(async (params, next) => {
  if (params.model === 'Article') {
    if (params.action === 'findMany') {
      params.args.where = {
        ...params.args.where,
        deletedAt: null,
      };
    }
    if (params.action === 'delete') {
      params.action = 'update';
      params.args.data = {
        deletedAt: new Date()
      };
    }
  }
  return next(params);
});
SQL view approach
-- View handles the filtering
CREATE VIEW active_articles AS
SELECT * FROM articles
WHERE deleted_at IS NULL;

-- All queries automatically exclude
-- deleted rows
SELECT * FROM active_articles
WHERE beat = 'politics';

-- Delete = update
UPDATE articles
SET deleted_at = NOW()
WHERE id = 42;

-- Permanent purge (after 90 days)
DELETE FROM articles
WHERE deleted_at < NOW() - INTERVAL '90 days';

Soft deletes add complexity but save careers. Every newsroom has a story about an editor who deleted the wrong article 5 minutes before deadline. With soft deletes, recovery is one UPDATE away. Without them, you're restoring from backups and hoping they're recent.

Audit Trails with Triggers

An audit trail records who changed what and when. It's essential for newsrooms (editorial accountability), healthcare (HIPAA), finance (SOX compliance), and any app where you need to answer "who edited this article at 3am?"

-- Audit log table: stores who changed what and when
CREATE TABLE audit_log (
  id          SERIAL PRIMARY KEY,
  table_name  TEXT NOT NULL,
  row_id      INTEGER NOT NULL,
  action      TEXT NOT NULL,  -- 'INSERT', 'UPDATE', 'DELETE'
  old_data    JSONB,          -- row before change
  new_data    JSONB,          -- row after change
  changed_by  TEXT,
  changed_at  TIMESTAMP DEFAULT NOW()
);

-- Generic trigger function (works for any table)
CREATE FUNCTION audit_trigger_fn()
RETURNS TRIGGER AS $$
BEGIN
  IF TG_OP = 'INSERT' THEN
    INSERT INTO audit_log (table_name, row_id, action, new_data, changed_by)
    VALUES (TG_TABLE_NAME, NEW.id, 'INSERT', to_jsonb(NEW),
      current_setting('app.current_user', true));
    RETURN NEW;
  ELSIF TG_OP = 'UPDATE' THEN
    INSERT INTO audit_log (table_name, row_id, action, old_data, new_data, changed_by)
    VALUES (TG_TABLE_NAME, NEW.id, 'UPDATE', to_jsonb(OLD), to_jsonb(NEW),
      current_setting('app.current_user', true));
    RETURN NEW;
  ELSIF TG_OP = 'DELETE' THEN
    INSERT INTO audit_log (table_name, row_id, action, old_data, changed_by)
    VALUES (TG_TABLE_NAME, OLD.id, 'DELETE', to_jsonb(OLD),
      current_setting('app.current_user', true));
    RETURN OLD;
  END IF;
END;
$$ LANGUAGE plpgsql;

-- Attach to any table
CREATE TRIGGER articles_audit
AFTER INSERT OR UPDATE OR DELETE ON articles
FOR EACH ROW EXECUTE FUNCTION audit_trigger_fn();

Querying the Audit Trail

-- Set user in your API route (lasts one transaction)
-- await prisma.$executeRaw`SET LOCAL app.current_user = ${userId}`;

-- Who edited article #42?
SELECT action, changed_by, changed_at,
  old_data->>'title' AS old_title,
  new_data->>'title' AS new_title
FROM audit_log
WHERE table_name = 'articles' AND row_id = 42
ORDER BY changed_at DESC;

-- What did user 'trevor' change today?
SELECT table_name, row_id, action, changed_at
FROM audit_log
WHERE changed_by = 'trevor' AND changed_at::DATE = CURRENT_DATE;

Database Migrations Strategy

You already know migrations from Prisma: npx prisma migrate dev. But what's actually happening? A migration is a versioned SQL file that changes your database schema. Each migration builds on the previous one, like Git commits for your database structure.

Prisma migrations
// 1. Change schema.prisma
model Article {
  id    Int    @id
  title String
  slug  String @unique // NEW!
}

// 2. npx prisma migrate dev --name add-slug
// Creates a SQL migration file
Generated SQL
-- The actual migration file
ALTER TABLE articles ADD COLUMN slug TEXT;

CREATE UNIQUE INDEX articles_slug_key
ON articles (slug);

-- Prisma tracks applied migrations
-- in a _prisma_migrations table

Safe Migration Practices

  • Never modify an applied migration. Create a new one instead. Migrations are immutable once pushed, like Git commits.
  • Add columns as nullable first. Backfill existing rows, then add the NOT NULL constraint in a second migration.
  • Don't drop columns immediately. Stop reading the column in your app first, deploy, then drop it in a later migration.
-- Safe way to add a required column (two-step)
-- Migration 1: Add nullable column
ALTER TABLE articles ADD COLUMN slug TEXT;

-- Migration 2 (after backfilling):
UPDATE articles SET slug = lower(replace(title, ' ', '-')) WHERE slug IS NULL;
ALTER TABLE articles ALTER COLUMN slug SET NOT NULL;
CREATE UNIQUE INDEX articles_slug_unique ON articles (slug);

Custom SQL in Prisma migrations: After generating a migration, edit the migration.sql file to add views, functions, triggers, and indexes that Prisma can't generate. Then run npx prisma migrate dev to apply.

Practical — Production Newsroom Database

Let's put everything together. Here's a production-ready schema for a newsroom CMS that uses every pattern from this lesson:

-- Core tables with soft deletes and search
CREATE TABLE articles (
  id            SERIAL PRIMARY KEY,
  title         TEXT NOT NULL,
  slug          TEXT UNIQUE,
  body          TEXT NOT NULL,
  beat          TEXT,
  published     BOOLEAN DEFAULT false,
  published_at  TIMESTAMP,
  views         INTEGER DEFAULT 0,
  metadata      JSONB DEFAULT '{}',
  author_id     INTEGER REFERENCES users(id),
  search_vector tsvector,
  deleted_at    TIMESTAMP,
  created_at    TIMESTAMP DEFAULT NOW(),
  updated_at    TIMESTAMP DEFAULT NOW()
);

-- Indexes: search, metadata, filtered for active rows
CREATE INDEX articles_search_idx ON articles USING GIN(search_vector);
CREATE INDEX articles_metadata_idx ON articles USING GIN(metadata);
CREATE INDEX articles_published_idx ON articles (published, published_at DESC)
  WHERE deleted_at IS NULL;

-- Auto-update search vector on insert/update
CREATE TRIGGER article_search_trigger
BEFORE INSERT OR UPDATE ON articles
FOR EACH ROW EXECUTE FUNCTION update_article_search();

-- Audit trigger (uses generic audit function from earlier)
CREATE TRIGGER articles_audit
AFTER INSERT OR UPDATE OR DELETE ON articles
FOR EACH ROW EXECUTE FUNCTION audit_trigger_fn();

-- Soft-delete views
CREATE VIEW active_articles AS
SELECT * FROM articles WHERE deleted_at IS NULL;

-- Dashboard materialized view
CREATE MATERIALIZED VIEW newsroom_dashboard AS
SELECT
  u.name AS author, u.beat,
  COUNT(*) FILTER (WHERE a.published) AS published,
  COUNT(*) FILTER (WHERE NOT a.published) AS drafts,
  SUM(a.views) AS total_views,
  MAX(a.published_at) AS latest_publish
FROM users u
LEFT JOIN active_articles a ON u.id = a.author_id
WHERE u.role = 'journalist'
GROUP BY u.id, u.name, u.beat;
CREATE UNIQUE INDEX ON newsroom_dashboard (author);
-- Sample queries against this production schema

-- 1. Full-text search with ranking
SELECT title, ts_rank(search_vector,
    websearch_to_tsquery('english', 'city budget cuts')) AS relevance
FROM active_articles
WHERE search_vector @@ websearch_to_tsquery('english', 'city budget cuts')
ORDER BY relevance DESC LIMIT 10;

-- 2. Cursor-based pagination
SELECT id, title, published_at, views FROM active_articles
WHERE published = true AND id < 500
ORDER BY id DESC LIMIT 20;

-- 3. JSONB metadata query
SELECT title, metadata->>'source' AS source FROM active_articles
WHERE metadata @> '{"tags": ["breaking"]}' AND published = true;

-- 4. Soft-delete (audit trail captured by trigger)
UPDATE articles SET deleted_at = NOW() WHERE id = 42;

-- 5. Refresh dashboard on a schedule
REFRESH MATERIALIZED VIEW CONCURRENTLY newsroom_dashboard;

Quick Check

1. Why is cursor-based pagination faster than OFFSET/LIMIT for deep pages?

OFFSET forces the database to scan and discard N rows before returning results. Cursor pagination uses a WHERE clause on an indexed column (e.g., WHERE id < 95), which jumps directly to the right position using the index. No scanning or discarding needed.

2. What's the difference between -> and ->> for JSONB?

-> returns the JSON element (with quotes for strings, preserving JSON type). ->> returns the value as plain text. Use ->> when you want to compare with text values: WHERE metadata->>'source' = 'AP'.

3. What's the advantage of database-level audit trails (triggers) over application-level ones (Prisma middleware)?

Database triggers capture ALL changes to the table, no matter where they come from (app, admin tool, migration script, another service). Application-level audit only captures changes made through that specific app. If an admin runs a raw SQL update, only triggers will catch it.

4. Why should you add new required columns in two migrations instead of one?

If you add a NOT NULL column to a table with existing rows, the migration fails (existing rows have no value for the new column). The safe approach: (1) add the column as nullable, (2) backfill existing rows with a default value, (3) add the NOT NULL constraint.

5. When would you use websearch_to_tsquery instead of to_tsquery?

websearch_to_tsquery accepts plain language input (like what a user types into a search box). It handles quoted phrases, OR keywords, and - for exclusion automatically. to_tsquery requires explicit operators (&, |, !) and will error on plain text input.
