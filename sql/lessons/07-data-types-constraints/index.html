07 Data Types & Constraints — Designing Bulletproof Schemas

← Back to lessons

In Prisma, you pick String, Int, Boolean and call it a day. In raw SQL, you have a richer set of data types and constraints that give you finer control over what goes into your database. This lesson is about designing schemas that reject bad data before it ever becomes a problem.

The Mental Model

Think of data types and constraints as two layers of defense:

  • Data types are like TypeScript types — they define what kind of value a column can hold (number, text, date, etc.).
  • Constraints are like validation rules — they define which values are valid (not null, unique, within a range, etc.).

Together, they form a contract. Any data that violates the contract gets rejected at the database level, regardless of what your application code does. Even if your frontend validation has a bug, even if someone hits your API directly, the database is the last line of defense.

Why this matters: In a Prisma/Next.js app, you might validate data in your API route and trust the ORM to enforce types. But what about direct database access? What about migrations that bypass your app? What about that one SQL script someone runs in production? Database-level constraints are the safety net that catches everything.

Numeric Types

JavaScript has one number type. PostgreSQL has several, each optimized for different use cases:

SQL TypeRangeStorageUse Case

Prisma types
model Article {
  id       Int     @id @default(autoincrement())
  views    Int

PostgreSQL types
CREATE TABLE articles (
  id       SERIAL PRIMARY KEY,
  views    INTEGER,

Never use FLOAT or REAL for money. Floating-point math introduces rounding errors: 0.1 + 0.2 = 0.30000000000000004. Use NUMERIC(10,2) for money — it stores exact decimal values. NUMERIC(10,2) means up to 10 total digits with 2 after the decimal point: max value 99,999,999.99.

Text Types

SQL TypeDescriptionWhen to Use

VARCHAR(n)Variable-length with max limitWhen you want to enforce a max length (titles, names)
TEXTVariable-length, no limitArticle bodies, descriptions, any long text
 
CREATE TABLE articles (
  id         SERIAL PRIMARY KEY,
  slug       VARCHAR(100) NOT NULL,   -- URL slug: limited length
  title      VARCHAR(255) NOT NULL,   -- Title: reasonable max
  subtitle   VARCHAR(500),            -- Optional subtitle
  body       TEXT NOT NULL,            -- Article body: no limit needed
  state_code CHAR(2)                  -- Always exactly 2 characters
);

In PostgreSQL, TEXT and VARCHAR perform identically. Unlike some other databases, PostgreSQL doesn't store TEXT less efficiently than VARCHAR. The only difference is VARCHAR(n) enforces a max length. Use VARCHAR(n) when you want that enforcement; use TEXT when you don't care about length.

model Article {

  slug  String   @db.VarChar(100)
  body  String
}
PostgreSQL
-- Prisma's String maps to TEXT by default
-- Use @db.VarChar(n) to get VARCHAR(n)
title TEXT,
slug  VARCHAR(100),
body  TEXT

Boolean, Date, and Timestamp

BOOLEAN

PostgreSQL booleans accept multiple representations, but they all store as true/false:

-- All of these are valid
INSERT INTO articles (published) VALUES (true);
INSERT INTO articles (published) VALUES ('yes');
INSERT INTO articles (published) VALUES ('t');
INSERT INTO articles (published) VALUES (1);

-- In queries, use true/false
SELECT * FROM articles WHERE published = true;

DATE and TIMESTAMP

SQL TypeStoresExamplePrisma Type
DATEDate only (no time)'2024-03-15'DateTime @db.Date
TIMETime only (no date)'14:30:00'Not natively supported
TIMESTAMPDate + time, no timezone'2024-03-15 14:30:00'DateTime
TIMESTAMPTZDate + time + timezone'2024-03-15 14:30:00+00'DateTime (default)

Always use TIMESTAMPTZ (TIMESTAMP WITH TIME ZONE) for timestamps. Without the timezone, you'll run into bugs when your server, your database, and your users are in different timezones. Prisma's DateTime maps to TIMESTAMPTZ by default — they got this one right.

CREATE TABLE articles (
  id            SERIAL PRIMARY KEY,
  title         VARCHAR(255) NOT NULL,
  published_at  TIMESTAMPTZ,          -- when it went live (with timezone)
  embargo_date  DATE,                  -- date only, no time needed
  created_at    TIMESTAMPTZ DEFAULT NOW(),
  updated_at    TIMESTAMPTZ DEFAULT NOW()
);

Date arithmetic

-- Articles published in the last 7 days
SELECT * FROM articles
WHERE published_at > NOW() - INTERVAL '7 days';

-- Articles published this year
SELECT * FROM articles
WHERE published_at >= DATE_TRUNC('year', NOW());

-- How many days old is each article?
SELECT title,
  published_at,
  NOW() - published_at AS age,
  EXTRACT(DAY FROM NOW() - published_at) AS days_old
FROM articles;

UUID — Universal Unique Identifiers

A UUID is a 128-bit identifier that looks like a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11. Unlike SERIAL (auto-incrementing integers), UUIDs can be generated anywhere — client-side, server-side, across multiple databases — without conflicts.

SERIAL (auto-increment)
-- Pros: short, readable, ordered
-- Cons: sequential (reveals count),
--   can't generate outside the DB
CREATE TABLE articles (
  id SERIAL PRIMARY KEY
);
-- id: 1, 2, 3, 4, 5...
UUID
-- Pros: globally unique, no collisions,
--   can generate client-side
-- Cons: larger, not human-readable
CREATE TABLE articles (
  id UUID PRIMARY KEY
    DEFAULT gen_random_uuid()
);
-- id: a0eebc99-9c0b-4ef8-...
Prisma (UUID)
model Article {
  id String @id @default(uuid())
}
PostgreSQL (UUID)
CREATE TABLE articles (
  id UUID PRIMARY KEY
    DEFAULT gen_random_uuid()
);

When to use UUID vs SERIAL: Use UUID when IDs are exposed in URLs (prevents enumeration attacks), when you need to generate IDs client-side before inserting, or in distributed systems. Use SERIAL when performance and simplicity matter most (UUID indexes are larger and slightly slower).

JSONB — Structured Data Inside a Column

PostgreSQL's JSONB (Binary JSON) lets you store JSON objects directly in a column. It's like embedding a NoSQL document inside a relational table.

CREATE TABLE articles (
  id       SERIAL PRIMARY KEY,
  title    VARCHAR(255) NOT NULL,
  metadata JSONB DEFAULT '{}'
);

-- Insert with JSONB data
INSERT INTO articles (title, metadata) VALUES (
  'Election Results',
  '{"tags": ["politics", "election"], "word_count": 1200, "featured": true}'
);

-- Query JSONB fields
SELECT title,
  metadata->>'word_count' AS word_count,       -- ->> returns text
  metadata->'tags' AS tags,                     -- -> returns JSON
  (metadata->>'featured')::BOOLEAN AS featured  -- cast to boolean
FROM articles
WHERE metadata->>'featured' = 'true';

The key operators for JSONB:

OperatorReturnsExample

->>Text valuemetadata->>'word_count' returns '1200' (as text)
@>Contains checkmetadata @> '{"featured": true}'

Prisma (JSON fields)
model Article {
  id       Int  @id @default(autoincrement())
  title    String
  metadata Json @default("{}")
}

// Querying JSON in Prisma
await prisma.article.findMany({
  where: {
    metadata: {
      path: ['featured'],
      equals: true,
    },
  },
});
PostgreSQL (JSONB)
CREATE TABLE articles (
  id       SERIAL PRIMARY KEY,
  title    VARCHAR(255) NOT NULL,
  metadata JSONB DEFAULT '{}'
);

-- Same query in raw SQL
SELECT * FROM articles
WHERE metadata @> '{"featured": true}';

-- Or using the ->> operator
SELECT * FROM articles
WHERE metadata->>'featured' = 'true';

When to use JSONB vs separate columns: Use JSONB for flexible, schema-less data that varies between rows (metadata, settings, tags). Use regular columns for data you frequently query, filter, or join on. If you find yourself always querying the same JSONB key, it probably should be its own column.

Constraints — NOT NULL and DEFAULT

Now we move from data types (what kind of data) to constraints (what rules the data must follow).

NOT NULL

NOT NULL means a column must have a value — you can't insert a row without providing it (unless there's a DEFAULT).

Prisma (required by default)
model Article {
  title     String      // required (NOT NULL)
  subtitle  String?     // optional (nullable)
}
PostgreSQL (nullable by default)
CREATE TABLE articles (
  title    VARCHAR(255) NOT NULL, -- required
  subtitle VARCHAR(255)           -- nullable
);

Prisma and SQL have opposite defaults. In Prisma, fields are required by default; you add ? to make them optional. In SQL, columns are nullable by default; you add NOT NULL to make them required. This is one of the biggest gotchas when going from Prisma to raw SQL.

DEFAULT

DEFAULT provides a value when none is specified during INSERT:

CREATE TABLE articles (
  id           SERIAL PRIMARY KEY,
  title        VARCHAR(255) NOT NULL,
  published    BOOLEAN DEFAULT false,
  views        INTEGER DEFAULT 0,
  status       VARCHAR(20) DEFAULT 'draft',
  created_at   TIMESTAMPTZ DEFAULT NOW(),
  updated_at   TIMESTAMPTZ DEFAULT NOW()
);

-- These two INSERTs produce the same result:
INSERT INTO articles (title) VALUES ('Breaking Story');
INSERT INTO articles (title, published, views, status, created_at, updated_at)
VALUES ('Breaking Story', false, 0, 'draft', NOW(), NOW());

UNIQUE and CHECK Constraints

UNIQUE

UNIQUE ensures no two rows have the same value in that column. The database will reject any INSERT or UPDATE that would create a duplicate.

CREATE TABLE users (
  id    SERIAL PRIMARY KEY,
  email VARCHAR(255) NOT NULL UNIQUE,  -- no duplicate emails
  slug  VARCHAR(100) NOT NULL UNIQUE   -- no duplicate URL slugs
);

-- This works:
INSERT INTO users (email, slug) VALUES ('sarah@news.com', 'sarah-chen');

-- This fails with an error:
INSERT INTO users (email, slug) VALUES ('sarah@news.com', 'sarah-c');
-- ERROR: duplicate key value violates unique constraint "users_email_key"

Composite UNIQUE

You can enforce uniqueness across a combination of columns:

Prisma
model ArticleTag {
  articleId Int
  tagId     Int

  @@unique([articleId, tagId])
}
PostgreSQL
CREATE TABLE article_tags (
  article_id INTEGER NOT NULL,
  tag_id     INTEGER NOT NULL,
  UNIQUE (article_id, tag_id)
);

CHECK — Custom Validation

CHECK lets you define arbitrary conditions that must be true for every row. It's like adding validation rules at the database level.

CREATE TABLE articles (
  id         SERIAL PRIMARY KEY,
  title      VARCHAR(255) NOT NULL,
  word_count INTEGER CHECK (word_count > 0),
  rating     NUMERIC(2,1) CHECK (rating >= 0 AND rating <= 5),
  status     VARCHAR(20) CHECK (status IN ('draft', 'review', 'published', 'archived'))
);

-- This works:
INSERT INTO articles (title, word_count, rating, status)
VALUES ('Good Story', 500, 4.5, 'draft');

-- These all fail:
INSERT INTO articles (title, word_count) VALUES ('Bad', -10);
-- ERROR: violates check constraint (word_count > 0)

INSERT INTO articles (title, rating) VALUES ('Bad', 6.0);
-- ERROR: violates check constraint (rating >= 0 AND rating <= 5)

INSERT INTO articles (title, status) VALUES ('Bad', 'deleted');
-- ERROR: violates check constraint (status IN (...))

CHECK is the database equivalent of Zod validation. In your Next.js app, you probably validate with Zod or similar at the API layer. CHECK constraints add the same rules at the database layer — defense in depth. Even if someone bypasses your API, the database rejects invalid data.

Zod validation (API layer)
const articleSchema = z.object({
  title: z.string().min(1),
  wordCount: z.number().positive(),
  rating: z.number().min(0).max(5),
  status: z.enum([
    'draft', 'review',
    'published', 'archived'
  ]),
});
CHECK constraints (DB layer)
CREATE TABLE articles (
  title VARCHAR(255) NOT NULL
    CHECK (LENGTH(title) > 0),
  word_count INTEGER
    CHECK (word_count > 0),
  rating NUMERIC(2,1)
    CHECK (rating >= 0 AND rating <= 5),
  status VARCHAR(20)
    CHECK (status IN ('draft', 'review',
      'published', 'archived'))
);

Primary Keys — SERIAL vs UUID

Every table needs a primary key — a column (or combination of columns) that uniquely identifies each row. You've seen two approaches:

-- Option 1: Auto-incrementing integer (most common)
CREATE TABLE articles (
  id SERIAL PRIMARY KEY
);

-- Option 2: UUID (globally unique)
CREATE TABLE articles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid()
);

-- Option 3: Composite primary key (for junction/join tables)
CREATE TABLE article_tags (
  article_id INTEGER NOT NULL,
  tag_id     INTEGER NOT NULL,
  PRIMARY KEY (article_id, tag_id)
);

A primary key automatically gets:

  • NOT NULL — can't be empty
  • UNIQUE — can't be duplicated
  • B-tree index — fast lookups

Foreign Keys — Referential Integrity

A foreign key is a column that references the primary key of another table. It's how you create relationships between tables, and it's what makes relational databases relational.

Prisma (relations)
model Article {
  id       Int  @id @default(autoincrement())
  title    String
  authorId Int
  author   User @relation(fields: [authorId],
    references: [id])
}

model User {
  id       Int       @id @default(autoincrement())
  name     String
  articles Article[]
}
PostgreSQL (foreign keys)
CREATE TABLE users (
  id   SERIAL PRIMARY KEY,
  name VARCHAR(255) NOT NULL
);

CREATE TABLE articles (
  id        SERIAL PRIMARY KEY,
  title     VARCHAR(255) NOT NULL,
  author_id INTEGER NOT NULL
    REFERENCES users(id)
);

The foreign key constraint guarantees referential integrity: you can't insert an article with an author_id that doesn't exist in the users table.

-- This works (user with id=1 exists):
INSERT INTO articles (title, author_id) VALUES ('Breaking News', 1);

-- This fails (no user with id=999):
INSERT INTO articles (title, author_id) VALUES ('Ghost Story', 999);
-- ERROR: insert or update on table "articles" violates foreign key constraint
-- Key (author_id)=(999) is not present in table "users".

Prisma handles this for you. When you define a @relation, Prisma creates the foreign key constraint in the migration. But understanding the underlying SQL helps you debug constraint violation errors — which are among the most common errors you'll see in production.

CASCADE, SET NULL, RESTRICT

What happens when you delete a user who has articles? The foreign key constraint needs to know what to do with the orphaned articles. This is called the referential action.

ActionOn DELETEOn UPDATEWhen to Use
RESTRICTBlock the deleteBlock the updateDefault. Prevents accidental data loss.
CASCADEDelete child rows tooUpdate child FK tooWhen children have no meaning without parent (comments on a deleted post).
SET NULLSet FK to NULLSet FK to NULLWhen children can exist without parent (articles by a deleted user).
SET DEFAULTSet FK to its defaultSet FK to its defaultRare. When you want a fallback value.
NO ACTIONSame as RESTRICT (check deferred)Same as RESTRICTWhen using deferred constraints.
 
-- RESTRICT: can't delete a user who has articles
CREATE TABLE articles (
  id        SERIAL PRIMARY KEY,
  author_id INTEGER REFERENCES users(id) ON DELETE RESTRICT
);

-- CASCADE: deleting a user deletes all their articles
CREATE TABLE articles (
  id        SERIAL PRIMARY KEY,
  author_id INTEGER REFERENCES users(id) ON DELETE CASCADE
);

-- SET NULL: deleting a user sets author_id to NULL
CREATE TABLE articles (
  id        SERIAL PRIMARY KEY,
  author_id INTEGER REFERENCES users(id) ON DELETE SET NULL
);
Prisma (onDelete)
model Article {
  id       Int  @id @default(autoincrement())
  authorId Int?
  author   User? @relation(
    fields: [authorId],
    references: [id],
    onDelete: SetNull
  )
}
// Options: Cascade, Restrict,
// NoAction, SetNull, SetDefault
PostgreSQL
CREATE TABLE articles (
  id        SERIAL PRIMARY KEY,
  author_id INTEGER
    REFERENCES users(id)
    ON DELETE SET NULL
    ON UPDATE CASCADE
);

-- ON DELETE: what happens when
--   the referenced row is deleted
-- ON UPDATE: what happens when
--   the referenced PK changes

Be careful with CASCADE. ON DELETE CASCADE can create a chain reaction: deleting one row might delete thousands of related rows across multiple tables. Always know your cascade paths. In a newsroom schema, you probably want SET NULL for articles (keep the article, just clear the author) and CASCADE for comments (if the article goes, comments go too).

Practical: Designing a Newsroom Schema

Let's put it all together and design a properly typed and constrained schema for a newsroom CMS:

-- Users / authors
CREATE TABLE users (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email      VARCHAR(255) NOT NULL UNIQUE,
  name       VARCHAR(255) NOT NULL,
  role       VARCHAR(20) NOT NULL DEFAULT 'writer'
             CHECK (role IN ('writer', 'editor', 'admin')),
  bio        TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Sections (news, opinion, sports, etc.)
CREATE TABLE sections (
  id   SERIAL PRIMARY KEY,
  name VARCHAR(100) NOT NULL UNIQUE,
  slug VARCHAR(100) NOT NULL UNIQUE
);

-- Articles
CREATE TABLE articles (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  title          VARCHAR(255) NOT NULL CHECK (LENGTH(title) > 0),
  slug           VARCHAR(255) NOT NULL UNIQUE,
  body           TEXT NOT NULL,
  excerpt        VARCHAR(500),
  author_id      UUID NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
  section_id     INTEGER NOT NULL REFERENCES sections(id) ON DELETE RESTRICT,
  status         VARCHAR(20) NOT NULL DEFAULT 'draft'
                 CHECK (status IN ('draft', 'review', 'published', 'archived')),
  word_count     INTEGER CHECK (word_count >= 0),
  views          INTEGER NOT NULL DEFAULT 0 CHECK (views >= 0),
  featured       BOOLEAN NOT NULL DEFAULT false,
  metadata       JSONB DEFAULT '{}',
  published_at   TIMESTAMPTZ,
  created_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Tags
CREATE TABLE tags (
  id   SERIAL PRIMARY KEY,
  name VARCHAR(100) NOT NULL UNIQUE,
  slug VARCHAR(100) NOT NULL UNIQUE
);

-- Article-Tag junction table (many-to-many)
CREATE TABLE article_tags (
  article_id UUID NOT NULL REFERENCES articles(id) ON DELETE CASCADE,
  tag_id     INTEGER NOT NULL REFERENCES tags(id) ON DELETE CASCADE,
  PRIMARY KEY (article_id, tag_id)
);

-- Comments
CREATE TABLE comments (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  body       TEXT NOT NULL CHECK (LENGTH(body) > 0),
  article_id UUID NOT NULL REFERENCES articles(id) ON DELETE CASCADE,
  author_id  UUID REFERENCES users(id) ON DELETE SET NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

Notice the design decisions:

  • UUIDs for user-facing IDs (articles, users, comments) — prevents enumeration
  • SERIAL for internal IDs (sections, tags) — simpler and faster
  • RESTRICT on articles-to-users — don't accidentally delete a journalist's work
  • CASCADE on article_tags and comments — if the article goes, these go too
  • SET NULL on comments.author_id — keep the comment even if the user is deleted
  • CHECK constraints on status, word_count, views, title length — database-level validation
  • JSONB metadata for flexible, per-article data that varies (SEO settings, social media info, etc.)

Quick Check

1. In Prisma, fields are required by default. What's the default in PostgreSQL?

In PostgreSQL, columns are nullable by default. You must explicitly add NOT NULL to make them required. This is the opposite of Prisma.

2. Why should you never use REAL or FLOAT for money?

Floating-point types introduce rounding errors (e.g., 0.1 + 0.2 = 0.30000000000000004). Use NUMERIC(10,2) for exact decimal storage. Same reason you'd use a Decimal library instead of JavaScript's Number for financial calculations.

3. What's the difference between ON DELETE CASCADE and ON DELETE SET NULL?

CASCADE deletes the child rows (e.g., deleting an article deletes its comments). SET NULL keeps the child rows but sets the foreign key column to NULL (e.g., deleting a user sets the comment's author_id to NULL). Use CASCADE when children are meaningless without the parent; SET NULL when they can stand alone.

4. What does JSONB's ->> operator return, and how is it different from ->?

-> returns the value as a JSON element (preserving its type). ->> returns the value as text. Use ->> when you want to compare or display the value; use -> when you need to chain further JSON operations.

5. Why use TIMESTAMPTZ instead of TIMESTAMP?

TIMESTAMPTZ stores timezone information, so the same moment in time is represented correctly regardless of the server's timezone setting. TIMESTAMP (without TZ) can produce bugs when your server, database, or users are in different timezones.

Cheat Sheet

-- Numeric types
INTEGER                          -- whole numbers (most common)
SERIAL                           -- auto-incrementing integer
BIGINT                           -- very large integers
NUMERIC(10,2)                    -- exact decimal (money!)

-- Text types
VARCHAR(n)                       -- variable-length with max
TEXT                             -- variable-length, no max
CHAR(n)                          -- fixed-length

-- Date/Time types
DATE                             -- date only
TIMESTAMPTZ                      -- date + time + timezone (use this!)
INTERVAL                         -- duration

-- Other types
BOOLEAN                          -- true/false
UUID                             -- universally unique ID
JSONB                            -- binary JSON data

-- Constraints
NOT NULL                         -- column must have a value
UNIQUE                           -- no duplicate values
CHECK (condition)                -- custom validation rule
DEFAULT value                    -- value when none provided
PRIMARY KEY                      -- unique + not null + indexed
REFERENCES table(col)            -- foreign key

-- Foreign key actions
ON DELETE RESTRICT               -- block delete (default)
ON DELETE CASCADE                -- delete children too
ON DELETE SET NULL               -- set FK to NULL
ON UPDATE CASCADE                -- update FK when parent PK changes

-- UUID generation
DEFAULT gen_random_uuid()        -- PostgreSQL 13+

-- JSONB operators
column->'key'                    -- get JSON element
column->>'key'                   -- get as text
column @> '{"key": "val"}'      -- contains check
column ? 'key'                   -- key exists
