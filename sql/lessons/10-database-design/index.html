10 Views, Functions & Triggers

← Back to lessons

Think of views as saved queries and functions as stored JavaScript. You've been writing queries that you run once. Now you're going to package them up so the database remembers them — and can even run code automatically when data changes.

Views — Saved Queries You Can Reuse

Here's the mental model: a view is a saved SELECT query that acts like a virtual table. It doesn't store data itself — it runs the query every time you read from it. Think of it like a computed property in a React component: it derives its value from underlying data.

React computed value
// Derived from state, recalculated on render
const publishedArticles = articles
  .filter(a => a.published)
  .map(a => ({
    title: a.title,
    author: a.author.name,
    views: a.views,
  }));
SQL View (same idea)
-- Saved query, re-runs on every SELECT
CREATE VIEW published_articles AS
SELECT
  a.title,
  u.name AS author,
  a.views
FROM articles a
JOIN users u ON a.author_id = u.id
WHERE a.published = true;

Once created, you query a view exactly like a table:

-- Use it just like a table
SELECT * FROM published_articles;

-- Filter it, sort it, join it
SELECT * FROM published_articles
WHERE views > 1000
ORDER BY views DESC
LIMIT 10;

Why views matter: Instead of copying a complex 20-line query into every API route, you create it once as a view. Then every query is just SELECT * FROM my_view. It's the DRY principle applied to SQL.

Creating and Managing Views

-- Create a view
CREATE VIEW active_journalists AS
SELECT id, name, email, beat
FROM users
WHERE role = 'journalist' AND active = true;

-- Replace an existing view (update the query)
CREATE OR REPLACE VIEW active_journalists AS
SELECT id, name, email, beat, hire_date
FROM users
WHERE role = 'journalist' AND active = true;

-- Delete a view
DROP VIEW active_journalists;

-- Delete only if it exists (no error if missing)
DROP VIEW IF EXISTS active_journalists;

Views don't store data. They store the query definition. Every time you SELECT FROM a view, PostgreSQL runs the underlying query fresh. The data always reflects the current state of the base tables.

Updatable Views

Simple views (single table, no aggregation, no DISTINCT) can be written to. You can INSERT, UPDATE, and DELETE through them:

-- This view is updatable because it's a simple filter on one table
CREATE VIEW draft_articles AS
SELECT id, title, body, author_id, created_at
FROM articles
WHERE published = false;

-- Insert through the view (goes into articles table)
INSERT INTO draft_articles (title, body, author_id)
VALUES ('New Story', 'Working on it...', 3);

-- Update through the view
UPDATE draft_articles
SET title = 'Updated Draft Title'
WHERE id = 15;

-- Delete through the view
DELETE FROM draft_articles WHERE id = 15;

A view is automatically updatable when it meets these conditions:

  • References exactly one table (no JOINs)
  • No aggregate functions (COUNT, SUM, etc.)
  • No DISTINCT, GROUP BY, HAVING, UNION
  • No window functions

Watch out for the "disappearing row" trap. If you INSERT a row through draft_articles with published = true, the row goes into the articles table but won't appear in the view (because the view filters for published = false). Use WITH CHECK OPTION to prevent this:

-- WITH CHECK OPTION prevents inserts/updates that would
-- make the row invisible to the view
CREATE VIEW draft_articles AS
SELECT id, title, body, author_id, created_at
FROM articles
WHERE published = false
WITH CHECK OPTION;

-- This now FAILS because published=true wouldn't be in the view
INSERT INTO draft_articles (title, body, author_id, published)
VALUES ('Story', 'Content', 3, true);
-- ERROR: new row violates check option for view "draft_articles"

Materialized Views — Cached Query Results

Regular views re-run the query every time. Materialized views actually store the results — like caching. Think of it as the difference between a computed property and useMemo in React:

Regular View
-- Re-runs query every time
-- Always fresh, but can be slow
CREATE VIEW article_stats AS
SELECT
  author_id,
  COUNT(*) AS total,
  AVG(views) AS avg_views
FROM articles
GROUP BY author_id;
Materialized View
-- Stores results physically
-- Fast reads, but can be stale
CREATE MATERIALIZED VIEW article_stats AS
SELECT
  author_id,
  COUNT(*) AS total,
  AVG(views) AS avg_views
FROM articles
GROUP BY author_id;

The trade-off is simple: materialized views are fast to read but can be stale. You manually refresh them:

-- Refresh the data (re-runs the query, replaces stored results)
REFRESH MATERIALIZED VIEW article_stats;

-- Refresh without locking reads (other queries can still read old data)
REFRESH MATERIALIZED VIEW CONCURRENTLY article_stats;
-- Note: CONCURRENTLY requires a unique index on the materialized view

When to use materialized views: Dashboard queries that aggregate millions of rows but don't need real-time data. A newsroom dashboard showing "articles per beat this month" doesn't need to be recalculated on every page load — refreshing every 5 minutes is fine.

Stores dataNo (virtual)Yes (physical)
Always freshYesNo (must refresh)

User-Defined Functions

Functions in SQL are like functions in JavaScript — they take inputs, do something, and return a result. You write them once, then call them from any query. In PostgreSQL, you write functions using PL/pgSQL, which is basically SQL with variables, loops, and conditionals bolted on.

JavaScript function
function wordCount(text) {
  if (!text) return 0;
  return text
    .trim()
    .split(/\s+/)
    .length;
}
PostgreSQL function
CREATE FUNCTION word_count(text_input TEXT)
RETURNS INTEGER AS $$
BEGIN
  IF text_input IS NULL THEN
    RETURN 0;
  END IF;
  RETURN array_length(
    string_to_array(trim(text_input), ' '),
    1
  );
END;
$$ LANGUAGE plpgsql;

Let's break down the anatomy:

  • CREATE FUNCTION name(params) — defines the function name and inputs
  • RETURNS type — what type the function returns (like TypeScript return type)
  • $$ — dollar-quoting, marks the start/end of the function body (avoids escaping issues with single quotes)
  • BEGIN ... END — the function body
  • LANGUAGE plpgsql — tells PostgreSQL which language the body is written in

Now use it in queries:

-- Use the function in SELECT
SELECT title, word_count(body) AS words
FROM articles
ORDER BY words DESC;

-- Use it in WHERE
SELECT title FROM articles
WHERE word_count(body) > 500;

Simple SQL Functions

For simple calculations, you can skip PL/pgSQL and use a plain SQL function — it's cleaner:

-- Simple SQL function (no BEGIN/END needed)
CREATE FUNCTION reading_time(body TEXT)
RETURNS INTEGER AS $$
  SELECT GREATEST(1, word_count(body) / 200);
$$ LANGUAGE sql;

-- Use it
SELECT title, reading_time(body) AS minutes
FROM articles;

Functions That Return Tables

-- Return multiple rows (like an API endpoint)
CREATE FUNCTION get_articles_by_beat(beat_name TEXT)
RETURNS TABLE(title TEXT, author TEXT, views INTEGER) AS $$
BEGIN
  RETURN QUERY
  SELECT a.title, u.name, a.views
  FROM articles a
  JOIN users u ON a.author_id = u.id
  WHERE a.beat = beat_name AND a.published = true
  ORDER BY a.views DESC;
END;
$$ LANGUAGE plpgsql;

-- Call it (use like a table)
SELECT * FROM get_articles_by_beat('politics');

-- Filter the results further
SELECT * FROM get_articles_by_beat('politics')
WHERE views > 5000;

PL/pgSQL Basics

PL/pgSQL gives you the programming constructs you're used to from JavaScript. Here's a quick translation guide:

Variables

CREATE FUNCTION calculate_bonus(employee_id INTEGER)
RETURNS NUMERIC AS $$
DECLARE
  -- Variables go in the DECLARE block
  emp_salary NUMERIC;
  bonus_rate NUMERIC;
  result NUMERIC;
BEGIN
  -- Assign with :=  or SELECT INTO
  SELECT salary INTO emp_salary
  FROM employees WHERE id = employee_id;

  -- Conditional logic
  IF emp_salary > 100000 THEN
    bonus_rate := 0.10;
  ELSIF emp_salary > 60000 THEN
    bonus_rate := 0.15;
  ELSE
    bonus_rate := 0.20;
  END IF;

  result := emp_salary * bonus_rate;
  RETURN result;
END;
$$ LANGUAGE plpgsql;

Loops

CREATE FUNCTION generate_slugs()
RETURNS VOID AS $$
DECLARE
  article_row RECORD;
BEGIN
  -- FOR loop over query results
  FOR article_row IN
    SELECT id, title FROM articles WHERE slug IS NULL
  LOOP
    UPDATE articles
    SET slug = lower(replace(article_row.title, ' ', '-'))
    WHERE id = article_row.id;
  END LOOP;
END;
$$ LANGUAGE plpgsql;

When to use PL/pgSQL vs application code: Use PL/pgSQL when the logic needs to be close to the data (bulk operations, complex calculations on many rows). Use your Next.js app for business logic, API responses, and anything that needs external services. A good rule: if you'd need to fetch 10,000 rows to your app just to update them, do it in a function instead.

Triggers — Automatic Database Reactions

A trigger is code that runs automatically when something happens to a table. Think of it like an event listener in JavaScript:

JavaScript event listener
// Run code when something happens
document.addEventListener('click', (event) => {
  console.log('Clicked!', event.target);
});

// Middleware that runs before save
app.pre('save', function(next) {
  this.updatedAt = new Date();
  next();
});
SQL Trigger (same concept)
-- Run code when a row is modified
CREATE TRIGGER update_timestamp
BEFORE UPDATE ON articles
FOR EACH ROW
EXECUTE FUNCTION set_updated_at();

-- The trigger function
CREATE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

Triggers have two parts: (1) the trigger function that defines what to do, and (2) the trigger that defines when to do it.

Trigger Timing

OptionWhen It RunsUse Case
BEFORE INSERTBefore a new row is addedValidate or modify incoming data
AFTER INSERTAfter a new row is addedLog the change, notify systems
BEFORE UPDATEBefore a row is changedAuto-set updated_at
AFTER UPDATEAfter a row is changedAudit trail, sync to other table
BEFORE DELETEBefore a row is removedPrevent deletion, archive instead
AFTER DELETEAfter a row is removedLog what was deleted

NEW and OLD

Inside a trigger function, you have access to two special variables:

  • NEW — the row after the change (available in INSERT and UPDATE triggers)
  • OLD — the row before the change (available in UPDATE and DELETE triggers)
-- Example: auto-generate a slug when an article is created
CREATE FUNCTION generate_article_slug()
RETURNS TRIGGER AS $$
BEGIN
  -- NEW refers to the row being inserted
  NEW.slug := lower(replace(NEW.title, ' ', '-'));
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER article_slug_trigger
BEFORE INSERT ON articles
FOR EACH ROW
EXECUTE FUNCTION generate_article_slug();
-- Example: log article status changes
CREATE FUNCTION log_publish_event()
RETURNS TRIGGER AS $$
BEGIN
  -- OLD = before, NEW = after
  IF OLD.published = false AND NEW.published = true THEN
    INSERT INTO article_events (article_id, event_type, occurred_at)
    VALUES (NEW.id, 'published', NOW());
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER on_article_publish
AFTER UPDATE ON articles
FOR EACH ROW
EXECUTE FUNCTION log_publish_event();

BEFORE triggers must RETURN NEW (or RETURN OLD, or NULL to cancel the operation). AFTER triggers can return NULL since the row is already written. If a BEFORE trigger returns NULL, the INSERT/UPDATE/DELETE is cancelled.

Prisma and Views

Prisma has added support for views, but it's more limited than raw SQL. Here's how they compare:

Prisma view support
// schema.prisma
// Mark a model as a view
view PublishedArticles {
  id     Int    @id
  title  String
  author String
  views  Int

  @@map("published_articles")
}

// Query it like a model
const articles = await prisma
  .publishedArticles.findMany({
    where: { views: { gt: 1000 } },
  });
Raw SQL (full power)
-- Create the view
CREATE VIEW published_articles AS
SELECT a.id, a.title, u.name AS author,
       a.views
FROM articles a
JOIN users u ON a.author_id = u.id
WHERE a.published = true;

-- Query it
SELECT * FROM published_articles
WHERE views > 1000;

-- Plus: materialized views, functions,
-- triggers, updatable views, etc.
-- None of these exist in Prisma.

Prisma can't create views, functions, or triggers. You define these in raw SQL migrations. Prisma's view keyword only lets you read from views that already exist. For creating them, you'll use prisma migrate with custom SQL files or prisma.$queryRaw.

// Using raw SQL in your Next.js app for things Prisma can't do
// Call a function
const stats = await prisma.$queryRaw`
  SELECT * FROM get_articles_by_beat(${beat})
`;

// Refresh a materialized view (e.g., from a cron job)
await prisma.$executeRaw`
  REFRESH MATERIALIZED VIEW article_stats
`;

Practical — Newsroom Dashboard Views

Let's build the views and functions a real newsroom CMS dashboard would need:

-- 1. Dashboard overview: articles per author with stats
CREATE MATERIALIZED VIEW author_dashboard AS
SELECT
  u.id AS author_id,
  u.name AS author_name,
  u.beat,
  COUNT(*) AS total_articles,
  COUNT(*) FILTER (WHERE a.published = true) AS published,
  COUNT(*) FILTER (WHERE a.published = false) AS drafts,
  COALESCE(SUM(a.views), 0) AS total_views,
  ROUND(AVG(a.views), 0) AS avg_views,
  MAX(a.created_at) AS last_article_date
FROM users u
LEFT JOIN articles a ON u.id = a.author_id
WHERE u.role = 'journalist'
GROUP BY u.id, u.name, u.beat;

-- Create index for fast lookups
CREATE UNIQUE INDEX ON author_dashboard (author_id);

-- 2. Trending articles (published in last 7 days, high views)
CREATE VIEW trending_articles AS
SELECT
  a.id,
  a.title,
  u.name AS author,
  a.beat,
  a.views,
  a.published_at,
  a.views / GREATEST(1,
    EXTRACT(EPOCH FROM NOW() - a.published_at) / 3600
  ) AS views_per_hour
FROM articles a
JOIN users u ON a.author_id = u.id
WHERE a.published = true
  AND a.published_at > NOW() - INTERVAL '7 days'
ORDER BY views_per_hour DESC;
-- 3. Function: daily editorial report
CREATE FUNCTION daily_report(report_date DATE DEFAULT CURRENT_DATE)
RETURNS TABLE(
  metric TEXT,
  value BIGINT
) AS $$
BEGIN
  RETURN QUERY
  SELECT 'Articles published'::TEXT,
    COUNT(*)::BIGINT
  FROM articles
  WHERE published_at::DATE = report_date AND published = true;

  RETURN QUERY
  SELECT 'Total views today'::TEXT,
    COALESCE(SUM(views), 0)::BIGINT
  FROM articles
  WHERE published_at::DATE = report_date;

  RETURN QUERY
  SELECT 'New drafts'::TEXT,
    COUNT(*)::BIGINT
  FROM articles
  WHERE created_at::DATE = report_date AND published = false;

  RETURN QUERY
  SELECT 'Active authors'::TEXT,
    COUNT(DISTINCT author_id)::BIGINT
  FROM articles
  WHERE created_at::DATE = report_date;
END;
$$ LANGUAGE plpgsql;

-- Use it
SELECT * FROM daily_report();
SELECT * FROM daily_report('2025-12-01');
-- 4. Trigger: auto-set published_at when article is published
CREATE FUNCTION set_published_at()
RETURNS TRIGGER AS $$
BEGIN
  IF OLD.published = false AND NEW.published = true THEN
    NEW.published_at := NOW();
  END IF;
  IF OLD.published = true AND NEW.published = false THEN
    NEW.published_at := NULL;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER auto_published_at
BEFORE UPDATE ON articles
FOR EACH ROW
EXECUTE FUNCTION set_published_at();

Quick Check

1. What's the difference between a view and a materialized view?

A regular view is a stored query that runs fresh every time you SELECT from it. A materialized view stores the query results physically — fast to read but can be stale. You must REFRESH MATERIALIZED VIEW to update it.

2. What does RETURN NEW do in a BEFORE trigger function?

It tells PostgreSQL to proceed with the INSERT/UPDATE using the (possibly modified) row. If you return NULL instead, the operation is cancelled. You can modify fields on NEW before returning it.

3. Can Prisma create views and triggers?

No. Prisma can read from existing views (using the view keyword in the schema), but it cannot create views, materialized views, functions, or triggers. You create those with raw SQL, typically in migration files.

4. What are OLD and NEW in trigger functions?

OLD is the row before the change (available in UPDATE and DELETE). NEW is the row after the change (available in INSERT and UPDATE). They let you compare what changed and modify the incoming data.

Practice Exercise

You have tables: articles (id, title, body, beat, published, views, author_id, created_at, updated_at) and users (id, name, email, role, beat, active).

A: Create a view called editor_queue that shows unpublished articles with the author's name, ordered by creation date (oldest first).

CREATE VIEW editor_queue AS
SELECT a.id, a.title, u.name AS author,
       a.beat, a.created_at
FROM articles a
JOIN users u ON a.author_id = u.id
WHERE a.published = false
ORDER BY a.created_at ASC;

B: Create a materialized view called beat_performance that shows total articles, total views, and average views per beat.

CREATE MATERIALIZED VIEW beat_performance AS
SELECT
  beat,
  COUNT(*) AS total_articles,
  SUM(views) AS total_views,
  ROUND(AVG(views), 0) AS avg_views
FROM articles
WHERE published = true
GROUP BY beat;

C: Write a function article_summary(article_id INTEGER) that returns the title, author name, word count, and reading time (words / 200, minimum 1 minute).

CREATE FUNCTION article_summary(aid INTEGER)
RETURNS TABLE(title TEXT, author TEXT,
              words INTEGER, read_minutes INTEGER) AS $$
BEGIN
  RETURN QUERY
  SELECT a.title, u.name,
    array_length(string_to_array(trim(a.body), ' '), 1),
    GREATEST(1, array_length(
      string_to_array(trim(a.body), ' '), 1) / 200)
  FROM articles a
  JOIN users u ON a.author_id = u.id
  WHERE a.id = aid;
END;
$$ LANGUAGE plpgsql;

D: Create a trigger that automatically sets updated_at to the current timestamp whenever an article is modified.

CREATE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at := NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER auto_update_timestamp
BEFORE UPDATE ON articles
FOR EACH ROW
EXECUTE FUNCTION set_updated_at();

Cheat Sheet

-- Views
CREATE VIEW name AS SELECT ...;
CREATE OR REPLACE VIEW name AS SELECT ...;
DROP VIEW IF EXISTS name;

-- Materialized Views
CREATE MATERIALIZED VIEW name AS SELECT ...;
REFRESH MATERIALIZED VIEW name;
REFRESH MATERIALIZED VIEW CONCURRENTLY name;

-- Functions (PL/pgSQL)
CREATE FUNCTION name(params) RETURNS type AS $$
DECLARE
  var_name type;
BEGIN
  -- logic here
  RETURN value;
END;
$$ LANGUAGE plpgsql;

-- Functions (plain SQL)
CREATE FUNCTION name(params) RETURNS type AS $$
  SELECT expression;
$$ LANGUAGE sql;

-- Table-returning functions
CREATE FUNCTION name(params)
RETURNS TABLE(col1 type, col2 type) AS $$
BEGIN
  RETURN QUERY SELECT ...;
END;
$$ LANGUAGE plpgsql;

-- Triggers
CREATE FUNCTION trigger_fn() RETURNS TRIGGER AS $$
BEGIN
  -- NEW = incoming row, OLD = previous row
  NEW.column := value;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER name
BEFORE|AFTER INSERT|UPDATE|DELETE ON table
FOR EACH ROW
EXECUTE FUNCTION trigger_fn();

-- Drop
DROP FUNCTION IF EXISTS name(params);
DROP TRIGGER IF EXISTS name ON table;
