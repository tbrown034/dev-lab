02 Filtering & Selecting

← Back to lessons

Filtering a DataFrame is just a SQL WHERE clause wearing different syntax. If you can write SELECT * FROM users WHERE age > 30 or users.filter(u => u.age > 30), you already understand boolean indexing — this lesson maps both of those mental models onto pandas and warns you about the two gotchas that bite everyone.

Boolean Indexing — The WHERE Clause of Pandas

The core idea: you build a column of True/False values, then hand it back to the DataFrame to keep only the True rows. SQL hides this behind WHERE; JavaScript hides it behind a callback. Pandas makes the boolean mask visible.

SQL
SELECT *
FROM   df
WHERE  age > 30;
Pandas
# df["age"] > 30 is a Series of True/False
mask = df["age"] > 30
df[mask]

# or inline, the idiom you'll see everywhere:
df[df["age"] > 30]
JavaScript
const rows = data.filter(
  row => row.age > 30
);
Pandas
rows = df[df["age"] > 30]

The mental model: df["age"] > 30 does not return rows. It returns a boolean Series — one True or False per row. When you put that Series inside df[...], pandas keeps every row where the value is True. Think of it as a stencil laid over the DataFrame. SQL's WHERE and JS's .filter() do the same thing, they just never show you the mask.

The comparison is vectorized: the > runs against all rows at once, no loop required. This is why pandas is fast — the comparison happens in compiled C, not a Python for loop.

# All the usual comparisons produce masks
df[df["status"] == "active"]
df[df["price"] <= 100]
df[df["country"] != "US"]
df[df["name"].str.startswith("A")]   # string methods via .str
df[df["category"].isin(["A", "B"])]  # like SQL IN (...)
df[df["email"].isna()]               # like SQL IS NULL

Two handy SQL translations: SQL's IN (...) becomes .isin([...]), and SQL's IS NULL becomes .isna() (with .notna() for IS NOT NULL). Both return boolean Series, so they drop straight into df[...].

Combining Conditions — &, |, and Parentheses

Multiple conditions are where pandas diverges sharply from both SQL and JavaScript. You do not use and/or. You use &, |, and ~ — and every condition must be wrapped in its own parentheses.

SQL
SELECT *
FROM   df
WHERE  age > 30
  AND  country = 'US';
Pandas
df[(df["age"] > 30) & (df["country"] == "US")]
#   ^------------^   ^---------------------^
#   each condition wrapped in parentheses
JavaScript
data.filter(
  row => row.age > 30 || row.country === "US"
);
Pandas
df[(df["age"] > 30) | (df["country"] == "US")]

Translation rule: AND / && → &. OR / || → |. NOT / ! → ~. These are bitwise operators applied element-by-element across the two boolean Series, which is exactly what you want here. The plain English and/or keywords do not work on Series — they raise an error.

The parentheses gotcha that bites everyone. Write df[(df["age"] > 30) & (df["country"] == "US")], never df[df["age"] > 30 & df["country"] == "US"]. In Python, & binds tighter than > and ==, so without parentheses Python tries to evaluate 30 & df["country"] first and throws a confusing TypeError. Operator precedence, not pandas, is the culprit. Wrap every condition.

Why and/or fail. Python's and and or need a single True/False, but a boolean Series has many values, so pandas raises "The truth value of a Series is ambiguous." That error message almost always means: you typed and where you needed &, or you forgot a pair of parentheses.

# Negation with ~ (the NOT operator)
df[~df["status"].isin(["closed", "archived"])]

# Three conditions: still parenthesize each one
df[
    (df["age"] > 30)
    & (df["country"] == "US")
    & (df["active"])
]

.loc vs .iloc — Label vs Position

Pandas gives you two selectors, and the difference is one of the most important things in the library. .loc selects by label (the index value and column name). .iloc selects by integer position (like a normal array).

JavaScript (position)
// arrays are position-only
data[0];        // first row
data[2];        // third row
data.slice(0, 5);   // first five
Pandas
# .iloc = position, like a JS array
df.iloc[0]        # first row
df.iloc[2]        # third row
df.iloc[0:5]      # first five rows

# .loc = label, the index value itself
df.loc[0]         # row whose index label IS 0
df.loc["alice"]   # row whose index label is "alice"

The mental model: the i in .iloc stands for integer position — it always counts 0, 1, 2 ... no matter what the index looks like. .loc uses the actual index labels and column names, which is closer to SQL: you ask for a primary-key value and a named column. If your DataFrame uses the default integer index, the two look identical — but the moment you set a custom index, they diverge.

Selecting Rows and Columns Together

The real power of .loc and .iloc is selecting rows and columns in one shot: df.loc[rows, columns]. This is the pandas answer to SELECT col1, col2 FROM df WHERE ....

SQL
SELECT name, age
FROM   df
WHERE  age > 30;
Pandas
# rows by condition, columns by label
df.loc[df["age"] > 30, ["name", "age"]]
# .loc takes [row selector, column selector]
df.loc[df["active"], "email"]          # one column, filtered rows
df.loc[5:10, "name":"email"]           # label slice INCLUDES 10 and "email"
df.loc[:, ["name", "age"]]             # all rows, two columns

# .iloc takes [row positions, column positions]
df.iloc[0:3, 0:2]                      # first 3 rows, first 2 cols
df.iloc[0:3, 0:2]                      # position slice EXCLUDES 3 and 2
df.iloc[-1]                            # last row, like data.at(-1)

Label slices are inclusive; position slices are not. This is the subtle one. df.loc[5:10] includes the row labeled 10. But df.iloc[5:10] stops before position 10 — exactly like Python list slicing and JS .slice(). Same numbers, different last row. When in doubt, remember: .loc is labels (inclusive), .iloc is positions (exclusive end).

Selecting Columns — The SELECT List

Picking columns is the pandas version of choosing which fields to SELECT. There are two shapes, and the difference matters.

SQL
-- one column
SELECT name FROM df;

-- several columns
SELECT name, age, email FROM df;
Pandas
# one column -> a Series (1-D)
df["name"]

# several columns -> a DataFrame (2-D)
df[["name", "age", "email"]]

Single brackets vs double brackets. df["name"] (single) returns a one-dimensional Series. df[["name"]] (double) returns a DataFrame with one column. The inner [...] is just a Python list of column names — that is why selecting many columns is df[["a", "b", "c"]]. If you want a table back, use double brackets.

# Drop columns instead of listing the ones you keep
df.drop(columns=["internal_id", "raw_blob"])

# Select by data type
df.select_dtypes(include="number")     # numeric columns only

# Rename while selecting (chain it)
df[["name", "age"]].rename(columns={"name": "full_name"})

.query() — Filtering That Reads Like SQL

If the parentheses and & symbols feel noisy, .query() lets you write the filter as a string that looks remarkably like a WHERE clause. You pass a condition, and you can use plain and/or inside it.

SQL
SELECT *
FROM   df
WHERE  age > 30
  AND  country = 'US';
Pandas (.query)
df.query("age > 30 and country == 'US'")

# compare to the bracket version:
df[(df["age"] > 30) & (df["country"] == "US")]

Inside .query(), the gotchas relax. You write column names bare (no df["..."]), and you can use and/or/not as words. Pandas parses the string, so the operator-precedence trap from earlier does not apply here. It is often the most readable choice for long filters.

# Reference a Python variable with @
threshold = 30
df.query("age > @threshold")

# IN-style membership
df.query("category in ['A', 'B']")

# Column names with spaces need backticks
df.query("`sign up date` > '2024-01-01'")

.query() trades safety for readability. Because the condition is a string, typos in column names surface only at runtime, and your editor cannot autocomplete or type-check them. For one-off exploration it is lovely; inside production code, the explicit df[mask] form is easier to lint and refactor.

The SettingWithCopyWarning Trap

This is the warning every pandas beginner hits, and it is genuinely confusing the first time. It appears when you filter a DataFrame and then try to assign into the result — pandas cannot tell whether you are editing a copy or the original, so it warns you.

# The trap: chained indexing then assignment
df[df["age"] > 30]["bonus"] = 1000
# SettingWithCopyWarning! The change may silently vanish.

Why it happens. df[df["age"] > 30] may return a copy of the matching rows, not a view into df. So ["bonus"] = 1000 writes into that throwaway copy, and your real DataFrame is untouched. The two square-bracket operations in a row — "chained indexing" — are the root cause. Pandas warns because the outcome is undefined.

The fix is to do the row selection and the column assignment in a single .loc call, so pandas knows you mean the original DataFrame.

Wrong (chained)
df[df["age"] > 30]["bonus"] = 1000

# two [] in a row = chained indexing
# warns, and often does nothing
Right (single .loc)
df.loc[df["age"] > 30, "bonus"] = 1000

# one call, rows AND column together
# pandas edits df in place, no warning

The other safe fix: make a real copy. If you actually want a separate filtered DataFrame to work on, say so explicitly: seniors = df[df["age"] > 30].copy(). Now seniors["bonus"] = 1000 is unambiguous — you are editing seniors, never df, and no warning appears. Use .loc to edit the original; use .copy() to branch off a new one.

Putting It All Together

A realistic slice of analysis: load a customer table, filter it the way you would in SQL, pick the columns you care about, and safely add a derived column. This is the daily-driver workflow.

import pandas as pd

df = pd.read_csv("customers.csv")

# 1. Filter: active US customers who spent over $500
#    SQL: WHERE active = 1 AND country = 'US' AND total > 500
high_value = df[
    (df["active"])
    & (df["country"] == "US")
    & (df["total_spend"] > 500)
]

# 2. Select just the columns we want to report on
#    SQL: SELECT name, email, total_spend
report = high_value[["name", "email", "total_spend"]].copy()

# 3. Add a derived column safely with .loc (no warning)
report.loc[report["total_spend"] > 1000, "tier"] = "platinum"
report.loc[report["total_spend"] <= 1000, "tier"] = "gold"

# 4. The same filter, written with .query() for readability
high_value_q = df.query(
    "active and country == 'US' and total_spend > 500"
)

# 5. Position-based peek at the result
print(report.iloc[0:5])          # first five rows
print(report.loc[:, "name"])     # just the name column

Notice the discipline: every condition is parenthesized, the new DataFrame is created with .copy(), and the derived column is written through .loc. Those three habits eliminate the two warnings and the one TypeError that account for most beginner pandas pain.

Check Your Understanding

Quick Quiz

1. What does df["age"] > 30 actually return, and what makes df[df["age"] > 30] work?

It returns a boolean Series — one True/False per row. Passing that Series back into df[...] (boolean indexing) keeps only the rows where the value is True, exactly like a SQL WHERE clause or JS .filter().

2. Why does df[df["age"] > 30 & df["country"] == "US"] throw an error?

Operator precedence. In Python, & binds tighter than > and ==, so Python tries to evaluate 30 & df["country"] first. Wrap each condition in its own parentheses: df[(df["age"] > 30) & (df["country"] == "US")]. Also remember to use & not and.

3. What is the difference between .loc and .iloc?

.loc selects by label (the index value and column name); its slices are inclusive of the end. .iloc selects by integer position (counting 0, 1, 2... like a JS array); its slices exclude the end, like normal Python/JS slicing. The i stands for integer position.

4. What is the difference between df["name"] and df[["name"]]?

Single brackets return a one-dimensional Series. Double brackets (a list of column names) return a DataFrame. Use double brackets when you want a table back or are selecting more than one column: df[["name", "age"]].

5. How do you avoid a SettingWithCopyWarning when assigning to filtered rows?

Do the row selection and column assignment in one .loc call: df.loc[df["age"] > 30, "bonus"] = 1000 instead of the chained df[df["age"] > 30]["bonus"] = 1000. If you want a separate DataFrame, branch off with .copy() first.
