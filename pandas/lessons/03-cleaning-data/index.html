03 Cleaning Data

← Back to lessons

Real data is filthy, and cleaning it is the job. Nobody tells you this in tutorials, but roughly 80% of analysis work is fixing missing values, wrong types, bad dates, duplicate rows, and inconsistent strings. This lesson is the unglamorous craft — the same defensive thinking you already do with SQL NULLs and sloppy JSON, translated into pandas.

Missing Data — NaN, isna, notna

Every dataset has holes. In SQL it's NULL. In JavaScript it's null or undefined. In pandas it's NaN (a special float) or None, and pandas treats them as the same thing when you ask "is this missing?"

SQL
-- Find rows with a missing email
SELECT *
FROM users
WHERE email IS NULL;

-- Find rows that DO have one
SELECT *
FROM users
WHERE email IS NOT NULL;

-- Count the gaps
SELECT COUNT(*) - COUNT(email)
FROM users;
Pandas
import pandas as pd

# Boolean mask of missing values
df["email"].isna()      # True where NaN/None

# Rows that DO have an email
df[df["email"].notna()]

# Count the gaps per column
df.isna().sum()         # a count for every column

Translation rule: IS NULL → .isna() and IS NOT NULL → .notna(). The big difference from SQL: .isna() returns a whole column of True/False (a boolean mask), not a filter you bolt onto a query. You then feed that mask back into df[...] to select rows, exactly like the filtering you saw in the last lesson.

NaN is not equal to anything — not even itself. Just like SQL's NULL = NULL is never true, df["x"] == None does not work in pandas. NaN == NaN is False. Always use .isna() / .notna() to test for missing values, never == or !=.

Dropping and Filling — dropna, fillna

Once you can find the holes, you decide: throw the rows away, or patch them. SQL handles this with WHERE filters and COALESCE. Pandas gives you two purpose-built methods.

SQL
-- Drop rows missing a key field
SELECT *
FROM orders
WHERE customer_id IS NOT NULL;

-- Patch missing values with a default
SELECT
  name,
  COALESCE(country, 'Unknown') AS country
FROM users;
Pandas
# Drop rows where ANY column is missing
df.dropna()

# Drop only rows missing a specific column
df.dropna(subset=["customer_id"])

# Patch one column (COALESCE equivalent)
df["country"] = df["country"].fillna("Unknown")

# Patch with a computed value
df["age"] = df["age"].fillna(df["age"].median())

The arguments worth knowing: dropna(how="all") drops a row only if every cell is missing (the default how="any" drops if one is). dropna(axis=1) drops missing columns instead of rows. dropna(thresh=3) keeps rows with at least 3 non-missing values. Reach for subset most often — blanket dropna() quietly deletes more than you expect.

The mental model: .fillna() is pandas COALESCE. You can fill with a constant, with a per-column dict (df.fillna({"age": 0, "country": "Unknown"})), or with a strategy like forward-fill (df.ffill(), which carries the last seen value down — handy for time series). Filling with the median or mean is the analyst's default for numeric gaps.

Types — dtypes and astype

Here is the bug that bites every newcomer: a CSV column of numbers that pandas reads as text, so "10" + "2" gives "102" instead of 12. Columns have types (dtypes), and you fix them with astype. JavaScript devs know this pain — it's the same as "5" * 1 coercion games.

JavaScript
// Coerce a string to a number
const n = Number("42");        // 42
const f = parseFloat("3.14");  // 3.14

// Check the type of a value
typeof n;                      // "number"

// To string
String(42);                    // "42"
Pandas
# See the type of every column
df.dtypes

# Coerce a whole column to int / float
df["age"] = df["age"].astype(int)
df["price"] = df["price"].astype(float)

# Coerce to string
df["zip"] = df["zip"].astype(str)
Pandas dtype SQL type What it holds
int64 INTEGER Whole numbers (cannot hold NaN)
float64 FLOAT / REAL Decimals, and the only numeric type that holds NaN
object VARCHAR / TEXT Usually strings (the catch-all type)
bool BOOLEAN True / False
datetime64[ns] TIMESTAMP Real dates and times
category ENUM Repeated labels stored efficiently

A numeric column with one missing value becomes float64, not int64. Integers in pandas cannot hold NaN, so the moment a gap appears, the column is promoted to float and your 5 shows as 5.0. If you truly need nullable integers, use the modern dtype: df["age"].astype("Int64") (capital I). That's the pandas extension type that does support missing values.

When astype(int) throws on dirty data, reach for pd.to_numeric(df["x"], errors="coerce"). It turns anything unparseable into NaN instead of crashing the whole conversion — the pandas equivalent of wrapping Number() in a try/catch and defaulting to null.

Parsing Dates — to_datetime

Dates arrive as strings: "2024-03-14", "03/14/2024", "March 14, 2024". Until you parse them, they are just text and you cannot sort, subtract, or extract the month. This is new Date() in JavaScript and CAST(... AS DATE) in SQL.

SQL / JavaScript
-- SQL: cast text to a real date
SELECT CAST(signup AS DATE)
FROM users;

-- SQL: pull out a part
SELECT EXTRACT(YEAR FROM signup)
FROM users;
Pandas
# Parse a string column into real dates
df["signup"] = pd.to_datetime(df["signup"])

# Now the .dt accessor unlocks date parts
df["signup"].dt.year     # 2024
df["signup"].dt.month    # 3
df["signup"].dt.day_name()  # "Thursday"

# Subtract dates to get a duration
df["age_days"] = (pd.Timestamp.now() - df["signup"]).dt.days

The mental model: once a column is datetime64, the .dt accessor is your toolbox — .dt.year, .dt.month, .dt.dayofweek, .dt.date, and more. It mirrors JavaScript's date.getFullYear() and SQL's EXTRACT, but it runs across the whole column at once.

Ambiguous formats: is 03/04/2024 March 4th or April 3rd? Pass format= to be explicit: pd.to_datetime(df["d"], format="%m/%d/%Y"). For messy real-world columns, add errors="coerce" so unparseable values become NaT (the datetime version of NaN) instead of raising.

Duplicates — duplicated, drop_duplicates

Double-counted rows quietly inflate every total you compute. In SQL you'd use SELECT DISTINCT or a GROUP BY. Pandas gives you two precise tools: one to find dupes, one to remove them.

SQL
-- Get unique rows
SELECT DISTINCT *
FROM events;

-- Unique by a specific key
SELECT DISTINCT user_id, day
FROM logins;
Pandas
# Boolean mask: True for repeat rows
df.duplicated()

# Drop exact duplicate rows
df.drop_duplicates()

# Dedupe on specific columns, keep the first seen
df.drop_duplicates(subset=["user_id", "day"], keep="first")

Translation rule: SELECT DISTINCT → .drop_duplicates(). The pandas version is more flexible: subset= picks which columns define a duplicate, and keep= chooses which copy survives — "first" (default), "last", or False (drop every copy of any duplicated row). Run df.duplicated().sum() first to see how many you'd lose before you delete anything.

Renaming Columns

Source data ships with column names like "Customer Name " (trailing space) or "DOB". Clean names make the rest of your code readable. This is SQL's AS alias and a JavaScript object key rename rolled together.

SQL
-- Alias a column on the way out
SELECT
  cust_nm  AS customer_name,
  dob      AS birth_date
FROM customers;
Pandas
# Rename specific columns by mapping old to new
df = df.rename(columns={
    "cust_nm": "customer_name",
    "dob": "birth_date",
})

# Bulk clean: lowercase, strip, snake_case all columns
df.columns = (
    df.columns.str.strip().str.lower().str.replace(" ", "_")
)

That bulk-clean line is worth memorizing. df.columns.str.strip().str.lower().str.replace(" ", "_") standardizes every header in one pass — no trailing spaces, no capitals, no spaces in names. It's the first thing many analysts run after loading a CSV.

String Cleaning — the .str Accessor

Text columns are where the real mess lives: stray whitespace, mixed case, "N/A" typed by hand, dollar signs glued to numbers. The .str accessor applies a string method to every cell at once. If you know JavaScript string methods, you already know these — they just run vectorized over the column.

JavaScript (one value)
const s = "  Hello, World!  ";

s.trim();                  // "Hello, World!"
s.toLowerCase();
s.includes("World");       // true
s.replace("World", "JS");
s.startsWith("  H");
s.split(", ");
Pandas (whole column)
col = df["title"]

col.str.strip()                  # trim every cell
col.str.lower()
col.str.contains("World")        # boolean mask
col.str.replace("World", "PD")
col.str.startswith("  H")
col.str.split(", ")

The mental model: .str turns a single-value string method into a column-wide one. "x".lower() lowercases one string; df["c"].str.lower() lowercases ten thousand. The method names match Python's built-in string methods almost exactly, so there is little new to learn — just remember the .str prefix.

A Realistic String Cleanup

# A price column read in as text: "$1,299.00"
df["price"] = (
    df["price"]
      .str.replace("$", "", regex=False)
      .str.replace(",", "", regex=False)
      .astype(float)            # now a real number: 1299.0
)

# Normalize a category column riddled with inconsistency
df["state"] = df["state"].str.strip().str.upper()
# "  ny ", "Ny", "NY " all become "NY"

# Treat hand-typed "N/A" and "" as real missing values
df["notes"] = df["notes"].replace(["N/A", "", "null"], pd.NA)

Watch the regex= flag on .str.replace(). By default pandas treats the pattern as a regular expression, so characters like $, ., and ( have special meaning. When you mean them literally — stripping a real dollar sign — pass regex=False, or escape them. Forgetting this is a classic silent bug.

Custom Logic — apply and map

When the built-in methods can't express your rule, you write a function and run it over the data. This is exactly JavaScript's Array.map(). Pandas has two flavors: .map() for a single column (a Series), and .apply() for a column or across rows.

JavaScript
const prices = [10, 20, 30];

// Transform each value
const withTax = prices.map(p => p * 1.07);

// Bucket each value with a function
const tiers = prices.map(p =>
  p >= 25 ? "high" : "low"
);

// Look up via a mapping object
const codes = { US: "United States" };
rows.map(r => codes[r.country]);
Pandas
# Transform each value in a column
df["with_tax"] = df["price"].map(lambda p: p * 1.07)

# Bucket each value with a function
def tier(p):
    return "high" if p >= 25 else "low"

df["tier"] = df["price"].apply(tier)

# Look up via a mapping dict (map shines here)
codes = {"US": "United States"}
df["country_name"] = df["country"].map(codes)

Translation rule: array.map(fn) → series.apply(fn) or series.map(fn). The quirk: .map() also accepts a plain dict for lookups (like indexing a JS object), while .apply() only takes a function. For whole-row logic that touches several columns, use df.apply(fn, axis=1), where fn receives one row at a time.

Prefer vectorized methods over apply when you can. df["price"] * 1.07 is far faster than df["price"].apply(lambda p: p * 1.07) because it runs in compiled code, not a Python loop. Use apply for genuinely custom logic; use plain arithmetic and .str methods for everything they already cover.

Putting It All Together

A realistic cleanup pipeline — the kind you'd write the moment a messy CSV lands on your desk. It standardizes headers, fixes types, parses dates, patches gaps, normalizes strings, and drops duplicates, in the order you'd actually do them.

import pandas as pd

def clean_signups(path):
    """Load a raw signups CSV and return a tidy DataFrame."""
    df = pd.read_csv(path)

    # 1. Standardize column names
    df.columns = (
        df.columns.str.strip().str.lower().str.replace(" ", "_")
    )

    # 2. Treat hand-typed blanks as real missing values
    df = df.replace(["N/A", "", "null", "-"], pd.NA)

    # 3. Fix types — strip the "$" and commas off price, then to float
    df["price"] = (
        df["price"].str.replace(r"[$,]", "", regex=True)
                   .pipe(pd.to_numeric, errors="coerce")
    )

    # 4. Parse the signup date; bad dates become NaT, not crashes
    df["signup_date"] = pd.to_datetime(
        df["signup_date"], errors="coerce"
    )

    # 5. Normalize the state column: trim + uppercase
    df["state"] = df["state"].str.strip().str.upper()

    # 6. Patch missing values sensibly
    df["country"] = df["country"].fillna("Unknown")
    df["price"] = df["price"].fillna(df["price"].median())

    # 7. Drop rows with no usable identity, then exact duplicates
    df = df.dropna(subset=["email"])
    df = df.drop_duplicates(subset=["email"], keep="last")

    # 8. Add a derived column for downstream analysis
    df["signup_year"] = df["signup_date"].dt.year

    return df

if __name__ == "__main__":
    clean = clean_signups("data/raw_signups.csv")
    print(clean.dtypes)
    print(f"{len(clean)} clean rows ready")

Note .pipe() in step 3. It threads the Series into a function that isn't a method — x.pipe(pd.to_numeric, errors="coerce") reads left to right like the rest of the chain, instead of wrapping it as pd.to_numeric(x, ...). It's the pandas answer to a pipe operator, keeping long cleanups readable top to bottom.

Check Your Understanding

Quick Quiz

1. SQL uses IS NULL to find missing values. What's the pandas equivalent, and why can't you use == None?

Use .isna() (and .notna() for the inverse). You can't use == None because NaN is never equal to anything, not even itself — exactly like SQL's NULL = NULL never being true. .isna() returns a boolean mask you can feed back into df[...].

2. You read a CSV and a numeric column shows up with dtype object. What happened, and how do you fix it?

Pandas read the numbers as text (often because of stray characters like $ or commas, or hand-typed values like "N/A"). Clean the strings with .str.replace(), then convert with .astype(float) or, more safely, pd.to_numeric(col, errors="coerce") which turns anything unparseable into NaN instead of raising.

3. After parsing a column with pd.to_datetime(), how do you pull out just the year? How does this compare to SQL and JavaScript?

Use the .dt accessor: df["signup"].dt.year. It's the column-wide version of SQL's EXTRACT(YEAR FROM signup) and JavaScript's date.getFullYear(). The .dt accessor only works once the column is a real datetime64 dtype, not while it's still text.

4. What's the difference between .map() and .apply() on a column, and when would you avoid both?

Both transform values like JavaScript's Array.map(). .map() works on a single Series and also accepts a plain dict for lookups; .apply() takes a function and can run across rows with axis=1. Avoid both when a vectorized operation will do — df["price"] * 1.07 is much faster than .apply(lambda p: p * 1.07) because it skips the Python loop.

5. SELECT DISTINCT user_id, day dedupes in SQL. Write the pandas equivalent and explain the keep argument.

df.drop_duplicates(subset=["user_id", "day"]). The keep argument chooses which copy survives: "first" (default) keeps the earliest, "last" keeps the most recent, and False drops every copy of any duplicated row entirely. Run df.duplicated().sum() first to see how many rows you'd remove.
