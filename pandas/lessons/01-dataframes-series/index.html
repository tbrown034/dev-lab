01 DataFrames & Series

← Back to lessons

A DataFrame is a SQL result set you can hold in a variable. If you already think in SQL queries and arrays of objects, pandas is mostly new names for ideas you know. This lesson maps the two core types — Series and DataFrame — onto a SQL table and a JavaScript array of objects, then covers loading a CSV and inspecting what you got. Assume import pandas as pd everywhere.

The Two Types You'll Use All Day

Pandas has exactly two data structures you need to learn first. A Series is one column. A DataFrame is a whole table. Everything else builds on these two.

SQL
-- A DataFrame is the result of:
SELECT * FROM users;

-- name  | age | city
-- Alice |  30 | NYC
-- Bob   |  25 | LA

-- A Series is a single column:
SELECT age FROM users;

-- 30
-- 25
Pandas
import pandas as pd

# A DataFrame: the whole table
df = pd.DataFrame({
    "name": ["Alice", "Bob"],
    "age": [30, 25],
    "city": ["NYC", "LA"],
})

# A Series: one column
df["age"]
# 0    30
# 1    25
# Name: age, dtype: int64

The mental model: A DataFrame is a SQL table or a JavaScript array of objects. A Series is one column of that table — a labeled, typed array. Pull a single column out of a DataFrame and you get a Series. Stack several Series side by side and you get a DataFrame. That's the whole relationship.

JavaScript
// A DataFrame is like this array
const users = [
  { name: "Alice", age: 30, city: "NYC" },
  { name: "Bob",   age: 25, city: "LA"  },
];

// A Series is like one field
// pulled across every row:
const ages = users.map(u => u.age);
// [30, 25]
Pandas
df = pd.DataFrame([
    {"name": "Alice", "age": 30, "city": "NYC"},
    {"name": "Bob",   "age": 25, "city": "LA"},
])

# Same column, far less ceremony
ages = df["age"]
# 0    30
# 1    25
# Name: age, dtype: int64

You can build a DataFrame from either shape. Pass a dict of columns ({"age": [30, 25]}) or a list of row dicts ([{"age": 30}, {"age": 25}]). Pandas figures out the rest. The column-dict form maps to how a database stores data; the row-list form maps to JSON you'd get from an API.

Loading Data — pd.read_csv

You'll rarely type data by hand. In practice you load a file, and the file is almost always a CSV. One function does it.

SQL / shell
-- Postgres: load CSV into a table
COPY users
FROM 'users.csv'
WITH (FORMAT csv, HEADER true);

-- Then query it
SELECT * FROM users;
Pandas
import pandas as pd

# Load CSV straight into a DataFrame
df = pd.read_csv("users.csv")

# The header row becomes column names
# automatically. No schema to declare.

The CSV's first row becomes your column names, and pandas guesses a type for each column. There are arguments for the messy real world too.

# Common read_csv arguments you'll actually use
pd.read_csv("data.csv")                     # the 90% case
pd.read_csv("data.tsv", sep="\t")           # tab-separated
pd.read_csv("data.csv", header=None)        # file has no header row
pd.read_csv("data.csv", index_col="id")     # use a column as the index
pd.read_csv("data.csv", usecols=["name", "age"])  # only these columns
pd.read_csv("data.csv", nrows=100)          # just the first 100 rows
pd.read_csv("data.csv", parse_dates=["created_at"])  # parse a date column

Translation rule: fetch().then(r => r.json()) or COPY ... FROM → pd.read_csv(path). You don't loop, you don't declare a schema, and you don't write a parser. One call returns a DataFrame ready to query. There are siblings for other formats too: pd.read_json, pd.read_excel, pd.read_sql, and pd.read_parquet.

read_csv guesses types, and it can guess wrong. A ZIP code column like 07105 may load as the integer 7105, dropping the leading zero. An ID column may become a float if it has any blanks. Always check df.dtypes after loading, and force a type with dtype={"zip": str} when it matters.

First Five Things You Do After Loading

Once you have a DataFrame, you look at it before you touch it. These five calls are muscle memory for anyone who works with data. Each one answers a question you'd otherwise write a query for.

head() and tail() — peek at the rows

SQL
-- Postgres / SQLite
SELECT * FROM users LIMIT 5;

-- Last rows is awkward in SQL:
SELECT * FROM users
ORDER BY id DESC LIMIT 5;
Pandas
df.head()      # first 5 rows (default)
df.head(10)    # first 10 rows

df.tail()      # last 5 rows
df.tail(3)     # last 3 rows

shape — how big is it

SQL
-- Row count
SELECT COUNT(*) FROM users;

-- Column count means querying
-- information_schema. Annoying.
Pandas
df.shape       # (rows, columns) e.g. (1000, 4)
df.shape[0]    # row count: 1000
df.shape[1]    # column count: 4
len(df)        # also row count: 1000

shape is an attribute, not a method. No parentheses: it's df.shape, not df.shape(). Same with df.columns, df.index, and df.dtypes. The things that compute something — head(), info(), describe() — are methods and need the parentheses.

info() — the schema and the nulls

df.info() is the closest thing to DESCRIBE users in SQL. It prints every column, its type, and how many non-null values it holds — your fastest read on data quality.

df.info()

# <class 'pandas.core.frame.DataFrame'>
# RangeIndex: 1000 entries, 0 to 999
# Data columns (total 4 columns):
#  #   Column  Non-Null Count  Dtype
# ---  ------  --------------  -----
#  0   name    1000 non-null   object
#  1   age     987 non-null    float64  <- 13 missing!
#  2   city    1000 non-null   object
#  3   signup  1000 non-null   object

The mental model: info() answers "what columns exist, what type is each, and where are the holes?" The "Non-Null Count" column is the one to watch. If a column says 987 non-null out of 1000 rows, you have 13 missing values to deal with before you trust any math on it.

describe() — the summary stats

SQL
SELECT
  COUNT(age),
  AVG(age),
  MIN(age),
  MAX(age),
  STDDEV(age)
FROM users;
Pandas
df.describe()

# All of that, for every numeric
# column at once:
#        age
# count  987.0
# mean    34.2
# std      9.1
# min     18.0
# 25%     27.0
# 50%     33.0   <- the median
# 75%     41.0
# max     65.0

describe() skips text columns by default. It only summarizes numeric columns. To summarize text (object) columns instead — getting count, unique values, top value, and frequency — pass df.describe(include="object"), or df.describe(include="all") for everything.

dtypes — what type is each column

df.dtypes

# name      object   <- text (and anything mixed)
# age        int64
# height   float64
# active      bool
# signup    object   <- a date stored as text!
# dtype: object

object almost always means "text," but really means "Python object." When pandas can't pick a clean numeric type, it falls back to object. A date column read from CSV shows up as object until you parse it. A number column with stray commas or a "N/A" string also lands as object — and then math silently breaks. Seeing object where you expected a number is a red flag.

The inspection toolkit at a glance

You want… SQL Pandas
First few rows LIMIT 5 df.head()
Row count COUNT(*) len(df) / df.shape[0]
Columns and types DESCRIBE table df.info() / df.dtypes
Summary statistics AVG, MIN, MAX… df.describe()
Column names query information_schema df.columns
Distinct values in a column SELECT DISTINCT col df["col"].unique()
Counts per value GROUP BY col + COUNT df["col"].value_counts()

Selecting Columns — One Bracket vs Two

This is the single most important syntax detail in this lesson, and it trips up everyone once. The number of brackets decides whether you get back a Series or a DataFrame.

SQL
-- One column
SELECT age FROM users;

-- Several columns
SELECT name, age FROM users;
Pandas
# One column -> a Series (single brackets)
df["age"]

# Several columns -> a DataFrame (double!)
df[["name", "age"]]

The translation rule that matters most: single brackets with one name (df["age"]) return a Series — one column, flattened. Double brackets with a list of names (df[["name", "age"]]) return a DataFrame — even if that list has only one name. The inner [ ] is a Python list; the outer [ ] is the selection. df[["age"]] is a one-column DataFrame, not a Series.

# Prove it to yourself with type()
type(df["age"])        # pandas.core.series.Series
type(df[["age"]])      # pandas.core.frame.DataFrame

# Why it matters: a Series has different methods
df["age"].mean()       # 34.2  (a single number)
df["age"].max()        # 65
df["age"].value_counts()

# Selecting multiple columns keeps the table shape
df[["name", "age", "city"]].head()

Dot access works, until it doesn't. You can also write df.age instead of df["age"], and you'll see it in tutorials. Avoid it. It breaks on column names with spaces (df.first name is a syntax error), on names that collide with pandas methods (a column called count or shape), and you can never use it to create a column. Bracket access (df["age"]) always works. Use it.

The Index — The Part SQL Doesn't Have

Here is the one concept with no clean SQL equivalent: every DataFrame and every Series has an index, a label for each row. By default it's just integers 0, 1, 2, … like array positions, but it can be anything — a user ID, a date, a country name.

JavaScript
// An array has positions 0, 1, 2...
const users = [
  { name: "Alice", age: 30 },
  { name: "Bob",   age: 25 },
];
users[0];  // { name: "Alice", age: 30 }

// But positions are all you get.
// You can't make "alice" a key
// without converting to an object.
Pandas
# Default index is positions: 0, 1, 2...
df.index        # RangeIndex(start=0, stop=2, step=1)

# But you can set a meaningful index
df = df.set_index("name")

# Now rows are labeled by name
df.loc["Alice"]   # the whole Alice row

# Go back to integer positions
df = df.reset_index()

The mental model: the index is a row label that travels with the data. Think of it as a permanent PRIMARY KEY that pandas keeps attached to every row, even after you filter or sort. When you pull out a Series with df["age"], it carries the same index, so you always know which row each value came from. This is why alignment "just works" later when you combine columns — pandas matches on the index, not on position.

# The index shows up everywhere you print a Series
df["age"]
# name             <- the index name
# Alice    30
# Bob      25
# Name: age, dtype: int64

# Useful index operations
df.index                 # the row labels
df.set_index("id")       # promote a column to the index
df.reset_index()         # demote the index back to a column
df.reset_index(drop=True)  # drop it entirely, renumber 0..n

Setting the index returns a new DataFrame. Most pandas methods don't modify in place — df.set_index("id") hands you a new DataFrame and leaves the original alone. So you reassign: df = df.set_index("id"). This is the same idea as arr.map(...) in JavaScript returning a new array rather than mutating the old one.

Putting It All Together

A realistic first pass at a dataset — the exact sequence you'd run the moment a CSV lands on your desk. The goal is not to analyze anything yet, just to understand what you're holding.

import pandas as pd

# 1. Load it. Force the zip column to stay a string
#    so leading zeros survive.
df = pd.read_csv("subscribers.csv", dtype={"zip": str})

# 2. How big is it?
print(df.shape)          # (12450, 6)

# 3. What do the rows look like?
print(df.head())

# 4. What are the columns, types, and missing values?
df.info()
#  0   email      12450 non-null  object
#  1   name       12450 non-null  object
#  2   age        11203 non-null  float64   <- 1247 missing
#  3   zip        12450 non-null  object
#  4   plan       12450 non-null  object
#  5   signup     12450 non-null  object    <- a date as text

# 5. Summary stats on the numbers
print(df.describe())     # count/mean/min/max for age

# 6. Look at one column (a Series) and its spread
print(df["plan"].value_counts())
# free       8120
# pro        3890
# team        440

# 7. Grab just the columns you care about (a DataFrame)
core = df[["email", "plan", "age"]]
print(core.head())

# 8. Make email the row label, since it's unique
df = df.set_index("email")
print(df.loc["alice@example.com"])  # that one row

This eight-step pass is the habit to build. Load, check shape, peek with head, run info, run describe, eyeball a key column with value_counts. You do it before every analysis. It takes thirty seconds and it catches the wrong-dtype, missing-data, and leading-zero problems that would otherwise corrupt everything downstream.

Check Your Understanding

Quick Quiz

1. What's the difference between a Series and a DataFrame?

A DataFrame is the whole table — like a SQL result set or a JS array of objects. A Series is a single column of that table: a labeled, typed array. Pull one column out of a DataFrame and you get a Series.

2. What does df["age"] return, and how is it different from df[["age"]]?

df["age"] (single brackets, one name) returns a Series. df[["age"]] (double brackets — a list of names) returns a one-column DataFrame. The inner brackets are a Python list, so any time you pass a list of column names you get a DataFrame back, even with just one name in it.

3. You loaded a CSV and a column you expected to be numeric shows object in df.dtypes. What likely happened?

object means "text / mixed Python object." The column probably has non-numeric junk in it — stray commas, a "N/A" string, currency symbols — so pandas couldn't pick a clean numeric type. Math on it will silently break. Clean it, then convert with something like pd.to_numeric.

4. Which of these need parentheses: shape, info, describe, dtypes, head?

Methods need parentheses: info(), describe(), head(). Attributes do not: shape, dtypes (also columns, index). Rule of thumb: if it computes something it's a method; if it's a property of the object it's an attribute.

5. What is the index, and why does SQL not really have an equivalent?

The index is a label for each row that travels with the data — like a permanent primary key pandas keeps attached even after filtering or sorting. SQL rows have no inherent order or persistent label you carry around; you re-derive identity each query. In pandas the index lets pandas align data by label rather than position. By default it's the integers 0, 1, 2…, but you can set it to any column with df.set_index(...).
