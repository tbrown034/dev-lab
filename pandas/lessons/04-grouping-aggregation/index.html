04 Grouping & Aggregation

← Back to lessons

GROUP BY is the question you ask of almost every dataset. Total sales per region, average score per student, count of orders per customer — all of it is "split the rows into buckets, then summarize each bucket." If you know SQL's GROUP BY, you already understand the idea. This lesson maps that idea onto Pandas groupby and agg, beside the SQL you already write.

The Mental Model — Split, Apply, Combine

Every grouped summary follows the same three steps. SQL hides them inside one statement. Pandas exposes them, which is confusing at first and powerful later.

Split, apply, combine. First split the rows into groups by some key. Then apply a function to each group (sum, mean, count). Then combine the per-group results into one new table. SQL does all three in SELECT region, SUM(sales) ... GROUP BY region. Pandas does the same with df.groupby("region")["sales"].sum().

Here is the most basic version — one group key, one aggregation — side by side.

SQL
-- Total sales per region
SELECT region, SUM(sales) AS total
FROM orders
GROUP BY region;
Pandas
# Total sales per region
df.groupby("region")["sales"].sum()

# region
# East     1200
# West      950
# Name: sales, dtype: int64

The result is indexed by the group key, not numbered. SQL hands you a flat table with a region column. Pandas hands you a Series whose index is the region. If you want a normal column back, add .reset_index() — you will do this constantly. More on that below.

groupby — The Core Verb

df.groupby("col") by itself does almost nothing — it just defines the buckets. The work happens when you select columns and call an aggregation. Think of groupby as the SQL GROUP BY clause and the method after it as the function inside SELECT.

# A GroupBy object — no computation yet
g = df.groupby("region")

# Now apply an aggregation
g["sales"].sum()       # SUM(sales) per region
g["sales"].mean()      # AVG(sales) per region
g["sales"].count()     # COUNT(sales) — non-null only
g["sales"].size()      # COUNT(*) — all rows incl. nulls
g["sales"].max()       # MAX(sales) per region
g["sales"].median()    # no direct SQL equivalent

# Group by multiple keys (composite GROUP BY)
df.groupby(["region", "product"])["sales"].sum()

Translation rule: GROUP BY region → .groupby("region"). The aggregate function in your SELECT → the method you chain after selecting a column. SUM(x) → ["x"].sum(). COUNT(*) → .size(). COUNT(x) → ["x"].count(). The order is flipped: SQL writes the function first, Pandas writes the grouping first.

JavaScript Has No Real Equivalent

If you have grouped data in plain JavaScript, you know it is a chore. There is no GROUP BY — you build a lookup object by hand. Seeing it makes you appreciate what Pandas gives you for free.

JavaScript
// Total sales per region — manual reduce
const totals = {};
for (const row of orders) {
  totals[row.region] =
    (totals[row.region] || 0) + row.sales;
}
// { East: 1200, West: 950 }

// Object.groupBy (newer runtimes) helps split,
// but you still reduce each bucket yourself
const groups = Object.groupBy(
  orders, (o) => o.region
);
Pandas
# One line does split + apply + combine
df.groupby("region")["sales"].sum()



# And multiple stats just as easily
df.groupby("region")["sales"].agg(
    ["sum", "mean", "count"]
)

agg — Multiple Aggregations at Once

In SQL you list several aggregate functions in one SELECT. In Pandas, .agg() is how you do the same — pass it a list of function names and you get one column per function.

SQL
SELECT
  region,
  SUM(sales)   AS total,
  AVG(sales)   AS avg,
  COUNT(*)     AS n,
  MAX(sales)   AS biggest
FROM orders
GROUP BY region;
Pandas
df.groupby("region")["sales"].agg(
    ["sum", "mean", "count", "max"]
)

#          sum   mean  count  max
# region
# East    1200  400.0      3  600
# West     950  475.0      2  500

You can also aggregate several columns at once by passing a dict — "this function on that column." This is the close cousin of writing different aggregates on different columns in SQL.

# Different aggregation per column
df.groupby("region").agg({
    "sales": "sum",        # SUM(sales)
    "units": "mean",       # AVG(units)
    "order_id": "count",   # COUNT(order_id)
})

# Multiple functions on the same column
df.groupby("region").agg({
    "sales": ["sum", "mean", "max"]
})

Passing a dict to agg can give you messy column names. When you ask for several functions on one column, you get a MultiIndex on the columns — tuples like ("sales", "sum"). That is awkward to work with. The clean fix is named aggregation, covered next, which lets you name every output column directly.

Named Aggregation — Clean Output Columns

This is the modern, preferred way to aggregate. It maps almost perfectly onto SQL's AS aliases: you say "this output column equals this function applied to that input column."

SQL
SELECT
  region,
  SUM(sales)  AS total_sales,
  AVG(sales)  AS avg_sales,
  COUNT(*)    AS order_count
FROM orders
GROUP BY region;
Pandas (named agg)
df.groupby("region").agg(
    total_sales=("sales", "sum"),
    avg_sales=("sales", "mean"),
    order_count=("sales", "count"),
)

#         total_sales  avg_sales  order_count
# region
# East           1200      400.0            3
# West            950      475.0            2

Translation rule: SQL's SUM(sales) AS total_sales becomes total_sales=("sales", "sum"). The output name is on the left of the =; the tuple is (input_column, function). This is the cleanest way to write multi-column aggregations and the one to reach for by default. Each keyword becomes a flat, well-named column — no MultiIndex mess.

You can pass your own function too, not just string names — useful for things SQL cannot express in one call.

# Custom function: range of sales within each group
df.groupby("region").agg(
    spread=("sales", lambda s: s.max() - s.min()),
    total=("sales", "sum"),
)

Getting a Flat Table Back — reset_index

The single biggest surprise coming from SQL: the group key ends up as the index, not a column. SQL always returns a flat result set. Pandas does not, unless you ask.

Indexed result (default)
g = df.groupby("region").agg(
    total=("sales", "sum")
)

#         total
# region        <- region is the INDEX
# East     1200
# West      950

g["region"]   # KeyError! It's the index
Flat result (like SQL)
g = df.groupby("region").agg(
    total=("sales", "sum")
).reset_index()

#   region  total   <- region is a COLUMN
# 0   East   1200
# 1   West    950

g["region"]   # works now

Shortcut: pass as_index=False to groupby to skip the reset_index() step: df.groupby("region", as_index=False)["sales"].sum(). Same flat result, one fewer call. Use whichever reads more clearly — both are common in real code.

Filtering Groups — The HAVING Equivalent

SQL's HAVING filters after grouping — "keep only the regions whose total exceeds 1000." There are two ways to do this in Pandas, and which you pick depends on whether you want to filter on the aggregated result or on the raw groups.

The most common path: aggregate first, then filter the result with normal boolean indexing — exactly the filtering you learned in Lesson 02.

SQL (HAVING)
SELECT region, SUM(sales) AS total
FROM orders
GROUP BY region
HAVING SUM(sales) > 1000;
Pandas (aggregate, then filter)
summary = (
    df.groupby("region", as_index=False)
      .agg(total=("sales", "sum"))
)

summary[summary["total"] > 1000]

#   region  total
# 0   East   1200

The second path keeps the original rows. .filter() on a GroupBy returns the full rows of the groups that pass a test — not an aggregated table. Handy when you want every row belonging to a qualifying group.

# Keep all rows from regions whose total sales > 1000
df.groupby("region").filter(lambda g: g["sales"].sum() > 1000)

# Returns original-shape rows, only from qualifying groups

WHERE vs HAVING still matters in Pandas. Filter the DataFrame before groupby to drop rows (that is WHERE). Filter the aggregated summary after to drop whole groups (that is HAVING). Mixing them up gives wrong numbers in both SQL and Pandas.

transform vs aggregate — The One That Confuses Everyone

This distinction has no clean SQL-92 equivalent, which is why it trips people up. The closest analogy is a SQL window function (SUM(...) OVER (PARTITION BY ...)).

.agg() collapses each group to one row. .transform() keeps every row but fills it with its group's aggregate value. Use transform when you want to compare each row to its group — "what fraction of my region's total is this order?"

aggregate (collapses)
df.groupby("region")["sales"].sum()

# One row per region:
# East    1200
# West     950

# Fewer rows than you started with
transform (keeps shape)
df["region_total"] = (
    df.groupby("region")["sales"]
      .transform("sum")
)

# Same number of rows as df. Each row
# now carries its region's total, so you
# can compute a share:
df["share"] = df["sales"] / df["region_total"]

Here is the SQL window function that does the same thing — the value is attached to every row without collapsing them.

SQL (window function)
SELECT
  *,
  SUM(sales) OVER (
    PARTITION BY region
  ) AS region_total
FROM orders;
Pandas (transform)
df["region_total"] = (
    df.groupby("region")["sales"]
      .transform("sum")
)

The rule of thumb: if the answer should have one row per group, use agg. If the answer should have the same rows you started with plus a per-group value, use transform. agg is GROUP BY; transform is OVER (PARTITION BY ...).

value_counts — The Quick COUNT GROUP BY

So often you just want "how many of each?" — value_counts() is the shorthand. It is a grouped count, sorted, in one call. It maps to the most common reporting query there is.

SQL
SELECT region, COUNT(*) AS n
FROM orders
GROUP BY region
ORDER BY n DESC;
Pandas
df["region"].value_counts()

# region
# East    3
# West    2
# Name: count, dtype: int64

# As proportions instead of counts
df["region"].value_counts(normalize=True)

# Count across two columns
df.value_counts(["region", "product"])

Useful flags: normalize=True gives proportions instead of raw counts (each value's share of the total). dropna=False counts missing values too, instead of silently skipping them — handy when nulls are exactly what you are hunting for.

pivot_table — Grouping in Two Dimensions

Sometimes you want the group result laid out as a grid: one variable down the rows, another across the columns. That is a pivot table — the same thing spreadsheets do, and the same thing you would get from a GROUP BY on two keys followed by reshaping.

SQL (long form)
-- Sales by region AND product,
-- as a long two-key GROUP BY
SELECT region, product, SUM(sales) AS total
FROM orders
GROUP BY region, product;
-- You'd reshape this in the app layer
-- to get a grid.
Pandas (grid form)
df.pivot_table(
    index="region",      # rows
    columns="product",   # columns
    values="sales",      # cells
    aggfunc="sum",       # how to combine
)

# product  Widget  Gadget
# region
# East        700     500
# West        950     NaN

The mental model: pivot_table is a two-dimensional groupby. index= and columns= together are your GROUP BY keys; values= and aggfunc= are your aggregate. The difference is purely shape: the second key spreads across columns instead of stacking into rows. NaN appears wherever a combination has no data.

# Fill empty cells, add row/column totals
df.pivot_table(
    index="region",
    columns="product",
    values="sales",
    aggfunc="sum",
    fill_value=0,      # replace NaN with 0
    margins=True,      # adds an "All" total row + column
)

# Multiple aggregations at once
df.pivot_table(
    index="region",
    values="sales",
    aggfunc=["sum", "mean", "count"],
)

Translation Cheat Sheet

SQL Pandas What it does
GROUP BY region .groupby("region") Split rows into buckets
SUM(sales) ["sales"].sum() Total per group
AVG(sales) ["sales"].mean() Average per group
COUNT(*) .size() Rows per group (incl. nulls)
COUNT(sales) ["sales"].count() Non-null rows per group
SUM(x) AS total total=("x", "sum") Named aggregation
HAVING SUM(x) > n summary[summary["total"] > n] Filter groups after aggregating
COUNT(*) ... GROUP BY x ORDER BY n DESC df["x"].value_counts() Quick sorted counts
SUM(x) OVER (PARTITION BY g) .groupby("g")["x"].transform("sum") Per-group value on every row
two-key GROUP BY + reshape df.pivot_table(...) Grouped grid (rows × columns)

Worked Example — Analyzing a Sales Dataset

Let us put it together on a small dataset and actually read the answers out loud, the way you would in a real analysis. Assume import pandas as pd is already done.

import pandas as pd

df = pd.DataFrame({
    "region":  ["East", "East", "East", "West", "West"],
    "product": ["Widget", "Widget", "Gadget", "Widget", "Gadget"],
    "sales":   [300, 400, 500, 950, 0],
    "units":   [3, 4, 5, 9, 0],
})

Question 1: How much did each region sell, and how many orders did each place? Named aggregation, flat output.

summary = (
    df.groupby("region", as_index=False)
      .agg(
          total_sales=("sales", "sum"),
          avg_sales=("sales", "mean"),
          orders=("sales", "count"),
      )
)

#   region  total_sales  avg_sales  orders
# 0   East         1200      400.0       3
# 1   West          950      475.0       2

Reading it: East sold more in total (1200 vs 950) across more orders (3 vs 2), but West has the higher average order value (475 vs 400). Volume lives in the East; bigger individual deals live in the West.

Question 2: Which regions cleared 1000 in total sales? This is HAVING — filter the summary.

summary[summary["total_sales"] > 1000]

#   region  total_sales  avg_sales  orders
# 0   East         1200      400.0       3

Reading it: only East clears the bar. West, despite its strong average, falls short on total.

Question 3: What share of its region's total does each order represent? This needs transform, because the answer has one value per original row.

df["region_total"] = (
    df.groupby("region")["sales"].transform("sum")
)
df["share"] = (df["sales"] / df["region_total"]).round(2)

#   region product  sales  units  region_total  share
# 0   East  Widget    300      3          1200   0.25
# 1   East  Widget    400      4          1200   0.33
# 2   East  Gadget    500      5          1200   0.42
# 3   West  Widget    950      9           950   1.00
# 4   West  Gadget      0      0           950   0.00

Reading it: in the East the three orders are reasonably balanced, with the Gadget order the largest at 42 percent. The West is lopsided — one Widget order is the entire region.

Question 4: Lay sales out as a region-by-product grid. A pivot table.

df.pivot_table(
    index="region",
    columns="product",
    values="sales",
    aggfunc="sum",
    fill_value=0,
)

# product  Gadget  Widget
# region
# East        500     700
# West          0     950

Reading it: the grid makes the gap obvious. West sells Widgets and effectively no Gadgets; East has a real mix. If you were deciding where to push Gadget marketing, the West is the open territory.

This is the whole loop of exploratory analysis. Aggregate to see the totals, filter to find the standouts, transform to compare each row to its group, pivot to see two dimensions at once. Four moves answer most "what is going on in this data?" questions — and every one of them has a SQL twin you already knew.

Check Your Understanding

Quick Quiz

1. Write the Pandas equivalent of SELECT region, SUM(sales) AS total FROM orders GROUP BY region, returning a flat table.

df.groupby("region", as_index=False).agg(total=("sales", "sum")). The as_index=False keeps region as a column instead of moving it to the index (otherwise you would add .reset_index()). Named aggregation gives you the total alias cleanly.

2. What is the difference between .agg("sum") and .transform("sum") on a GroupBy?

.agg("sum") collapses each group to a single row — the result has one row per group (like GROUP BY). .transform("sum") keeps every original row and fills each with its group's sum — the result has the same number of rows as the input (like a SQL window function, SUM(...) OVER (PARTITION BY ...)).

3. SQL's HAVING SUM(sales) > 1000 — how do you express it in Pandas?

Aggregate first, then filter the result with boolean indexing: s = df.groupby("region", as_index=False).agg(total=("sales", "sum")); s[s["total"] > 1000]. HAVING filters groups after aggregating, so you filter the summary — not the original rows (that would be WHERE).

4. Why does df.groupby("region")["sales"].sum() not let you write result["region"] afterward?

Because the group key becomes the index of the result, not a column. SQL always returns flat tables, but Pandas puts the grouping key in the index by default. Call .reset_index() (or use as_index=False) to turn region back into a normal column.

5. When would you reach for pivot_table instead of a plain groupby?

When you want the result laid out as a two-dimensional grid — one variable down the rows (index=), another across the columns (columns=). It is a two-key groupby reshaped into a matrix. A plain groupby(["a", "b"]) gives the same numbers stacked in a long format; pivot_table spreads the second key across columns for readability.
