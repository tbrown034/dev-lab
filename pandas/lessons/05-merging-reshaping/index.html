05 Merging & Reshaping

← Back to lessons

Real analysis almost never lives in one table. You pull customers from one place, orders from another, and you need them side by side. This lesson covers combining DataFrames with pd.merge and pd.concat, then reshaping between wide and long with melt and pivot — all mapped directly to the SQL JOINs you already know.

pd.merge — It's Just a SQL JOIN

If you can write a SQL JOIN, you can already merge in pandas. The mental model is identical: take two tables, line up rows where a key matches, and stitch the columns together.

SQL
-- Join customers to their orders
SELECT *
FROM customers c
INNER JOIN orders o
  ON c.customer_id = o.customer_id;
Pandas
import pandas as pd

# Join customers to their orders
result = pd.merge(
    customers,
    orders,
    on="customer_id",
    how="inner",
)

Translation rule: FROM a JOIN b ON a.key = b.key → pd.merge(a, b, on="key", how=...). The first DataFrame is the left table, the second is the right. The on argument is your ON clause, and how picks the join type. That's the whole API.

The Four Join Types

The how argument maps one-to-one onto SQL join types. Same names, same behavior, same edge cases with unmatched rows turning into nulls.

SQL Pandas how= Keeps
INNER JOIN "inner" (default) Only rows matched in both tables
LEFT JOIN "left" All left rows; right cols null if no match
RIGHT JOIN "right" All right rows; left cols null if no match
FULL OUTER JOIN "outer" All rows from both; nulls where no match
SQL
-- Every customer, even with no orders
SELECT *
FROM customers c
LEFT JOIN orders o
  ON c.customer_id = o.customer_id;

-- Everyone from both sides
SELECT *
FROM customers c
FULL OUTER JOIN orders o
  ON c.customer_id = o.customer_id;
Pandas
# Every customer, even with no orders
pd.merge(
    customers, orders,
    on="customer_id",
    how="left",
)

# Everyone from both sides
pd.merge(
    customers, orders,
    on="customer_id",
    how="outer",
)

SQLite has no RIGHT or FULL OUTER JOIN — you fake them by flipping the tables or using UNION. Pandas supports all four directly with how=. If you came up on SQLite, how="right" and how="outer" will feel like new tools.

Join Keys — on, left_on, right_on

When both tables share a column name, use on. When the key columns are named differently, you spell out each side — exactly like writing a.id = b.user_id in SQL.

SQL
-- Same column name on both sides
... ON c.customer_id = o.customer_id

-- Different names per table
SELECT *
FROM customers c
JOIN orders o
  ON c.id = o.cust_id;

-- Composite key (two columns)
... ON a.region = b.region
   AND a.year = b.year;
Pandas
# Same column name on both sides
pd.merge(c, o, on="customer_id")

# Different names per table
pd.merge(
    customers, orders,
    left_on="id",
    right_on="cust_id",
)

# Composite key (list of columns)
pd.merge(
    a, b,
    on=["region", "year"],
)

The mental model: on= is shorthand for "the key column has the same name in both tables." If the names differ, switch to left_on= and right_on=. A composite key is just a list: on=["region", "year"]. There is no separate AND keyword — the list is the AND.

Overlapping non-key columns get suffixed. If both tables have a name column that isn't a join key, pandas keeps both and renames them name_x and name_y. Control this with suffixes=("_cust", "_order") so you don't end up debugging mystery _x columns later.

Index Joins and the .join() Shortcut

DataFrames also have a .join() method that merges on the index by default. It is handy when your key already lives in the index, but for column keys, plain pd.merge is clearer.

# Merge on a column from the index of the right frame
customers.merge(orders, left_on="customer_id", right_index=True)

# .join() defaults to index-on-index, left join
left_df.join(right_df, how="left")    # aligns on the index

Guarding Against Bad Joins

The classic JOIN bug is duplicate keys silently fanning out your rows — a one-to-one join that quietly becomes one-to-many. Pandas gives you two safety rails that SQL makes you check by hand.

# Assert the relationship you expect; raises if it's wrong
pd.merge(customers, orders, on="customer_id", validate="one_to_many")

# Add a column showing where each row came from
merged = pd.merge(
    customers, orders,
    on="customer_id",
    how="outer",
    indicator=True,
)
merged["_merge"]   # "left_only", "right_only", or "both"

validate= is your seatbelt. Options are "one_to_one", "one_to_many", "many_to_one", and "many_to_many". If the data violates your assumption, the merge raises instead of silently producing a bloated table. SQL has no equivalent — you find out from a wrong row count.

pd.concat — Stacking, Not Joining

Merging combines tables side by side on a key. Concatenating glues them end to end. If merge is a JOIN, then concat is a SQL UNION ALL — or, going the other way, gluing columns together.

SQL
-- Stack rows from two tables
SELECT * FROM sales_jan
UNION ALL
SELECT * FROM sales_feb;

-- UNION (not ALL) dedupes;
-- concat keeps everything by default
Pandas
# Stack rows (axis=0 is the default)
all_sales = pd.concat([sales_jan, sales_feb])

# Reset the index so rows are 0..n again
all_sales = pd.concat(
    [sales_jan, sales_feb],
    ignore_index=True,
)

# Glue columns side by side instead
wide = pd.concat([df_a, df_b], axis=1)

The mental model: pd.concat takes a list of DataFrames, not two positional ones. axis=0 (default) stacks rows like UNION ALL; axis=1 stacks columns side by side. Columns line up by name when stacking rows; missing columns become NaN rather than an error.

concat keeps the original index by default. Stack two 100-row frames and you get index labels 0–99 twice over. That breaks .loc lookups and confuses joins downstream. Pass ignore_index=True unless you have a deliberate reason to preserve the old index.

merge vs concat at a glance

You want to... Tool SQL analogy
Match rows on a key, combine columns pd.merge JOIN
Stack rows from same-shaped tables pd.concat(axis=0) UNION ALL
Glue columns by position/index pd.concat(axis=1) (no clean SQL analogy)

Reshaping — Wide vs Long

Two layouts hold the same data. Wide spreads a variable across columns; long (tidy) stacks it into rows. You reshape constantly — long is great for grouping and plotting, wide is great for human reading and side-by-side comparison.

# WIDE: one row per city, a column per year
   city    2022   2023   2024
0  Austin   840    910    975
1  Denver   720    760    810

# LONG: one row per city-year observation
   city    year   pop
0  Austin  2022   840
1  Austin  2023   910
2  Austin  2024   975
3  Denver  2022   720
...

The mental model: if you find yourself naming columns 2022, 2023, 2024 — values masquerading as column names — you are in wide format. melt turns those headers back into data (wide → long). pivot does the reverse, promoting a column's values into headers (long → wide).

melt — Wide to Long

melt takes columns you want to keep fixed (id_vars) and unpivots the rest into two columns: one for the old header names, one for the values.

wide = pd.DataFrame({
    "city": ["Austin", "Denver"],
    "2022": [840, 720],
    "2023": [910, 760],
    "2024": [975, 810],
})

long = wide.melt(
    id_vars="city",            # columns to keep as-is
    var_name="year",           # name for the old headers
    value_name="pop",          # name for the values
)
# -> columns: city, year, pop  (6 rows)

There is a JavaScript parallel here too. Reshaping wide to long is the same instinct as flatMap-ing each object into one record per measured value.

JavaScript
const wide = [
  { city: "Austin", 2022: 840, 2023: 910 },
  { city: "Denver", 2022: 720, 2023: 760 },
];

const years = ["2022", "2023"];

const long = wide.flatMap((row) =>
  years.map((y) => ({
    city: row.city,
    year: y,
    pop: row[y],
  }))
);
Pandas
wide = pd.DataFrame({
    "city": ["Austin", "Denver"],
    "2022": [840, 720],
    "2023": [910, 760],
})

long = wide.melt(
    id_vars="city",
    var_name="year",
    value_name="pop",
)
# one line vs a nested map

Tidy data rule of thumb: each variable is a column, each observation is a row. Long format is "tidy," and it is what groupby, seaborn, and most plotting libraries expect. When a function complains it cannot find your "year" column, you are probably handing it wide data — melt first.

pivot — Long to Wide

pivot is the inverse of melt. You pick which column supplies the new row labels (index), which supplies the new column headers (columns), and which fills the cells (values).

wide_again = long.pivot(
    index="city",        # becomes the row labels
    columns="year",      # its values become headers
    values="pop",        # fills the cells
)
# columns are now 2022, 2023, 2024; index is city

pivot fails on duplicate index/column pairs. If two rows share the same city and year, pandas does not know which value to place in that one cell and raises an error. When duplicates exist and you want them aggregated (summed, averaged), use pivot_table with an aggfunc instead.

pivot (no aggregation)
# Requires unique index/column pairs
long.pivot(
    index="city",
    columns="year",
    values="pop",
)
pivot_table (aggregates dupes)
# Collapses duplicates with aggfunc
long.pivot_table(
    index="city",
    columns="year",
    values="pop",
    aggfunc="sum",
)

Translation rule: pivot_table is essentially a GROUP BY rendered as a grid. index and columns are your grouping keys, values is the measure, and aggfunc is the aggregate (SUM, AVG, COUNT). Anyone who has built a pivot table in a spreadsheet already knows the muscle.

Putting It All Together

A realistic flow — combine two datasets, then reshape the result for analysis. Say you have customers in one frame and their monthly orders in another, and you want a wide grid of revenue per customer per month.

import pandas as pd

# --- Two source datasets ---
customers = pd.DataFrame({
    "customer_id": [1, 2, 3],
    "name": ["Ada", "Lin", "Omar"],
    "region": ["West", "East", "West"],
})

orders = pd.DataFrame({
    "customer_id": [1, 1, 2, 2, 3],
    "month": ["Jan", "Feb", "Jan", "Feb", "Jan"],
    "revenue": [120, 90, 200, 150, 75],
})

# --- 1. MERGE: attach customer info to each order (LEFT JOIN) ---
df = pd.merge(
    orders,
    customers,
    on="customer_id",
    how="left",
    validate="many_to_one",   # many orders, one customer
)
# df now has: customer_id, month, revenue, name, region

# --- 2. PIVOT: revenue grid, one row per customer, a column per month ---
grid = df.pivot_table(
    index="name",
    columns="month",
    values="revenue",
    aggfunc="sum",
    fill_value=0,             # no order that month -> 0
)
#         Feb  Jan
# name
# Ada      90  120
# Lin     150  200
# Omar      0   75

# --- 3. MELT: flatten the grid back to tidy rows for plotting ---
tidy = grid.reset_index().melt(
    id_vars="name",
    var_name="month",
    value_name="revenue",
)
# tidy: one row per name-month, ready for groupby or seaborn

# --- 4. CONCAT: append a new month of data later ---
march = pd.DataFrame({
    "name": ["Ada", "Lin", "Omar"],
    "month": ["Mar", "Mar", "Mar"],
    "revenue": [110, 175, 60],
})
tidy = pd.concat([tidy, march], ignore_index=True)

Notice the round trip. You merge to enrich, pivot to make a human-readable report, then melt back to tidy form for the next computation. Wide for eyes, long for code. Moving fluidly between the two is most of what day-to-day pandas reshaping actually is.

Check Your Understanding

Quick Quiz

1. What is the pandas equivalent of SELECT * FROM a LEFT JOIN b ON a.id = b.id?

pd.merge(a, b, on="id", how="left"). The first argument is the left table, on is the join key, and how="left" keeps every left row, filling right-side columns with NaN where there is no match.

2. The two tables use different names for the join key — id on the left, cust_id on the right. How do you merge them?

Use left_on and right_on: pd.merge(left, right, left_on="id", right_on="cust_id"). The on shortcut only works when the key column has the same name on both sides.

3. When would you use pd.concat instead of pd.merge?

When you are stacking, not matching. pd.concat glues frames end to end — rows with axis=0 (like SQL UNION ALL), or columns with axis=1. merge is for combining columns by matching a key. Remember ignore_index=True when stacking rows.

4. You have columns named 2022, 2023, 2024 and need one row per year instead. Which function, and what direction?

melt — it goes wide to long. Use df.melt(id_vars="city", var_name="year", value_name="pop"). The year headers become data in a new year column, and the cell values land in pop.

5. Why might pivot raise an error where pivot_table succeeds?

pivot requires unique index/column pairs — one value per cell. If duplicates exist, it cannot decide which value to use and raises. pivot_table handles duplicates by collapsing them with an aggfunc like "sum" or "mean".
