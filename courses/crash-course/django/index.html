Django What It Is & How to Talk About It

← Back to crash course
Texas Tribune Interview Prep

Django is a Python web framework. That's the whole sentence. It takes a URL someone visits, runs some Python, and sends back a web page or JSON data. The Tribune uses it to run their CMS, their data apps, and the backend that feeds their D3 graphics. Here's what you need to know to talk about it.

What is Django?

Django is a web framework written in Python. When someone visits a website powered by Django, here's what happens:

Someone visits a URL → Django matches it to a function → That function pulls data from the database → Fills in an HTML template → Sends the page back

That's the whole framework. Everything else — models, views, templates, the admin panel — is just Django's name for each step in that flow.

The analogy: Think of Django like a newsroom's assignment desk. A request comes in (the URL). The assignment desk routes it to the right reporter (the view). The reporter pulls records from the filing cabinet (the database/models). Then writes the story using a standard layout (the template). The finished page goes out to the reader.

How to say it: "Django handles the backend — the database, the URL routing, the API endpoints, the admin interface. It's a Python framework that's been around since 2005, so it's very mature and battle-tested. It's especially popular in newsrooms because of how fast you can build content management tools with it."

Why Newsrooms Love Django

Django has been the default backend for news organizations for years. The Washington Post, The New York Times, the Tribune — tons of newsrooms use it. Here's why:

  • The admin panel is free. You define your data (stories, election results, county data) and Django auto-generates a full editing interface. Reporters and editors can create, edit, search, and filter content without a developer building anything. This is Django's killer feature for newsrooms.
  • It's fast to build with. Django comes with authentication, database management, URL routing, form handling, and security all built in. You're not assembling 15 npm packages — it's all there.
  • Python is the newsroom language. Data journalists already use Python for analysis (pandas, Jupyter notebooks). Django lets them use the same language for the web app.
  • It handles data well. Django's ORM (object-relational mapper) lets you query databases in Python instead of raw SQL. That makes data pipelines — like importing election results from the AP — straightforward.
  • "Batteries included" philosophy. Django's motto. Unlike more minimal frameworks, Django ships with everything you need out of the box.

How to say it: "What makes Django a natural fit for newsrooms is the admin panel. You define your data models, and Django gives you a full content management interface for free. Editors can publish stories, update election results, manage event listings — all without engineering support. On the developer side, the ORM makes it straightforward to build data pipelines for things like ingesting AP election results."

Django's Four Pieces

Django organizes code into four main parts. If someone mentions any of these, you'll know what they're talking about:

The Core Pieces

Models
Your database tables, defined as Python classes. A Story model has fields like headline, body, pub_date, published. Each field becomes a column in the database. You never write SQL to create tables — Django generates it from your Python code.
Views
The Python functions that handle requests. A view receives a URL request, does something (usually queries the database), and returns a response (usually an HTML page or JSON data). In other frameworks, these are called "controllers" or "route handlers."
Templates
HTML files with placeholders that Django fills in with data. Think of them like mail merge: you write the layout once with blank spots, and Django fills in the specific story headline, date, author, etc. for each page.
URLs (URL conf)
A list that maps URL patterns to view functions. "When someone visits /stories/, run the story_list view. When they visit /stories/texas-budget/, run the story_detail view." This is explicit — you write it out, nothing is magic.

That pattern has a name: MTV (Model-Template-View). It's Django's version of the more common MVC (Model-View-Controller) pattern. The naming is slightly different, but the concept is identical: separate your data, your logic, and your presentation.

Don't mix up: Django "views" are not like React views/components. A Django view is a backend function that handles a request — more like an API route handler. The "template" is the part that's closer to what React developers think of as a "view."

More Vocabulary You Should Know

Terms That Might Come Up

ORM (Object-Relational Mapper)
Django's way of talking to the database in Python instead of SQL. Instead of writing SELECT * FROM stories WHERE published = true, you write Story.objects.filter(published=True). Same result, but in Python. It works with PostgreSQL, MySQL, SQLite — you can switch databases without changing your code.
Migrations
When you change a model (add a field, rename something), Django generates a "migration" — a script that updates the database to match. The whole team commits migrations to git, so everyone's database stays in sync. You run makemigrations to generate them and migrate to apply them.
Django Admin
The auto-generated editing interface. You register a model and instantly get a web-based CRUD (create, read, update, delete) panel. Editors use it to manage content. You can customize it with search, filters, custom actions, and inline editing. This is Django's biggest selling point for newsrooms.
Django REST Framework (DRF)
An add-on library for building JSON APIs. If your D3 chart needs data from the backend, DRF is how you serve it. It handles serialization (converting database objects to JSON), pagination, filtering, and authentication. Most Django shops use it.
Management Commands
Custom terminal commands you write in Django. In a newsroom, these are your data pipelines: "import election results from this CSV," "pull the latest AP data," "scrape the state legislature website." They're scheduled with cron jobs or run manually. Think of them as scripts that have full access to your database.
Middleware
Code that runs on every request before it reaches your view, and on every response before it leaves. Authentication checks, security headers, logging. You don't usually write middleware yourself, but you should know it exists as a layer.
WSGI / ASGI
How Django talks to the web server in production. WSGI is the traditional (synchronous) standard. ASGI is the newer async version. You don't need to know the details — just know that in production, Django runs behind something like Gunicorn (a WSGI server) or Uvicorn (ASGI), not the built-in runserver.
Apps
Django projects are made of "apps" — self-contained modules. A Tribune project might have a stories app, an elections app, and a data_visuals app. Each has its own models, views, and URLs. They're meant to be reusable and independent.

Django vs. Other Frameworks

If they compare it to something you know, here's how to orient:

Framework How It Compares
Express.js Minimal — you assemble everything yourself. Django is "batteries included" (auth, ORM, admin all built in).
Next.js Full-stack JavaScript/React. Django is Python-only backend. They can work together: Django serves APIs, Next.js renders the frontend.
Flask Python, but minimal like Express. Flask gives you routing and that's about it. Django gives you the whole kitchen. Flask is better for tiny projects; Django is better for big ones like a CMS.
Ruby on Rails The closest equivalent. Both are "batteries included," opinionated, have ORMs, auto-generated admin tools, and follow similar patterns. Django is to Python what Rails is to Ruby.
WordPress Different level entirely. WordPress is a ready-made CMS. Django is a framework you build a CMS with. Much more flexible, but more engineering required.

How to say it: "Django's sweet spot is when you need a structured backend with good data management. Compared to something like Flask, it's more opinionated but gives you a lot for free — especially the admin panel and the ORM. For a newsroom that manages a lot of structured content, that tradeoff makes sense."

How D3 and Django Work Together

This is probably the most useful thing you can talk about in the interview — because it shows you understand the full stack:

Django serves JSON data via API → D3 fetches that JSON in the browser → D3 draws the visualization

They're two completely separate layers that talk through HTTP. The Django backend doesn't know or care that D3 is drawing a chart — it just serves data. The D3 frontend doesn't know or care that Django is talking to a PostgreSQL database — it just fetches JSON.

This separation is powerful:

  • A backend developer can change the database structure without touching the chart code
  • A graphics developer can redesign the visualization without touching the API
  • The same API can feed a D3 chart, a mobile app, and an internal tool
  • On election night, Django ingests new results from the AP. The D3 chart polls the API and updates automatically. Two independent systems, working together.

How to say it: "What I find interesting about the D3-Django stack is how clean the separation is. Django owns the data layer — models, migrations, data ingestion. D3 owns the presentation layer — the visual encoding, the interactivity, the responsiveness. They meet at the API. That means on election night, the data pipeline team can focus on getting results into Django, and the graphics team can focus on the D3 chart rendering correctly. Neither team is blocked by the other."

The Admin Panel — Talk About This

If there's one Django feature to talk about with enthusiasm, it's the admin.

Here's the concept: you define a database table in Python (a "model"). You add one line of code to register it with the admin. Django then auto-generates a full web interface where non-technical users can:

  • Create, edit, and delete records
  • Search and filter by any field
  • Sort by date, author, status
  • Bulk-select and perform actions ("publish selected stories")

No frontend code. No React. No forms. Just register the model and it works.

For a newsroom, this means:

  • Reporters can publish stories without bugging a developer
  • Editors can update election results by hand if the automated pipeline breaks
  • Data journalists can manage datasets through a web UI instead of raw SQL
  • Developers spend their time on features, not building CRUD interfaces

How to say it: "The admin is one of the reasons I'm drawn to Django for newsroom work. In a fast-moving environment, you don't always have time to build custom interfaces. The admin gives editors and reporters a way to manage content immediately, and developers can customize it over time — add search, filters, custom actions, whatever the team needs."

Data Pipelines in Django

Newsrooms ingest a lot of external data — election results from the AP, census data, school ratings, campaign finance records. Django handles this with management commands: custom scripts that can read files, call APIs, and save data to the database.

The concept:

  1. Write a Python script that reads a CSV or calls an API
  2. Use Django's ORM to save each row to the database
  3. Run it from the terminal: python manage.py import_election_results
  4. Schedule it with a cron job to run automatically (e.g., every 5 minutes on election night)

The D3 frontend then reads from that same database via the API. Data flows in from the AP, Django stores it, D3 displays it. A complete pipeline.

How to say it: "I've been learning about Django's management commands as a way to handle data ingestion — pulling in external data sources and loading them into the database. I can see how that'd be critical on election night: you'd have a command polling the AP results feed, saving to the database, and the D3 graphics reading from the API in near-real time."

Smart Questions You Could Ask Them

  • "What does your Django stack look like? Is it primarily serving templates or mostly acting as an API backend for JavaScript frontends?"
  • "How do you handle data ingestion for election results — is that a Django management command pipeline, or something separate?"
  • "How much do reporters and editors interact with the Django admin directly?"
  • "Are you using Django REST Framework for your APIs, or something else?"
  • "What's the deployment setup — are you on Heroku, AWS, or something custom?"
  • "How do you handle the relationship between your CMS content and the interactive graphics?"

Your Advantage — Don't Forget This

You're a journalist who codes. Not a developer who reads the news. That distinction matters.

Most developers the Tribune interviews can write Python and build APIs. They can't do what you can do:

  • You know what makes a good story. You know which data point is the lead, what context readers need, and what questions they'll ask.
  • You understand the editorial process. Deadline pressure, fact-checking, source verification, editorial judgment about what to publish.
  • You can talk to reporters. You speak their language. You know what "the nut graf" is. You know why a graphic needs a clear takeaway, not just cool interactivity.
  • You're learning the technical side. That trajectory — journalist → developer — is exactly what organizations like the Tribune value, because it's rare.

If you say one thing in this interview, say this: "I bring editorial judgment to the development process. I've been in newsrooms. I know what makes a graphic useful to readers, not just technically impressive. The code skills I'm building are how I deliver on that editorial instinct at scale."
