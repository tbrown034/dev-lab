06 Next.js App Router — The File System Is the API

← Back to lessons

You've used the App Router. Now understand its architecture. Layouts, loading states, error boundaries, parallel routes, and intercepting routes — each is a carefully designed primitive that solves a specific UX problem. This lesson maps those problems to those solutions.

The File Convention System

App Router uses special filenames as declarations. Each file in a route directory serves a specific architectural purpose:

app/
  layout.tsx       → Root layout (wraps all pages, never re-renders on navigation)
  page.tsx         → Home page (/)
  loading.tsx      → Suspense fallback for the page
  error.tsx        → Error boundary for the page
  not-found.tsx    → 404 page

  articles/
    layout.tsx     → Articles layout (persists across article navigation)
    page.tsx       → Articles list (/articles)
    loading.tsx    → Loading state for articles list

    [slug]/
      page.tsx     → Single article (/articles/budget-bill-passes)
      loading.tsx  → Loading state for individual article
      error.tsx    → Error boundary for individual article

  (dashboard)/     → Route group (parentheses = no URL segment)
    analytics/
      page.tsx     → /analytics (not /dashboard/analytics)
    settings/
      page.tsx     → /settings

Each special file maps to a React concept. layout.tsx is a component that wraps its children and preserves state across navigations. loading.tsx is a Suspense boundary. error.tsx is an Error Boundary. page.tsx is the actual content. You're not learning Next.js conventions — you're learning React patterns expressed as files.

Layouts: Persistent UI That Never Re-Renders

A layout wraps its children. When you navigate between pages that share a layout, the layout does not re-render. Its state persists.

// app/articles/layout.tsx
export default function ArticlesLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <div className="articles-layout">
      <aside>
        <ArticlesSidebar />  {/* Stays mounted during navigation */}
      </aside>
      <main>
        {children}  {/* This swaps when navigating between articles */}
      </main>
    </div>
  );
}

When the user navigates from /articles/climate-report to /articles/budget-bill, only the page.tsx content swaps. The sidebar, any scroll position in the sidebar, any expanded/collapsed state — all preserved. This is not a hack. It's the architecture.

Root layout is special. app/layout.tsx is the only required layout. It must include <html> and <body> tags. It wraps every page in your application and is the place for global providers, analytics scripts, and fonts. It never unmounts during a client-side navigation.

Loading States: Automatic Suspense Boundaries

Creating a loading.tsx file automatically wraps the corresponding page.tsx in a React Suspense boundary. The loading component shows instantly while the page's async data loads.

// app/articles/[slug]/loading.tsx
export default function ArticleLoading() {
  return (
    <div className="article-skeleton">
      <div className="skeleton-title" />
      <div className="skeleton-byline" />
      <div className="skeleton-body">
        <div className="skeleton-line" />
        <div className="skeleton-line" />
        <div className="skeleton-line short" />
      </div>
    </div>
  );
}

// What Next.js generates behind the scenes:
<Suspense fallback={<ArticleLoading />}>
  <ArticlePage />
</Suspense>

The beauty of this: the layout renders immediately. The sidebar is usable. Only the page content shows a loading state. The user gets a responsive, interactive shell while the expensive data fetching happens.

Error Boundaries: Granular Error Handling

error.tsx creates an error boundary around the page. When an error occurs, only the affected section shows the error UI — the rest of the application continues working.

// app/articles/[slug]/error.tsx
'use client'; // Error components MUST be client components

export default function ArticleError({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  return (
    <div className="error-state">
      <h2>Something went wrong loading this article</h2>
      <p>{error.message}</p>
      <button onClick={() => reset()}>
        Try again
      </button>
    </div>
  );
}

Error boundaries must be Client Components. This seems counterintuitive, but it makes sense: error boundaries need to catch errors during rendering and maintain state (the error itself, the "try again" handler). These are browser-side concerns. The reset() function re-renders the error boundary's children, retrying the failed render.

Parallel Routes: Multiple Pages in One Layout

Parallel routes let a single layout render multiple "slots" simultaneously. Each slot has its own loading and error states. Think of it as a multi-pane dashboard where each pane loads independently.

app/
  dashboard/
    layout.tsx          → Renders @analytics, @activity, and children
    page.tsx            → Main content
    @analytics/
      page.tsx          → Analytics pane (loads independently)
      loading.tsx       → Analytics skeleton
      error.tsx         → Analytics error state
    @activity/
      page.tsx          → Activity feed pane
      loading.tsx       → Activity skeleton
// app/dashboard/layout.tsx
export default function DashboardLayout({
  children,
  analytics,
  activity,
}: {
  children: React.ReactNode;
  analytics: React.ReactNode;
  activity: React.ReactNode;
}) {
  return (
    <div className="dashboard-grid">
      <main>{children}</main>
      <aside className="analytics">{analytics}</aside>
      <aside className="activity">{activity}</aside>
    </div>
  );
}

Parallel routes solve the "waterfall" problem. Without them, if your dashboard needs data from three APIs, they load sequentially: fetch A, render, fetch B, render, fetch C, render. With parallel routes, each slot is its own Suspense boundary. All three fetch simultaneously, and each pane renders as soon as its data arrives. The slowest API doesn't block the fastest.

Intercepting Routes: Modals Done Right

Intercepting routes let you "intercept" a navigation and show a different view — typically a modal. The URL updates, but you stay on the current page with the intercepted content in a modal. If the user refreshes or shares the URL, they get the full page version.

app/
  articles/
    page.tsx                → Article list
    [slug]/
      page.tsx              → Full article page (direct navigation)
    (.)articles/[slug]/
      page.tsx              → Article modal (intercepted from list)

The (.) prefix means "intercept routes at the same level." When the user clicks an article from the list, instead of navigating to the full article page, Next.js shows the article in a modal overlay. The URL changes to /articles/slug. If the user refreshes, they see the full page version at that URL.

// app/articles/(.)articles/[slug]/page.tsx — the modal version
import { Modal } from '@/components/modal';

export default async function ArticleModal({ params }) {
  const article = await getArticle(params.slug);

  return (
    <Modal>
      <h1>{article.title}</h1>
      <p>{article.excerpt}</p>
      <a href={`/articles/${params.slug}`}>Read full article</a>
    </Modal>
  );
}

This is the Instagram pattern. Click a photo from the grid: modal opens, URL changes. Refresh: see the full photo page. Share the URL: recipient sees the full page. This pattern used to require complex client-side routing logic. With intercepting routes, it's a file system convention.

Caching: The Most Misunderstood Part

Next.js App Router has multiple caching layers. Understanding them is critical for debugging "why isn't my data updating?"

CacheWhereWhatDuration
Request MemoizationServerDeduplicate identical fetches in one renderOne request lifecycle
Data CacheServerCached fetch() responsesUntil revalidated
Full Route CacheServerStatic HTML + RSC payloadUntil revalidated
Router CacheClientRSC payloads for visited routesSession (or 30s/5min)
// Control caching per-fetch
const articles = await fetch('https://api.example.com/articles', {
  next: { revalidate: 60 },  // Revalidate every 60 seconds
});

const user = await fetch('https://api.example.com/user', {
  cache: 'no-store',  // Never cache — always fresh
});

// Control caching at the route level
export const dynamic = 'force-dynamic'; // Opt out of static generation
export const revalidate = 3600;         // Revalidate every hour
// Revalidate from Server Actions
'use server';

export async function publishArticle(formData: FormData) {
  await db.article.create({ /* ... */ });

  revalidatePath('/articles');     // Invalidate specific path
  revalidateTag('articles');       // Invalidate by cache tag
}

Check Your Understanding

1. What happens to a layout component when the user navigates between its child pages?

Nothing. The layout does not re-render. Its state persists. Only the children (page.tsx content) swaps. This is why layouts are ideal for navigation sidebars, persistent headers, and shared state that shouldn't reset on navigation.

2. What React concept does loading.tsx implement?

A Suspense boundary. Next.js wraps the page component in <Suspense fallback={<Loading />}>. The loading component renders immediately while the async page component resolves. This is React Suspense expressed as a file convention.

3. Why must error.tsx be a Client Component?

Error boundaries need to catch rendering errors and maintain state (the error object, the reset function). They need event handlers for the "try again" button. These are all client-side concerns that require "use client." Server Components can't use hooks or event handlers.

4. How do parallel routes prevent loading waterfalls?

Each parallel route slot (@analytics, @activity) is its own Suspense boundary. They fetch data simultaneously and render independently. One slow API doesn't block others. Each slot shows its own loading state and resolves when ready.

5. Your article list shows stale data after publishing a new article. Which caches might be involved?

Three possible caches: the Data Cache (server-side cached fetch response), the Full Route Cache (cached static page), and the Router Cache (client-side cached RSC payload). You need to call revalidatePath('/articles') or revalidateTag('articles') in your publish server action to bust the server caches, and the router cache will pick up the fresh data on the next navigation.
