01 How LLMs Work — The Mental Model

← Back to lessons

No code in this lesson. Just concepts. You need to understand what's happening inside the box before you start building on top of it.

The Mental Model: Superhuman Autocomplete

Every LLM does exactly one thing: predict the next token. That's it. When you type "The capital of France is" and Claude responds "Paris," it's not "knowing" geography. It's predicting that "Paris" is the most likely next token based on the patterns it learned during training.

This is the single most important concept in this entire course. Every behavior you see from an LLM — creative writing, code generation, reasoning, mistakes, hallucinations — emerges from next-token prediction. If you internalize this, everything else makes sense.

Think of it like this: your phone's keyboard predicts the next word. An LLM does the same thing, but trained on a significant fraction of all human text, with billions of parameters that capture incredibly nuanced patterns. The difference is scale, not kind.

Tokens: The Atoms of Language

LLMs don't read words. They read tokens — chunks of text that are usually 3–4 characters long. A word like "understanding" might be split into ["under", "stand", "ing"]. A common word like "the" is a single token.

Why does this matter?

  • Pricing — you pay per token (input + output). Knowing token count helps you estimate costs.
  • Context limits — models have a maximum token window (Claude: 200K tokens). That's how much text fits in one conversation.
  • Speed — output is generated one token at a time. Longer responses = more time.

Rule of thumb: 1 token is roughly 3/4 of a word. 100 tokens is roughly 75 words. A full page of text is about 300–400 tokens. Claude's 200K context window fits roughly a 500-page book.

Tokenization in practice

// What you type:
"The journalist investigated the corruption scandal."

// What the model sees (approximately):
["The", " journalist", " investigated", " the", " corruption", " scandal", "."]

// Each token maps to a number (the model works with numbers):
[464, 28782, 28685, 272, 26164, 24090, 13]

Notice: spaces are included in tokens. Punctuation is often its own token. Rare words get split into multiple tokens. This is why LLMs sometimes struggle with spelling — they literally don't see individual letters.

Journalism parallel: Tokenization is like how a wire service breaks a story into standard chunks for transmission. The chunks aren't always where you'd expect the breaks to be, but the system needs them to be consistent and predictable.

Attention: How the Model "Reads"

The breakthrough that made modern LLMs possible is called attention (from the 2017 paper "Attention Is All You Need"). Here's the mental model:

When predicting the next token, the model doesn't just look at the previous word. It looks at every previous token and decides how much weight to give each one. Some tokens are highly relevant to the prediction; others aren't.

Without attention (old approach)
// Read left to right, one word at a time
// By the time you reach the end,
// you've "forgotten" the beginning

"The reporter who covered the
 city hall beat for 20 years
 and won three Pulitzers
 finally ___"

// Hard to predict: what was
// the subject again?
With attention (transformer approach)
// Look at ALL tokens at once
// Weight them by relevance

"The REPORTER who covered the
 city hall beat for 20 years
 and won three Pulitzers
 finally ___"

// "reporter" gets high attention
// weight — it's the subject.
// Prediction: "retired" or "quit"

Attention lets the model connect tokens that are far apart in the text. "The reporter ... finally retired" — the model gives high weight to "reporter" even though it's 15 tokens back, because it's the grammatical subject.

Why this matters for you: Attention is why prompt structure matters. When you put important instructions at the beginning or end of a prompt, the model can attend to them strongly. When you bury key context in the middle of a long document, attention may be weaker. This directly affects your prompting strategy in Lesson 02.

Transformers: The Architecture

A transformer is the neural network architecture that uses attention. It's the "T" in GPT (Generative Pre-trained Transformer). Claude is also a transformer. Here's what you need to know:

  1. Input: Your prompt gets tokenized into numbers.
  2. Embedding: Each token number gets converted into a high-dimensional vector (a list of ~4,000 numbers that represent its meaning).
  3. Attention layers: The vectors pass through many layers of attention (Claude has dozens). Each layer refines the model's understanding of how tokens relate to each other.
  4. Output: The final layer produces a probability distribution over all possible next tokens. The model picks one (or samples randomly from the top candidates).
  5. Repeat: The chosen token is appended to the input, and the whole process runs again for the next token.

Key insight: The model generates text one token at a time. When Claude writes a 500-word response, it runs this entire process ~375 times. Each time, it "sees" everything it's already written plus your original prompt. This is why streaming exists — you can show each token as it's generated instead of waiting for the full response.

Context Windows: The Model's Working Memory

The context window is the total amount of text (in tokens) the model can "see" at once. It includes your system prompt, the conversation history, any documents you've pasted in, AND the model's own response so far.

// Claude's context window: ~200,000 tokens

[System prompt]           ~500 tokens
[Conversation history]    ~2,000 tokens
[Document you pasted]     ~50,000 tokens
[Claude's response so far] ~1,000 tokens
─────────────────────────────────────────
Total used:               ~53,500 tokens
Remaining:                ~146,500 tokens

Critical misconception: The context window is NOT long-term memory. Every API call starts fresh. The model doesn't "remember" previous conversations unless you explicitly include them in the prompt. When you chat with Claude in a web interface, the UI is sending the entire conversation history with every message.

Think of it like a desk. The context window is how big the desk is. You can spread out papers (tokens) on it. But when you leave the room and come back, the desk is cleared. If you want to continue, you have to put the papers back out.

Temperature: Creativity vs. Precision

When the model predicts the next token, it doesn't just pick the #1 most likely option. It produces a probability distribution and samples from it. Temperature controls how this sampling works:

temperature: 0 (deterministic)
// Always picks the highest-
// probability token.
// Same input = same output.
// Good for: fact extraction,
// classification, structured data

"The capital of France is ___"
→ "Paris" (100% of the time)
temperature: 1 (creative)
// Samples more broadly from
// the probability distribution.
// Same input = varied output.
// Good for: creative writing,
// brainstorming, diverse suggestions

"Write a headline about ___"
→ Different every time

Practical rule: Use temperature: 0 for tasks where you want consistency (data extraction, classification, code generation). Use temperature: 0.7–1.0 for creative tasks (writing, brainstorming). You'll set this in the API calls starting in Lesson 03.

How Models Are Trained (The 30-Second Version)

Training happens in phases:

  1. Pre-training: The model reads a massive amount of text (books, websites, code, articles) and learns to predict the next token. This gives it general language understanding and knowledge. Takes months on thousands of GPUs.
  2. Fine-tuning (RLHF): Human raters rank model responses from best to worst. The model is adjusted to produce responses humans prefer. This is what makes Claude helpful and safe rather than just a raw text predictor.
  3. The cutoff: The model's knowledge stops at its training data cutoff. It doesn't know about events after that date unless you tell it in the prompt.

Journalism parallel: Pre-training is like a reporter reading every newspaper ever published. Fine-tuning is like an editor teaching them the house style, what's publishable, and what's not. The cutoff is why you always need to fact-check — the model's "knowledge" is frozen at a point in time.

Hallucinations: Why Models Make Stuff Up

LLMs sometimes generate text that sounds authoritative but is factually wrong. This isn't a bug — it's a fundamental consequence of how they work:

  • The model predicts the most likely next token. Sometimes "most likely" is wrong.
  • It has no internal fact-checker. It doesn't "know" things the way a database does.
  • It fills in gaps with plausible-sounding text, just like a student writing an essay about a book they skimmed.

This is why lessons 05–06 (embeddings and RAG) exist. The solution to hallucination is to give the model the actual source material instead of relying on what it "remembers" from training.

For journalists, this is critical. Never publish AI-generated facts without verification. Use AI to draft, summarize, and surface leads — but always verify against primary sources. Lesson 11 covers the full ethical framework.

Check Your Understanding

1. What does an LLM actually do, at the most fundamental level?

It predicts the next token based on all the tokens before it. Everything else (conversation, reasoning, code generation) emerges from this one operation repeated millions of times.

2. What's the approximate relationship between tokens and words?

1 token is roughly 3/4 of a word, or about 3-4 characters. 100 tokens is about 75 words.

3. Why does attention matter for prompt engineering?

Attention determines how much weight the model gives to different parts of your prompt. Important instructions should be positioned where they get strong attention (beginning or end). Context buried in the middle of long documents may get weaker attention.

4. Does Claude "remember" your previous conversations?

No. Each API call starts with an empty context window. The chat interface creates the illusion of memory by sending the full conversation history with every message. The model has no persistent memory between calls.

5. When should you use temperature 0 vs. temperature 1?

Temperature 0 for deterministic tasks (data extraction, classification, factual Q&A). Temperature 0.7-1.0 for creative tasks (writing, brainstorming). Higher temperature = more randomness in token selection.

6. Why do LLMs hallucinate?

Because they predict the most likely next token, not the most truthful one. They have no internal fact-checker. They fill in gaps with plausible-sounding text. The solution is RAG: giving the model actual source material to reference.

Exercise: Token Estimation

Before you touch the API, practice thinking in tokens:

  1. Take one of your published articles (or any 1,000-word article).
  2. Estimate how many tokens it contains using the 3/4 rule.
  3. If you sent this article as context + a 200-word prompt + received a 500-word response, estimate the total tokens consumed.
  4. At $3 per million input tokens and $15 per million output tokens (Claude Sonnet pricing), estimate the cost of that single API call.

This exercise grounds the abstract concepts in real numbers you'll work with every day.
