07 Agents — AI That Reasons and Acts

← Back to lessons

An agent is a loop, not a call. Instead of "send prompt, get response," an agent reasons about what to do, does it, observes the result, and decides what to do next. Repeat until done.

Mental Model: The Intern with a Checklist

Imagine you hand an intern a task: "Research the mayor's campaign donors and write a summary." The intern doesn't do this in one step. They:

  1. Think: "I need to find donor records. Let me search the FEC database."
  2. Act: Search the database.
  3. Observe: "Found 47 records. The top donors are..."
  4. Think: "I should cross-reference with our articles about these donors."
  5. Act: Search the article archive.
  6. Observe: "Found 3 articles mentioning donor conflicts of interest..."
  7. Think: "I have enough to write the summary."
  8. Act: Write the summary.

That's an agent. Think → Act → Observe → repeat. The model is the "thinker," your tools are the "actions," and the tool results are the "observations."

The key difference from tool use (Lesson 04): Tool use is one round — the model calls a tool and responds. An agent is multiple rounds of reasoning + tool use, where each round's observations inform the next round's thinking. The model decides when it has enough information to stop.

The ReAct Pattern

ReAct (Reasoning + Acting) is the most common agent pattern. The model alternates between reasoning about the task and taking actions:

// ReAct prompt structure
const system = `You are a research agent for investigative journalists.

You have access to these tools:
- search_articles: Search the article archive
- search_public_records: Search government databases
- get_document: Retrieve a specific document

For each step:
1. THINK: Explain your reasoning about what to do next
2. ACT: Use a tool to get information
3. OBSERVE: The tool result will be provided

Continue until you have enough information to answer.
Then provide your final answer with citations.`;

// The agent loop (same as Lesson 04's tool loop, but with intent)
async function runAgent(task) {
  let messages = [{ role: "user", content: task }];
  let steps = 0;
  const MAX_STEPS = 10;

  while (steps < MAX_STEPS) {
    const response = await client.messages.create({
      model: "claude-sonnet-4-6",
      max_tokens: 2048,
      system,
      tools,
      messages
    });

    steps++;

    if (response.stop_reason === "tool_use") {
      // Execute all tool calls
      const toolResults = [];
      for (const block of response.content) {
        if (block.type === "tool_use") {
          const result = await executeTools(block.name, block.input);
          toolResults.push({
            type: "tool_result",
            tool_use_id: block.id,
            content: JSON.stringify(result)
          });
        }
      }

      messages.push({ role: "assistant", content: response.content });
      messages.push({ role: "user", content: toolResults });
      continue;
    }

    // Agent decided it's done
    const text = response.content.find(b => b.type === "text");
    return { answer: text?.text, steps };
  }

  return { answer: "Max steps reached", steps };
}

Always set a max steps limit. Without it, a confused agent could loop forever, burning through your API budget. 10–20 steps is typical. If the agent can't solve the problem in that many steps, it probably needs better tools or a clearer task.

Four Agent Patterns

1. ReAct (Reasoning + Acting)

What we just built. The model reasons, acts, observes, and repeats. Best for open-ended research tasks.

2. Plan-then-Execute

The model first creates a plan, then executes each step. Better for complex, multi-stage tasks.

// Plan-then-execute prompt
const system = `You are a research agent.

When given a task:
1. First, create a numbered PLAN with specific steps
2. Then execute each step one at a time
3. After each step, check if the plan needs adjustment
4. When all steps are complete, synthesize the results`;

// The model will first output:
// "Plan:
//  1. Search for campaign finance records
//  2. Identify top 10 donors
//  3. Cross-reference with article archive
//  4. Check for conflicts of interest
//  5. Write summary
//
// Executing Step 1..."

3. Reflection

After generating a response, a second pass critiques and improves it. Good for quality-sensitive tasks.

// Reflection pattern: two calls
async function reflectiveAgent(task) {
  // First pass: generate
  const draft = await client.messages.create({
    model: "claude-sonnet-4-6",
    max_tokens: 2048,
    system: "You are a research assistant. Generate a thorough answer.",
    messages: [{ role: "user", content: task }]
  });

  // Second pass: critique and improve
  const improved = await client.messages.create({
    model: "claude-sonnet-4-6",
    max_tokens: 2048,
    system: `You are a critical editor. Review this draft for:
- Unsupported claims (flag them)
- Missing context or nuance
- Logical gaps
- Factual accuracy concerns
Then rewrite an improved version.`,
    messages: [{ role: "user", content: `Draft:\n${draft.content[0].text}\n\nOriginal task: ${task}` }]
  });

  return improved.content[0].text;
}

4. Orchestrator

A top-level agent that delegates subtasks to specialized sub-agents. For complex workflows.

// Orchestrator: one model coordinates, specialized models execute
async function orchestrator(task) {
  const plan = await client.messages.create({
    model: "claude-sonnet-4-6",
    max_tokens: 1024,
    system: `Break this task into subtasks. Return JSON:
{ "subtasks": [{ "type": "research|analysis|writing", "description": "..." }] }`,
    messages: [{ role: "user", content: task }]
  });

  const subtasks = JSON.parse(plan.content[0].text).subtasks;
  const results = [];

  for (const subtask of subtasks) {
    // Each subtask gets a specialized system prompt
    const systemPrompts = {
      research: "You are a thorough researcher. Find and cite specific facts.",
      analysis: "You are a data analyst. Identify patterns and anomalies.",
      writing: "You are a journalist. Write clear, factual prose in AP Style."
    };

    const result = await client.messages.create({
      model: "claude-sonnet-4-6",
      max_tokens: 1024,
      system: systemPrompts[subtask.type],
      messages: [{ role: "user", content: subtask.description + "\n\nContext: " + results.join("\n") }]
    });

    results.push(result.content[0].text);
  }

  return results;
}

Agent Safety

Agents can do real things. If your tools can write files, send emails, or make API calls, a confused agent can cause real damage. Always:

  • Sandbox dangerous actions — require human approval for write operations
  • Set step limits — prevent infinite loops
  • Log everything — trace the agent's reasoning for debugging
  • Use read-only tools first — searching is safe; writing isn't
  • Test with mock tools — simulate before connecting real systems

Check Your Understanding

1. What's the difference between tool use and an agent?

Tool use is typically one round: the model calls a tool, gets the result, and responds. An agent is multiple rounds of reasoning + tool use, where each round's observations inform the next. The model autonomously decides what to do and when to stop.

2. What does ReAct stand for, and how does the pattern work?

Reasoning + Acting. The model alternates: (1) THINK about what to do, (2) ACT by calling a tool, (3) OBSERVE the result. Repeat until the task is complete. It's a loop of deliberate reasoning interleaved with concrete actions.

3. Why is a step limit critical for agents?

Without a limit, a confused or stuck agent could loop forever, making API calls (costing money) without making progress. The limit is a safety valve that forces the agent to stop and report what it found, even if it didn't fully complete the task.

4. When would you use the reflection pattern?

For quality-sensitive tasks where accuracy matters more than speed: fact-checking, editorial review, legal analysis. The second pass catches errors, unsupported claims, and logical gaps that the first pass missed. It doubles the cost but significantly improves quality.
