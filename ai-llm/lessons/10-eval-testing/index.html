10 Eval & Testing — Proving It Works

← Back to lessons

You can't unit test vibes. But you can build evaluation systems that tell you whether your AI is getting better or worse with every change you make.

Mental Model: Unit Tests for Vibes

In traditional software, you write assert(add(2, 3) === 5). The output is deterministic. With LLMs, the output varies every time. You can't assert exact strings. Instead, you measure qualities across many examples:

  • Does the response answer the question? (relevance)
  • Are the facts correct? (accuracy)
  • Does it follow the format instructions? (compliance)
  • Is it the right length? (conciseness)
  • Did it refuse to answer when it should have? (safety)

The journalism parallel: You don't evaluate a reporter by whether they wrote the exact sentence you imagined. You evaluate them on accuracy, sourcing, clarity, and adherence to editorial standards. LLM evals work the same way.

The Structure of an Eval

// An eval has three parts:

// 1. TEST CASES: input + expected output (or criteria)
const testCases = [
  {
    input: "Summarize: The city council voted 7-2 to approve a $50M budget.",
    expected: {
      mentions_vote_count: true,
      mentions_amount: true,
      max_words: 30,
      no_hallucinated_details: true
    }
  },
  {
    input: "Summarize: [long article about education reform]",
    expected: {
      mentions_key_policy: true,
      mentions_stakeholders: true,
      max_words: 50
    }
  }
];

// 2. RUNNER: calls the model for each test case
async function runEval(testCases, config) {
  const results = [];
  for (const tc of testCases) {
    const response = await client.messages.create({
      model: config.model,
      max_tokens: config.maxTokens,
      system: config.systemPrompt,
      messages: [{ role: "user", content: tc.input }]
    });
    results.push({
      input: tc.input,
      output: response.content[0].text,
      expected: tc.expected,
      usage: response.usage
    });
  }
  return results;
}

// 3. SCORER: grades each result against criteria
function scoreResults(results) {
  return results.map(r => {
    const text = r.output.toLowerCase();
    const scores = {};

    if (r.expected.mentions_vote_count) {
      scores.vote_count = text.includes("7-2") ? 1 : 0;
    }
    if (r.expected.mentions_amount) {
      scores.amount = (text.includes("50") || text.includes("$50m")) ? 1 : 0;
    }
    if (r.expected.max_words) {
      scores.conciseness = r.output.split(/\s+/).length <= r.expected.max_words ? 1 : 0;
    }

    const total = Object.values(scores);
    return {
      ...r,
      scores,
      overall: total.reduce((a, b) => a + b, 0) / total.length
    };
  });
}

Types of Evaluations

1. Exact Match (simplest)

// For classification, extraction, or yes/no questions
function exactMatch(output, expected) {
  return output.trim().toLowerCase() === expected.trim().toLowerCase();
}

// "Is this claim true or false?"
// Expected: "false"
// Score: 1 if output === "false", else 0

2. Contains Check

// For checking that key facts appear in the response
function containsAll(output, requiredTerms) {
  const text = output.toLowerCase();
  return requiredTerms.every(term => text.includes(term.toLowerCase()));
}

// Summary must mention: "7-2", "budget", "approved"

3. LLM-as-Judge (most powerful)

// Use a model to evaluate another model's output
async function llmJudge(question, response, criteria) {
  const judgment = await client.messages.create({
    model: "claude-sonnet-4-20250514",
    max_tokens: 512,
    system: `You are an evaluation judge. Score the response on a scale of 1-5
for each criterion. Respond with ONLY valid JSON.`,
    messages: [{
      role: "user",
      content: `Question: ${question}
Response: ${response}

Score each criterion (1-5):
${criteria.map(c => `- ${c}`).join("\n")}

JSON format: { "scores": { "criterion_name": score }, "reasoning": "..." }`
    }]
  });

  return JSON.parse(judgment.content[0].text);
}

// Usage
const eval = await llmJudge(
  "Summarize the budget article",
  modelResponse,
  ["accuracy", "conciseness", "completeness", "sourcing"]
);

LLM-as-judge is a metapattern: You're using AI to evaluate AI. This sounds circular, but it works well in practice because evaluating is easier than generating. A model can reliably score whether a summary is accurate even if it couldn't have written a perfect summary itself. The key is specific, rubric-based criteria.

Prompt Versioning

Every change to a prompt is a potential regression. Track prompt versions the same way you track code versions:

// prompts/summarizer.js
export const VERSIONS = {
  "v1": {
    system: "Summarize the article in 2 sentences.",
    date: "2025-01-15",
    notes: "Initial version"
  },
  "v2": {
    system: "Summarize the article in 2 sentences. Always mention the primary source. Use AP Style.",
    date: "2025-01-20",
    notes: "Added sourcing requirement and style guide"
  },
  "v3": {
    system: `Summarize the article in 2 sentences.
Rules:
- Always mention the primary source by name
- Include the most important number/statistic
- Use AP Style
- Past tense`,
    date: "2025-01-25",
    notes: "Added statistics and tense requirements"
  }
};

// Run eval across versions to compare
async function compareVersions(testCases) {
  const results = {};
  for (const [version, config] of Object.entries(VERSIONS)) {
    results[version] = await runEval(testCases, {
      model: "claude-sonnet-4-20250514",
      maxTokens: 256,
      systemPrompt: config.system
    });
  }

  // Print comparison table
  for (const [version, scores] of Object.entries(results)) {
    const avg = scores.reduce((sum, s) => sum + s.overall, 0) / scores.length;
    console.log(`${version}: ${(avg * 100).toFixed(1)}% average score`);
  }
}

The workflow: Change prompt → run eval → compare scores to previous version → keep the change if scores improve, revert if they don't. This is regression testing for AI. Without it, you're guessing whether your changes helped.

Check Your Understanding

1. Why can't you use traditional unit tests for LLM outputs?

LLM outputs are non-deterministic (even at temperature 0, minor variations occur). You can't assert exact strings. Instead, you evaluate qualities (accuracy, relevance, compliance) across many examples using scoring functions.

2. How does LLM-as-judge work, and why isn't it circular?

You use a model to score another model's output against specific criteria. It's not circular because evaluating (scoring) is a simpler task than generating. A model can reliably judge whether a summary is accurate even if it couldn't write a perfect summary itself. The key is specific, rubric-based criteria.

3. Why is prompt versioning important?

Every prompt change is a potential regression. Without versioning and evals, you can't tell if a change improved or degraded quality. Prompt versioning + eval comparison is the equivalent of running tests after a code change.

4. What's the minimum eval you should have before deploying an AI feature?

At least 20-30 test cases covering normal inputs, edge cases, and failure modes. Each test case should have clear pass/fail criteria. Run the eval on every prompt change and compare to the baseline. Without this, you're shipping blind.
