02 Prompt Engineering — Talking to Models

← Back to lessons

The most important skill in AI development. A great prompt can make a mediocre model perform brilliantly. A bad prompt can make the best model useless.

Mental Model: The Briefing Document

As a journalist, you've briefed editors, sources, and producers. You know that the quality of the brief determines the quality of the output. Prompting works exactly the same way.

A prompt is a briefing document for an AI. It tells the model: who you are, what you need, what format to use, what to avoid, and what good output looks like. The more precise the brief, the better the result.

The mindset shift: Stop thinking "How do I ask this question?" Start thinking "How do I brief this assistant so they can't get it wrong?"

System Prompts: Setting the Stage

The system prompt is a special message that sets the model's behavior for the entire conversation. It's like the standing orders for a newsroom: always active, always in effect.

Weak system prompt
"You are a helpful assistant."

// Too vague. The model has no
// constraints, no personality,
// no specific expertise.
Strong system prompt
"You are a fact-checking assistant
for an investigative journalism team.

Rules:
- Always cite sources by name and date
- Flag claims that can't be verified
- Use AP Style
- Never speculate — say 'unverified'
  if you can't confirm
- Keep responses under 200 words"

// Specific role, clear rules,
// constraints, style guide.

The system prompt goes in the system parameter of the API call (we'll see this in Lesson 03). It's separate from the user's messages and gets special treatment by the model.

Pro tip: Write your system prompt like a job description. Include: the role, the audience, the format, what to do, and what NOT to do. The "what NOT to do" is often more important than the positive instructions.

Few-Shot Prompting: Show, Don't Tell

Instead of explaining what you want, show the model examples. This is called few-shot prompting (zero-shot = no examples, one-shot = one example, few-shot = several examples).

Zero-shot (just instructions)
"Extract the person's name, title,
and organization from this text."

// The model guesses the format.
// Sometimes JSON, sometimes prose,
// sometimes a table.
Few-shot (with examples)
"Extract structured data:

Input: 'Sarah Chen, a senior
reporter at The Post'
Output: {
  name: 'Sarah Chen',
  title: 'senior reporter',
  org: 'The Post'
}

Input: 'interviewed Mayor Johnson'
Output: {
  name: 'Johnson',
  title: 'Mayor',
  org: null
}

Input: [YOUR TEXT HERE]"

// Exact format. Every time.

Why this works: Remember from Lesson 01 — the model predicts the next token based on patterns. When you show it a pattern (input → output, input → output), it continues that pattern. It's literally what it's designed to do.

Chain-of-Thought: Making the Model Show Its Work

For complex tasks, telling the model to think step by step dramatically improves accuracy. This is called chain-of-thought (CoT) prompting.

// WITHOUT chain-of-thought:
"Is this claim true or false: 'The US national debt
exceeded $30 trillion for the first time in 2022.'"
→ "True." // Might be wrong. No reasoning shown.

// WITH chain-of-thought:
"Analyze this claim step by step:
1. Identify the specific factual assertions
2. State what you know about each assertion
3. Note any uncertainties
4. Provide your assessment

Claim: 'The US national debt exceeded $30 trillion
for the first time in 2022.'"

→ "Step 1: The claim asserts that (a) the debt
exceeded $30T and (b) this happened first in 2022.
Step 2: The US national debt crossed $30T in
January 2022 according to Treasury data...
Step 3: My training data confirms this timeline...
Step 4: This claim appears accurate."

The model's reasoning is visible, checkable, and more likely to be correct. It's the difference between a source saying "yes" and a source saying "yes, because X, Y, and Z."

When to use CoT: Math problems, fact-checking, multi-step reasoning, classification with edge cases, and any task where you'd want to see the work. For simple tasks (reformatting text, translation), CoT adds unnecessary tokens and cost.

Structured Output: Getting Parseable Results

When you're building applications (not chatting), you need output your code can parse. The trick: be extremely specific about the format.

// Prompt that produces parseable output:
`Analyze this article and respond with ONLY valid JSON.
No markdown, no explanation, just the JSON object.

Schema:
{
  "headline": "string - a suggested headline",
  "summary": "string - 2 sentence summary",
  "sources_cited": ["string array of source names"],
  "fact_check_flags": ["string array of claims to verify"],
  "tone": "neutral | positive | negative | mixed"
}

Article:
${articleText}`

The pattern: Specify the exact schema. Say "respond with ONLY valid JSON." Include an example of the expected shape. This is so important that Lesson 08 is entirely devoted to structured output patterns.

Common Prompting Mistakes

  1. Being vague. "Summarize this" vs. "Write a 2-sentence summary for a news brief, AP Style, past tense."
  2. No examples. If the format matters, show the model what you want. Don't describe it; demonstrate it.
  3. Ignoring the system prompt. Put persistent instructions in the system prompt, not repeated in every user message.
  4. Asking for perfection. "Make sure there are absolutely zero errors" doesn't help. Specific checks do: "Verify all dates against the source material."
  5. Overloading one prompt. If you need five different things, make five API calls. Don't ask the model to do everything at once.

Check Your Understanding

1. What's the difference between a system prompt and a user prompt?

The system prompt sets the model's overall behavior and personality for the entire conversation (like standing orders). The user prompt is the specific request or input for that turn. System prompts are set once; user prompts change every message.

2. Why does few-shot prompting work so well?

Because LLMs predict the next token by continuing patterns. When you show input-output examples, the model recognizes the pattern and continues it with your new input. You're leveraging the model's core mechanism rather than fighting it.

3. When should you NOT use chain-of-thought prompting?

For simple, well-defined tasks like text reformatting, translation, or simple extraction. CoT adds unnecessary tokens (cost and latency) when the task doesn't require multi-step reasoning.

4. You want the model to return JSON. What's the most reliable way to ensure it?

Specify the exact schema, say "respond with ONLY valid JSON, no markdown," and provide an example of the expected output shape. For maximum reliability, use the API's JSON mode (Lesson 08).

Exercise: Prompt Engineering for Journalism

Write three prompts (system + user) for these journalism tasks. Focus on specificity and constraints:

  1. Source extractor: Given a news article, extract every named source with their title, organization, and what they said (direct or paraphrased).
  2. Headline generator: Given a lead paragraph, generate 5 headline options in different styles (breaking news, feature, analytical, clickbait, SEO-optimized).
  3. Fact-check assistant: Given a claim, break it into verifiable sub-claims and rate each on a confidence scale.
