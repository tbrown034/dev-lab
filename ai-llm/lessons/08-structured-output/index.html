08 Structured Output — Reliable Data from Text

← Back to lessons

When you're building applications, you need data, not prose. This lesson covers every technique for getting reliable, parseable, schema-validated output from Claude.

Mental Model: The Form vs. The Essay

When you interview a source, you get unstructured information — quotes, anecdotes, context. But your CMS needs structured data: headline, byline, section, tags, body. Structured output is the process of converting freeform AI responses into data your code can reliably consume.

The spectrum: At one end is freeform text (chat responses). At the other end is perfectly typed, schema-validated JSON. Most production AI features need something closer to the JSON end. This lesson teaches you how to get there.

Level 1: Prompt-Based JSON

The simplest approach: ask for JSON in the prompt.

Fragile approach
"Extract the sources from
this article."

// Response might be:
// - A bulleted list
// - A paragraph
// - JSON
// - A table
// Unpredictable!
Reliable approach
"Extract sources from this article.
Respond with ONLY a JSON array.
No markdown, no explanation.

Schema:
[{
  name: string,
  title: string | null,
  org: string | null,
  quote_type: 'direct' | 'paraphrase'
}]"

// Consistent JSON every time
const response = await client.messages.create({
  model: "claude-sonnet-4-20250514",
  max_tokens: 1024,
  system: `You are a data extraction assistant.
Always respond with ONLY valid JSON. No markdown code fences.
No explanation before or after the JSON.`,
  messages: [{
    role: "user",
    content: `Extract all named sources from this article.

Schema: [{ "name": "string", "title": "string|null", "org": "string|null", "quote_type": "direct|paraphrase" }]

Article:
${articleText}`
  }]
});

// Parse the response
const sources = JSON.parse(response.content[0].text);
console.log(sources);
// [{ name: "Sarah Chen", title: "spokesperson", org: "City Hall", quote_type: "direct" }, ...]

The "no markdown" instruction matters. Without it, the model often wraps JSON in ```json ... ``` code fences, which breaks JSON.parse(). Be explicit: "No markdown code fences, no explanation, just the JSON."

Level 2: Tool-Based Structured Output

A more reliable approach: define a "tool" whose input schema IS your desired output format. The model is forced to fill the schema:

const response = await client.messages.create({
  model: "claude-sonnet-4-20250514",
  max_tokens: 1024,
  system: "Extract structured data from news articles.",
  tools: [{
    name: "save_article_analysis",
    description: "Save the structured analysis of a news article",
    input_schema: {
      type: "object",
      properties: {
        headline: { type: "string", description: "The article's headline" },
        summary: { type: "string", description: "2-3 sentence summary" },
        sources: {
          type: "array",
          items: {
            type: "object",
            properties: {
              name: { type: "string" },
              title: { type: "string" },
              org: { type: "string" },
              quote_type: { type: "string", enum: ["direct", "paraphrase", "background"] }
            },
            required: ["name", "quote_type"]
          }
        },
        topics: {
          type: "array",
          items: { type: "string" },
          description: "Key topics covered"
        },
        sentiment: {
          type: "string",
          enum: ["positive", "negative", "neutral", "mixed"]
        }
      },
      required: ["headline", "summary", "sources", "topics", "sentiment"]
    }
  }],
  tool_choice: { type: "tool", name: "save_article_analysis" },
  messages: [{ role: "user", content: `Analyze this article:\n\n${articleText}` }]
});

// The response is guaranteed to match the schema
const analysis = response.content.find(b => b.type === "tool_use").input;
console.log(analysis.headline);
console.log(analysis.sources);
console.log(analysis.sentiment);

tool_choice: { type: "tool", name: "..." } is the key. This forces the model to call that specific tool, meaning its output MUST conform to the tool's input schema. It's the most reliable way to get structured output from Claude.

Level 3: Zod Schema Validation

Even with tool-based output, you should validate the data before using it. Zod is the industry standard for TypeScript/JavaScript schema validation:

import { z } from "zod";

// Define the schema
const SourceSchema = z.object({
  name: z.string(),
  title: z.string().nullable(),
  org: z.string().nullable(),
  quote_type: z.enum(["direct", "paraphrase", "background"])
});

const ArticleAnalysis = z.object({
  headline: z.string().min(1),
  summary: z.string().min(10),
  sources: z.array(SourceSchema),
  topics: z.array(z.string()).min(1),
  sentiment: z.enum(["positive", "negative", "neutral", "mixed"])
});

// Validate the model's output
function parseAnalysis(raw) {
  const result = ArticleAnalysis.safeParse(raw);

  if (result.success) {
    return result.data;  // Typed, validated data
  } else {
    console.error("Validation failed:", result.error.issues);
    return null;  // Handle gracefully
  }
}

// Usage
const toolOutput = response.content.find(b => b.type === "tool_use").input;
const validated = parseAnalysis(toolOutput);

if (validated) {
  // Safe to use — all fields are typed and validated
  console.log(`${validated.sources.length} sources found`);
  console.log(`Sentiment: ${validated.sentiment}`);
}

Why validate at all if the tool schema constrains output? Defense in depth. Models occasionally produce malformed output, especially with complex nested schemas. Zod catches these edge cases and gives you clear error messages instead of runtime crashes in production.

Practical Extraction Patterns

Pattern: Entity Extraction

// Extract people, organizations, locations from an article
const entities = {
  name: "extract_entities",
  description: "Extract named entities from text",
  input_schema: {
    type: "object",
    properties: {
      people: { type: "array", items: { type: "object", properties: {
        name: { type: "string" },
        role: { type: "string" },
        first_mention: { type: "string", description: "The sentence where they first appear" }
      }, required: ["name"] }},
      organizations: { type: "array", items: { type: "string" } },
      locations: { type: "array", items: { type: "string" } },
      dates: { type: "array", items: { type: "object", properties: {
        date: { type: "string" },
        context: { type: "string" }
      }}}
    },
    required: ["people", "organizations", "locations"]
  }
};

Pattern: Claim Extraction for Fact-Checking

// Extract verifiable claims from an article
const claims = {
  name: "extract_claims",
  description: "Extract factual claims that can be verified",
  input_schema: {
    type: "object",
    properties: {
      claims: { type: "array", items: { type: "object", properties: {
        claim: { type: "string", description: "The factual assertion" },
        source_in_article: { type: "string", description: "Who made this claim" },
        verifiability: { type: "string", enum: ["easily_verifiable", "requires_research", "subjective"] },
        suggested_sources: { type: "array", items: { type: "string" }, description: "Where to verify" }
      }, required: ["claim", "verifiability"] }}
    },
    required: ["claims"]
  }
};

When Parsing Fails: Retry Strategies

async function extractWithRetry(text, schema, maxRetries = 2) {
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      const response = await client.messages.create({
        model: "claude-sonnet-4-20250514",
        max_tokens: 2048,
        tools: [schema],
        tool_choice: { type: "tool", name: schema.name },
        messages: [{ role: "user", content: text }]
      });

      const output = response.content.find(b => b.type === "tool_use").input;
      const validated = ArticleAnalysis.safeParse(output);

      if (validated.success) return validated.data;

      console.warn(`Attempt ${attempt + 1} validation failed:`, validated.error.issues);
    } catch (error) {
      console.warn(`Attempt ${attempt + 1} error:`, error.message);
    }
  }

  return null; // All retries failed
}

Check Your Understanding

1. What's the most reliable way to get structured JSON from Claude?

Define a tool whose input_schema matches your desired output format, and use tool_choice to force the model to call that tool. The model's output must conform to the schema. Then validate with Zod for defense in depth.

2. What does tool_choice: { type: "tool", name: "..." } do?

It forces the model to call that specific tool instead of choosing whether to use a tool or respond with text. This guarantees the output matches the tool's input schema.

3. Why validate with Zod even when using tool-based output?

Defense in depth. Models can occasionally produce output that technically matches the JSON schema but has semantic issues (empty strings where content is expected, wrong enum values, etc.). Zod catches these edge cases before they cause runtime errors.

4. When should you use prompt-based JSON vs. tool-based structured output?

Use prompt-based JSON for quick prototyping or when the schema is very simple. Use tool-based output for production code, complex schemas, or when reliability is critical. Tool-based is always more reliable but requires more setup.
