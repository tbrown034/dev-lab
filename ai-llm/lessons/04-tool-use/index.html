04 Tool Use — Giving the Model Hands

← Back to lessons

The model can't browse the web, query databases, or send emails. But you can give it tools that do. This is what turns a chatbot into an application.

Mental Model: A Phone and a Rolodex

Imagine a brilliant researcher locked in a room with no internet. Alone, they can only answer from memory. But if you give them a phone and a list of contacts (a rolodex), they can call experts, look up databases, and verify facts. The researcher decides who to call and what to ask. You handle the actual dialing.

That's tool use. You define the tools (functions the model can call). The model decides when and how to call them. Your code executes the functions and feeds the results back.

Critical distinction: The model doesn't execute tools itself. It says "I want to call function X with arguments Y." Your code actually runs the function, gets the result, and sends it back to the model. The model then uses that result to continue its response.

The Tool Use Loop

// The flow:
// 1. You send a message + tool definitions
// 2. Model responds with a tool_use block (instead of text)
// 3. You execute the tool and send the result back
// 4. Model uses the result to generate its final answer
// 5. If the model needs more tools, repeat from step 2

User: "What's the weather in Austin, TX?"
          ↓
Model: tool_use → get_weather({ city: "Austin, TX" })
          ↓
Your code: calls weather API → { temp: 85, condition: "sunny" }
          ↓
Model: "It's currently 85°F and sunny in Austin, TX."

Defining Tools

Tools are defined as JSON schemas in the API call. Each tool has a name, description, and input schema:

const tools = [
  {
    name: "search_articles",
    description: "Search the newspaper's article archive by keyword, date range, or author. Returns matching article titles and snippets.",
    input_schema: {
      type: "object",
      properties: {
        query: {
          type: "string",
          description: "Search keywords or phrase"
        },
        author: {
          type: "string",
          description: "Filter by author name (optional)"
        },
        date_from: {
          type: "string",
          description: "Start date in YYYY-MM-DD format (optional)"
        },
        date_to: {
          type: "string",
          description: "End date in YYYY-MM-DD format (optional)"
        }
      },
      required: ["query"]
    }
  },
  {
    name: "get_article",
    description: "Retrieve the full text of an article by its ID.",
    input_schema: {
      type: "object",
      properties: {
        article_id: {
          type: "string",
          description: "The unique article identifier"
        }
      },
      required: ["article_id"]
    }
  }
];

The description is a prompt. The model uses the tool description to decide when to use it. Write descriptions like you're explaining the tool to a smart intern: what it does, what it returns, when to use it. Vague descriptions lead to wrong tool choices.

Full Example: Article Research Assistant

import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic();

// Your tool implementations
function searchArticles({ query, author, date_from, date_to }) {
  // In production, this queries your database
  return [
    { id: "art-001", title: "City Council Approves Budget", date: "2025-01-15", snippet: "The council voted 7-2..." },
    { id: "art-002", title: "Budget Shortfall Raises Concerns", date: "2025-01-20", snippet: "Critics warn that..." }
  ];
}

function getArticle({ article_id }) {
  return { id: article_id, text: "Full article text here...", author: "Jane Smith", published: "2025-01-15" };
}

// Map tool names to functions
const toolHandlers = {
  search_articles: searchArticles,
  get_article: getArticle
};

// The conversation loop
async function chat(userMessage) {
  let messages = [{ role: "user", content: userMessage }];

  while (true) {
    const response = await client.messages.create({
      model: "claude-sonnet-4-20250514",
      max_tokens: 1024,
      system: "You are a research assistant for investigative journalists. Use the available tools to find and cite specific articles.",
      tools,
      messages
    });

    // If the model wants to use a tool
    if (response.stop_reason === "tool_use") {
      // Find the tool_use block in the response
      const toolUse = response.content.find(b => b.type === "tool_use");

      // Execute the tool
      const handler = toolHandlers[toolUse.name];
      const result = handler(toolUse.input);

      // Add the model's response + tool result to messages
      messages.push({ role: "assistant", content: response.content });
      messages.push({
        role: "user",
        content: [{
          type: "tool_result",
          tool_use_id: toolUse.id,
          content: JSON.stringify(result)
        }]
      });

      // Loop back — model will process the result
      continue;
    }

    // Model is done (no more tool calls)
    const text = response.content.find(b => b.type === "text");
    return text?.text;
  }
}

const answer = await chat("Find articles about the city budget from January 2025");
console.log(answer);

The loop pattern is everything. Send message → check if model wants tools → execute tools → send results back → repeat until the model returns text. This exact pattern powers every AI agent and tool-using application. Master it here; use it everywhere.

Multiple Tool Calls in One Turn

The model can request multiple tools at once. The response content array will have multiple tool_use blocks:

// Model might respond with:
response.content = [
  { type: "text", text: "Let me search for both topics..." },
  { type: "tool_use", id: "call_1", name: "search_articles", input: { query: "budget" } },
  { type: "tool_use", id: "call_2", name: "search_articles", input: { query: "zoning" } }
];

// You must return a tool_result for EACH tool_use, matching by id
messages.push({ role: "assistant", content: response.content });
messages.push({
  role: "user",
  content: [
    { type: "tool_result", tool_use_id: "call_1", content: JSON.stringify(budgetResults) },

Error Handling in Tools

// If a tool fails, return an error in the tool_result
messages.push({
  role: "user",
  content: [{
    type: "tool_result",
    tool_use_id: toolUse.id,
    content: JSON.stringify({ error: "Article not found" }),
    is_error: true
  }]
});

// The model will see the error and adapt:
// "I wasn't able to find that article. Let me try a different search..."

Always handle errors gracefully. Don't let your tool throw an unhandled exception. Catch errors and return them as tool_result with is_error: true. The model is surprisingly good at recovering from tool errors and trying alternative approaches.

Check Your Understanding

1. Does the model actually execute tool functions?

No. The model requests a tool call by returning a tool_use block with the function name and arguments. YOUR code executes the function and sends the result back. The model never runs code directly.

2. What happens after you send a tool result back to the model?

The model processes the result and either (a) makes another tool call if it needs more information, or (b) generates a text response using the tool results. You loop until the model stops requesting tools.

3. Why is the tool description so important?

The model uses the description to decide when and how to use the tool. A vague description leads to wrong tool selection and bad arguments. Write descriptions like you're explaining the tool to a smart colleague.

4. How do you handle tool errors?

Catch the error in your tool handler, return a tool_result with is_error: true and a description of what went wrong. The model will see the error and adapt its approach.
