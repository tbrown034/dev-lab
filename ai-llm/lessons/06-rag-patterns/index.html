06 RAG Patterns — Grounding AI in Facts

← Back to lessons

RAG is the cure for hallucination. Instead of relying on what the model memorized, you give it the actual source material. It's the difference between a closed-book and an open-book exam.

Mental Model: The Open-Book Exam

Without RAG, the model answers from memory (training data). It might be wrong, outdated, or confidently fabricated. With RAG, you retrieve relevant documents and include them in the prompt. The model answers based on the actual text you provided.

RAG = Retrieval-Augmented Generation. Three steps: (1) user asks a question, (2) you retrieve relevant documents from your database, (3) you send those documents + the question to the model. The model generates an answer grounded in real sources.

// The RAG flow:

User question: "What did the city council decide about the budget?"
         ↓
Step 1: RETRIEVE — search your article database for relevant documents
         ↓
Found: [article_1: "Council voted 7-2 to approve...",
        article_2: "Budget includes $5M for..."]
         ↓
Step 2: AUGMENT — inject documents into the prompt
         ↓
Prompt: "Based on ONLY these sources, answer the question:
         [source 1] [source 2]
         Question: What did the city council decide?"
         ↓
Step 3: GENERATE — model answers using the provided sources
         ↓
Response: "The city council voted 7-2 to approve the budget,
          which includes $5M for infrastructure (Source 1)."

Chunking: Breaking Documents into Pieces

You can't embed and retrieve entire articles if they're thousands of words. You need to break them into chunks — smaller pieces that each contain a coherent unit of information.

Bad: fixed-size chunks
// Split every 500 characters
// Might break mid-sentence
// Loses context

"The council voted 7-2 to ap"
"prove the budget. Mayor John"
"son said 'This is a historic"

// Chunk boundary breaks
// the key quote!
Good: semantic chunks
// Split by paragraph or section
// Overlap for context
// Keep metadata

Chunk 1: "The council voted 7-2 to
approve the $50M budget..."
{ source: "art-001", para: 1 }

Chunk 2: "Mayor Johnson said 'This
is a historic investment...'"
{ source: "art-001", para: 2 }

// Each chunk is self-contained
// Practical chunking function
function chunkArticle(article, { chunkSize = 500, overlap = 50 } = {}) {
  const paragraphs = article.text.split("\n\n");
  const chunks = [];
  let currentChunk = "";

  for (const para of paragraphs) {
    if ((currentChunk + para).length > chunkSize && currentChunk) {
      chunks.push({
        text: currentChunk.trim(),
        source: article.id,
        title: article.title,
        date: article.date
      });
      // Keep overlap for context continuity
      const words = currentChunk.split(" ");
      currentChunk = words.slice(-overlap / 4).join(" ") + "\n\n" + para;
    } else {
      currentChunk += (currentChunk ? "\n\n" : "") + para;
    }
  }

  if (currentChunk.trim()) {
    chunks.push({
      text: currentChunk.trim(),
      source: article.id,
      title: article.title,
      date: article.date
    });
  }

  return chunks;
}

Chunk size tradeoffs: Smaller chunks (200–400 tokens) are more precise in retrieval but lose context. Larger chunks (500–1000 tokens) keep more context but may include irrelevant information. Start with ~400 tokens and adjust based on your retrieval quality.

The Complete RAG Pipeline

import Anthropic from "@anthropic-ai/sdk";
import { ChromaClient } from "chromadb";

const claude = new Anthropic();
const chroma = new ChromaClient();

// Step 0 (once): Index your documents
async function indexArticles(articles) {
  const collection = await chroma.getOrCreateCollection({ name: "newsroom" });
  const chunks = articles.flatMap(a => chunkArticle(a));

  await collection.add({
    ids: chunks.map((_, i) => `chunk-${i}`),
    documents: chunks.map(c => c.text),
    metadatas: chunks.map(c => ({
      source: c.source,
      title: c.title,
      date: c.date
    }))
  });

  return collection;
}

// Step 1-3: The RAG query function
async function ragQuery(question, collection) {
  // RETRIEVE: find relevant chunks
  const results = await collection.query({
    queryTexts: [question],
    nResults: 5
  });

  // AUGMENT: build the prompt with sources
  const sources = results.documents[0].map((doc, i) => {
    const meta = results.metadatas[0][i];
    return `[Source: ${meta.title}, ${meta.date}]\n${doc}`;
  }).join("\n\n---\n\n");

  // GENERATE: ask Claude with the sources
  const response = await claude.messages.create({
    model: "claude-sonnet-4-20250514",
    max_tokens: 1024,
    system: `You are a research assistant for journalists.
Answer questions based ONLY on the provided sources.
Always cite which source you're drawing from.
If the sources don't contain enough information, say so.
Never make up facts not in the sources.`,
    messages: [{
      role: "user",
      content: `Sources:\n${sources}\n\nQuestion: ${question}`
    }]
  });

  return response.content[0].text;
}

The critical instruction: "Answer based ONLY on the provided sources." This is what prevents hallucination. Without this constraint, the model will happily blend source material with its training data, which defeats the whole point.

Citation Patterns

For journalism, citations aren't optional. Here are three approaches:

// Pattern 1: Inline citations (simplest)
system: `Cite sources inline like [Source: Title, Date].`

// Output: "The council voted 7-2 [Source: Budget Article, 2025-01-15]"


// Pattern 2: Numbered footnotes (cleanest)
system: `Number each source [1], [2], etc.
Add a "Sources" section at the end.`

// Output: "The council voted 7-2 [1]...
// Sources:
// [1] Budget Article, Jan 15, 2025
// [2] Council Meeting Notes, Jan 14, 2025"


// Pattern 3: Structured JSON (most parseable)
system: `Return JSON with claims and their source references.`

// Output: {
//   "answer": "The council voted 7-2...",
//   "citations": [
//     { "claim": "voted 7-2", "source": "art-001", "quote": "..." }
//   ]
// }

Check Your Understanding

1. What are the three steps of RAG?

Retrieve (find relevant documents from your database), Augment (inject those documents into the prompt), Generate (the model answers based on the provided sources).

2. Why do you need to chunk documents?

Long documents can't be effectively embedded as a single vector (the meaning gets diluted). Chunking creates smaller, focused pieces where each chunk has a clear topic. This makes retrieval more precise — you find the specific paragraph that answers the question, not just the right article.

3. What's the most important instruction for preventing hallucination in RAG?

"Answer based ONLY on the provided sources." Without this, the model blends source material with training data. You also need "If the sources don't contain enough information, say so" to prevent the model from filling gaps with made-up content.

4. Why is chunk overlap useful?

Overlap ensures that information at chunk boundaries isn't lost. If a key fact spans two paragraphs, the overlap captures context from both. Without it, relevant sentences at the edges of chunks might lose their context.
