01 R Basics & Vectors

← Back to lessons

R thinks in vectors, not loops, and counts from one, not zero. If you know Python lists or JavaScript arrays, most of R will feel familiar — but two things will trip you up early: everything is vectorized by default, and indexing starts at 1. This lesson translates R's core data type from the languages you already use.

Assignment — <- vs =

The first thing that looks strange in R is the arrow. Where Python and JavaScript use =, idiomatic R uses <-. Both work for assignment, but the arrow is the convention you'll see everywhere.

Python
# Assignment with =
name = "Trevor"
age = 38
is_admin = True

# Reassign freely
age = age + 1
R
# Idiomatic assignment with <-
name <- "Trevor"
age <- 38
is_admin <- TRUE

# = also works, but <- is the convention
age = 38

# Reassign freely
age <- age + 1

Why two operators? In R, = is reserved for passing arguments inside function calls, like plot(x = 1, y = 2). Using <- for assignment keeps the two ideas visually distinct. The arrow even points in the direction the value flows: x <- 5 means "5 goes into x". There is a rightward version too: 5 -> x, though almost nobody uses it.

RStudio shortcut: press Alt + - (minus) and RStudio types <- with spaces around it. You'll use it thousands of times, so learn it on day one.

The Console and RStudio

R ships with a plain console, but almost everyone works in RStudio — an IDE built specifically for R. Think of it the way you think of a Python REPL plus VS Code fused into one window.

The console evaluates expressions immediately and prints the result. There is no print() needed at the top level — just type an expression and R echoes the answer back.

Python (REPL)
>>> 2 + 2
4
>>> x = 10
>>> x          # bare name prints in REPL
10
>>> print(x)   # explicit print
10
R (console)
> 2 + 2
[1] 4
> x <- 10
> x            # bare name prints
[1] 10
> print(x)     # explicit print
[1] 10

What is that [1]? R prints a position marker before output because R has no scalars — a single number is really a vector of length one. The [1] tells you the first value shown is element one of the vector. When a vector wraps across lines, each new line starts with the index of its first element. Once you accept "everything is a vector," R stops feeling weird.

RStudio layout: four panes — the script editor (top left), the console (bottom left), environment and history (top right), and files, plots, packages, and help (bottom right). Write code in the editor, send a line to the console with Cmd/Ctrl + Enter. Comments use #, the same as Python.

Vectors — the c() Function

The vector is R's fundamental data structure. It is an ordered collection of values, all of the same type. You build one with c(), which stands for "combine".

JavaScript
// Arrays with brackets
const nums = [10, 20, 30, 40];
const names = ["Alice", "Bob"];
const flags = [true, false, true];

// Length
nums.length;   // 4

// Mixed types allowed
const mixed = [1, "two", true];
R
# Vectors with c()
nums <- c(10, 20, 30, 40)
names <- c("Alice", "Bob")
flags <- c(TRUE, FALSE, TRUE)

# Length
length(nums)   # 4

# Mixed types? No — R coerces (see below)
mixed <- c(1, "two", TRUE)
# becomes c("1", "two", "TRUE")

The mental model: a JavaScript array or Python list is a generic container that holds anything. An R vector is closer to a typed column in a spreadsheet or a SQL column — every element shares one type. If you need a true mixed bag, R has list(), which behaves like a Python list. But the workhorse is the atomic vector built with c().

c() flattens. Combining vectors does not nest them: c(c(1, 2), c(3, 4)) gives c(1, 2, 3, 4), not a nested structure. To nest, use list().

Vectorized Arithmetic

This is the single biggest mindset shift coming from Python or JavaScript. In R, math operators apply to every element at once. You almost never write a loop to transform a vector.

Python
prices = [10, 20, 30]

# Add tax with a loop or comprehension
with_tax = [p * 1.07 for p in prices]
# [10.7, 21.4, 32.1]

# Element-wise add of two lists
a = [1, 2, 3]
b = [10, 20, 30]
sums = [x + y for x, y in zip(a, b)]
# [11, 22, 33]
R
prices <- c(10, 20, 30)

# Add tax — operates on the whole vector
with_tax <- prices * 1.07
# c(10.7, 21.4, 32.1)

# Element-wise add of two vectors
a <- c(1, 2, 3)
b <- c(10, 20, 30)
sums <- a + b
# c(11, 22, 33)

The translation rule: in Python you reach for a list comprehension or map(); in R you just apply the operator to the vector. prices * 1.07, nums > 20, sqrt(nums) — all run element by element automatically. This is faster than a manual loop because the work happens in compiled C under the hood.

Recycling — the Helpful Trap

When two vectors differ in length, R "recycles" the shorter one to match the longer. A scalar (length-one vector) recycles to every element, which is exactly why prices * 1.07 works.

c(1, 2, 3, 4) + c(10, 20)
# c(11, 22, 13, 24)  — 10,20 recycled to 10,20,10,20

c(1, 2, 3) + c(10, 20)
# c(11, 22, 13) with a WARNING:
# longer object length is not a multiple of
# shorter object length

Recycling can hide bugs. If lengths divide evenly, R recycles silently and you may not notice the two vectors were the wrong sizes. If they do not divide evenly, you get a warning but still get a result. Always check length() when results look off — this has no equivalent in Python or JavaScript, so it surprises everyone.

Atomic Types

R has a handful of atomic vector types. The three you will use constantly are numeric, character, and logical. Check a type with class() or typeof().

R type Python JavaScript Example
numeric (double) float number 3.14, 42
integer int number 42L (the L suffix)
character str string "hello"
logical bool boolean TRUE, FALSE
complex complex N/A 2+3i
Python
type(42)        # int
type(3.14)      # float
type("hi")      # str
type(True)      # bool

# Booleans are True / False
flag = True
R
class(42)       # "numeric"
class(42L)      # "integer" (L = integer)
class("hi")     # "character"
class(TRUE)     # "logical"

# Booleans are TRUE / FALSE (all caps)
flag <- TRUE
# T and F are shortcuts, but avoid them

Numbers are doubles by default. In R, 42 is numeric (a double), not an integer. You only get an integer with the L suffix: 42L. Coming from Python, where 42 is an int, this catches people. For most analysis work it does not matter — R does the math the same way either way.

Indexing — R Counts from 1

Here is the surprise that bites every developer arriving from Python or JavaScript: R indexes from 1, not 0. The first element is [1], not [0]. SQL counts from 1 too (think SUBSTR), so if you know SQL this feels less alien.

Python (0-based)
nums = [10, 20, 30, 40]

nums[0]      # 10  (first)
nums[1]      # 20
nums[-1]     # 40  (last)
nums[1:3]    # [20, 30]  (slice)

# Index out of range raises IndexError
nums[99]     # IndexError
R (1-based)
nums <- c(10, 20, 30, 40)

nums[1]      # 10  (first!)
nums[2]      # 20
nums[length(nums)]  # 40  (last)
nums[2:3]    # c(20, 30)  (range)

# Out of range returns NA, no error
nums[99]     # NA

The translation rule: take any Python index and add one. Python's nums[0] becomes R's nums[1]. Python's nums[2] becomes R's nums[3]. Slicing also shifts: Python's half-open nums[1:3] (stops before 3) becomes R's inclusive nums[2:3] (includes 3). R ranges include both endpoints.

Out of bounds gives NA, not an error. Where Python raises IndexError and JavaScript returns undefined, R hands back NA (missing value) and keeps going. That is convenient but dangerous — a typo in an index produces a silent NA instead of a loud crash. Check your lengths.

Negative Indexing Means "Drop"

This is a real gotcha. In Python, a negative index counts from the end. In R, a negative index removes that element.

Python
nums = [10, 20, 30, 40]

nums[-1]    # 40  (last element)
nums[-2]    # 30  (second to last)

# To drop the first element:
nums[1:]    # [20, 30, 40]
R
nums <- c(10, 20, 30, 40)

nums[-1]    # c(20, 30, 40)  — drops element 1!
nums[-2]    # c(10, 30, 40)  — drops element 2

# Drop multiple at once
nums[-c(1, 2)]   # c(30, 40)
nums[c(-1, -4)]  # c(20, 30)

Negative index = exclusion in R. nums[-1] does not get the last element — it returns everything except the first. To grab the last element, use nums[length(nums)] or tail(nums, 1). This is the index difference most likely to produce wrong results that still run.

Logical and Named Indexing

R can also subset with a logical vector — keep every position that is TRUE. This pairs perfectly with vectorized comparisons and is how filtering works throughout R.

nums <- c(10, 20, 30, 40)

# Comparison returns a logical vector
nums > 20            # c(FALSE, FALSE, TRUE, TRUE)

# Use it to filter — like SQL WHERE
nums[nums > 20]      # c(30, 40)

# Combine conditions with & (and) and | (or)
nums[nums > 10 & nums < 40]   # c(20, 30)

Think SQL WHERE. nums[nums > 20] reads like "select from nums where value > 20". This logical-subsetting pattern is the foundation of filtering data frames in the next lesson, so it is worth getting comfortable now. Note R uses single & and | for element-wise logic on vectors.

Named Vectors

Vector elements can carry names, letting you index by label instead of position. A named vector sits halfway between a Python list and a Python dict.

Python (dict)
scores = {
    "alice": 90,
    "bob": 85,
    "carol": 95,
}

scores["alice"]      # 90
list(scores.keys())  # ['alice','bob','carol']
list(scores.values())
R (named vector)
scores <- c(alice = 90, bob = 85, carol = 95)

scores["alice"]   # alice: 90
names(scores)     # c("alice","bob","carol")
unname(scores)    # c(90, 85, 95)

# Still vectorized — names come along
scores * 2        # alice 180, bob 170, ...

Add names after the fact with names(x) <- c(...). That assigning-into-a-function syntax looks odd at first, but it is common in R: names(scores) <- c("a", "b", "c") renames the elements in place.

Generating Vectors — seq() and rep()

You rarely type long vectors by hand. R has compact tools for building sequences and repeated patterns. The colon : is the quickest way to make a run of integers.

Python
# Range (0-based, stop exclusive)
list(range(1, 6))      # [1, 2, 3, 4, 5]
list(range(0, 10, 2))  # [0, 2, 4, 6, 8]

# Repeat
[0] * 5                # [0,0,0,0,0]
["a", "b"] * 3         # a,b,a,b,a,b
R
# Colon — inclusive on both ends
1:5                    # c(1, 2, 3, 4, 5)
seq(0, 8, by = 2)      # c(0, 2, 4, 6, 8)
seq(0, 1, length.out = 5)  # 0, .25, .5, .75, 1

# Repeat
rep(0, 5)              # c(0,0,0,0,0)
rep(c("a", "b"), times = 3)   # a,b,a,b,a,b
rep(c("a", "b"), each = 3)    # a,a,a,b,b,b

Key difference: Python's range(1, 6) stops before 6, giving 1 through 5. R's 1:6 is inclusive, giving 1 through 6. The colon is the everyday tool; reach for seq() when you need a step size (by) or a fixed count of points (length.out). rep()'s times repeats the whole vector, while each repeats each element in place.

Missing Values — NA and NULL

R has first-class support for missing data, which is a big reason statisticians love it. NA means "a value exists but is unknown". NULL means "nothing is here at all".

Python / SQL
# Python's stand-ins
missing = None       # absence of value
import math
nan = math.nan       # not-a-number

# SQL has a true missing marker
-- SELECT * FROM t WHERE col IS NULL
R
x <- c(10, NA, 30)    # NA = missing, like SQL NULL

is.na(x)             # c(FALSE, TRUE, FALSE)

# NA is contagious in math
sum(x)               # NA
sum(x, na.rm = TRUE) # 40 — drop the NAs

# NULL = truly nothing, drops out
c(1, NULL, 3)        # c(1, 3) — NULL vanishes
length(NULL)         # 0

NA spreads through calculations. Any arithmetic touching an NA returns NA — 10 + NA is NA, and mean(x) on a vector with one NA is NA. This is intentional: R refuses to silently guess. Pass na.rm = TRUE to functions like sum(), mean(), and max() to ignore missing values. And never test with x == NA — that returns NA; always use is.na(x).

The mental model: NA is SQL's NULL — a known-unknown that propagates and must be handled explicitly. R's NULL is different: it is the empty object, closer to deleting a thing entirely. You will reach for NA constantly and NULL rarely.

Coercion — One Type Wins

Because a vector must be a single type, mixing types forces R to coerce everything to the most flexible common type. The hierarchy runs: logical → integer → numeric → character. Character always wins.

# Mix logical and numeric — logical becomes numeric
c(TRUE, FALSE, 3)     # c(1, 0, 3)

# Mix numeric and character — all become character
c(1, 2, "three")      # c("1", "2", "three")

# Explicit conversion functions
as.numeric("42")      # 42
as.character(42)      # "42"
as.logical(0)         # FALSE
as.logical("yes")     # NA — can't parse (no warning)

# Logicals act as 0/1 in math — handy
sum(c(TRUE, TRUE, FALSE, TRUE))   # 3 — counts TRUEs

That last trick is everywhere in R. Because TRUE is 1 and FALSE is 0, summing a logical vector counts how many conditions are true: sum(nums > 20) tells you how many values exceed 20, and mean(nums > 20) gives the proportion. It is R's idiom for "count rows matching a condition," much like COUNT(*) ... WHERE in SQL.

Failed numeric conversion gives NA, not an error. as.numeric("abc") returns NA with a warning rather than throwing, unlike Python's int("abc") which raises ValueError. After converting user input, check for NA before trusting the result.

Putting It All Together

A realistic snippet — the kind of quick analysis you would type into the console to summarize a small dataset, using only the tools from this lesson.

# Daily site visitors for one week (one NA: tracker was down)
visitors <- c(mon = 120, tue = 135, wed = NA,
              thu = 150, fri = 142, sat = 98, sun = 110)

# How many days did we actually record?
recorded <- sum(!is.na(visitors))      # 6

# Average daily visitors, ignoring the missing day
avg <- mean(visitors, na.rm = TRUE)    # 125.83...

# Which days beat the average? (logical subsetting)
# Comparison keeps NA positions, so exclude them with !is.na
busy <- visitors[visitors > avg & !is.na(visitors)]
busy                                    # tue 135, thu 150, fri 142
names(busy)                             # c("tue","thu","fri")

# Count busy days — sum a logical vector
sum(visitors > avg, na.rm = TRUE)       # 3

# Project 10% growth across the whole week (vectorized)
next_week <- round(visitors * 1.10)
next_week                               # mon 132, tue 148, wed NA, ...

# Drop the weekend with negative indexing
weekdays_only <- visitors[-c(6, 7)]
length(weekdays_only)                   # 5

No loops, no comprehensions — just vectors, named subsetting, and built-in functions that handle missing data. That is R's whole personality in one block.

Check Your Understanding

Quick Quiz

1. Coming from Python's nums[0], how do you get the first element of an R vector?

Use nums[1]. R indexes from 1, not 0. The translation rule is to add one to any Python index: Python's nums[0] becomes R's nums[1].

2. What does nums[-1] do in R, and how is it different from Python?

In R, nums[-1] returns everything except the first element — a negative index removes that position. In Python, nums[-1] returns the last element. To get the last element in R, use nums[length(nums)] or tail(nums, 1).

3. Why does c(1, 2, "three") become c("1", "2", "three")?

A vector must hold one type. When you mix types, R coerces everything to the most flexible common type. Character wins over numeric, so the numbers become strings. Use list() if you truly need mixed types.

4. How do you compute the mean of c(10, NA, 30) without getting NA?

Pass na.rm = TRUE: mean(c(10, NA, 30), na.rm = TRUE) gives 20. By default NA is contagious and the result would be NA. Also never test with x == NA — use is.na(x).

5. In R, how would you double every value in a vector without writing a loop?

Just multiply the vector: x * 2. R is vectorized, so the operator applies to every element automatically. Where Python uses a list comprehension [v * 2 for v in x], R applies the math directly to the whole vector.
