05 Statistics in R

← Back to lessons

R was built by statisticians, for statisticians. Where Python and JavaScript bolt statistics on through libraries, R bakes them into the language itself. This lesson covers the everyday tools — summary statistics, distributions, correlation, t-tests, and linear regression — translated from the Python, SQL, and JavaScript you already know.

Why R Was Built for This

In Python you reach for NumPy, SciPy, and statsmodels. In JavaScript you hunt for an npm package and hope it's maintained. In SQL you can manage AVG and STDDEV but little more. In R, the statistics are the standard library.

Python
# Import everything you need first
import numpy as np
from scipy import stats

x = [4, 8, 15, 16, 23, 42]

np.mean(x)        # 18.0
np.median(x)
np.std(x, ddof=1) # sample sd
stats.ttest_1samp(x, 0)
R
# No imports needed — it's all built in

x <- c(4, 8, 15, 16, 23, 42)

mean(x)       # 18
median(x)
sd(x)         # sample sd by default
t.test(x, mu = 0)

The mental model: in R, statistical functions are global and assume you mean the sample, not the population. sd() and var() divide by n - 1 by default — the same as NumPy's ddof=1. There's no separate import, no flag to remember. That default reflects R's origin: it expects you're analyzing a sample of data, which is what statisticians do all day.

Summary Statistics

The core descriptive functions are named exactly what you'd guess. They're vectorized, so they operate on a whole vector at once — no loops.

Python (NumPy)
import numpy as np

x = np.array([12, 7, 9, 15, 22, 7, 30])

np.mean(x)              # average
np.median(x)            # middle value
np.std(x, ddof=1)       # sample sd
np.var(x, ddof=1)       # sample variance
x.min(); x.max()
np.percentile(x, [25, 50, 75])

# Mode is not in NumPy core
from scipy import stats
stats.mode(x)
R
x <- c(12, 7, 9, 15, 22, 7, 30)

mean(x)                 # average
median(x)               # middle value
sd(x)                   # sample sd
var(x)                  # sample variance
min(x); max(x)
range(x)                # c(min, max) together
quantile(x, c(.25, .5, .75))

# No built-in mode() for stats —
# (mode() returns the data TYPE)

Gotcha: mode() in R is not the statistical mode. mode(x) returns the storage type of an object (like "numeric" or "character"), not the most common value. To get the statistical mode you compute it yourself: names(which.max(table(x))). This surprises everyone coming from other languages.

The summary() Function

R's summary() is the closest thing to Pandas' df.describe(). Call it on a vector for a five-number summary plus the mean. Call it on a whole data frame for a column-by-column breakdown.

x <- c(12, 7, 9, 15, 22, 7, 30)
summary(x)

#    Min. 1st Qu.  Median    Mean 3rd Qu.    Max.
#   7.000   8.000  12.000  14.571  18.500  30.000

# On a data frame, summarizes every column at once
summary(mtcars)   # mtcars is a built-in dataset

Translation rule: Pandas df.describe() → R summary(df). Pandas gives count, mean, std, and quartiles in a tidy table; R's summary() gives min, quartiles, median, mean, and max per column. R also reports counts of levels for categorical (factor) columns — something describe() skips by default.

Quantiles and Percentiles

scores <- c(55, 62, 71, 78, 84, 88, 91, 95, 99)

quantile(scores)                 # 0%, 25%, 50%, 75%, 100%
quantile(scores, 0.9)            # 90th percentile
quantile(scores, c(0.1, 0.5, 0.9))

IQR(scores)                      # interquartile range (Q3 - Q1)

SQL parallel: R's quantile(x, 0.5) is the same idea as SQL's PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY x). R just makes it a one-liner. And unlike AVG() in SQL, R's mean() propagates missing values — see the next warning.

Missing Values — NA Is Contagious

One missing value poisons the whole calculation unless you tell R to ignore it. This is by design: R refuses to silently hide gaps in your data.

SQL
-- NULLs are silently skipped
SELECT AVG(salary) FROM employees;

-- Returns the average of
-- the non-NULL rows only
R
salary <- c(50000, 62000, NA, 71000)

mean(salary)            # NA — one gap ruins it
mean(salary, na.rm = TRUE)   # 61000

sd(salary, na.rm = TRUE)
sum(is.na(salary))      # 1 — count the gaps

Remember na.rm = TRUE. Unlike SQL's aggregate functions, which skip NULLs automatically, R's mean(), sd(), median(), and friends return NA if any value is missing. Add na.rm = TRUE to remove the gaps before computing. Forgetting this is the single most common beginner mistake in R.

Distributions and Random Sampling

R ships with a consistent naming scheme for probability distributions. Every distribution has four functions, prefixed by a letter: r for random draws, d for density, p for cumulative probability, and q for quantiles.

Prefix Meaning Normal example
r Random draws rnorm(100)
d Density (PDF height) dnorm(0)
p Cumulative probability (CDF) pnorm(1.96)
q Quantile (inverse CDF) qnorm(0.975)
Python (NumPy / SciPy)
import numpy as np
from scipy import stats

rng = np.random.default_rng(42)

# 100 draws from Normal(mean=10, sd=2)
rng.normal(10, 2, size=100)

# Uniform, Poisson, binomial
rng.uniform(0, 1, 100)
rng.poisson(3, 100)
rng.binomial(10, 0.5, 100)

# CDF and inverse CDF
stats.norm.cdf(1.96)
stats.norm.ppf(0.975)
R
set.seed(42)

# 100 draws from Normal(mean=10, sd=2)
rnorm(100, mean = 10, sd = 2)

# Uniform, Poisson, binomial
runif(100, 0, 1)
rpois(100, lambda = 3)
rbinom(100, size = 10, prob = 0.5)

# CDF and inverse CDF
pnorm(1.96)       # ~0.975
qnorm(0.975)      # ~1.96

Translation rule: NumPy's rng.normal(mean, sd, size) → R's rnorm(size, mean, sd) — note the size comes first in R. SciPy's stats.norm.cdf() → R's pnorm(), and stats.norm.ppf() → R's qnorm(). Once you learn the r/d/p/q prefix system, every distribution works the same way: rpois, dpois, ppois, qpois, and so on.

Reproducibility with set.seed()

# Without a seed, results change every run
rnorm(3)   # different numbers each time

# With a seed, results are reproducible
set.seed(123)
rnorm(3)   # -0.5605 -0.2302  1.5587
set.seed(123)
rnorm(3)   # exactly the same again

set.seed() is R's np.random.seed(). Call it before any random operation so your analysis is reproducible. Anyone who runs your script with the same seed gets the same numbers — essential for sharing results and debugging.

Sampling from Existing Data

deck <- 1:52

sample(deck, 5)              # 5 cards, no replacement
sample(deck, 5, replace = TRUE)   # with replacement
sample(deck)                # shuffle the whole deck

# Bootstrap: resample your data with replacement
data <- c(4.1, 5.2, 3.8, 6.0, 4.9)
sample(data, length(data), replace = TRUE)

Correlation

Correlation measures how two variables move together, on a scale from -1 to 1. R's cor() handles a pair of vectors or an entire data frame at once.

Python (NumPy)
import numpy as np

hours = [1, 2, 3, 4, 5]
score = [52, 58, 67, 71, 79]

# Returns a correlation matrix
np.corrcoef(hours, score)[0, 1]
# 0.995...

# Spearman needs SciPy
from scipy import stats
stats.spearmanr(hours, score)
R
hours <- c(1, 2, 3, 4, 5)
score <- c(52, 58, 67, 71, 79)

# Returns a single number directly
cor(hours, score)
# 0.9952...

# Spearman is just an argument
cor(hours, score, method = "spearman")

# Correlation matrix of a whole frame
cor(mtcars)

The mental model: cor(x, y) gives you Pearson correlation by default. Switch to method = "spearman" for rank correlation, or "kendall" for Kendall's tau — no extra library. Pass a data frame and you get the full correlation matrix, like df.corr() in Pandas. And cor.test(x, y) adds a p-value and confidence interval to the estimate.

Correlation is not causation, and it only catches straight-line relationships. A correlation near zero does not mean two variables are unrelated — it means they're not linearly related. A perfect parabola can have a Pearson correlation of zero. Always plot the data before trusting a single number.

Hypothesis Testing with t.test()

A t-test asks whether a difference in means is likely real or just noise. R's t.test() covers the common cases through one function and prints a full, readable report.

Python (SciPy)
from scipy import stats

control = [4.1, 3.8, 4.5, 4.0, 3.9]
treat   = [5.2, 4.9, 5.5, 5.1, 5.0]

# Two-sample t-test
t, p = stats.ttest_ind(
    control, treat, equal_var=False
)
print(t, p)

# One-sample against a value
stats.ttest_1samp(control, 4.0)
R
control <- c(4.1, 3.8, 4.5, 4.0, 3.9)
treat   <- c(5.2, 4.9, 5.5, 5.1, 5.0)

# Two-sample t-test (Welch by default)
t.test(control, treat)


# One-sample against a value
t.test(control, mu = 4.0)

# Paired test for before/after data
t.test(before, after, paired = TRUE)

The output of t.test() is a printed report, not a bare number:

	Welch Two Sample t-test

data:  control and treat
t = -6.8034, df = 7.8034, p-value = 0.0001541
alternative hypothesis: true difference in means is not equal to 0
95 percent confidence interval:
 -1.4476785 -0.7123215
sample estimates:
mean of x mean of y
     4.06      5.14

Translation rule: SciPy's ttest_ind(a, b, equal_var=False) → R's t.test(a, b) — R defaults to the Welch test (unequal variances), which is the safer choice. SciPy's ttest_1samp(a, value) → R's t.test(a, mu = value). Where SciPy returns a tuple you unpack, R prints a labeled report and also returns a list you can index: t.test(...)$p.value.

Pull values out of the result. Save the test to a variable and grab pieces by name: result <- t.test(control, treat), then result$p.value, result$estimate, or result$conf.int. Almost every R modeling function returns a structured list like this.

Linear Regression with lm()

This is where R feels purpose-built. Fitting a model is one line, and the formula syntax — y ~ x — is a first-class part of the language, not a string you pass to a library.

Python (statsmodels)
import statsmodels.formula.api as smf

# Formula lives inside a string
model = smf.ols(
    "mpg ~ wt + hp",
    data=df
).fit()

print(model.summary())
model.params        # coefficients
model.predict(new_df)
R
# Formula is real R syntax, not a string
model <- lm(mpg ~ wt + hp, data = mtcars)




summary(model)
coef(model)         # coefficients
predict(model, newdata = new_df)

The formula mini-language: y ~ x means "model y as a function of x." Add predictors with +: y ~ x1 + x2. Include an interaction with *: y ~ x1 * x2 expands to both main effects plus their interaction. Use every other column with y ~ .. Remove the intercept with y ~ x - 1. This same syntax powers glm(), aov(), and most modeling functions in R.

Reading summary(model)

The summary is dense but every piece earns its place. Here's a fitted model and how to read it.

model <- lm(mpg ~ wt + hp, data = mtcars)
summary(model)

#
# Call:
# lm(formula = mpg ~ wt + hp, data = mtcars)
#
# Residuals:
#     Min      1Q  Median      3Q     Max
# -3.941  -1.600  -0.182   1.050   5.854
#
# Coefficients:
#              Estimate Std. Error t value Pr(>|t|)
# (Intercept) 37.22727    1.59879  23.285  < 2e-16 ***
# wt          -3.87783    0.63273  -6.129 1.12e-06 ***
# hp          -0.03177    0.00903  -3.519  0.00145 **
# ---
# Signif. codes:  0 '***' 0.001 '**' 0.01 '*' 0.05 '.' 0.1 ' ' 1
#
# Residual standard error: 2.593 on 29 degrees of freedom
# Multiple R-squared:  0.8268, Adjusted R-squared:  0.8148
# F-statistic: 69.21 on 2 and 29 DF,  p-value: 9.109e-12
Section What it tells you
Estimate The coefficient. Each 1-unit rise in wt drops mpg by ~3.88, holding hp fixed.
Std. Error Uncertainty on that estimate. Smaller is more precise.
t value Estimate divided by its standard error.
Pr(>|t|) The p-value for that predictor. The stars are a quick visual flag.
R-squared Share of variance explained. 0.83 means the model captures 83% of the variation in mpg.
Adjusted R-squared R-squared penalized for adding predictors. Use this when comparing models.
F-statistic Tests whether the model as a whole beats an intercept-only model.

Useful follow-ups after fitting: coef(model) for just the coefficients, confint(model) for their confidence intervals, fitted(model) for predictions on the training data, residuals(model) for the errors, and plot(model) for four diagnostic charts that check the model's assumptions.

A high R-squared does not mean the model is good. R will happily fit a line through anything, including data that violates every assumption behind linear regression. Check the residual plots from plot(model) for patterns, watch for influential outliers, and never extrapolate far beyond the range of your data. The math runs regardless of whether it should.

Putting It All Together

A realistic mini-analysis using R's built-in mtcars dataset — the kind of quick exploration you'd run before reporting anything.

# mtcars ships with R: 32 cars, fuel economy and specs
set.seed(7)

# 1. Describe the outcome we care about
summary(mtcars$mpg)
sd(mtcars$mpg)          # spread of fuel economy

# 2. Which specs move with mpg? (correlation matrix)
cor(mtcars[, c("mpg", "wt", "hp", "disp")])

# 3. Do manual cars (am == 1) get better mpg than automatics?
manual    <- mtcars$mpg[mtcars$am == 1]
automatic <- mtcars$mpg[mtcars$am == 0]

t.test(manual, automatic)
# p-value < 0.05 means the gap is unlikely to be noise

# 4. Model mpg from weight and horsepower
model <- lm(mpg ~ wt + hp, data = mtcars)
summary(model)

# 5. Predict mpg for a hypothetical car
new_car <- data.frame(wt = 3.0, hp = 120)
predict(model, newdata = new_car)   # ~ 21.8 mpg

# 6. Sanity-check the model's assumptions visually
plot(model)             # residuals, Q-Q, leverage, etc.

The whole workflow stayed in base R. Describe, correlate, test, model, predict, diagnose — no imports, no installs. That density is exactly why R remains the default in academic statistics, clinical trials, and econometrics, even as Python wins the broader programming world.

When to Reach for a Statistician

The honest part. Running t.test() or lm() is easy. Knowing whether the result means anything is the hard part, and that's a separate skill from writing R.

Get a real statistician involved when the stakes are high. If you're designing a clinical trial, sizing a study, choosing between competing models, dealing with messy missing-data mechanisms, or making a decision that affects health, money, or policy — the cost of a subtle error far outweighs the cost of a consult. Tools like R lower the barrier to running a test. They don't lower the barrier to interpreting one correctly.

Some honest limits worth internalizing early:

  • p-values are widely misread. A p-value is not the probability your hypothesis is true, and 0.05 is an arbitrary line. A "significant" result can still be tiny and meaningless.
  • Multiple comparisons inflate false positives. Run twenty tests and one will likely look significant by chance alone. There are corrections for this; know they exist.
  • Assumptions matter. Linear regression assumes roughly linear relationships, independent errors, and constant variance. Violate them and the numbers still print — they're just wrong.
  • Garbage in, garbage out. No test rescues biased sampling or a poorly designed study. The hardest statistical problems are upstream of any code.

Check Your Understanding

Quick Quiz

1. Why does mean(x) return NA, and how do you fix it?

Because x contains at least one NA, and R refuses to silently ignore missing values. Add na.rm = TRUE: mean(x, na.rm = TRUE). The same applies to sd(), median(), var(), and other summary functions. This differs from SQL, where aggregates skip NULLs automatically.

2. What do the r, d, p, and q prefixes mean in functions like rnorm and qnorm?

r = random draws (rnorm), d = density / PDF height (dnorm), p = cumulative probability / CDF (pnorm), q = quantile / inverse CDF (qnorm). The system is consistent across distributions: rpois, dbinom, punif, and so on. pnorm is SciPy's norm.cdf; qnorm is norm.ppf.

3. How is R's formula syntax different from passing a model spec in Python's statsmodels?

In R, mpg ~ wt + hp is real language syntax you pass directly to lm(). In statsmodels you pass the same idea as a string: "mpg ~ wt + hp". R's formulas are first-class objects, so + adds predictors, * adds an interaction, . means all other columns, and - 1 drops the intercept.

4. In summary(model), what does the R-squared value tell you, and why prefer adjusted R-squared when comparing models?

R-squared is the share of variance in the outcome the model explains (0 to 1). Adding any predictor — even a useless one — can only raise plain R-squared, so it's misleading for comparison. Adjusted R-squared penalizes extra predictors, so it can drop when you add a variable that doesn't earn its place. Use adjusted R-squared to compare models with different numbers of predictors.

5. Why does R default to the Welch t-test, and how do you run a paired test?

R's t.test(a, b) defaults to Welch's test, which does not assume the two groups have equal variances — the safer, more robust default (SciPy needs equal_var=False to match it). For before/after data on the same subjects, add paired = TRUE: t.test(before, after, paired = TRUE).
