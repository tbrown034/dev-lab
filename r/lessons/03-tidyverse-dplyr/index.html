03 The Tidyverse: dplyr

← Back to lessons

If you know SQL, you already know dplyr. dplyr is R's grammar of data manipulation: a handful of verbs that map almost one-to-one onto SQL clauses and Pandas methods. This lesson covers the pipe, the five core verbs — filter(), select(), mutate(), arrange(), summarise() — plus group_by() and count(), all shown beside the SQL and Pandas you already use.

Loading the Tidyverse

dplyr ships inside the tidyverse, a collection of packages that share a common design. You load it once per session. Throughout this lesson, assume library(dplyr) is already run and we have a tibble called employees.

# Install once, ever
install.packages("tidyverse")

# Load every session (like Python's import)
library(dplyr)

# A small tibble to work with
employees <- tibble(
  name   = c("Alice", "Bob", "Carol", "Dan", "Eve"),
  dept   = c("Eng", "Eng", "Sales", "Sales", "Eng"),
  salary = c(120, 95, 80, 70, 130),
  years  = c(5, 2, 4, 1, 8)
)

Translation rule: library(dplyr) is R's version of import pandas as pd — it brings the verbs into scope. The difference: after library(), the functions are bare (filter(), not pd.filter()). R dumps the package's functions straight into your namespace.

The Pipe — |> and %>%

The pipe takes the thing on its left and feeds it as the first argument to the function on its right. It turns nested calls inside-out into a top-to-bottom reading order — exactly like method chaining in JavaScript or Pandas.

JavaScript (method chaining)
const result = employees
  .filter(e => e.dept === "Eng")
  .map(e => ({ ...e, raise: e.salary * 1.1 }))
  .sort((a, b) => b.salary - a.salary);

// Read top to bottom, left to right
R (the pipe)
result <- employees |>
  filter(dept == "Eng") |>
  mutate(raise = salary * 1.1) |>
  arrange(desc(salary))

# Read top to bottom, same as JS chaining

There are two pipes in the wild. The native pipe |> has been built into base R since version 4.1 (2021). The older magrittr pipe %>% comes from the tidyverse and is still everywhere in existing code.

magrittr pipe (older)
employees %>%
  filter(dept == "Eng") %>%
  summarise(avg = mean(salary))

# Needs library(dplyr) or library(magrittr)
# Supports the . placeholder:
employees %>% lm(salary ~ years, data = .)
native pipe (modern)
employees |>
  filter(dept == "Eng") |>
  summarise(avg = mean(salary))

# Built into R itself, no package needed
# Placeholder is _ and needs a named arg:
employees |> lm(salary ~ years, data = _)

The mental model: x |> f(y) is just f(x, y). The pipe moves the left side into the first slot of the call. So employees |> filter(dept == "Eng") is literally filter(employees, dept == "Eng"). Every dplyr verb takes the data frame as its first argument precisely so the pipe works.

Which pipe should you use? Prefer the native |> for new code — no dependency, and it is now the default RStudio inserts with Ctrl+Shift+M. Reach for %>% only when you need its . placeholder in arbitrary positions, which the native pipe handles more strictly.

filter() — Keep Rows (SQL WHERE)

filter() keeps the rows where a condition is true. It is the WHERE clause, and the Pandas boolean mask.

SQL / Pandas
-- SQL
SELECT *
FROM employees
WHERE dept = 'Eng' AND salary > 100;
# Pandas
employees[
    (employees["dept"] == "Eng") &
    (employees["salary"] > 100)
]
R (dplyr)
employees |>
  filter(dept == "Eng", salary > 100)

# Commas mean AND. Or be explicit:
employees |>
  filter(dept == "Eng" & salary > 100)

# OR uses a single |
employees |>
  filter(dept == "Eng" | salary > 100)

Translation rule: SQL's WHERE and Pandas' boolean mask both become filter(). Inside filter() you write column names bare — no employees["dept"], no quotes around the column. dplyr knows to look those names up inside the data frame. Multiple comma-separated conditions are combined with AND.

Use == for equality, not =. A single = means assignment to a function argument. filter(dept = "Eng") throws an error; you want filter(dept == "Eng"). Same trap as C or JavaScript — double equals to compare.

select() — Pick Columns (SQL SELECT)

select() chooses which columns to keep, drop, or reorder. It is the column list in a SQL SELECT.

SQL / Pandas
-- SQL
SELECT name, salary FROM employees;
# Pandas
employees[["name", "salary"]]

# Drop a column
employees.drop(columns=["years"])
R (dplyr)
employees |> select(name, salary)

# Drop a column with a minus sign
employees |> select(-years)

# Rename while selecting
employees |> select(employee = name, salary)

select() also understands helper functions for matching column names by pattern, which is handy on wide tables.

employees |> select(starts_with("s"))     # salary
employees |> select(ends_with("e"))       # name
employees |> select(where(is.numeric))    # salary, years
employees |> select(name:salary)          # a range of columns

Tip: a leading minus drops columns. select(-years) keeps everything except years — cleaner than Pandas' .drop(columns=...) when you want most columns. To rename a column without dropping anything, use the dedicated rename(new = old) verb.

mutate() — Add or Change Columns

mutate() creates new columns or overwrites existing ones. SQL expresses this as a computed column in the SELECT list; Pandas uses assign() or bracket assignment.

SQL / Pandas
-- SQL
SELECT *,
       salary * 1000      AS annual,
       salary / years     AS per_year
FROM employees;
# Pandas
employees.assign(
    annual=employees["salary"] * 1000,
    per_year=employees["salary"] / employees["years"],
)
R (dplyr)
employees |>
  mutate(
    annual   = salary * 1000,
    per_year = salary / years
  )

The mental model: mutate() is vectorized, just like a Pandas Series or a SQL computed column. salary * 1000 runs against the whole column at once — no loop. You can also reference a column you just created earlier in the same mutate() call, top to bottom.

employees |>
  mutate(
    raise    = salary * 0.1,        # new column
    new_pay  = salary + raise,      # reuses raise above
    senior   = years >= 5,          # logical column
    band     = if_else(salary > 100, "high", "low")
  )

mutate() does not change the original. Like every dplyr verb, it returns a new tibble. employees |> mutate(...) leaves employees untouched unless you reassign with employees <- employees |> mutate(...). dplyr is functional, not in-place — the opposite of Pandas' inplace=True habit.

arrange() — Sort Rows (SQL ORDER BY)

arrange() sorts rows. Ascending by default; wrap a column in desc() for descending.

SQL / Pandas
-- SQL
SELECT * FROM employees
ORDER BY dept ASC, salary DESC;
# Pandas
employees.sort_values(
    ["dept", "salary"],
    ascending=[True, False],
)
R (dplyr)
employees |>
  arrange(dept, desc(salary))

# dept ascending, salary descending

Translation rule: ORDER BY col DESC → arrange(desc(col)). Pandas needs a parallel ascending=[True, False] list; dplyr puts the direction right on each column with desc(), which reads more clearly when you mix directions.

summarise() + group_by() — Aggregate (SQL GROUP BY)

This pairing is the heart of analysis. group_by() splits the data into groups; summarise() collapses each group to a single row. Together they are SQL's GROUP BY plus aggregate functions, and Pandas' groupby().agg().

SQL / Pandas
-- SQL
SELECT dept,
       COUNT(*)      AS n,
       AVG(salary)   AS avg_salary
FROM employees
GROUP BY dept;
# Pandas
(employees
    .groupby("dept")
    .agg(n=("salary", "size"),
         avg_salary=("salary", "mean"))
    .reset_index())
R (dplyr)
employees |>
  group_by(dept) |>
  summarise(
    n          = n(),
    avg_salary = mean(salary)
  )

The mental model: group_by() alone changes nothing visible — it just tags the tibble with grouping. The next summarise() then runs once per group. n() is dplyr's COUNT(*): it counts rows in the current group. Inside summarise(), ordinary functions like mean(), sum(), min(), and max() become aggregates.

Both spellings work: summarise() (British) and summarize() (American) are identical functions. The tidyverse accepts either.

Groups linger after summarise(). By default summarise() drops the last grouping level but may leave others in place, which can surprise later steps. Add .groups = "drop" to end ungrouped, or call ungroup() explicitly: summarise(n = n(), .groups = "drop").

count() — The Shortcut for Tallies

Counting rows per group is so common it gets its own verb. count(dept) is shorthand for group_by(dept) |> summarise(n = n()) — the equivalent of Pandas' value_counts().

SQL / Pandas
-- SQL
SELECT dept, COUNT(*) AS n
FROM employees
GROUP BY dept
ORDER BY n DESC;
# Pandas
employees["dept"].value_counts()
R (dplyr)
employees |> count(dept)

# Sort the tally by size, largest first
employees |> count(dept, sort = TRUE)

The Verb Map

Keep this table next to you while the verbs sink in. Read across: same idea, three dialects.

dplyr verb SQL clause Pandas equivalent
filter() WHERE boolean mask df[df.x > 1]
select() SELECT col1, col2 df[["a", "b"]]
mutate() SELECT expr AS new df.assign(new=...)
arrange() ORDER BY df.sort_values(...)
summarise() aggregate fns (AVG, SUM) df.agg(...)
group_by() GROUP BY df.groupby(...)
count() GROUP BY ... COUNT(*) df["x"].value_counts()
rename() AS alias df.rename(columns=...)
distinct() SELECT DISTINCT df.drop_duplicates()
slice_head(n) LIMIT n df.head(n)

The order of operations is the same too. SQL is written SELECT … FROM … WHERE … GROUP BY … ORDER BY but executed roughly FROM → WHERE → GROUP BY → SELECT → ORDER BY. A dplyr pipeline simply writes the steps in execution order, top to bottom. That is why dplyr often feels more honest than SQL about what actually happens.

Putting It All Together

A realistic multi-step pipeline. The question: for each department, among employees with at least two years of tenure, what is the headcount and average salary — sorted from highest-paid department down? Here it is in SQL first, then dplyr.

SQL
SELECT
    dept,
    COUNT(*)              AS headcount,
    ROUND(AVG(salary), 1) AS avg_salary
FROM employees
WHERE years >= 2
GROUP BY dept
HAVING COUNT(*) > 1
ORDER BY avg_salary DESC;
R (dplyr)
employees |>
  filter(years >= 2) |>
  group_by(dept) |>
  summarise(
    headcount  = n(),
    avg_salary = round(mean(salary), 1),
    .groups    = "drop"
  ) |>
  filter(headcount > 1) |>
  arrange(desc(avg_salary))

Notice the two filters. The first filter() runs before grouping — that is SQL's WHERE. The second runs after summarise() on the aggregated result — that is SQL's HAVING. dplyr does not need a separate keyword: a filter() after summarise() simply filters the grouped output. One verb, two roles, decided by position.

The same pipeline can be wrapped in a function so you can reuse it across data sets. dplyr verbs all take the data frame as their first argument, so a function that pipes is trivial to write.

library(dplyr)

dept_summary <- function(data, min_years = 2) {
  data |>
    filter(years >= min_years) |>
    group_by(dept) |>
    summarise(
      headcount  = n(),
      avg_salary = round(mean(salary), 1),
      top_earner = max(salary),
      .groups    = "drop"
    ) |>
    arrange(desc(avg_salary))
}

# Call it on any tibble with dept, years, salary columns
dept_summary(employees)
dept_summary(employees, min_years = 5)

# Result (min_years = 2):
# dept   headcount  avg_salary  top_earner
# Eng            3       115.0         130
# Sales          1        80.0          80

Why this reads well: each line is one verb, one transformation. You can comment out a single step to debug, or paste the pipeline up to any |> to inspect the intermediate result. That incremental, top-to-bottom workflow is the whole reason analysts reach for dplyr over base R or raw SQL strings.

Check Your Understanding

Quick Quiz

1. What does the pipe x |> f(y) actually do?

It rewrites the call as f(x, y) — the left side becomes the first argument of the function on the right. That is why every dplyr verb takes the data frame as its first argument: it makes the pipe chain cleanly, like method chaining in JavaScript or Pandas.

2. Which dplyr verb maps to SQL's WHERE, and which maps to SELECT col1, col2?

filter() keeps rows (SQL WHERE). select() keeps columns (the SQL column list). The names feel backwards if you think of Pandas, where .filter() works on labels — in dplyr, filter is always about rows.

3. Why does employees |> mutate(raise = salary * 0.1) not change employees?

dplyr is functional: every verb returns a new tibble and leaves the input untouched. To keep the result you must reassign: employees <- employees |> mutate(...). There is no inplace=True as in Pandas.

4. What is the difference between a filter() before group_by() and a filter() after summarise()?

A filter() before grouping is SQL's WHERE — it screens individual rows. A filter() after summarise() is SQL's HAVING — it screens the aggregated groups. dplyr uses the same verb for both; position decides the role.

5. What is count(dept) shorthand for, and what is the Pandas equivalent?

count(dept) is shorthand for group_by(dept) |> summarise(n = n()). It tallies rows per group. The Pandas equivalent is df["dept"].value_counts(). Add sort = TRUE to order the tally largest-first.
