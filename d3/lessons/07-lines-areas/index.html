07 Enter / Update / Exit — The Deep Dive

← Back to lessons

Mental Model: Think of a wedding seating chart. Enter = new guests arriving (need a seat). Update = guests already seated (maybe move them). Exit = guests leaving (clear their seat).

Why This Matters

In Lesson 04, you used .join("rect") and it "just worked." But what happens under the hood when data changes? Understanding enter/update/exit gives you control over animations, transitions, and complex data updates — the difference between a basic chart and a polished one.

The Three Groups

When you call .data(newData), D3 compares the new data to the existing elements and sorts everything into three groups:

  • Enter: Data items that have no matching element. Need to create new elements.
  • Update: Data items that match an existing element. May need to change attributes.
  • Exit: Elements that have no matching data item. Need to be removed.

.join() with three callbacks

svg.selectAll("rect")
  .data(newData, d => d.id)     // key function!
  .join(
    enter => enter.append("rect")  // NEW: create + animate in
      .attr("fill", "#4ade80")
      .attr("y", height)
      .attr("height", 0)
      .call(el => el.transition().duration(500)
        .attr("y", d => y(d.value))
        .attr("height", d => height - y(d.value))),

    update => update               // EXISTING: animate to new position
      .attr("fill", "#f97316")
      .call(el => el.transition().duration(500)
        .attr("y", d => y(d.value))
        .attr("height", d => height - y(d.value))),

    exit => exit                   // LEAVING: animate out + remove
      .attr("fill", "#ef4444")
      .call(el => el.transition().duration(300)
        .attr("y", height).attr("height", 0).remove())
  );

Key Functions: Name Tags for Data

By default, D3 matches data to elements by index (position). The first data item goes with the first element, etc. This works for static charts but breaks for animations.

// Without key: D3 matches by index (position)
.data(newData)              // data[0] → element[0], data[1] → element[1]

// With key: D3 matches by identity
.data(newData, d => d.id)   // finds the element whose data has the same id

Keys are like React's key prop. Without keys, D3 (like React) just matches by position. With keys, it can track individual items through data changes. This is what makes smooth animations possible when items are added, removed, or reordered.

Check Your Understanding

1. What are the three groups D3 creates when you call .data()?

Enter: data items without matching elements (need creation). Update: data items with matching elements (may need attribute changes). Exit: elements without matching data items (need removal). Together, these three groups account for every possible difference between old and new data.

2. What does a key function do, and why does it matter for animations?

A key function (.data(data, d => d.id)) tells D3 how to match data items to elements by identity rather than by index. Without keys, if you remove the first item, D3 thinks every element changed (indices shifted). With keys, D3 knows element #1 is the same item it was before, just in a new position. This enables smooth, meaningful animations.

3. How is .join("rect") different from .join(enter, update, exit)?

The string form .join("rect") is shorthand that uses defaults: enter appends the element, update does nothing special, exit removes immediately. The three-callback form gives you full control: you can animate entering elements, transition updates, and fade out exiting elements. Use the string form for simple charts, the callback form when you need animation control.

4. Why is the "general update pattern" (a reusable update function) important?

It lets you write one function that handles any data change: filtering, sorting, adding, removing. Call it whenever data changes, and the enter/update/exit pattern handles the DOM updates automatically. This is the standard pattern for interactive D3 charts — buttons, sliders, and filters all just call the update function with new data.

5. How do D3 key functions compare to React's key prop?

They serve the same purpose: helping the framework track which items in a list are the same across re-renders. React's key helps the virtual DOM diff. D3's key function helps the data join. Without keys, both default to matching by index, which causes incorrect animations and wasted re-renders when items are reordered or removed.

6. In the three-callback .join(), what happens to exiting elements if you don't call .remove()?

They stay in the DOM but are no longer bound to any data. This is a memory leak and visual bug. Always call .remove() on exit elements, typically at the end of a transition: exit.transition().duration(300).attr("opacity", 0).remove().

Recap

  1. Enter = new data needs new elements
  2. Update = existing data may need attribute changes
  3. Exit = old elements need removal
  4. Key functions (d => d.id) match by identity, not position
  5. .join(enter, update, exit) gives full control over animations
  6. General update pattern = one function handles all data changes
