05 REST APIs — Next.js API Routes to DRF

← Back to lessons

You already know API routes. You've built route.ts files in Next.js that handle GET, POST, PUT, DELETE. Django REST Framework does the same thing — but with more structure, automatic validation, browsable docs, and auth built in. Think of it as going from hand-rolling API routes to having a framework that does the heavy lifting.

The Mental Model — What DRF Actually Is

Django by itself serves HTML pages. When a request comes in, a view function renders a template and sends back HTML. That's great for server-rendered apps, but modern frontends (React, Next.js, mobile apps) want JSON.

Django REST Framework (DRF) is a third-party package that adds a JSON API layer on top of Django. It gives you serializers (for shaping data), views (for handling requests), authentication, permissions, pagination — everything you need to build a production API.

The analogy: If Django is like Next.js pages (server-rendered HTML), then DRF is like Next.js API routes (route.ts) — but with a whole framework around them. Imagine if every route.ts file came with automatic input validation, typed responses, auth middleware, and a browsable documentation page. That's DRF.

ConceptNext.jsDjango + DRF

Installation & Setup

DRF is a pip package. One install, a couple lines of config, and you're ready.

# Install DRF
pip install djangorestframework

# Optional but common extras
pip install django-filter          # filtering support
pip install djangorestframework-simplejwt  # JWT auth

Then add it to your Django settings:

# settings.py

INSTALLED_APPS = [
    "django.contrib.admin",
    "django.contrib.auth",
    "django.contrib.contenttypes",
    "django.contrib.sessions",
    "django.contrib.messages",
    "django.contrib.staticfiles",
    "rest_framework",              # Add this
    "django_filters",              # If using filtering
    "newsroom",                    # Your app
]

# Global DRF settings (like a default next.config.js for your API)
REST_FRAMEWORK = {
    "DEFAULT_PAGINATION_CLASS": "rest_framework.pagination.PageNumberPagination",
    "PAGE_SIZE": 20,
    "DEFAULT_AUTHENTICATION_CLASSES": [
        "rest_framework.authentication.SessionAuthentication",
        "rest_framework.authentication.TokenAuthentication",
    ],
    "DEFAULT_PERMISSION_CLASSES": [
        "rest_framework.permissions.IsAuthenticatedOrReadOnly",
    ],
}

The REST_FRAMEWORK dict is your global API config. It sets defaults for every endpoint — pagination, auth, permissions, throttling. Individual views can override any of these. Think of it like setting defaults in next.config.js that every route inherits.

Our Example: A Newsroom API

We'll build a real-world API throughout this lesson: a newsroom system with articles, authors, and a protected publish endpoint. Here's the model we're working with (you already know how to build these from Lesson 07):

# newsroom/models.py
from django.db import models
from django.contrib.auth.models import User

class Article(models.Model):
    STATUS_CHOICES = [
        ("draft", "Draft"),
        ("published", "Published"),
        ("archived", "Archived"),
    ]

    title = models.CharField(max_length=200)
    slug = models.SlugField(unique=True)
    body = models.TextField()
    author = models.ForeignKey(User, on_delete=models.CASCADE, related_name="articles")
    status = models.CharField(max_length=10, choices=STATUS_CHOICES, default="draft")
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        ordering = ["-created_at"]

    def __str__(self):
        return self.title

Serializers — Your API's Zod Schemas

In Next.js, when data comes into an API route, you have to manually validate it — maybe with Zod, maybe by hand. And when data goes out, you manually shape the JSON response. Serializers handle both directions automatically.

The mental model: A serializer is a Zod schema + a data transformer in one object. It validates incoming data (like z.object({ title: z.string() }).parse(body)) and it formats outgoing data (like picking which fields to include in your JSON response). One class, two jobs.

Next.js + Zod
// Validation schema
const ArticleSchema = z.object({
  title: z.string().min(1).max(200),
  slug: z.string(),
  body: z.string(),
  status: z.enum(["draft", "published"]),
});

// In your route.ts
export async function POST(req: Request) {
  const body = await req.json();
  const data = ArticleSchema.parse(body);
  // manually save to DB...
  // manually shape response...
  return NextResponse.json(data);
}
DRF Serializer
# newsroom/serializers.py
from rest_framework import serializers
from .models import Article

class ArticleSerializer(
    serializers.ModelSerializer
):
    class Meta:
        model = Article
        fields = [
            "id", "title", "slug", "body",
            "author", "status",
            "created_at", "updated_at",
        ]
        read_only_fields = [
            "id", "created_at", "updated_at"
        ]
# Validation + DB save + JSON output
# all handled automatically

How ModelSerializer Works

ModelSerializer reads your Django model and automatically generates fields that match. You just tell it which model to use and which fields to expose. It's like if Zod could read your Prisma schema and auto-generate the validation rules.

# The serializer does three things:

# 1. Serialize (Python object → JSON)
article = Article.objects.get(pk=1)
serializer = ArticleSerializer(article)
serializer.data
# {"id": 1, "title": "Breaking News", "slug": "breaking-news", ...}

# 2. Deserialize + Validate (JSON → Python, with validation)
serializer = ArticleSerializer(data=request.data)
serializer.is_valid(raise_exception=True)
# If invalid: raises 400 with {"title": ["This field is required."]}

# 3. Save (validated data → database)
serializer.save(author=request.user)
# Creates a new Article in the DB

Custom Field Validation

Just like Zod's .refine(), you can add custom validation rules to individual fields or to the whole object.

# newsroom/serializers.py
class ArticleSerializer(serializers.ModelSerializer):
    class Meta:
        model = Article
        fields = ["id", "title", "slug", "body", "author", "status",
                  "created_at", "updated_at"]
        read_only_fields = ["id", "created_at", "updated_at"]

    # Validate a single field: validate_<field_name>
    def validate_title(self, value):
        if value.isupper():
            raise serializers.ValidationError(
                "No ALL CAPS titles, this isn't a tabloid."
            )
        return value

    # Validate multiple fields together (like Zod .refine)
    def validate(self, data):
        if data.get("status") == "published" and not data.get("body"):
            raise serializers.ValidationError(
                "Published articles need a body. No empty stories."
            )
        return data

Validation naming convention: validate_title validates just the title field. validate (no suffix) validates the whole object — use it for cross-field rules. DRF discovers these methods automatically by name. No registration needed.

Nested Serializers

When your API returns an article, you probably want the author's name — not just their ID number. Nested serializers let you embed related objects in your response.

# newsroom/serializers.py
from django.contrib.auth.models import User

class AuthorSerializer(serializers.ModelSerializer):
    class Meta:
        model = User
        fields = ["id", "username", "first_name", "last_name"]

class ArticleSerializer(serializers.ModelSerializer):
    # Replace author ID with full author object
    author = AuthorSerializer(read_only=True)

    class Meta:
        model = Article
        fields = ["id", "title", "slug", "body", "author", "status",
                  "created_at", "updated_at"]

# Now the response looks like:
# {
#   "id": 1,
#   "title": "Breaking News",
#   "author": {
#     "id": 5,
#     "username": "trevor",
#     "first_name": "Trevor",
#     "last_name": "Brown"
#   },
#   ...
# }

Read-only vs writable nested serializers: When read_only=True, the nested author appears in GET responses but you still pass just the author ID on POST/PUT. For writable nested serializers (creating related objects together), you need to override .create() — but that's an advanced pattern. Start with read_only=True.

APIView — The Building Block

APIView is DRF's equivalent of a Next.js route.ts file. You write methods for each HTTP verb. It's explicit, easy to understand, and gives you full control.

Next.js API Route
// app/api/articles/route.ts

export async function GET() {
  const articles = await db.article
    .findMany();
  return NextResponse.json(articles);
}

export async function POST(
  req: Request
) {
  const body = await req.json();
  // validate manually...
  const article = await db.article
    .create({ data: body });
  return NextResponse.json(
    article, { status: 201 }
  );
}
DRF APIView
# newsroom/views.py
from rest_framework.views import APIView
from rest_framework.response import Response
from rest_framework import status

class ArticleList(APIView):
    def get(self, request):
        articles = Article.objects.all()
        serializer = ArticleSerializer(
            articles, many=True
        )
        return Response(serializer.data)

    def post(self, request):
        serializer = ArticleSerializer(
            data=request.data
        )
        serializer.is_valid(
            raise_exception=True
        )
        serializer.save(
            author=request.user
        )
        return Response(
            serializer.data,
            status=status.HTTP_201_CREATED
        )

And you wire it up in urls.py just like any Django view:

# newsroom/urls.py
from django.urls import path
from . import views

urlpatterns = [
    path("api/articles/", views.ArticleList.as_view()),
]

Detail View — Single Resource

For a single article endpoint (GET one, PUT to update, DELETE to remove), you write another APIView:

Next.js Dynamic Route
// app/api/articles/[id]/route.ts

export async function GET(
  req: Request,
  { params }: { params: { id: string } }
) {
  const article = await db.article
    .findUnique({
      where: { id: params.id }
    });
  if (!article)
    return NextResponse.json(
      { error: "Not found" },
      { status: 404 }
    );
  return NextResponse.json(article);
}

export async function PUT(req, { params }) {
  const body = await req.json();
  const article = await db.article
    .update({
      where: { id: params.id },
      data: body,
    });
  return NextResponse.json(article);
}

export async function DELETE(
  req, { params }
) {
  await db.article.delete({
    where: { id: params.id },
  });
  return new Response(null,
    { status: 204 });
}
DRF APIView
# newsroom/views.py
from django.shortcuts import get_object_or_404

class ArticleDetail(APIView):
    def get(self, request, pk):
        article = get_object_or_404(
            Article, pk=pk
        )
        serializer = ArticleSerializer(
            article
        )
        return Response(serializer.data)

    def put(self, request, pk):
        article = get_object_or_404(
            Article, pk=pk
        )
        serializer = ArticleSerializer(
            article, data=request.data
        )
        serializer.is_valid(
            raise_exception=True
        )
        serializer.save()
        return Response(serializer.data)

    def delete(self, request, pk):
        article = get_object_or_404(
            Article, pk=pk
        )
        article.delete()
        return Response(
            status=status.HTTP_204_NO_CONTENT
        )

# urls.py
# path("api/articles/<int:pk>/",
#      views.ArticleDetail.as_view())

Notice the pattern: Every APIView method follows the same flow: get the object, run it through the serializer, return the response. After writing two or three of these, the repetition becomes obvious. That's exactly why ViewSets exist — they eliminate this boilerplate.

ViewSets & Routers — CRUD on Autopilot

Here's where DRF pulls ahead of anything in the Next.js world. A ViewSet combines the list view and the detail view into one class. A Router automatically generates all the URL patterns. Together, they give you a complete CRUD API in about 10 lines of code.

The analogy: Imagine if in Next.js you could write one file and it automatically generated GET /api/articles, POST /api/articles, GET /api/articles/:id, PUT /api/articles/:id, PATCH /api/articles/:id, and DELETE /api/articles/:id — all with validation, error handling, and proper status codes. That's a ViewSet + Router.

Next.js (2 files, ~60 lines)
// app/api/articles/route.ts
export async function GET() { ... }
export async function POST() { ... }

// app/api/articles/[id]/route.ts
export async function GET() { ... }
export async function PUT() { ... }
export async function PATCH() { ... }
export async function DELETE() { ... }

// Plus: validation, error handling,
// auth checks in each function...
DRF (1 file, ~10 lines)
# newsroom/views.py
from rest_framework import viewsets
from .models import Article
from .serializers import ArticleSerializer

class ArticleViewSet(viewsets.ModelViewSet):
    queryset = Article.objects.all()
    serializer_class = ArticleSerializer

    def perform_create(self, serializer):
        serializer.save(
            author=self.request.user
        )

# That's it. Full CRUD. Validated.
# With proper status codes.

Wire it up with a Router to auto-generate URLs:

# newsroom/urls.py
from django.urls import path, include
from rest_framework.routers import DefaultRouter
from . import views

router = DefaultRouter()
router.register("articles", views.ArticleViewSet)
# router.register("authors", views.AuthorViewSet)  # add more resources

urlpatterns = [
    path("api/", include(router.urls)),
]

# This auto-generates:
# GET    /api/articles/         → list all articles
# POST   /api/articles/         → create article
# GET    /api/articles/{id}/    → retrieve one article
# PUT    /api/articles/{id}/    → full update
# PATCH  /api/articles/{id}/    → partial update
# DELETE /api/articles/{id}/    → delete article
# GET    /api/                  → browsable API root

ModelViewSet gives you all 6 CRUD operations. If you only want some, use mixins: mixins.ListModelMixin, mixins.CreateModelMixin, etc. Or use ReadOnlyModelViewSet for just list + retrieve (no create/update/delete). Mix and match like React components.

ViewSet Action Methods

Under the hood, ViewSets map HTTP verbs to method names. Understanding this map is key:

HTTP MethodURLViewSet MethodWhat It Does
GET/articles/list()Return all articles
POST/articles/create()Create new article
GET/articles/5/retrieve()Return article #5
PUT/articles/5/put() via update()Full update
PATCH/articles/5/partial_update()Partial update
DELETE/articles/5/destroy()Delete article

You can override any of these methods to customize behavior. And you can add custom actions beyond CRUD:

# newsroom/views.py
from rest_framework.decorators import action

class ArticleViewSet(viewsets.ModelViewSet):
    queryset = Article.objects.all()
    serializer_class = ArticleSerializer

    def perform_create(self, serializer):
        serializer.save(author=self.request.user)

    # Custom action: POST /api/articles/{id}/publish/
    @action(detail=True, methods=["post"])
    def publish(self, request, pk=None):
        article = self.get_object()
        article.status = "published"
        article.save()
        serializer = self.get_serializer(article)
        return Response(serializer.data)

    # Custom action: GET /api/articles/drafts/
    @action(detail=False, methods=["get"])
    def drafts(self, request):
        drafts = self.queryset.filter(status="draft", author=request.user)
        serializer = self.get_serializer(drafts, many=True)
        return Response(serializer.data)

detail=True vs detail=False: detail=True means the action operates on a single object — the URL includes an ID (/articles/5/publish/). detail=False means it operates on the collection — no ID (/articles/drafts/). Think of it as "does this need a specific record?"

from rest_framework.response import Response

@api_view(["GET"])

    return Response({"status": "ok", "version": "1.0.0"})

Authentication — Who Is Making This Request?

In Next.js, you probably use NextAuth or roll your own JWT solution. DRF gives you pluggable authentication backends. You pick one (or more), add it to settings, and every request automatically gets request.user.

The mental model: Authentication answers "who are you?" It's the DRF equivalent of your NextAuth session — after auth runs, request.user is either a real user or an anonymous user. DRF checks auth before your view code runs, just like Next.js middleware.

Token Authentication

The simplest option. Each user gets a token string. They send it in the Authorization header. This is the pattern most similar to how you'd manually implement auth with bearer tokens in Next.js.

# settings.py
INSTALLED_APPS = [
    ...
    "rest_framework.authtoken",    # Add this
]

REST_FRAMEWORK = {
    "DEFAULT_AUTHENTICATION_CLASSES": [
        "rest_framework.authentication.TokenAuthentication",
    ],
}

# Then run: python manage.py migrate
# (creates the authtoken table)
# Generate tokens for users
from rest_framework.authtoken.models import Token

token = Token.objects.create(user=some_user)
print(token.key)   # "9944b09199c62bcf9418ad846dd0e4bbdfc6ee4b"
Next.js (manual bearer token)
// Client sends:
fetch("/api/articles", {
  headers: {
    Authorization: `Bearer ${token}`,
  },
});

// Server checks:
export async function GET(req) {
  const token = req.headers
    .get("authorization")
    ?.split(" ")[1];
  const user = await verifyToken(token);
  if (!user) return new Response(
    "Unauthorized", { status: 401 }
  );
  // proceed...
}

JWT Authentication

For SPAs and mobile apps, JWT is more common. Use djangorestframework-simplejwt:

# settings.py
REST_FRAMEWORK = {
    "DEFAULT_AUTHENTICATION_CLASSES": [
        "rest_framework_simplejwt.authentication.JWTAuthentication",
    ],
}

# urls.py — adds /api/token/ and /api/token/refresh/
from rest_framework_simplejwt.views import (
    TokenObtainPairView,
    TokenRefreshView,
)

urlpatterns = [
    path("api/token/", TokenObtainPairView.as_view()),
    path("api/token/refresh/", TokenRefreshView.as_view()),
    path("api/", include(router.urls)),
]
# Client flow — same as any JWT setup you've used in Next.js

# 1. Login → POST /api/token/ with username + password
# Response: {"access": "eyJ...", "refresh": "eyJ..."}

# 2. Use access token for requests
# Authorization: Bearer eyJ...

# 3. When access token expires, refresh it
# POST /api/token/refresh/ with {"refresh": "eyJ..."}
# Response: {"access": "eyJ..."} (new access token)

Session vs Token vs JWT: SessionAuthentication uses cookies (good for same-site Django templates + API). TokenAuthentication uses a simple database-backed token (good for scripts and simple clients). JWTAuthentication uses stateless tokens (good for SPAs and mobile apps). You can enable multiple auth classes and DRF will try each one.

Permissions — What Are You Allowed to Do?

Authentication tells you who someone is. Permissions tell you what they can do. In Next.js, you'd write if (!session.user.isAdmin) return new Response("Forbidden", { status: 403 }) in every route. DRF gives you reusable permission classes.

Built-in Permission Classes

Permission ClassWhat It DoesNext.js Equivalent
AllowAnyNo restrictions at allNo auth check
IsAuthenticatedMust be logged inif (!session) return 401
IsAdminUserMust be staff/adminif (!session.user.isAdmin) return 403
IsAuthenticatedOrReadOnlyAnyone can GET; must be logged in to POST/PUT/DELETEManual check per method
# Apply permissions to a view
from rest_framework.permissions import IsAuthenticated, IsAdminUser

class ArticleViewSet(viewsets.ModelViewSet):
    queryset = Article.objects.all()
    serializer_class = ArticleSerializer
    permission_classes = [IsAuthenticated]  # Must be logged in for everything

Per-Action Permissions

Often you want different permissions for different actions. Anyone can read articles, but only authenticated users can create them, and only admins can delete them:

# newsroom/views.py
class ArticleViewSet(viewsets.ModelViewSet):
    queryset = Article.objects.all()
    serializer_class = ArticleSerializer

    def get_permissions(self):
        if self.action in ["create", "update", "partial_update"]:
            return [IsAuthenticated()]
        elif self.action == "destroy":
            return [IsAdminUser()]
        return [AllowAny()]  # list and retrieve are public

Custom Permissions

For real-world apps, you'll need custom rules — like "only the author can edit their own article." DRF makes this clean:

# newsroom/permissions.py
from rest_framework.permissions import BasePermission

class IsAuthorOrReadOnly(BasePermission):
    """
    Allow read access to anyone.
    Write access only to the article's author.
    """
    def has_object_permission(self, request, view, obj):
        # GET, HEAD, OPTIONS are safe methods → allow anyone
        if request.method in ["GET", "HEAD", "OPTIONS"]:
            return True
        # Write methods → only the author
        return obj.author == request.user
# Use it in the viewset
from .permissions import IsAuthorOrReadOnly

class ArticleViewSet(viewsets.ModelViewSet):
    queryset = Article.objects.all()
    serializer_class = ArticleSerializer
    permission_classes = [IsAuthenticated, IsAuthorOrReadOnly]
    # Both must pass: must be logged in AND must be the author (for writes)

has_permission vs has_object_permission: has_permission runs on every request (before the object is loaded) — use it for "is this user logged in?" checks. has_object_permission runs after loading the specific object — use it for "does this user own this object?" checks. Two levels of gatekeeping.

Pagination, Filtering & Search

Real APIs don't return every record at once. DRF handles pagination, filtering, and search with minimal config — features that would take serious work to build from scratch in Next.js API routes.

Pagination

If you set it in REST_FRAMEWORK settings (we already did), it applies globally. Your list endpoints automatically return paginated responses:

# With PAGE_SIZE = 20 in settings, a GET /api/articles/ returns:
{
    "count": 84,
    "next": "http://localhost:8000/api/articles/?page=2",
    "previous": null,
    "results": [
        {"id": 84, "title": "Latest Story", ...},
        {"id": 83, "title": "Another Story", ...},
        // ... 18 more articles
    ]
}

# Client paginates with: GET /api/articles/?page=2
Next.js (manual)
// app/api/articles/route.ts
export async function GET(req: Request) {
  const { searchParams } = new URL(
    req.url
  );
  const page = Number(
    searchParams.get("page") || 1
  );
  const pageSize = 20;
  const offset = (page - 1) * pageSize;

  const [articles, total] =
    await Promise.all([
      db.article.findMany({
        skip: offset,
        take: pageSize,
      }),
      db.article.count(),
    ]);

  return NextResponse.json({
    results: articles,
    count: total,
    next: offset + pageSize < total
      ? `?page=${page + 1}` : null,
  });
}
DRF (automatic)
# settings.py — that's it
REST_FRAMEWORK = {

Filtering & Search

With django-filter, you can add query parameter filtering to any endpoint:

# newsroom/views.py
from rest_framework import viewsets, filters
from django_filters.rest_framework import DjangoFilterBackend

class ArticleViewSet(viewsets.ModelViewSet):
    queryset = Article.objects.all()
    serializer_class = ArticleSerializer

    # Enable filtering, searching, and ordering
    filter_backends = [
        DjangoFilterBackend,       # exact-match filters
        filters.SearchFilter,       # text search
        filters.OrderingFilter,     # sort control
    ]

    # Filter by exact field values
    filterset_fields = ["status", "author"]
    # GET /api/articles/?status=published
    # GET /api/articles/?author=5

    # Search across text fields
    search_fields = ["title", "body"]
    # GET /api/articles/?search=breaking+news

    # Allow client to control sort order
    ordering_fields = ["created_at", "title"]
    # GET /api/articles/?ordering=-created_at

Compare to Next.js: In a Next.js API route, you'd parse searchParams, build a Prisma where clause, handle search with contains, and build orderBy — all manually. DRF does it with three config lines. The filter backends read query parameters and modify the queryset automatically.

Testing Your API

The Browsable API

DRF's secret weapon: visit any API endpoint in your browser and you get an interactive HTML page where you can test GET, POST, PUT, DELETE — with forms, syntax highlighting, and authentication. No Postman needed.

# Just run the dev server and visit the URL in your browser:
python manage.py runserver

# Open http://localhost:8000/api/articles/ in Chrome
# You'll see a beautiful HTML page with:
#   - The JSON response, syntax-highlighted
#   - A form to POST new articles
#   - Login/logout buttons
#   - Links to related endpoints
#   - Pagination controls

The browsable API is like built-in Swagger/Postman. It's enabled by default. When a browser requests your endpoint (sending Accept: text/html), DRF renders HTML. When an API client requests it (sending Accept: application/json), DRF returns JSON. Same endpoint, different formats.

Testing with curl

For quick terminal testing, curl is your friend:

# List articles

Automated Tests

DRF includes an APIClient that makes writing tests feel like writing fetch requests:

# newsroom/tests.py
from django.test import TestCase
from django.contrib.auth.models import User
from rest_framework.test import APIClient
from rest_framework import status
from .models import Article

class ArticleAPITest(TestCase):
    def setUp(self):
        self.client = APIClient()
        self.user = User.objects.create_user(
            username="trevor", password="testpass123"
        )
        self.client.force_authenticate(user=self.user)

    def test_create_article(self):
        response = self.client.post("/api/articles/", {
            "title": "Test Article",
            "slug": "test-article",
            "body": "This is a test.",
            "status": "draft",
        })
        self.assertEqual(response.status_code, status.HTTP_201_CREATED)
        self.assertEqual(response.data["title"], "Test Article")
        self.assertEqual(response.data["author"]["username"], "trevor")

    def test_list_articles(self):
        Article.objects.create(
            title="Story 1", slug="story-1", body="...", author=self.user
        )
        response = self.client.get("/api/articles/")
        self.assertEqual(response.status_code, status.HTTP_200_OK)
        self.assertEqual(len(response.data["results"]), 1)

    def test_unauthenticated_cannot_create(self):
        self.client.force_authenticate(user=None)  # log out
        response = self.client.post("/api/articles/", {
            "title": "Sneaky", "slug": "sneaky", "body": "...",
        })
        self.assertEqual(response.status_code, status.HTTP_401_UNAUTHORIZED)

force_authenticate is the testing equivalent of mocking a session. No need to actually log in, get a token, and pass headers. Just tell the test client "pretend this user is logged in." It's similar to how you might mock NextAuth in a test environment.

Putting It All Together — The Complete Newsroom API

Here's the full API, from models to URLs. This is everything you'd need for a newsroom backend that a Next.js frontend could consume.

File: newsroom/serializers.py

from rest_framework import serializers
from django.contrib.auth.models import User
from .models import Article


class AuthorSerializer(serializers.ModelSerializer):
    class Meta:
        model = User
        fields = ["id", "username", "first_name", "last_name"]


class ArticleSerializer(serializers.ModelSerializer):
    author = AuthorSerializer(read_only=True)

    class Meta:
        model = Article
        fields = [
            "id", "title", "slug", "body",
            "author", "status", "created_at", "updated_at",
        ]
        read_only_fields = ["id", "created_at", "updated_at"]

    def validate_title(self, value):
        if len(value) < 5:
            raise serializers.ValidationError("Title must be at least 5 characters.")
        return value

File: newsroom/permissions.py

from rest_framework.permissions import BasePermission


class IsAuthorOrReadOnly(BasePermission):
    def has_object_permission(self, request, view, obj):
        if request.method in ["GET", "HEAD", "OPTIONS"]:
            return True
        return obj.author == request.user

File: newsroom/views.py

from rest_framework import viewsets, filters, status
from rest_framework.decorators import action
from rest_framework.permissions import IsAuthenticated, AllowAny
from rest_framework.response import Response
from django_filters.rest_framework import DjangoFilterBackend
from .models import Article
from .serializers import ArticleSerializer
from .permissions import IsAuthorOrReadOnly


class ArticleViewSet(viewsets.ModelViewSet):
    queryset = Article.objects.all()
    serializer_class = ArticleSerializer
    permission_classes = [IsAuthorOrReadOnly]

    # Filtering, search, ordering
    filter_backends = [DjangoFilterBackend, filters.SearchFilter, filters.OrderingFilter]
    filterset_fields = ["status", "author"]
    search_fields = ["title", "body"]
    ordering_fields = ["created_at", "title"]

    def get_permissions(self):
        if self.action in ["create", "publish", "drafts"]:
            return [IsAuthenticated()]
        elif self.action in ["update", "partial_update", "destroy"]:
            return [IsAuthenticated(), IsAuthorOrReadOnly()]
        return [AllowAny()]

    def perform_create(self, serializer):
        serializer.save(author=self.request.user)

    @action(detail=True, methods=["post"])
    def publish(self, request, pk=None):
        """POST /api/articles/{id}/publish/ — publish a draft"""
        article = self.get_object()
        if article.author != request.user:
            return Response(
                {"detail": "Only the author can publish."},
                status=status.HTTP_403_FORBIDDEN,
            )
        if not article.body:
            return Response(
                {"detail": "Cannot publish an empty article."},
                status=status.HTTP_400_BAD_REQUEST,
            )
        article.status = "published"
        article.save()
        return Response(ArticleSerializer(article).data)

    @action(detail=False, methods=["get"])
    def drafts(self, request):
        """GET /api/articles/drafts/ — list current user's drafts"""
        drafts = self.queryset.filter(status="draft", author=request.user)
        page = self.paginate_queryset(drafts)
        if page is not None:
            serializer = self.get_serializer(page, many=True)
            return self.get_paginated_response(serializer.data)
        serializer = self.get_serializer(drafts, many=True)
        return Response(serializer.data)

File: newsroom/urls.py

from django.urls import path, include
from rest_framework.routers import DefaultRouter
from . import views

router = DefaultRouter()
router.register("articles", views.ArticleViewSet)

urlpatterns = [
    path("api/", include(router.urls)),
]

# Generated endpoints:
# GET    /api/articles/              — list (public, paginated, filterable)
# POST   /api/articles/              — create (auth required)
# GET    /api/articles/{id}/         — retrieve (public)
# PUT    /api/articles/{id}/         — update (author only)
# PATCH  /api/articles/{id}/         — partial update (author only)
# DELETE /api/articles/{id}/         — delete (author only)
# POST   /api/articles/{id}/publish/ — publish (author only)
# GET    /api/articles/drafts/       — user's drafts (auth required)

DRF trailing slashes: By default, DRF URLs end with /. So it's /api/articles/, not /api/articles. If a request comes without the trailing slash, Django redirects (302) to the version with it. This can cause confusion when testing with curl. Either always include the trailing slash, or set router.trailing_slash = False.

Consuming Your DRF API from Next.js

Once your DRF backend is running, your Next.js frontend talks to it exactly like any other REST API. Here's how the pieces connect:

// In your Next.js app — fetching from your Django API

// List articles
const res = await fetch("http://localhost:8000/api/articles/?status=published");
const data = await res.json();
// data.results = [{id: 1, title: "...", author: {username: "trevor"}, ...}]
// data.count = 84
// data.next = "http://localhost:8000/api/articles/?page=2"

// Create an article (with JWT)
const res = await fetch("http://localhost:8000/api/articles/", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "Authorization": `Bearer ${accessToken}`,
  },
  body: JSON.stringify({
    title: "New Story",
    slug: "new-story",
    body: "The full text...",
    status: "draft",
  }),
});

// Publish an article
await fetch(`http://localhost:8000/api/articles/${id}/publish/`, {
  method: "POST",
  headers: { "Authorization": `Bearer ${accessToken}` },
});

// Search
const res = await fetch(
  "http://localhost:8000/api/articles/?search=election&ordering=-created_at"
);

CORS heads-up: When your Next.js frontend (port 3000) calls your Django backend (port 8000), you'll hit CORS errors. Install django-cors-headers, add it to INSTALLED_APPS and MIDDLEWARE, and set CORS_ALLOWED_ORIGINS = ["http://localhost:3000"]. This is the same CORS dance you'd do with any separate frontend/backend setup.

Check Your Understanding

Quick Quiz

1. What is the DRF equivalent of a Zod schema in a Next.js API route?

A Serializer (specifically ModelSerializer). It validates incoming data, shapes outgoing data, and can read field definitions from your Django model automatically. Unlike Zod, it also handles saving to the database via serializer.save().

2. You need a full CRUD API for a Comment model in about 10 lines. What two DRF classes do you use?

ModelViewSet (handles all CRUD logic) and DefaultRouter (auto-generates the URL patterns). Register the viewset with the router: router.register("comments", CommentViewSet). You get list, create, retrieve, update, partial_update, and destroy automatically.

3. What's the difference between has_permission and has_object_permission on a custom permission class?

has_permission runs on every request before the view logic — use it for broad checks like "is the user authenticated?" has_object_permission runs after loading a specific object — use it for "does this user own this specific article?" The first is a gate at the entrance; the second is a check at each item.

4. How do you add a custom POST /api/articles/{id}/publish/ endpoint to an existing ArticleViewSet?

Use the @action decorator: @action(detail=True, methods=["post"]) on a method named publish. detail=True means it operates on a single object (so the URL includes the ID). The router automatically generates the URL pattern.

5. Your API returns all 10,000 articles in one response. How do you add pagination with one config change?

Add "DEFAULT_PAGINATION_CLASS": "rest_framework.pagination.PageNumberPagination" and "PAGE_SIZE": 20 to the REST_FRAMEWORK dict in settings.py. All list endpoints will automatically paginate, returning count, next, previous, and results in the response.

6. A request to your API returns 401 Unauthorized. What does that tell you vs a 403 Forbidden response?

401 means authentication failed — the user didn't provide credentials or the token is invalid. The DRF authentication layer rejected them. 403 means authentication succeeded (DRF knows who they are) but the permission layer denied them — they're logged in but not allowed to perform that action.
