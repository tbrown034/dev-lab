04 Python Object-Oriented Programming

← Back to lessons

You already know classes. JavaScript ES6 classes and Python classes are siblings — same inheritance model, same constructor pattern, same method syntax. The differences are small but important: explicit self, dunder methods, and Python's cleaner approach to properties.

Classes — The Basics

JavaScript
class Article {
  constructor(title, author) {
    this.title = title;
    this.author = author;
    this.published = false;
    this.views = 0;
  }

  publish() {
    this.published = true;
    return `Published: ${this.title}`;
  }

  addView() {
    this.views++;
  }
}

const a = new Article("Scoop", "Trevor");
a.publish();
console.log(a.title);  // "Scoop"
Python
class Article:
    def __init__(self, title, author):
        self.title = title
        self.author = author
        self.published = False
        self.views = 0


    def publish(self):
        self.published = True
        return f"Published: {self.title}"

    def add_view(self):
        self.views += 1


a = Article("Scoop", "Trevor")  # No 'new'!
a.publish()
print(a.title)  # "Scoop"

Three differences to internalize: (1) constructor() → __init__(). (2) this → self, but self must be the first parameter of every method. (3) No new keyword — just call the class like a function. Everything else works the same.

Forgetting self is the #1 class bug. Every instance method must include self as its first parameter. Forget it and you'll get TypeError: publish() takes 0 positional arguments but 1 was given. Python passes the instance automatically, but you must declare it explicitly.

self vs this — Explicit vs Implicit

This is the biggest conceptual difference. In JavaScript, this is implicit and its value depends on how the function is called (which causes endless bugs). In Python, self is explicit and always refers to the instance.

JavaScript
class Timer {
  constructor() {
    this.seconds = 0;
  }

  start() {
    // this changes in callbacks!
    setInterval(function() {
      this.seconds++;  // BUG: wrong this
    }, 1000);

    // Fix 1: arrow function
    setInterval(() => {
      this.seconds++;  // works
    }, 1000);

    // Fix 2: bind
    setInterval(function() {
      this.seconds++;
    }.bind(this), 1000);
  }
}
Python
class Timer:
    def __init__(self):
        self.seconds = 0

    def tick(self):
        self.seconds += 1

    def start(self):
        # self always means the instance
        # No binding issues ever
        self.tick()
        self.tick()
        print(self.seconds)  # 2

# Python's explicit self means:
# - No "this" binding bugs
# - No need for arrow functions
# - No .bind() calls
# - You always know what self is

The mental model: In JavaScript, this is a hidden, magic variable that changes based on calling context. In Python, self is just a regular parameter — the first argument of every method. When you call a.publish(), Python translates that to Article.publish(a). No magic. No surprises. This is why Python doesn't need arrow functions.

Instance vs Class Variables

JavaScript
class Dog {
  // Class field (ES2022)
  static species = "Canis familiaris";

  constructor(name) {
    // Instance fields
    this.name = name;
    this.tricks = [];
  }
}

// Class variable
Dog.species;          // "Canis familiaris"

// Instance variable
const d = new Dog("Rex");
d.name;               // "Rex"
Python
class Dog:
    # Class variable (shared by ALL instances)
    species = "Canis familiaris"

    def __init__(self, name):
        # Instance variables (unique per instance)
        self.name = name
        self.tricks = []


# Class variable
Dog.species           # "Canis familiaris"

# Instance variable
d = Dog("Rex")
d.name                # "Rex"

Mutable class variable trap! If you put tricks = [] as a class variable (outside __init__), ALL instances share that same list. One dog learns a trick, every dog knows it. Always define mutable attributes (lists, dicts) inside __init__ with self.tricks = [].

Dunder Methods — Python's Operator Overloading

"Dunder" means double underscore. These special methods are called automatically when you use operators, built-in functions, or language features on your objects. They're Python's version of JavaScript's toString() and [Symbol.iterator] — but far more powerful and consistent.

__str__ and __repr__ — String Representation

JavaScript
class Article {
  constructor(title, author) {
    this.title = title;
    this.author = author;
  }

  // For string conversion
  toString() {
    return `${this.title} by ${this.author}`;
  }
}

const a = new Article("Scoop", "Trevor");
console.log(`${a}`);  // "Scoop by Trevor"
console.log(a);        // Article { title:... }
Python
class Article:
    def __init__(self, title, author):
        self.title = title
        self.author = author

    # For print() / str() / f-strings
    def __str__(self):
        return f"{self.title} by {self.author}"

    # For debugging / REPL
    def __repr__(self):
        return f"Article('{self.title}', '{self.author}')"

a = Article("Scoop", "Trevor")
print(a)       # "Scoop by Trevor" (__str__)
a              # Article('Scoop', 'Trevor') (__repr__)

The mental model: __str__ is for humans (what print() shows). __repr__ is for developers (what the REPL shows). A good __repr__ looks like valid Python code that recreates the object. Django admin uses __str__ to display model records — defining it is one of the first things you'll do on every model.

__len__, __eq__, __lt__ — Operators and Built-ins

JavaScript
class Playlist {
  constructor(songs) {
    this.songs = songs;
  }

  // Can't make len() work — use getter
  get length() {
    return this.songs.length;
  }

  // == doesn't call methods
  // Must write custom .equals()
  equals(other) {
    return this.songs.length === other.songs.length
      && this.songs.every(
        (s, i) => s === other.songs[i]
      );
  }
}
Python
class Playlist:
    def __init__(self, songs):
        self.songs = songs

    # len(playlist) just works!
    def __len__(self):
        return len(self.songs)

    # == just works!
    def __eq__(self, other):
        return self.songs == other.songs

    # < works — enables sorted()
    def __lt__(self, other):
        return len(self.songs) < len(other.songs)

p = Playlist(["Song A", "Song B"])
len(p)  # 2

__iter__ — Making Objects Iterable

JavaScript
class NumberRange {
  constructor(start, end) {
    this.start = start;
    this.end = end;
  }

  [Symbol.iterator]() {
    let current = this.start;
    const end = this.end;
    return {
      next() {
        if (current <= end) {
          return { value: current++, done: false };
        }
        return { done: true };
      }
    };
  }
}

for (const n of new NumberRange(1, 5)) {
  console.log(n);  // 1, 2, 3, 4, 5
}
Python
class NumberRange:
    def __init__(self, start, end):
        self.start = start
        self.end = end

    def __iter__(self):
        current = self.start
        while current <= self.end:
            yield current  # pause & resume
            current += 1




for n in NumberRange(1, 5):
    print(n)  # 1, 2, 3, 4, 5

yield is Python's generator keyword. It's like return but pauses the function and resumes on the next iteration. JavaScript has function* generators too, but Python uses them more pervasively. Any function with yield becomes a generator.

Dunder Methods Reference

Dunder Method Triggered By JS Equivalent
__init__ MyClass() constructor()
__str__ print(obj), str(obj) toString()
__repr__ REPL, repr(obj) console.log(obj) output
__len__ len(obj) .length getter
__eq__ a == b Custom .equals()
__lt__ a < b, sorted() sort() comparator
__contains__ x in obj .includes()
__getitem__ obj[key] Proxy get handler
__iter__ for x in obj [Symbol.iterator]()
__call__ obj() Making an object callable
__add__ a + b No equivalent

The mental model: Dunder methods are Python's protocol system. When you write len(obj), Python calls obj.__len__(). When you write obj[key], Python calls obj.__getitem__(key). Every operator and built-in function has a corresponding dunder. This is how Django's ORM makes queryset[0:5] and item in queryset feel like native Python.

@property — Getters and Setters

Python's @property decorator lets you define methods that behave like attributes. Same concept as JavaScript's get and set.

JavaScript
class Circle {
  constructor(radius) {
    this._radius = radius;
  }

  get area() {
    return Math.PI * this._radius ** 2;
  }

  get diameter() {
    return this._radius * 2;
  }

  set diameter(d) {
    this._radius = d / 2;
  }
}

const c = new Circle(5);
c.area;        // 78.53 (no parens)
c.diameter;    // 10
c.diameter = 20;
c._radius;     // 10
Python
import math

class Circle:
    def __init__(self, radius):
        self._radius = radius

    @property
    def area(self):
        return math.pi * self._radius ** 2

    @property
    def diameter(self):
        return self._radius * 2

    @diameter.setter
    def diameter(self, d):
        self._radius = d / 2

c = Circle(5)
c.area         # 78.53 (no parens)
c.diameter     # 10
c.diameter = 20
c._radius      # 10

Translation rule: JS get propName() → Python @property decorator. JS set propName(val) → Python @propName.setter. Both let you access computed values without parentheses. Django uses @property on models for computed fields like full_name or is_overdue.

Inheritance & super()

JavaScript
class Shape {
  constructor(color) {
    this.color = color;
  }

  describe() {
    return `A ${this.color} shape`;
  }
}

class Circle extends Shape {
  constructor(color, radius) {
    super(color);
    this.radius = radius;
  }

  describe() {
    const base = super.describe();
    return `${base} (r=${this.radius})`;
  }

  area() {
    return Math.PI * this.radius ** 2;
  }
}

const c = new Circle("red", 5);
c instanceof Shape;  // true
Python
class Shape:
    def __init__(self, color):
        self.color = color

    def describe(self):
        return f"A {self.color} shape"


class Circle(Shape):
    def __init__(self, color, radius):
        super().__init__(color)
        self.radius = radius

    def describe(self):
        base = super().describe()
        return f"{base} (r={self.radius})"

    def area(self):
        return math.pi * self.radius ** 2


c = Circle("red", 5)
isinstance(c, Shape)  # True

Key differences: extends → parentheses class Child(Parent):. instanceof → isinstance(obj, Class). Python supports multiple inheritance (class Child(Parent1, Parent2):) which JavaScript does not — Django uses this extensively for mixins.

Multiple Inheritance — Django Mixins

from datetime import datetime

# Python supports multiple inheritance
class Timestamped:
    def __init__(self):
        self.created_at = datetime.now()

class Slugified:
    def slugify(self):
        return self.title.lower().replace(" ", "-")

# Combine multiple parents
class Article(Timestamped, Slugified):
    def __init__(self, title):
        super().__init__()
        self.title = title

a = Article("Breaking News")
a.created_at       # from Timestamped
a.slugify()        # from Slugified — "breaking-news"

The mental model: In JavaScript, you'd use composition or higher-order functions to mix in behavior. Python gives you multiple inheritance directly. Django's class-based views use this heavily: class MyView(LoginRequiredMixin, ListView): combines authentication with list rendering. Each parent class adds a capability.

@staticmethod and @classmethod

JavaScript
class User {
  constructor(name, email) {
    this.name = name;
    this.email = email;
  }

  // Static — factory method
  static fromJSON(json) {
    const data = JSON.parse(json);
    return new User(data.name, data.email);
  }

  // Static — utility
  static validateEmail(email) {
    return email.includes("@");
  }
}

User.validateEmail("t@b.com");
const u = User.fromJSON('...');
Python
import json

class User:
    def __init__(self, name, email):
        self.name = name
        self.email = email

    # classmethod — gets CLASS as first arg
    @classmethod
    def from_json(cls, json_str):
        data = json.loads(json_str)
        return cls(data["name"], data["email"])

    # staticmethod — gets nothing extra
    @staticmethod
    def validate_email(email):
        return "@" in email

User.validate_email("t@b.com")
u = User.from_json('...')

The distinction Python makes: @staticmethod = utility function namespaced to the class (no access to class or instance). @classmethod = receives the class itself as cls, enabling factory patterns that work with inheritance. Using cls() instead of User() means subclasses automatically get the right type back.

Dataclasses — TypeScript Interfaces Meet Classes

Dataclasses auto-generate __init__, __repr__, __eq__, and more from field declarations. They eliminate boilerplate for data-holding classes.

TypeScript
// Interface (compile-time only)
interface Article {
  title: string;
  author: string;
  views: number;
  published: boolean;
}

const a: Article = {
  title: "Scoop",
  author: "Trevor",
  views: 0,
  published: false
};

// Must check each field for equality
// No built-in toString
Python
from dataclasses import dataclass

@dataclass
class Article:
    title: str
    author: str
    views: int = 0
    published: bool = False

a = Article("Scoop", "Trevor")

# Auto-generated __repr__!
print(a)
# Article(title='Scoop', ...)

# Auto-generated __eq__!
a == Article("Scoop", "Trevor")  # True

Dataclass Advanced Features

from dataclasses import dataclass, field

@dataclass
class Config:
    host: str = "localhost"
    port: int = 8000
    debug: bool = False
    # Mutable defaults use field(default_factory=...)
    allowed_hosts: list[str] = field(default_factory=list)

# Frozen (immutable) — like Object.freeze
@dataclass(frozen=True)
class Point:
    x: float
    y: float

p = Point(1.0, 2.0)
# p.x = 3.0  — FrozenInstanceError!

# Ordered (enables sorting by fields)
@dataclass(order=True)
class Student:
    gpa: float
    name: str

students = [Student(3.5, "Alice"), Student(3.9, "Bob")]
sorted(students)  # sorts by gpa, then name

The mental model: @dataclass reads your type-annotated fields and writes the boilerplate: constructor, repr, equality. Think of it as TypeScript interfaces that actually do things at runtime. Django models are similar but more powerful — they also generate database tables and admin interfaces from field declarations.

Privacy Conventions

JavaScript
class Account {
  // Truly private (ES2022)
  #balance = 0;

  constructor(initial) {
    this.#balance = initial;
  }

  #validate(amount) {
    return amount > 0;
  }

  deposit(amount) {
    if (this.#validate(amount)) {
      this.#balance += amount;
    }
  }
}

const a = new Account(100);
// a.#balance — SyntaxError
Python
class Account:

    def __init__(self, initial):
        # _ = "private by convention"
        self._balance = initial

    def _validate(self, amount):
        return amount > 0

    def deposit(self, amount):
        if self._validate(amount):
            self._balance += amount



a = Account(100)
a._balance  # 100 — accessible!
# Python: "we're all consenting adults"

Python has no true privacy. _single_underscore = "treat as private" (convention). __double_underscore = name mangling (makes self.__x become self._ClassName__x — harder to access accidentally but still not truly private). JavaScript's # fields are genuinely inaccessible from outside.

Abstract Classes — Enforcing Interfaces

TypeScript
// Abstract class
abstract class Shape {
  abstract area(): number;
  describe() { return "I'm a shape"; }
}

// Interface
interface Serializable {
  serialize(): string;
}

class Circle extends Shape
    implements Serializable {
  constructor(public radius: number) {
    super();
  }
  area() { return Math.PI * this.radius ** 2; }
  serialize() { return JSON.stringify(this); }
}
Python
from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self) -> float: ...
    def describe(self): return "I'm a shape"

# No separate interface — use ABC or Protocol
from typing import Protocol

class Serializable(Protocol):
    def serialize(self) -> str: ...

class Circle(Shape):
    def __init__(self, radius: float):
        self.radius = radius
    def area(self): return 3.14159 * self.radius ** 2
    def serialize(self): return str(vars(self))

Runtime enforcement: TypeScript's abstract is compile-time only. Python's ABC enforces at runtime — trying to instantiate Shape() directly raises TypeError. Django uses abstract models (class Meta: abstract = True) for shared fields that don't create database tables.

Check Your Understanding

Quick Quiz

1. Why does every Python method need self as its first parameter?

self is the instance reference (like this in JS). When you call obj.method(), Python translates it to Class.method(obj), passing the instance explicitly. No binding surprises — self always means the instance.

2. What's the difference between __str__ and __repr__?

__str__ = human-readable (print(), f-strings). __repr__ = developer-readable (REPL, debugging). A good __repr__ shows how to recreate the object. Django admin uses __str__ to display model records in lists and dropdowns.

3. What does @dataclass auto-generate for you?

__init__ (constructor from fields), __repr__ (readable string), and __eq__ (field-by-field comparison). With order=True it adds comparison methods. With frozen=True it makes instances immutable.

4. What's the difference between @staticmethod and @classmethod?

@staticmethod receives no automatic arguments — it's a plain utility. @classmethod receives the class as cls, enabling factory methods that return the correct subclass type. Use cls() not ClassName() to support inheritance.

5. How do you make len(my_object) work on a custom class?

Define __len__(self) returning an integer. Python's len() calls obj.__len__() under the hood. Same pattern for __iter__ (for loops), __contains__ (in operator), __getitem__ (obj[key]).
