04 Templates & Forms — JSX to DTL, React Forms to Django Forms

← Back to lessons

You already write JSX and handle forms with React. Django does both server-side. This lesson maps Django's template language and form system to patterns you already know from React and Next.js.

Django Template Language (DTL) Overview

DTL is Django's built-in way to generate HTML dynamically. If JSX lets you write HTML inside JavaScript, DTL lets you write Python-like logic inside HTML. The big difference: DTL runs entirely on the server — no JavaScript ships to the browser.

Concept JSX / React Django DTL
Output a variable {title} {{ title }}
Conditional {show && <p>...</p>} {% if show %}<p>...</p>{% endif %}
Loop {items.map(i => ...)} {% for i in items %}...{% endfor %}
Layout wrapper layout.tsx + {children} {% extends "base.html" %} + {% block %}
Reusable component <Card /> {% include "card.html" %}

The mental model: DTL is JSX for people who want zero client-side JavaScript. Double curly braces {{ }} output values (like JSX's single curly braces). Curly-percent {% %} runs logic (like JSX's ternaries and .map calls). Everything renders on the server before the browser sees it.

Variables & Filters

Variables come from the "context" dictionary your view passes to the template. Filters transform them — think of filters like piping through utility functions.

React / JSX
// Variables from props/state
<h1>{title}</h1>
<p>{title.toUpperCase()}</p>
<p>{title.length} characters</p>
<p>{new Date(created).toLocaleDateString()}</p>
<p>{description || 'No description'}</p>
<div dangerouslySetInnerHTML=
  {{ __html: body }} />
Django Template
{# Variables from view context #}
<h1>{{ title }}</h1>
<p>{{ title|upper }}</p>
<p>{{ title|length }} characters</p>
<p>{{ created|date:"M d, Y" }}</p>
<p>{{ description|default:"No description" }}</p>
<div>{{ body|safe }}</div>

Common filters you'll use constantly:

Filter What It Does JS Equivalent
{{ x|lower }} Lowercase x.toLowerCase()
{{ x|truncatewords:30 }} Truncate to 30 words x.split(' ').slice(0,30).join(' ')
{{ x|date:"Y-m-d" }} Format date format(x, 'yyyy-MM-dd')
{{ x|default:"N/A" }} Fallback value x || 'N/A'
{{ x|slugify }} URL-safe slug x.toLowerCase().replace(/ /g, '-')
{{ x|safe }} Render raw HTML dangerouslySetInnerHTML
{{ list|join:", " }} Join array list.join(', ')

Security note: Django auto-escapes all {{ }} output to prevent XSS attacks. The |safe filter turns that off. Only use |safe on content you trust — just like React's dangerouslySetInnerHTML. The name is misleading; it means "I promise this is safe."

Template Tags — Logic in Templates

Tags ({% %}) add control flow to your templates. Here are the ones you'll use daily:

Conditionals

React / JSX
{user ? (
  <p>Welcome, {user.name}</p>
) : (
  <a href="/login">Log in</a>
)}

{posts.length > 0 && (
  <span>{posts.length} posts</span>
)}
Django Template
{% if user.is_authenticated %}
  <p>Welcome, {{ user.username }}</p>
{% else %}
  <a href="{% url 'login' %}">Log in</a>
{% endif %}

{% if posts %}
  <span>{{ posts|length }} posts</span>
{% endif %}

Loops

React / JSX
<ul>
  {posts.map((post, i) => (
    <li key={post.id}>
      {i + 1}. {post.title}
    </li>
  ))}
</ul>
{posts.length === 0 && (
  <p>No posts yet.</p>
)}
Django Template
<ul>
  {% for post in posts %}
    <li>
      {{ forloop.counter }}. {{ post.title }}
    </li>
  {% empty %}
    <li>No posts yet.</li>
  {% endfor %}
</ul>

forloop magic variables: Inside {% for %}, Django gives you forloop.counter (1-indexed), forloop.counter0 (0-indexed), forloop.first, forloop.last, and forloop.revcounter. The {% empty %} block handles empty lists — no need for a separate conditional.

Template Inheritance — Layouts in Django

Next.js nests layouts automatically via the file system. Django uses explicit inheritance with {% extends %} and {% block %}. Same result, more control.

The Base Template (Your Layout)

<!-- templates/base.html — like app/layout.tsx -->
<!DOCTYPE html>
<html>
<head>
  <title>{% block title %}My Site{% endblock %}</title>
  <link rel="stylesheet" href="{% static 'css/style.css' %}">
  {% block extra_css %}{% endblock %}
</head>
<body>
  <nav>
    <a href="/django/">Home</a>
    {% if user.is_authenticated %}
      <a href="{% url 'logout' %}">Logout</a>
    {% else %}
      <a href="{% url 'login' %}">Login</a>
    {% endif %}
  </nav>

  <main>
    {% block content %}
    <!-- child templates fill this in -->
    {% endblock %}
  </main>

  <footer>Copyright 2025</footer>
  {% block extra_js %}{% endblock %}
</body>
</html>

A Child Template

<!-- templates/blog/list.html — like app/blog/page.tsx -->
{% extends "base.html" %}

{% block title %}Blog | My Site{% endblock %}

{% block content %}
  <h1>All Posts</h1>
  {% for post in posts %}
    <article>
      <h2><a href="{% url 'blog-detail' post.slug %}">{{ post.title }}</a></h2>
      <p>{{ post.body|truncatewords:50 }}</p>
      <time>{{ post.created|date:"F j, Y" }}</time>
    </article>
  {% empty %}
    <p>No posts published yet.</p>
  {% endfor %}
{% endblock %}

Translation: {% extends "base.html" %} = "wrap me in the layout." {% block content %}...{% endblock %} = "this is where {children} goes." You can have multiple blocks (title, content, sidebar, extra_js) — it's like having multiple {children} slots, which React doesn't natively support without named slots or render props.

{% include %} — Reusable Components

React Component
// components/PostCard.tsx
function PostCard({ post }) {
  return (
    <div className="card">
      <h3>{post.title}</h3>
      <p>{post.excerpt}</p>
    </div>
  );
}

// Usage
<PostCard post={post} />
Django Include
<!-- templates/components/post_card.html -->
<div class="card">
  <h3>{{ post.title }}</h3>
  <p>{{ post.excerpt }}</p>
</div>

<!-- Usage -->
{% include "components/post_card.html" with post=post %}

Props = context variables. {% include "card.html" with post=post only %} — the with keyword passes variables (like props), and only restricts the included template to ONLY those variables (like isolating component scope). Without only, the included template inherits the entire parent context.

The {% url %} Tag — Type-Safe Links

In Next.js, you hardcode link paths. In Django, {% url %} generates URLs from route names. If you rename a URL pattern, all links update automatically.

Next.js Links
import Link from 'next/link';

// Hardcoded paths
<Link href="/blog">Blog</Link>
<Link href={`/blog/${post.slug}`}>
  {post.title}
</Link>
Django URL Tag
{# Named URL resolution #}
<a href="{% url 'blog-index' %}">Blog</a>
<a href="{% url 'blog-detail' post.slug %}">
  {{ post.title }}
</a>

Why this matters: If you change path('articles/', ...) to path('posts/', ...), every {% url 'blog-index' %} tag automatically resolves to the new path. No broken links. It's like having a centralized route map that your templates reference by name instead of hardcoded strings.

Django Forms — The Big Picture

In React, forms require: state management (useState), change handlers (onChange), validation (Zod/Yup), CSRF protection (custom), and rendering. Django handles all of this in one class.

React Form Stack
  • useState for each field
  • onChange handlers
  • Zod/Yup for validation
  • Manual CSRF tokens
  • Manual error display
  • ~50-100 lines per form
Django Form Class
  • Fields auto-generate HTML inputs
  • Validation built-in
  • CSRF automatic with {% csrf_token %}
  • Error messages auto-attached
  • Data cleaning/sanitization built-in
  • ~10-20 lines per form

Basic Form Class

React + Zod
// schemas/contact.ts
const ContactSchema = z.object({
  name: z.string().min(1),
  email: z.string().email(),
  message: z.string().min(10),
});

// Component: ~40 more lines of
// useState, onChange, onSubmit...
Django Form
# forms.py
from django import forms

class ContactForm(forms.Form):
    name = forms.CharField(max_length=100)
    email = forms.EmailField()
    message = forms.CharField(
        min_length=10,
        widget=forms.Textarea
    )
# That's it. Validation + HTML included.

The mental model: A Django Form class is like combining your Zod schema, your React component's form JSX, and your onChange handlers into ONE thing. Define the fields once; Django generates the HTML inputs, validates the data, and reports errors. The form IS the schema IS the renderer.

ModelForm — Forms from Models

The most powerful shortcut: ModelForm generates a form directly from your database model. It's like if Prisma could auto-generate your Zod schemas AND React form components.

# models.py
from django.db import models

class Article(models.Model):
    title = models.CharField(max_length=200)
    body = models.TextField()
    category = models.ForeignKey('Category', on_delete=models.CASCADE)
    published = models.BooleanField(default=False)
    created_at = models.DateTimeField(auto_now_add=True)
# forms.py — auto-generate a form from the model
from django.forms import ModelForm
from .models import Article

class ArticleForm(ModelForm):
    class Meta:
        model = Article
        fields = ['title', 'body', 'category', 'published']
        # created_at is excluded — auto-set by the model
        widgets = {
            'body': forms.Textarea(attrs={'rows': 10}),
            'title': forms.TextInput(attrs={'placeholder': 'Enter title...'}),
        }

ModelForm magic: It reads your model's field types and automatically creates the right HTML input. CharField becomes <input type="text">, BooleanField becomes a checkbox, ForeignKey becomes a <select> dropdown populated with the related objects. All validation constraints from the model (max_length, required, etc.) are inherited.

Rendering Forms in Templates

Django gives you multiple ways to render a form, from fully automatic to fully manual.

<!-- templates/article_form.html -->
{% extends "base.html" %}

{% block content %}
  <h1>Create Article</h1>

  <form method="post" enctype="multipart/form-data">
    {% csrf_token %}

    {# Option 1: Render entire form automatically #}
    {{ form.as_p }}

    {# Option 2: Render field by field for custom layout #}
    <div class="field">
      <label for="{{ form.title.id_for_label }}">Title</label>
      {{ form.title }}
      {% if form.title.errors %}
        <ul class="errors">
          {% for error in form.title.errors %}
            <li>{{ error }}</li>
          {% endfor %}
        </ul>
      {% endif %}
    </div>

    <button type="submit">Save</button>
  </form>
{% endblock %}
Render Method What It Does When to Use
{{ form.as_p }} Each field in a <p> tag Quick prototyping
{{ form.as_div }} Each field in a <div> Better for CSS styling
{{ form.as_table }} Table rows Admin-style layouts
{{ form.field_name }} Single field's HTML input Full control over layout

Never forget {% csrf_token %}! Every POST form needs it. Django will reject the submission with a 403 error otherwise. It's Django's built-in CSRF protection — something you'd need a library for in Next.js. Just put it right after the <form> tag.

Validation — clean() Methods

Django forms have three layers of validation, running in order. Think of it as a pipeline — data flows through each stage.

# forms.py
from django import forms
from django.core.exceptions import ValidationError

class ArticleForm(forms.ModelForm):
    class Meta:
        model = Article
        fields = ['title', 'body', 'category']

    # Layer 1: Field-level validation (like Zod .refine())
    def clean_title(self):
        title = self.cleaned_data['title']
        if 'clickbait' in title.lower():
            raise ValidationError("No clickbait titles allowed!")
        return title

    # Layer 2: Cross-field validation (like Zod .superRefine())
    def clean(self):
        cleaned = super().clean()
        title = cleaned.get('title', '')
        body = cleaned.get('body', '')
        if len(body) < len(title):
            raise ValidationError(
                "Article body must be longer than the title."
            )
        return cleaned
Zod Validation (JS)
const schema = z.object({
  title: z.string()
    .refine(t => !t.includes('clickbait'),
      { message: "No clickbait!" }),
  body: z.string(),
}).superRefine((data, ctx) => {
  if (data.body.length < data.title.length) {
    ctx.addIssue({
      code: 'custom',
      message: "Body too short",
    });
  }
});
Django Validation
# Validation pipeline:
# 1. Field type checks (auto)
#    CharField → is it a string?
#    EmailField → is it a valid email?
# 2. clean_fieldname() methods
#    Per-field custom logic
# 3. clean() method
#    Cross-field logic

# All errors auto-attach to fields
# and render in the template!

The View Pattern — Handling Form Submissions

Here's the standard Django pattern for handling a form in a view. This is the equivalent of a React form's onSubmit handler combined with the component itself:

# views.py
from django.shortcuts import render, redirect
from .forms import ArticleForm

def create_article(request):
    if request.method == 'POST':
        # Form submitted — bind data to form
        form = ArticleForm(request.POST, request.FILES)
        if form.is_valid():
            # Save and redirect (PRG pattern)
            article = form.save(commit=False)
            article.author = request.user
            article.save()
            return redirect('article-detail', slug=article.slug)
        # If invalid, fall through — form now has errors
    else:
        # GET request — show empty form
        form = ArticleForm()

    return render(request, 'articles/create.html', {'form': form})

The GET/POST pattern: One view handles both displaying the form (GET) and processing the submission (POST). On GET, you create an empty form. On POST, you bind the submitted data. If validation fails, the same template re-renders with errors pre-filled. It's like having your React component and your API route handler in one function — because server-rendered apps don't need separate client and server logic.

PRG = Post/Redirect/Get. After a successful form save, always redirect() instead of rendering a template. This prevents duplicate submissions when users refresh the page. It's a web pattern, not Django-specific — but Django makes it easy with redirect().

File Uploads

Django handles file uploads natively. No need for third-party upload libraries like in Next.js.

# models.py
class Article(models.Model):
    title = models.CharField(max_length=200)
    cover_image = models.ImageField(upload_to='covers/')  # needs Pillow
    attachment = models.FileField(upload_to='attachments/', blank=True)

# forms.py
from django import forms

class ArticleForm(forms.ModelForm):
    class Meta:
        model = Article
        fields = ['title', 'cover_image', 'attachment']

# views.py — request.FILES carries uploaded files
def create_article(request):
    if request.method == 'POST':
        form = ArticleForm(request.POST, request.FILES)  # <— request.FILES!
        if form.is_valid():
            form.save()
            return redirect('article-list')
    else:
        form = ArticleForm()
    return render(request, 'articles/create.html', {'form': form})

Two things to remember for file uploads: (1) The template form tag MUST include enctype="multipart/form-data". (2) The view MUST pass request.FILES to the form constructor. Forgetting either one means files silently don't upload — no error, just missing data.

Formsets & Widgets

Formsets let you edit multiple items on one page — like rendering an array of forms. Use modelformset_factory(Photo, fields=['caption', 'image'], extra=3) to generate a set. They handle indexing, validation, and saving automatically. Think of it as useState([{}, {}, {}]) in React, but server-managed.

Widgets control how a field renders as HTML. They're the "JSX props" of Django forms:

# forms.py — widgets configure the HTML output
class ArticleForm(forms.ModelForm):
    class Meta:
        model = Article
        fields = ['title', 'body', 'publish_date']
        widgets = {
            'title': forms.TextInput(attrs={
                'class': 'form-control',           # CSS class
                'placeholder': 'Article title...',  # placeholder
            }),
            'body': forms.Textarea(attrs={'rows': 15}),
            'publish_date': forms.DateInput(attrs={'type': 'date'}),  # HTML5 picker
        }

Widget = HTML, Field = validation. In React, you write <input type="date" className="form-control" />. In Django, you configure the same thing via the widget's attrs dictionary. Same HTML output, different authoring style.

Check Your Understanding

Quick Quiz

1. What's the Django equivalent of JSX's {variable}?

{{ variable }} — double curly braces for outputting values. Single curly braces are just literal text in Django templates.

2. How does Django's {% extends %} relate to Next.js layouts?

{% extends "base.html" %} is like being wrapped by layout.tsx. The base template defines {% block %} slots (like {children}), and child templates fill them in. The difference: it's explicit (you name the parent) rather than automatic (file-system based).

3. What does a ModelForm give you that a regular Form doesn't?

ModelForm auto-generates fields from your database model, inherits validation constraints, and provides a .save() method that creates/updates database records directly. A regular Form requires you to define every field manually and handle saving yourself.

4. Why must every Django POST form include {% csrf_token %}?

CSRF (Cross-Site Request Forgery) protection. The token proves the form submission came from your site, not a malicious third party. Django rejects POST requests without a valid token (403 error). It's built-in security that you'd need a library for in Next.js.

5. What are the three layers of Django form validation, in order?

(1) Field-type validation — automatic checks like "is this a valid email?" (2) clean_fieldname() methods — per-field custom logic. (3) clean() method — cross-field validation. Errors from any layer are collected and displayed in the template automatically.
