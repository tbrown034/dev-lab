06 Auth, Testing & Deployment — Ship It Like Production

← Back to lessons

You know NextAuth, Jest, and Vercel. Django has built-in equivalents for all three. This lesson covers authentication, testing, and deployment — everything you need to go from local dev to production.

Django Auth — Built-in Authentication

Next.js doesn't ship with auth — you reach for NextAuth/Auth.js, Clerk, or roll your own. Django includes a complete auth system out of the box: users, login, logout, passwords, permissions, groups.

Next.js Auth Stack
  • npm install next-auth
  • Configure providers (Google, GitHub...)
  • Create API route for auth
  • Session provider wrapper
  • Custom sign-in pages
  • Middleware for protection
  • ~100+ lines of setup
Django Auth (Built-in)
  • Already in INSTALLED_APPS
  • User model included
  • Login/logout views included
  • Password hashing automatic
  • Session management automatic
  • @login_required decorator
  • ~10 lines of setup

The mental model: Django auth is like if NextAuth came pre-installed, pre-configured, with a built-in User model, password hashing, session cookies, and permission system — and you just needed to point it at your templates. Zero provider configuration for username/password auth.

The User Model

# Django's built-in User model has these fields:
# username, password, email, first_name, last_name,
# is_active, is_staff, is_superuser, date_joined, last_login

# Using it in views:
from django.contrib.auth.models import User

# Create a user
user = User.objects.create_user(
    username='trevor',
    email='trevor@newsroom.com',
    password='securepassword123'
)

# Check in a view
def dashboard(request):
    if request.user.is_authenticated:
        print(request.user.username)  # 'trevor'
        print(request.user.email)     # 'trevor@newsroom.com'

Custom User models: For real projects, always create a custom User model before your first migration. Extend AbstractUser (add fields) or AbstractBaseUser (full control). Changing the User model after migrations exist is painful — like trying to change your Prisma schema's primary key type after you have data.

Login & Logout Views

NextAuth Setup
// app/api/auth/[...nextauth]/route.ts
import NextAuth from 'next-auth';
import Credentials from
  'next-auth/providers/credentials';

export const handler = NextAuth({
  providers: [
    Credentials({
      authorize(credentials) {
        // manual validation...
      }
    })
  ],
  pages: {
    signIn: '/login',
  }
});
Django Auth URLs
# urls.py — that's it
from django.contrib.auth import views as auth

urlpatterns = [
    path('login/', auth.LoginView.as_view(
        template_name='login.html'
    ), name='login'),
    path('logout/', auth.LogoutView.as_view(),
        name='logout'),
    path('password-change/',
        auth.PasswordChangeView.as_view(),
        name='password_change'),
]
# Login, logout, password reset — done.

Protecting Views — @login_required

Next.js Middleware
// middleware.ts
import { auth } from './auth';

export default auth((req) => {
  if (!req.auth &&
      req.nextUrl.pathname
        .startsWith('/dashboard')) {
    return Response.redirect(
      new URL('/login', req.url)
    );
  }
});
Django Decorator
# views.py
from django.contrib.auth.decorators \
    import login_required

@login_required
def dashboard(request):
    # Only authenticated users reach here
    # Others redirect to LOGIN_URL
    return render(request,
        'dashboard.html',
        {'user': request.user}
    )

One decorator vs. a middleware file. Django's @login_required goes directly on the view function. No separate middleware file, no route matching patterns. If the user isn't authenticated, Django redirects to LOGIN_URL (set in settings.py, defaults to /accounts/login/). After login, it redirects back to the original page automatically.

Permissions & Groups

# Django auto-creates permissions for every model:
# blog.add_article, blog.change_article, blog.delete_article, blog.view_article

# Check permissions in views
from django.contrib.auth.decorators import permission_required

@permission_required('blog.add_article', raise_exception=True)
def create_article(request):
    # Only users with "add_article" permission can access
    ...

# Check in templates
# {% if perms.blog.add_article %}
#   <a href="{% url 'create-article' %}">New Article</a>
# {% endif %}

# Groups — bundle permissions together
from django.contrib.auth.models import Group

editors = Group.objects.create(name='Editors')
editors.permissions.add(
    Permission.objects.get(codename='add_article'),
    Permission.objects.get(codename='change_article'),
)
# Assign user to group
user.groups.add(editors)

Newsroom mental model: Groups are like roles. Create an "Editors" group with edit/publish permissions and a "Reporters" group with create-only permissions. Assign users to groups instead of managing individual permissions. It's like role-based access control (RBAC) built right into the framework.

Testing with pytest-django

Django ships with its own test framework built on Python's unittest. But most teams use pytest-django for cleaner syntax. If you know Jest, pytest feels natural.

Jest + React Testing Library
// __tests__/api.test.ts
import { GET } from '@/app/api/posts/route';

describe('Posts API', () => {
  it('returns posts', async () => {
    const res = await GET();
    const data = await res.json();

    expect(res.status).toBe(200);
    expect(data.posts).toHaveLength(2);
  });
});
pytest-django
# tests/test_views.py
import pytest
from django.test import Client

@pytest.mark.django_db
def test_posts_list():
    client = Client()
    response = client.get('/api/posts/')

    assert response.status_code == 200
    data = response.json()
    assert len(data['posts']) == 2

The Test Client — Your Supertest/Fetch

# The Django test Client simulates HTTP requests — no server needed
from django.test import Client

client = Client()

# GET request
response = client.get('/blog/')
assert response.status_code == 200
assert 'My Blog' in response.content.decode()

# POST request with data
response = client.post('/blog/create/', {
    'title': 'Test Article',
    'body': 'This is a test.',
})
assert response.status_code == 302  # redirect on success

# Login and test protected views
client.login(username='trevor', password='testpass123')
response = client.get('/dashboard/')
assert response.status_code == 200  # now accessible

No server needed! Django's test Client sends HTTP requests directly to your views without starting a server. It's like if Jest could call your Next.js API routes directly. Each test uses a fresh database that's destroyed after the test — no test pollution.

Testing Models

# tests/test_models.py
import pytest
from blog.models import Article
from django.contrib.auth.models import User

@pytest.mark.django_db
def test_article_creation():
    user = User.objects.create_user(username='trevor', password='pass')
    article = Article.objects.create(
        title='Election Results 2024',
        body='The results are in...',
        author=user,
    )
    assert article.title == 'Election Results 2024'
    assert article.author.username == 'trevor'
    assert str(article) == 'Election Results 2024'  # tests __str__

@pytest.mark.django_db
def test_article_slug_auto_generated():
    article = Article.objects.create(title='Breaking News Today')
    assert article.slug == 'breaking-news-today'

Fixtures & Factories

Jest Fixtures
// test-utils.ts
function createMockUser(overrides = {}) {
  return {
    id: 1,
    name: 'Trevor',
    email: 'trevor@test.com',
    ...overrides,
  };
}

// In test
const user = createMockUser({
  name: 'Editor'
});
factory_boy (Python)
# tests/factories.py
import factory
from django.contrib.auth.models import User

class UserFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = User
    username = factory.Sequence(
        lambda n: f'user{n}')
    email = factory.LazyAttribute(
        lambda o: f'{o.username}@test.com')

# In test
user = UserFactory(username='editor')
# pytest fixtures — like Jest's beforeEach but more powerful
import pytest
from tests.factories import UserFactory, ArticleFactory

@pytest.fixture
def editor():
    user = UserFactory(username='editor')
    editors_group = Group.objects.create(name='Editors')
    user.groups.add(editors_group)
    return user

@pytest.fixture
def sample_articles(editor):
    return ArticleFactory.create_batch(5, author=editor)

# Use fixtures as function arguments — pytest injects them
@pytest.mark.django_db
def test_editor_can_see_articles(editor, sample_articles):
    client = Client()
    client.force_login(editor)
    response = client.get('/dashboard/')
    assert len(response.context['articles']) == 5

pytest fixtures vs. Jest beforeEach: In Jest, beforeEach runs setup for every test in the file. pytest fixtures are declared as function parameters — each test explicitly requests what it needs. A test that takes (editor, sample_articles) gets both injected automatically. More explicit, more composable.

What to Test

Layer What to Test Jest Equivalent
Models Validation, methods, str(), properties Unit tests for data layer
Views Status codes, redirects, context data API route / handler tests
Forms Validation, clean methods, error messages Schema validation tests
Templates Correct content rendered Component render tests (RTL)
Integration Full request/response cycles E2E tests (Playwright/Cypress)

Common mistake: Don't test Django's built-in functionality. You don't need to test that CharField accepts strings — Django already tests that. Focus on YOUR custom logic: custom validation, business rules, permissions, and edge cases. Test what you wrote, not what Django wrote.

Deployment — From Dev to Production

Deploying Django is conceptually the same as deploying Next.js, but with different tools at each step.

Next.js / Vercel
  • next build
  • Node.js server
  • Static files via CDN
  • vercel deploy
  • Environment variables in dashboard
  • Auto-scaling, serverless
Django / Any Platform
  • collectstatic
  • Gunicorn WSGI server
  • Static files via nginx/WhiteNoise/S3
  • git push to Railway/Render/Fly
  • Environment variables via .env or platform
  • Process-based scaling

WSGI & ASGI — Django's "Node Server"

The mental model: WSGI is to Django what Node.js is to Next.js. It's the server that actually runs your Python code. In development, runserver handles this. In production, you use Gunicorn (like PM2 for Node). ASGI is the async version — use it with Uvicorn or Daphne when you need WebSockets or async views.

# Development
python manage.py runserver  # built-in dev server (like next dev)

# Production — NEVER use runserver in production
pip install gunicorn
gunicorn mysite.wsgi:application --bind 0.0.0.0:8000 --workers 4

# Async production
pip install uvicorn
uvicorn mysite.asgi:application --host 0.0.0.0 --port 8000

Static Files — collectstatic

In development, Django serves static files automatically. In production, you need to collect them all into one directory and serve them efficiently.

# settings.py
STATIC_URL = '/static/'
STATIC_ROOT = BASE_DIR / 'staticfiles'   # where collectstatic puts files
STATICFILES_DIRS = [BASE_DIR / 'static'] # where your source static files live

# Run before deploying
python manage.py collectstatic
# Copies ALL static files (CSS, JS, images) from your apps
# and STATICFILES_DIRS into STATIC_ROOT

# Serve with WhiteNoise (simplest option)
pip install whitenoise

# settings.py
MIDDLEWARE = [
    'django.middleware.security.SecurityMiddleware',
    'whitenoise.middleware.WhiteNoiseMiddleware',  # add after security
    ...
]
STATICFILES_STORAGE = 'whitenoise.storage.CompressedManifestStaticFilesStorage'

WhiteNoise is Django's "Vercel for static files." It serves static files directly from your Python process with compression and caching headers. For most projects, it's all you need. For high-traffic sites, offload to nginx, a CDN, or S3 — like how Vercel puts static assets on its CDN edge network.

Environment Variables & Security

# Install python-decouple (like dotenv for Python)
pip install python-decouple

# .env (NEVER commit this)
SECRET_KEY=your-super-secret-key-here
DEBUG=False
DATABASE_URL=postgres://user:pass@host:5432/mydb
ALLOWED_HOSTS=mysite.com,www.mysite.com

# settings.py
from decouple import config

SECRET_KEY = config('SECRET_KEY')
DEBUG = config('DEBUG', default=False, cast=bool)
ALLOWED_HOSTS = config('ALLOWED_HOSTS', cast=lambda v: v.split(','))

Production checklist: (1) DEBUG = False — NEVER run debug mode in production (exposes stack traces). (2) Set a real SECRET_KEY (used for CSRF, sessions, password hashing). (3) Set ALLOWED_HOSTS to your domain(s). (4) Use PostgreSQL, not SQLite. (5) Run python manage.py check --deploy — Django will flag security issues.

Database Configuration for Production

# Install PostgreSQL adapter
pip install psycopg2-binary dj-database-url

# settings.py
import dj_database_url

# Development fallback to SQLite
DATABASES = {
    'default': dj_database_url.config(
        default='sqlite:///' + str(BASE_DIR / 'db.sqlite3')
    )
}
# Set DATABASE_URL in production:
# DATABASE_URL=postgres://user:pass@host:5432/mydb

Railway Deployment (Quickest Path)

# 1. Create requirements.txt
pip freeze > requirements.txt

# 2. Create Procfile (tells Railway how to start)
# Procfile
web: gunicorn mysite.wsgi:application --bind 0.0.0.0:$PORT

# 3. Create runtime.txt (optional)
python-3.12.0

# 4. Deploy
# Connect your GitHub repo to Railway
# Set environment variables in dashboard
# Railway auto-detects Django and deploys

# 5. Run migrations on the deployed database
railway run python manage.py migrate
railway run python manage.py createsuperuser

Vercel vs. Railway: Vercel auto-deploys your Next.js app from a GitHub push. Railway does the same for Django. Connect your repo, set env vars, push to main — it builds, migrates, and deploys. The mental model is identical; only the config file names change (vercel.json vs. Procfile, .env.local vs. Railway env vars).

Production Settings Pattern

A common pattern: split settings into base, development, and production. Like having different .env.local and .env.production files in Next.js.

mysite/
  settings/
    __init__.py ......... empty
    base.py ............ shared settings
    development.py ..... DEBUG=True, SQLite
    production.py ...... DEBUG=False, Postgres, security 
# settings/production.py
from .base import *

DEBUG = False
ALLOWED_HOSTS = config('ALLOWED_HOSTS', cast=lambda v: v.split(','))

# Security settings
SECURE_SSL_REDIRECT = True
SESSION_COOKIE_SECURE = True
CSRF_COOKIE_SECURE = True
SECURE_HSTS_SECONDS = 31536000

# Use it with: DJANGO_SETTINGS_MODULE=mysite.settings.production

Check Your Understanding

Quick Quiz

1. What's the Django equivalent of wrapping a Next.js page with an auth check in middleware?

The @login_required decorator on the view function. It redirects unauthenticated users to the login page and returns them to the original URL after login. For permission checks, use @permission_required('app.permission_name').

2. How is Django's test Client different from making HTTP requests with fetch?

The test Client calls your views directly in-process — no server is started, no network requests are made. It's faster and gives you access to internal details like response.context (the template variables) and response.templates (which templates were rendered). Real HTTP calls only see the response body.

3. What does python manage.py collectstatic do, and when do you run it?

It copies all static files (CSS, JS, images) from your apps and STATICFILES_DIRS into one STATIC_ROOT directory. Run it before deployment. In development, Django's dev server serves static files automatically, so you don't need it locally. It's like next build for static assets.

4. Why should you never use runserver in production?

runserver is single-threaded, unoptimized, auto-reloads on file changes, and hasn't been audited for security. Use Gunicorn (WSGI) or Uvicorn (ASGI) in production — they handle multiple concurrent requests, are battle-tested, and don't expose development features. It's like running next dev in production instead of next start.

5. What's the difference between a pytest fixture and Jest's beforeEach?

Jest's beforeEach runs the same setup for every test in a describe block. pytest fixtures are declared as function parameters — each test explicitly requests only the fixtures it needs. This is more explicit (you see what each test depends on) and more composable (fixtures can depend on other fixtures, building a dependency graph).
