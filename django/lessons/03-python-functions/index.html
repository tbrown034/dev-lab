03 Python Functions & Modules

← Back to lessons

Functions are functions. The concepts are identical — what changes is the syntax for defining them, the argument-passing superpowers Python adds, and how you organize code into modules.

def vs function / const arrow

JavaScript gives you three ways to define functions: function declarations, const arrow functions, and method shorthand. Python gives you one: def.

JavaScript
// Function declaration (hoisted)
function greet(name) {
  return `Hello, ${name}!`;
}

// Arrow function (not hoisted)
const greet = (name) => {
  return `Hello, ${name}!`;
};

// Arrow one-liner (implicit return)
const double = (n) => n * 2;

// No return = returns undefined
function log(msg) {
  console.log(msg);
}
Python
# def — the one and only way
def greet(name):
    return f"Hello, {name}!"



# Functions are first-class objects
greet_fn = greet  # assign to variable
greet_fn("Trevor")


# Lambda for one-liners (see below)
double = lambda n: n * 2

# No return = returns None
def log(msg):
    print(msg)

The mental model: Python doesn't need arrow functions because it doesn't have this-binding issues. In JavaScript, arrow functions exist largely to solve this scoping problems. Python uses explicit self parameters, so that problem never arises. One function syntax is enough.

Default Arguments

JavaScript
function createUser(
  name,
  role = "viewer",
  active = true
) {
  return { name, role, active };
}

createUser("Trevor");
// { name: "Trevor", role: "viewer", active: true }

createUser("Trevor", "admin");
// { name: "Trevor", role: "admin", active: true }

// Skip middle args? Can't easily.
// Need: createUser("Trevor", undefined, false)
Python
def create_user(
    name,
    role="viewer",
    active=True
):
    return {"name": name, "role": role, "active": active}

create_user("Trevor")
# {"name": "Trevor", "role": "viewer", "active": True}

create_user("Trevor", "admin")
# {"name": "Trevor", "role": "admin", "active": True}

# Skip middle args? Use keyword arguments!
create_user("Trevor", active=False)
# {"name": "Trevor", "role": "viewer", "active": False}

Translation rule: Python's keyword arguments solve the "skip a default parameter" problem elegantly. In JS, you pass undefined to skip. In Python, you name the argument: f(x, active=False). This is a superpower you'll miss when you go back to JS.

Mutable default argument trap! Never use a mutable value (list, dict) as a default argument. Python evaluates defaults once at function definition, not on each call. This is the single most common Python gotcha for ALL beginners.

WRONG
# BUG: all calls share the same list!
def add_item(item, items=[]):
    items.append(item)
    return items

add_item("a")  # ["a"]
add_item("b")  # ["a", "b"] !!!!
# Expected ["b"], got ["a", "b"]
RIGHT
# Use None sentinel + create inside
def add_item(item, items=None):
    if items is None:
        items = []
    items.append(item)
    return items

add_item("a")  # ["a"]
add_item("b")  # ["b"] — correct!

*args and **kwargs — Rest & Spread

Python splits JavaScript's ...rest into two concepts: *args for positional arguments (like a rest array) and **kwargs for keyword arguments (like a rest object).

*args — Rest Positional Arguments

JavaScript
// ...rest collects remaining args
function sum(...numbers) {
  return numbers.reduce((a, b) => a + b, 0);
}

sum(1, 2, 3);      // 6
sum(10, 20);       // 30

// Spread into function call
const nums = [1, 2, 3];
sum(...nums);      // 6
Python
# *args collects remaining positional args
def total(*numbers):
    return sum(numbers)  # numbers is a tuple


total(1, 2, 3)     # 6
total(10, 20)      # 30

# Spread into function call
nums = [1, 2, 3]
total(*nums)       # 6

**kwargs — Rest Keyword Arguments

JavaScript
// Destructured options object
function createServer(options) {
  const { host, port, debug, ...rest } = options;
  console.log(host, port, debug);
  console.log("Extra:", rest);
}

createServer({
  host: "localhost",
  port: 8000,
  debug: true,
  timeout: 30
});
Python
# **kwargs collects remaining keyword args
def create_server(**kwargs):
    host = kwargs.get("host", "localhost")
    port = kwargs.get("port", 8000)
    debug = kwargs.get("debug", False)
    print(host, port, debug)
    print("Extra:", kwargs)

create_server(
    host="localhost",
    port=8000,
    debug=True,
    timeout=30
)

Combining *args and **kwargs

# The full argument order (memorize this):
def func(positional, default="val", *args, **kwargs):
    print(positional)   # required arg
    print(default)      # has default
    print(args)         # tuple of extra positional args
    print(kwargs)       # dict of extra keyword args

func("a", "b", "c", "d", x=1, y=2)
# positional = "a"
# default = "b"
# args = ("c", "d")
# kwargs = {"x": 1, "y": 2}

Translation rule: ...rest (array) → *args (tuple). ...rest (object) → **kwargs (dict). Spread array into call: fn(...arr) → fn(*list). Spread object into call: fn({...obj}) → fn(**dict). One star for sequences, two stars for key-value pairs. Always.

Lambda — One-Line Arrow Functions

Python's lambda is a limited version of JavaScript's arrow function. It can only contain a single expression (no statements, no multi-line logic).

JavaScript
// Arrow functions — unlimited
const double = (n) => n * 2;
const add = (a, b) => a + b;
const greet = (name) => `Hi, ${name}`;

// Used inline constantly
const evens = nums.filter(n => n % 2 === 0);
const names = users.map(u => u.name);
users.sort((a, b) => a.age - b.age);

// Multi-line arrow (full function body)
const process = (data) => {
  const cleaned = data.trim();
  return cleaned.toUpperCase();
};
Python
# Lambda — single expression only
double = lambda n: n * 2
add = lambda a, b: a + b
greet = lambda name: f"Hi, {name}"

# Used inline
evens = list(filter(lambda n: n % 2 == 0, nums))
names = list(map(lambda u: u["name"], users))
users.sort(key=lambda u: u["age"])

# Multi-line? Use def (no lambda equivalent)
def process(data):
    cleaned = data.strip()
    return cleaned.upper()

In practice: Pythonistas rarely use lambda with map() and filter() because list comprehensions are more readable. The main use for lambda is in sort(key=lambda ...) and callback arguments. If you need more than one expression, just use def.

Docstrings — Built-in Documentation

Python has a first-class way to document functions. The first string in a function body becomes its docstring — accessible at runtime via help() and your editor's intellisense.

JavaScript / JSDoc
/**
 * Calculate the area of a rectangle.
 * @param {number} width - The width
 * @param {number} height - The height
 * @returns {number} The area
 */
function area(width, height) {
  return width * height;
}

// JSDoc is just a comment — not accessible
// at runtime (needs external tools)
Python
def area(width, height):
    """Calculate the area of a rectangle.

    Args:
        width: The width of the rectangle.
        height: The height of the rectangle.

    Returns:
        The area as a number.
    """
    return width * height

# Accessible at runtime!
help(area)         # prints the docstring
area.__doc__       # the raw string

The mental model: A docstring is a triple-quoted string as the first line of a function, class, or module. It's not a comment — it's a real string that Python attaches to the object. Your editor reads it for autocomplete tooltips. Django uses them for admin documentation. Always write them.

Type Hints — Python's TypeScript

Python has optional type annotations, just like TypeScript. They don't enforce anything at runtime (unless you use a tool like mypy), but they give you editor autocomplete, error checking, and documentation.

TypeScript
// Basic types
let name: string = "Trevor";
let age: number = 38;
let active: boolean = true;

// Function types
function greet(name: string): string {
  return `Hello, ${name}!`;
}

// Arrays
let scores: number[] = [95, 87, 92];

// Objects / interfaces
interface User {
  name: string;
  age: number;
  email?: string;  // optional
}

// Union types
let id: string | number = "abc";
Python (type hints)
# Basic types
name: str = "Trevor"
age: int = 38
active: bool = True

# Function types
def greet(name: str) -> str:
    return f"Hello, {name}!"

# Lists
scores: list[int] = [95, 87, 92]

# Dicts (closest to interfaces)
from typing import TypedDict

class User(TypedDict):
    name: str
    age: int
    email: str  # use NotRequired for optional

# Union types
id: str | int = "abc"  # Python 3.10+

Key difference: TypeScript types are enforced at compile time and stripped from the output. Python type hints are completely ignored at runtime — they're just metadata. Use mypy (a linter) to check them. Django and FastAPI use them heavily for auto-generating forms and API docs.

Common Type Patterns

TypeScript
// Optional (nullable)
function find(id: number): User | null { }

// Callback type
function apply(fn: (x: number) => number) { }

// Generic
function first<T>(arr: T[]): T { }

// Tuple
let pair: [string, number] = ["age", 38];
Python
# Optional (nullable)
def find(id: int) -> User | None: ...

# Callable type
from typing import Callable
def apply(fn: Callable[[int], int]): ...

# Generic
def first[T](arr: list[T]) -> T: ... # 3.12+

# Tuple
pair: tuple[str, int] = ("age", 38)

Translation rule: string → str. number → int or float. boolean → bool. null → None. T[] → list[T]. Return type : → ->. The biggest win: Django reads your type hints to auto-generate form validation and API schemas.

The Import System

Python's import is conceptually similar to JavaScript's, but with important differences in how modules are resolved.

JavaScript (ES Modules)
// Default import
import React from 'react';

// Named imports
import { useState, useEffect } from 'react';

// Import all as namespace
import * as d3 from 'd3';

// Rename on import
import { render as renderDOM } from 'react-dom';

// Dynamic import
const module = await import('./utils.js');

// Relative paths required for local files
import { helper } from './utils.js';
Python
# Import entire module
import os

# Named imports
from datetime import datetime, timedelta

# Import with alias
import numpy as np

# Rename on import
from os.path import join as path_join

# Dynamic import
module = __import__("utils")

# Relative imports (within a package)
from . import utils
from .utils import helper

No default exports! Python has no concept of export default. Every name you import is a named import. import os imports the module itself as a namespace (like import * as os). from os import path imports a specific name (like import { path } from 'os').

Module Organization

JavaScript
// File: utils/helpers.js
export function formatDate(d) { ... }
export function slugify(s) { ... }

// File: utils/index.js (barrel)
export { formatDate } from './helpers.js';

// Using it
import { formatDate } from './utils';

// index.js = default entry point
Python
# File: utils/helpers.py
def format_date(d): ...
def slugify(s): ...

# File: utils/__init__.py (barrel)
from .helpers import format_date

# Using it
from utils import format_date

# __init__.py = default entry point

Translation rule: index.js → __init__.py. A folder with an __init__.py file becomes a Python "package" (like a folder with index.js becomes a module in Node). The __init__.py can be empty — it just marks the folder as importable.

The Virtual Environment Workflow

# 1. Create a virtual environment (once per project)
python -m venv venv

# 2. Activate it (every terminal session)
source venv/bin/activate          # macOS/Linux
# Your prompt changes: (venv) $

# 3. Install packages (goes into venv, not global)
pip install django requests

# 4. Save your dependencies
pip freeze > requirements.txt

# 5. Deactivate when done
deactivate

# Teammate clones your repo:
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt   # installs everything

The mental model: A virtual environment is an isolated copy of Python with its own packages. It's what node_modules/ does for Node — keeps project dependencies separate. Without a venv, pip install installs globally, which is like npm install -g for everything. Always use a venv.

__name__ == "__main__"

This is a pattern you'll see in every Python file that's meant to be both imported and run directly. It's the "should I execute this code?" guard.

JavaScript
// No direct equivalent in JS/ESM
// In CommonJS Node.js:

// utils.js
function helper() { return 42; }

if (require.main === module) {
  // Only runs if this file is executed directly
  // Not when imported by another file
  console.log(helper());
}

module.exports = { helper };
Python
# utils.py
def helper():
    return 42

if __name__ == "__main__":
    # Only runs if: python utils.py
    # NOT when: from utils import helper
    print(helper())




# When run directly: __name__ = "__main__"
# When imported: __name__ = "utils"

The mental model: When Python runs a file directly, it sets __name__ to "__main__". When the file is imported, __name__ is set to the module name. The if __name__ == "__main__": guard means "only run this code if I'm the entry point, not when I'm being imported as a library." You'll see this in every Django management command.

First-Class Functions — Same as JS

Functions in Python are first-class objects, just like in JavaScript. You can pass them around, return them, and store them in data structures.

JavaScript
// Passing functions as arguments
function apply(fn, value) {
  return fn(value);
}

apply(Math.sqrt, 16);  // 4

// Returning functions (closures)
function multiplier(factor) {
  return (n) => n * factor;
}

const triple = multiplier(3);
triple(5);  // 15

// Functions in arrays
const ops = [Math.sqrt, Math.floor];
ops.forEach(fn => console.log(fn(4.7)));
Python
# Passing functions as arguments
def apply(fn, value):
    return fn(value)

import math
apply(math.sqrt, 16)   # 4.0

# Returning functions (closures)
def multiplier(factor):
    def inner(n):
        return n * factor
    return inner

triple = multiplier(3)
triple(5)  # 15

# Functions in lists
ops = [math.sqrt, math.floor]
for fn in ops: print(fn(4.7))

Key difference in closures: Python closures can read outer variables, but can't reassign them without the nonlocal keyword. In JavaScript, closures can freely modify outer let variables. In Python: nonlocal count then count += 1.

Decorators — A Preview

Decorators are heavily used in Django. They're functions that wrap other functions to add behavior. You'll see them everywhere.

JavaScript
// Higher-order function pattern
function withLogging(fn) {
  return function(...args) {
    console.log(`Calling ${fn.name}`);
    const result = fn(...args);
    console.log(`Result: ${result}`);
    return result;
  };
}

const add = withLogging((a, b) => a + b);
add(2, 3);
// Calling anonymous
// Result: 5
Python
# Decorator — same concept, cleaner syntax
def with_logging(fn):
    def wrapper(*args, **kwargs):
        print(f"Calling {fn.__name__}")
        result = fn(*args, **kwargs)
        print(f"Result: {result}")
        return result
    return wrapper

@with_logging          # the @ syntax!
def add(a, b):
    return a + b

add(2, 3)
# Calling add
# Result: 5

The mental model: @decorator above a function is just syntactic sugar for fn = decorator(fn). It's a higher-order function that wraps another function. In Django, you'll see @login_required, @csrf_exempt, @api_view — they all work this way. The @ symbol is not magic; it's just calling a function.

Check Your Understanding

Quick Quiz

1. What's wrong with def add_tag(tag, tags=[]) and how do you fix it?

The default [] is created once at definition time and shared across all calls. Fix: def add_tag(tag, tags=None): then if tags is None: tags = [] inside the function. This is Python's most infamous gotcha.

2. What's the Python equivalent of JavaScript's ...rest for collecting function arguments?

*args for positional arguments (collected as a tuple) and **kwargs for keyword arguments (collected as a dict). def f(*args, **kwargs) accepts anything.

3. What does if __name__ == "__main__": do?

It guards code that should only run when the file is executed directly (python file.py), not when it's imported by another file. When run directly, __name__ is "__main__"; when imported, it's the module name.

4. What's the Python equivalent of index.js in a module folder?

__init__.py — it marks a folder as an importable Python package and acts as the default entry point when you import the folder name. It can be empty or contain re-exports.

5. What does the @decorator syntax do?

@decorator above def fn(): is syntactic sugar for fn = decorator(fn). It passes your function into the decorator and replaces it with the return value. It's a higher-order function pattern with cleaner syntax.
