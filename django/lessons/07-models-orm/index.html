07 Models & ORM — Prisma to Django ORM

← Back to lessons

You already know Prisma schemas and database queries. This lesson maps every Prisma concept to its Django equivalent — models are your schema, the ORM is your query client, and migrations work almost identically. You'll feel right at home.

The Mental Model — What Are Models?

In Prisma, you define your database schema in a .prisma file, then Prisma generates a client you use to query. Django flips the order: you define your schema as Python classes (called models), and Django generates both the database tables and the query API from those classes.

Models are the M in Django's MTV (Model-Template-View) pattern. Think of it this way:

ConceptPrisma / Next.jsDjango
Schema definitionschema.prisma filePython classes in models.py
Query clientprisma.user.findMany()User.objects.all()
Migrationsnpx prisma migrate devpython manage.py makemigrations + migrate
Database viewerPrisma StudioDjango Admin (built-in, way more powerful)

The key insight: A Django model is a single Python class that is simultaneously your schema definition, your TypeScript type, AND your query builder. One class does the job of three separate things in the Prisma/Next.js world. Every model class maps to one database table. Every attribute maps to one column.

Defining Models — Your Schema in Python

Let's build a newsroom database. We'll start with an Article model and compare it directly to what you'd write in Prisma.

Prisma
model Article {
  id        Int      @id @default(autoincrement())
  title     String
  slug      String   @unique
  body      String
  status    String   @default("draft")
  wordCount Int?
  publishedAt DateTime?
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}
Django (models.py)
# newsroom/models.py
from django.db import models

class Article(models.Model):
    title = models.CharField(max_length=200)
    slug = models.SlugField(unique=True)
    body = models.TextField()
    status = models.CharField(
        max_length=20, default="draft"
    )
    word_count = models.IntegerField(
        null=True, blank=True
    )
    published_at = models.DateTimeField(
        null=True, blank=True
    )
    created_at = models.DateTimeField(
        auto_now_add=True
    )
    updated_at = models.DateTimeField(
        auto_now=True
    )

Where's the id? Django automatically adds an id field (auto-incrementing integer primary key) to every model. You never need to define it yourself unless you want a custom primary key. Same concept as Prisma's @id @default(autoincrement()), just invisible.

null vs blank — a Django gotcha. null=True means the database column allows NULL. blank=True means Django forms will accept an empty value. For optional text fields, you typically want both. For optional numbers and dates, you only need null=True. This two-flag system trips up everyone at first.

Common Field Options

# Options you'll use constantly
title = models.CharField(
    max_length=200,         # required for CharField
    unique=True,            # Prisma's @unique
    db_index=True,          # Prisma's @@index
    default="Untitled",     # Prisma's @default
    null=True,              # Prisma's ? (nullable)
    blank=True,             # allow empty in forms
    help_text="The headline",  # shows in admin UI
    verbose_name="Headline",   # human-readable label
    choices=[               # dropdown in admin
        ("draft", "Draft"),
        ("published", "Published"),
        ("archived", "Archived"),
    ],
)

The choices pattern: Instead of a separate enum, Django uses a list of (db_value, display_label) tuples directly on the field. The first value goes in the database, the second shows up in forms and the admin panel. It's inline and simple — no extra file or import needed.

The Meta Class — Table-Level Options

Prisma uses @@ decorators for table-level settings (indexes, unique constraints, table name). Django uses an inner Meta class.

Prisma
model Article {
  // ... fields ...

  @@map("articles")
  @@unique([slug, status])
  @@index([published_at])
}
Django
class Article(models.Model):
    # ... fields ...

    class Meta:
        db_table = "articles"
        unique_together = [
            ["slug", "status"]
        ]
        indexes = [
            models.Index(
                fields=["published_at"]
            ),
        ]
        ordering = ["-created_at"]
        verbose_name_plural = "Articles"

ordering is a lifesaver. Set it once in Meta, and every query returns results in that order by default. The - prefix means descending. No more adding .order_by() to every single query. Prisma has no equivalent — you must pass orderBy every time.

The __str__ Method — Your Model's Display Name

Always define __str__ on your models. It controls how objects appear in the admin panel, the shell, and debug output. Without it, you'll see Article object (1) everywhere.

class Article(models.Model):
    title = models.CharField(max_length=200)
    # ... other fields ...

    def __str__(self):
        return self.title

    # Now the admin shows "Breaking News: City Budget Approved"
    # instead of "Article object (3)"

Migrations — Schema Changes Without Tears

Migrations are how Django syncs your Python model classes with the actual database schema. If you've used Prisma migrations, the concept is identical: detect changes, generate a migration file, apply it.

Prisma
# 1. Edit schema.prisma
# 2. Generate + apply migration
npx prisma migrate dev \
  --name add_articles

# That single command does both
# steps. Prisma figures it out.
Django
# 1. Edit models.py
# 2. Generate migration file
python manage.py makemigrations

# 3. Apply migration to database
python manage.py migrate

# Two steps: generate, then apply.
# You can review the migration
# file before applying it.

Why two steps? Django separates "detect changes" (makemigrations) from "apply changes" (migrate) so you can inspect the generated migration file before touching the database. In production, you commit migration files to git and run migrate on deploy. This is exactly like Prisma's migration files, but you get to review them first.

The typical workflow:

# You changed a model (added a field, renamed something, etc.)

$ python manage.py makemigrations
# Migrations for 'newsroom':
#   newsroom/migrations/0002_article_word_count.py
#     - Add field word_count to article

$ python manage.py migrate
# Operations to perform:
#   Apply all migrations: newsroom
# Running migrations:
#   Applying newsroom.0002_article_word_count... OK

Common mistake: Forgetting to run makemigrations before migrate. If you change a model and only run migrate, nothing happens because Django doesn't know about your changes yet. Always: edit model, makemigrations, migrate. In that order.

The ORM — CRUD Operations

Django's ORM is the query builder that comes free with every model. Prisma gives you prisma.article.create(). Django gives you Article.objects.create(). The .objects manager is your entry point for all queries — think of it as the equivalent of prisma.article.

Creating Objects

Prisma
// Create one record
const article = await prisma.article.create({
  data: {
    title: "City Budget Approved",
    slug: "city-budget-approved",
    body: "The city council...",
    status: "published",
  },
});

// Create and return the object
console.log(article.id); // auto-set
console.log(article.createdAt);
Django
# Method 1: .create() — one line
article = Article.objects.create(
    title="City Budget Approved",
    slug="city-budget-approved",
    body="The city council...",
    status="published",
)

# Already saved to the database
print(article.id)         # auto-set
print(article.created_at)
# Method 2: build, then save (like Prisma's build pattern)
article = Article(
    title="New Park Opens Downtown",
    slug="new-park-opens",
    body="A ribbon-cutting ceremony...",
)
# Object exists in memory but NOT in the database yet
article.status = "draft"   # can modify before saving
article.save()             # NOW it hits the database

.create() vs .save(): Use .create() when you have all the data upfront — it's one step. Use the constructor + .save() when you need to build the object incrementally or run logic before saving. Both achieve the same result: a new row in the database.

Querying — Reading Data

This is where Django's ORM really shines. Every query returns a QuerySet — a lazy, chainable, database query that doesn't execute until you actually need the data. Think of it like a Promise that represents a SQL query.

Prisma
// Get all records
const articles = await prisma.article
  .findMany();

// Get one by unique field
const article = await prisma.article
  .findUnique({
    where: { slug: "city-budget" },
  });

// Filter records
const published = await prisma.article
  .findMany({
    where: { status: "published" },
  });

// Exclude records
const notDrafts = await prisma.article
  .findMany({
    where: {
      NOT: { status: "draft" },
    },
  });
Django
# Get all records
articles = Article.objects.all()


# Get one by unique field
article = Article.objects.get(
    slug="city-budget"
)


# Filter records
published = Article.objects.filter(
    status="published"
)


# Exclude records
not_drafts = Article.objects.exclude(
    status="draft"
)

QuerySets are lazy. Article.objects.filter(status="published") does NOT hit the database. It builds a query. The SQL only runs when you iterate, slice, call list(), or access results. This means you can chain multiple filters without triggering multiple database queries. It's like building up a Prisma query object before awaiting it.

.get() vs .filter(): .get() returns a single object and raises DoesNotExist if nothing matches (or MultipleObjectsReturned if more than one matches). .filter() returns a QuerySet (possibly empty). Use .get() when you expect exactly one result. Use .filter() when you expect zero or more.

Field Lookups — The Double-Underscore Magic

In Prisma, you nest filter conditions inside objects: { contains: "budget" }. In Django, you use double-underscore suffixes on field names. This is one of Django's most distinctive patterns.

Prisma
// Contains
where: { title: { contains: "budget" } }

// Greater than
where: { word_count: { gt: 1000 } }

// Less than or equal
where: { word_count: { lte: 500 } }

// In list
where: { status: { in: ["draft", "review"] } }

// Is null
where: { published_at: null }

// Starts with
where: { title: { startsWith: "Breaking" } }

// Date range
where: {
  created_at: {
    gte: new Date("2025-01-01"),
    lt: new Date("2026-01-01"),
  }
}
Django
# Contains
.filter(title__contains="budget")

# Greater than
.filter(word_count__gt=1000)

# Less than or equal
.filter(word_count__lte=500)

# In list
.filter(status__in=["draft", "review"])

# Is null
.filter(published_at__isnull=True)

# Starts with
.filter(title__startswith="Breaking")

# Date range
.filter(
    created_at__gte="2025-01-01",
    created_at__lt="2026-01-01",
)

Full lookup reference:

LookupDjango SyntaxSQL Equivalent
Exact matchfield__exact=val (or just field=val)WHERE field = val
Case-insensitivefield__iexact=valWHERE LOWER(field) = LOWER(val)
Containsfield__contains=valWHERE field LIKE '%val%'
Case-insensitive containsfield__icontains=valWHERE field ILIKE '%val%'
Starts withfield__startswith=valWHERE field LIKE 'val%'
Ends withfield__endswith=valWHERE field LIKE '%val'
Greater thanfield__gt=valWHERE field > val
Greater/equalfield__gte=valWHERE field >= val
Less thanfield__lt=valWHERE field < val
Less/equalfield__lte=valWHERE field <= val
In listfield__in=[a, b, c]WHERE field IN (a, b, c)
Is nullfield__isnull=TrueWHERE field IS NULL
Date rangefield__range=(start, end)WHERE field BETWEEN start AND end
Yearfield__year=2025WHERE YEAR(field) = 2025

The double-underscore pattern: field__lookup=value. The field name, two underscores, then the lookup type. It reads like English: title__contains="budget" means "title contains budget." Once this clicks, you'll never want to go back to nested filter objects. The i prefix (like icontains, iexact) means case-insensitive — incredibly useful for search.

Chaining QuerySets

Because QuerySets are lazy, you can chain filters together. Each chain narrows the results further. The database query only runs once, at the end.

Prisma
// Build up complex filters
const results = await prisma.article
  .findMany({
    where: {
      AND: [
        { status: "published" },
        { word_count: { gt: 500 } },
        { title: {
            contains: "city"
          }
        },
      ],
    },
    orderBy: { published_at: "desc" },
    take: 10,
  });
Django
# Chain filters — AND by default
results = (
    Article.objects
    .filter(status="published")
    .filter(word_count__gt=500)
    .filter(title__icontains="city")
    .order_by("-published_at")
    [:10]
)

# Or combine in one .filter() call:
results = Article.objects.filter(
    status="published",
    word_count__gt=500,
    title__icontains="city",
).order_by("-published_at")[:10]

Multiple .filter() calls are AND. .filter(a=1).filter(b=2) is WHERE a=1 AND b=2. You can also pass multiple keyword arguments to a single .filter() for the same effect. For OR queries, use Django's Q objects (covered below).

OR Queries with Q Objects

Prisma uses OR: [...] inside where. Django uses Q objects combined with the | (pipe) operator.

from django.db.models import Q

# Articles that are published OR have more than 1000 words
Article.objects.filter(
    Q(status="published") | Q(word_count__gt=1000)
)

# Complex: (published AND recent) OR (draft AND by me)
Article.objects.filter(
    (Q(status="published") & Q(created_at__year=2025)) |
    (Q(status="draft") & Q(author__name="Trevor"))
)

# NOT: exclude published articles
Article.objects.filter(~Q(status="published"))

Q objects are Django's boolean algebra for queries. | for OR, & for AND, ~ for NOT. They let you build any SQL WHERE clause, no matter how complex. You only need Q objects for OR — simple AND filters don't need them.

Updating Objects

Prisma
// Update one record
await prisma.article.update({
  where: { id: 1 },
  data: { status: "published" },
});

// Update many records
await prisma.article.updateMany({
  where: { status: "draft" },
  data: { status: "archived" },
});
Django
# Update one: fetch, modify, save
article = Article.objects.get(id=1)
article.status = "published"
article.save()

# Bulk update (no fetch needed!)
Article.objects.filter(
    status="draft"
).update(status="archived")

Bulk .update() is fast. It runs a single SQL UPDATE statement. It does NOT load objects into Python, does NOT call .save(), and does NOT trigger signals. Use it for mass updates. Use the fetch-modify-save pattern when you need model logic (custom save methods, signals) to run.

Deleting Objects

Prisma
// Delete one
await prisma.article.delete({
  where: { id: 1 },
});

// Delete many
await prisma.article.deleteMany({
  where: { status: "archived" },
});
Django
# Delete one
article = Article.objects.get(id=1)
article.delete()

# Delete many (bulk)
Article.objects.filter(
    status="archived"
).delete()

Django cascades by default. If you delete an Author who has articles, Django will delete all their articles too (like Prisma's onDelete: Cascade). You can change this behavior on the ForeignKey field with the on_delete parameter. Always set on_delete intentionally.

Relationships — Connecting Models

This is where things get interesting. Prisma models relationships with @relation decorators. Django models them with dedicated field types. The concepts are identical; the syntax is different.

ForeignKey — One-to-Many (like Prisma's @relation)

Prisma
model Author {
  id       Int       @id @default(autoincrement())
  name     String
  articles Article[]
}

model Article {
  id       Int    @id @default(autoincrement())
  title    String
  author   Author @relation(fields: [authorId],
                   references: [id])
  authorId Int
}
Django
class Author(models.Model):
    name = models.CharField(max_length=100)

    def __str__(self):
        return self.name


class Article(models.Model):
    title = models.CharField(max_length=200)
    author = models.ForeignKey(
        Author,
        on_delete=models.CASCADE,
        related_name="articles",
    )

Django is way less boilerplate here. In Prisma, you define both sides of the relationship (the Article[] on Author AND the @relation on Article) plus the foreign key column. In Django, you define the ForeignKey on one side and Django figures out the rest. The related_name="articles" gives you the reverse lookup: author.articles.all().

# Using ForeignKey relationships

# Forward: get the author of an article
article = Article.objects.get(id=1)
article.author          # Author object
article.author.name     # "Jane Smith"

# Reverse: get all articles by an author
author = Author.objects.get(name="Jane Smith")
author.articles.all()   # QuerySet of all Jane's articles

# Create with relationship
Article.objects.create(
    title="New Story",
    author=author,        # pass the Author object
)

# Or use the ID directly
Article.objects.create(
    title="New Story",
    author_id=author.id,  # Django auto-creates _id field
)

# Filter across relationships (double-underscore!)
Article.objects.filter(author__name="Jane Smith")
Article.objects.filter(author__name__icontains="jane")

The double-underscore crosses relationships. author__name means "follow the author FK, then look at the name field." You can chain across multiple relationships: article__author__email__contains="@nyt". This is one of Django ORM's most powerful features — Prisma needs nested where objects for the same thing.

on_delete Options

DjangoPrisma EquivalentWhat It Does
models.CASCADEonDelete: CascadeDelete article when author is deleted
models.PROTECTonDelete: RestrictPrevent deleting author who has articles
models.SET_NULLonDelete: SetNullSet author to NULL (field must be nullable)

ManyToManyField — Many-to-Many

Articles can have multiple categories, and categories can contain multiple articles. In Prisma, you'd model this with an implicit many-to-many relation. Django handles it the same way.

Prisma
model Article {
  id         Int        @id @default(autoincrement())
  title      String
  categories Category[]
}

model Category {
  id       Int       @id @default(autoincrement())
  name     String    @unique
  articles Article[]
}

// Prisma auto-creates a join table
Django
class Category(models.Model):
    name = models.CharField(
        max_length=50, unique=True
    )

    def __str__(self):
        return self.name


class Article(models.Model):
    title = models.CharField(max_length=200)
    categories = models.ManyToManyField(
        Category,
        related_name="articles",
        blank=True,
    )
# Django auto-creates a join table too
# Working with many-to-many relationships

article = Article.objects.get(id=1)
politics = Category.objects.get(name="Politics")
local = Category.objects.get(name="Local News")

# Add categories to an article
article.categories.add(politics, local)

# Remove a category
article.categories.remove(politics)

# Set exact categories (replaces all existing)
article.categories.set([politics, local])

# Clear all categories
article.categories.clear()

# Query: all categories for an article
article.categories.all()

# Reverse: all articles in a category
politics.articles.all()

# Filter across M2M: articles in the "Politics" category
Article.objects.filter(categories__name="Politics")

M2M methods: .add(), .remove(), .set(), .clear(). These four methods manage the join table for you. In Prisma, you'd use connect, disconnect, and set inside an update call. Django's approach is more direct — you call methods on the relationship itself.

OneToOneField — One-to-One

Use OneToOneField when each record in one table maps to exactly one record in another. Common for profile/settings extensions.

from django.contrib.auth.models import User

class AuthorProfile(models.Model):
    user = models.OneToOneField(
        User,
        on_delete=models.CASCADE,
        related_name="profile",
    )
    bio = models.TextField(blank=True)
    twitter_handle = models.CharField(
        max_length=50, blank=True
    )

# Usage: like ForeignKey but returns object, not QuerySet
user = User.objects.get(username="trevor")
user.profile             # AuthorProfile object (not a QuerySet)
user.profile.bio         # "Journalist turned developer"

profile = AuthorProfile.objects.get(id=1)
profile.user             # User object
profile.user.email       # "trevor@example.com"

Putting It Together — The Newsroom Database

Here's the complete model set for a newsroom app. This is what a real Django models.py looks like.

# newsroom/models.py
from django.db import models
from django.contrib.auth.models import User


class Category(models.Model):
    name = models.CharField(max_length=50, unique=True)
    slug = models.SlugField(unique=True)
    description = models.TextField(blank=True)

    class Meta:
        verbose_name_plural = "Categories"
        ordering = ["name"]

    def __str__(self):
        return self.name


class Author(models.Model):
    user = models.OneToOneField(
        User, on_delete=models.CASCADE, related_name="author_profile"
    )
    bio = models.TextField(blank=True)
    headshot = models.URLField(blank=True)

    def __str__(self):
        return self.user.get_full_name() or self.user.username


class Article(models.Model):

    STATUS_CHOICES = [
        ("draft", "Draft"),
        ("review", "In Review"),
        ("published", "Published"),
        ("archived", "Archived"),
    ]

    title = models.CharField(max_length=200)
    slug = models.SlugField(unique=True)
    body = models.TextField()
    summary = models.TextField(max_length=500, blank=True)
    status = models.CharField(
        max_length=20, choices=STATUS_CHOICES, default="draft"
    )
    word_count = models.IntegerField(default=0)
    author = models.ForeignKey(
        Author, on_delete=models.CASCADE, related_name="articles"
    )
    categories = models.ManyToManyField(
        Category, related_name="articles", blank=True
    )
    published_at = models.DateTimeField(null=True, blank=True)
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        ordering = ["-published_at", "-created_at"]
        indexes = [
            models.Index(fields=["status", "published_at"]),
            models.Index(fields=["slug"]),
        ]

    def __str__(self):
        return self.title

QuerySet Power Tools

Beyond basic CRUD, Django's QuerySets have methods that handle sorting, aggregation, and data shaping — things that would require raw SQL or extra libraries in many other frameworks.

Ordering

# Order by field (ascending)
Article.objects.order_by("title")

# Descending: prefix with -
Article.objects.order_by("-published_at")

# Multiple fields: first by status, then by date
Article.objects.order_by("status", "-published_at")

# Random order (useful for "featured" picks)
Article.objects.order_by("?")

Limiting & Slicing

Prisma
// First 10
prisma.article.findMany({ take: 10 })

// Skip 10, take 10 (pagination)
prisma.article.findMany({
  skip: 10,
  take: 10,
})

// First matching record
prisma.article.findFirst({
  where: { status: "published" },
})
Django
# First 10 (Python slice syntax!)
Article.objects.all()[:10]

# Skip 10, take 10 (pagination)
Article.objects.all()[10:20]


# First matching record
Article.objects.filter(
    status="published"
).first()

QuerySets use Python slice syntax for LIMIT/OFFSET. [:10] is LIMIT 10. [10:20] is OFFSET 10 LIMIT 10. You already know this syntax from Python lists. Django translates it to efficient SQL behind the scenes.

.values() and .values_list() — Selecting Specific Columns

Prisma
// Select specific fields
await prisma.article.findMany({
  select: {
    title: true,
    status: true,
  },
});
// [{title: "...", status: "..."}]

// Get just one field as array
const titles = articles
  .map(a => a.title);
Django
# Returns list of dicts
Article.objects.values(
    "title", "status"
)

# [{"title": "...", "status": "..."}]

# Returns list of tuples
Article.objects.values_list("title")
# [("Title 1",), ("Title 2",)]

# Flat list of single field
Article.objects.values_list(
    "title", flat=True
)
# ["Title 1", "Title 2", "Title 3"]

Aggregation — Database-Level Math

from django.db.models import Count, Avg, Sum, Max, Min

# Aggregate: returns a single dict of computed values
Article.objects.aggregate(
    total=Count("id"),
    avg_words=Avg("word_count"),
    max_words=Max("word_count"),
    total_words=Sum("word_count"),
)
# {"total": 150, "avg_words": 820.5, "max_words": 3200, "total_words": 123075}

# Annotate: adds a computed column to each row
authors = Author.objects.annotate(
    article_count=Count("articles")
)
for author in authors:
    print(f"{author} has written {author.article_count} articles")

# Group by: annotate + values = SQL GROUP BY
Article.objects.values("status").annotate(
    count=Count("id"),
    avg_length=Avg("word_count"),
).order_by("status")
# [{"status": "draft", "count": 23, "avg_length": 450.2},
#  {"status": "published", "count": 98, "avg_length": 1120.8}]

.aggregate() vs .annotate(): Think of aggregate as a "summary" — it collapses the entire QuerySet into a single dictionary. Think of annotate as "add a column" — it attaches a computed value to each object in the QuerySet. When you combine .values() with .annotate(), you get SQL GROUP BY. This is where Django's ORM can replace most raw SQL you'd ever write.

Other Useful QuerySet Methods

# Count without loading objects
Article.objects.filter(status="published").count()  # 98

# Check if any records exist
Article.objects.filter(status="published").exists()  # True

# Distinct values (SQL DISTINCT)
Article.objects.values_list("author__name", flat=True).distinct()

# Get or create (avoids duplicate checks)
category, created = Category.objects.get_or_create(
    name="Breaking News",
    defaults={"slug": "breaking-news"},
)
# created is True if it was just made, False if it already existed

# Update or create (upsert)
article, created = Article.objects.update_or_create(
    slug="city-budget-approved",
    defaults={
        "title": "City Budget Approved (Updated)",
        "status": "published",
    },
)

# Bulk create (insert many rows efficiently)
Article.objects.bulk_create([
    Article(title="Story 1", slug="story-1", body="...", author=author),
    Article(title="Story 2", slug="story-2", body="...", author=author),
    Article(title="Story 3", slug="story-3", body="...", author=author),
])

get_or_create and update_or_create are Django's version of Prisma's upsert. They're incredibly useful for import scripts, API endpoints, and seeding data. The defaults dict contains fields that get set on creation (or on update for update_or_create), while the other kwargs are used for the lookup.

Real-World Query Examples

Here are queries you'd actually write in a newsroom app, showing how naturally the ORM handles journalism-domain problems:

# Homepage: latest 10 published articles with their authors
Article.objects.filter(
    status="published"
).select_related("author").order_by("-published_at")[:10]

# Author page: all articles by a specific author
author = Author.objects.get(user__username="trevor")
author.articles.filter(status="published").order_by("-published_at")

# Category page: articles in "Politics" category
Article.objects.filter(
    categories__name="Politics",
    status="published",
).distinct()

# Search: articles matching a keyword
Article.objects.filter(
    Q(title__icontains=query) | Q(body__icontains=query),
    status="published",
)

# Dashboard: article counts by status
Article.objects.values("status").annotate(
    count=Count("id")
).order_by("status")

# Most prolific authors (top 5 by article count)
Author.objects.annotate(
    num_articles=Count("articles")
).order_by("-num_articles")[:5]

# Articles published this month
from django.utils import timezone
this_month = timezone.now().replace(day=1)
Article.objects.filter(
    published_at__gte=this_month,
    status="published",
)

select_related prevents N+1 queries. Without it, accessing article.author in a loop fires a separate SQL query for each article. select_related("author") does a SQL JOIN so all the data comes in one query. Use select_related for ForeignKey/OneToOne and prefetch_related for ManyToMany. This is like Prisma's include.

Check Your Understanding

Quick Quiz

1. In Prisma you write prisma.article.findMany({ where: { status: "published" } }). What's the Django equivalent?

Article.objects.filter(status="published") — The .objects manager is your query entry point (like prisma.article), and .filter() is findMany with a where clause.

2. How do you query articles whose author's name contains "Jane" (filtering across a relationship)?

Article.objects.filter(author__name__icontains="jane") — The double-underscore crosses the ForeignKey relationship (author__name) and then applies the lookup (__icontains).

3. What's the difference between .aggregate() and .annotate()?

.aggregate() collapses the entire QuerySet into a single summary dict (e.g., total count, average). .annotate() adds a computed value to each object in the QuerySet (e.g., adding an article_count to each Author).

4. What two commands do you run after changing a model in Django?

python manage.py makemigrations (detect changes and create migration file) then python manage.py migrate (apply migration to database). Always in that order. makemigrations is like prisma migrate dev --create-only, and migrate is like prisma migrate deploy.

5. You have a ForeignKey from Article to Author. How do you get all articles by a specific author using the reverse relationship?

author.articles.all() — The related_name="articles" on the ForeignKey creates the reverse accessor. Without related_name, you'd use author.article_set.all() (Django's default naming convention).

6. What does select_related("author") do, and why is it important?

It performs a SQL JOIN to fetch the related Author data in the same query. Without it, accessing article.author in a loop fires a separate database query per article (the N+1 problem). It's Django's version of Prisma's include: { author: true }.
