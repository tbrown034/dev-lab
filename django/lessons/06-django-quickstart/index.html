06 Django Quickstart — Next.js to Django

← Back to lessons

You know Next.js. Django solves the same problems differently. This lesson maps every Django concept to something you already understand from Next.js.

What is Django?

Django is a "batteries included" Python web framework. It gives you routing, an ORM (database layer), templates, authentication, an admin panel, form handling, and more — all built-in.

If Next.js is "React + routing + API routes + SSR," Django is "Python + routing + ORM + templates + auth + admin + forms + ..." It ships with even MORE built-in features than Next.js.

Next.js gives you
  • File-based routing
  • API routes (server functions)
  • SSR / SSG
  • Middleware
  • Image optimization
  • You add: Prisma, NextAuth, Zod...
Django gives you
  • URL routing (explicit)
  • Views (like API routes)
  • Server-rendered templates
  • Middleware
  • ORM (like Prisma, built-in)
  • Auth, admin panel, forms, CSRF...

The mental model: Django is like if Next.js came with Prisma, NextAuth, React Hook Form, and an auto-generated admin dashboard all baked in. The tradeoff? You get more out of the box, but you follow Django's patterns (not your own).

Project vs. App

Django has a unique concept that Next.js doesn't: the project/app split. Understanding this is crucial.

  • Project = the entire website. Like your Next.js app. Contains settings, root URL config.
  • App = a feature module within the project. Like a route group in Next.js (e.g., (blog)/, (auth)/, (dashboard)/).

Creating a Project

Next.js
# Create a new project
npx create-next-app@latest mysite

# Start dev server
cd mysite
npm run dev
Django
# Create a new project
django-admin startproject mysite

# Start dev server
cd mysite
python manage.py runserver

Creating an App (Feature Module)

# Inside your project directory
python manage.py startapp blog     # Creates a "blog" feature module
python manage.py startapp accounts # Creates an "accounts" feature module
python manage.py startapp api      # Creates an "api" feature module

Think of it like this: Your Next.js app is the project. Each route group or feature area (blog, auth, dashboard) would be a Django "app." A single Django project usually has 3-10 apps. Each app has its own models, views, URLs, and templates — nicely isolated.

File Structure

After running django-admin startproject mysite and python manage.py startapp blog, here's what you get:

mysite/ ............... project root (like your Next.js root)
  manage.py ............ CLI tool — like package.json scripts
  mysite/ .............. project config (same name, nested)
    __init__.py ........ marks this as a Python package
    settings.py ........ next.config.js + .env combined
    urls.py ............ root router — like app/layout.tsx routing
    wsgi.py ............ production server entry point
    asgi.py ............ async server entry point
  blog/ ................ your "blog" app
    __init__.py
    models.py .......... Prisma schema equivalent
    views.py ........... API routes / page handlers
    urls.py ............ app-level routes (you create this)
    admin.py ........... admin panel config
    apps.py ............ app metadata
    tests.py ........... test file
    migrations/ ........ database migrations (like prisma migrate)

File-by-File Translation

Django File Next.js Equivalent What It Does
manage.py package.json scripts CLI commands: runserver, migrate, createsuperuser, etc.
settings.py next.config.js + .env All configuration: database, installed apps, middleware, secrets
urls.py app/ directory structure URL routing — but explicit, not file-based
views.py route.ts / page.tsx Request handlers. Return HTML or JSON.
models.py schema.prisma Database schema as Python classes
admin.py No equivalent! Free admin dashboard — register a model, get full CRUD UI
migrations/ prisma/migrations/ Database migration history

The admin panel is Django's secret weapon. Register a model and Django auto-generates a full CRUD interface — create, read, update, delete — with search, filters, pagination. It's like getting a free Retool/Forest Admin for every model. Journalists love it for data entry.

Your First View

A Django "view" is a function that takes a request and returns a response. If you've written a Next.js API route, you already understand this.

Next.js API Route
// app/api/hello/route.ts
import { NextResponse } from 'next/server';

export async function GET(request: Request) {
  return NextResponse.json({
    message: 'Hello, World!'
  });
}
Django View
# blog/views.py
from django.http import JsonResponse

def hello(request):
    return JsonResponse({
        'message': 'Hello, World!'
    })

They're structurally identical: a function that receives a request object and returns a response. The difference is how you connect them to URLs.

Returning HTML Instead of JSON

Next.js Page Component
// app/page.tsx
export default function Home() {
  return (
    <div>
      <h1>Welcome!</h1>
      <p>This is my site.</p>
    </div>
  );
}
Django View + Template
# blog/views.py
from django.shortcuts import render

def home(request):
    return render(request, 'home.html', {
        'title': 'Welcome!'
    })

The key insight: In Next.js, the JSX IS the template (component = template + logic). In Django, they're separated: the view handles logic and the template handles HTML. This is the classic MVC/MVT pattern — Model, View, Template.

URL Routing

Next.js uses file-based routing — the file path IS the URL. Django uses explicit routing — you define URL patterns in urls.py files.

Next.js (File-Based)
File structure = URL pattern:

app/page.tsx           → /
app/about/page.tsx     → /about
app/blog/page.tsx      → /blog
app/blog/[slug]/page.tsx → /blog/:slug
app/api/posts/route.ts → /api/posts
Django (Explicit)
# mysite/urls.py (root)
from django.urls import path, include

urlpatterns = [
    path('', views.home),           # /
    path('about/', views.about),    # /about/
    path('blog/', include('blog.urls')), # /blog/*
]

Two-Level Routing

Django typically uses two levels of URL configuration:

# 1. Root URLs — mysite/urls.py
# This is like your app/layout.tsx that includes all route groups
from django.urls import path, include
from django.contrib import admin

urlpatterns = [
    path('admin/', admin.site.urls),         # Built-in admin
    path('blog/', include('blog.urls')),     # Blog app routes
    path('api/', include('api.urls')),       # API app routes
]
# 2. App URLs — blog/urls.py (you create this file)
# These are all prefixed with /blog/ from the root config
from django.urls import path
from . import views

urlpatterns = [
    path('', views.blog_index, name='blog-index'),           # /blog/
    path('<slug:slug>/', views.blog_detail, name='blog-detail'), # /blog/my-post/
]

URL parameters: Next.js uses [slug] in file names. Django uses <slug:slug> in the URL pattern. Same concept — capturing a dynamic segment. Django also has <int:id>, <str:name>, <uuid:pk> with built-in type validation.

Connecting Views to URLs

Here's the full flow — from URL request to view function to response:

# blog/views.py
from django.shortcuts import render
from django.http import JsonResponse

def blog_index(request):
    # Like: export default function BlogPage()
    posts = [
        {"title": "First Post", "slug": "first-post"},
        {"title": "Second Post", "slug": "second-post"},
    ]
    return render(request, 'blog/index.html', {'posts': posts})

def blog_detail(request, slug):
    # slug comes from the URL pattern <slug:slug>
    # Like: export default function PostPage({ params }: { params: { slug: string } })
    return render(request, 'blog/detail.html', {'slug': slug})

Translation: In Next.js, params.slug comes from the file path. In Django, slug is a function argument that comes from the URL pattern. Both achieve the same thing — the URL /blog/my-article/ passes "my-article" to your handler.

Templates

Django templates are HTML files with special tags for dynamic content. Think of them as JSX but server-rendered, with different syntax for variables and logic.

React/JSX
<div>
  <h1>{title}</h1>
  <ul>
    {posts.map(post => (
      <li key={post.id}>
        <a href={`/blog/${post.slug}`}>
          {post.title}
        </a>
      </li>
    ))}
  </ul>
  {posts.length === 0 && (
    <p>No posts yet.</p>
  )}
</div>
Django Template
<div>
  <h1>{{ title }}</h1>
  <ul>
    {% for post in posts %}
      <li>
        <a href="/blog/{{ post.slug }}">
          {{ post.title }}
        </a>
      </li>
    {% endfor %}
  </ul>
  {% if not posts %}
    <p>No posts yet.</p>
  {% endif %}
</div>

Syntax map: JSX {variable} → Django {{ variable }}. JSX {condition && ...} → Django {% if condition %}...{% endif %}. JSX {arr.map(...)} → Django {% for item in arr %}...{% endfor %}. Double curly braces for values, curly-percent for logic.

Template Inheritance (Like Layouts)

Next.js has layout.tsx files that wrap child pages. Django has template inheritance — same concept, different syntax.

Next.js Layout
// app/layout.tsx
export default function Layout({
  children
}: {
  children: React.ReactNode
}) {
  return (
    <html>
      <body>
        <nav>My Site</nav>
        {children}
        <footer>2024</footer>
      </body>
    </html>
  );
}
Django Base Template
<!-- templates/base.html -->
<html>
  <body>
    <nav>My Site</nav>
    {% block content %}
    <!-- child content goes here -->
    {% endblock %}
    <footer>2024</footer>
  </body>
</html>
<!-- templates/blog/index.html — "extends" is like being wrapped by the layout -->
{% extends "base.html" %}

{% block content %}
  <h1>Blog</h1>
  <p>This replaces the {% block content %} in base.html</p>
{% endblock %}

Translation: Next.js layout.tsx with {children} → Django base.html with {% block content %}{% endblock %}. Child pages use {% extends "base.html" %} instead of being auto-wrapped. More explicit, but the same "shell + content" pattern.

Settings

settings.py is the nerve center of your Django project. It combines what Next.js splits between next.config.js, .env, and various config files.

# mysite/settings.py — the important parts

# Like NODE_ENV === 'development'
DEBUG = True

# Like NEXTAUTH_URL / allowed origins
ALLOWED_HOSTS = ['localhost', '127.0.0.1', 'mysite.com']

# Like your package.json dependencies (but for Django apps)
INSTALLED_APPS = [
    'django.contrib.admin',       # Admin panel — free!
    'django.contrib.auth',        # Authentication — free!
    'django.contrib.contenttypes',
    'django.contrib.sessions',
    'django.contrib.messages',
    'django.contrib.staticfiles', # Static file serving
    'blog',                       # Your apps go here
    'api',
]

# Like your Prisma DATABASE_URL
DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.sqlite3',  # or postgresql
        'NAME': BASE_DIR / 'db.sqlite3',
    }
}

# Secret key — like your .env NEXTAUTH_SECRET
SECRET_KEY = 'django-insecure-change-me-in-production'

# Where templates live
TEMPLATES = [
    {
        'DIRS': [BASE_DIR / 'templates'],
        # ...
    },
]

Important: INSTALLED_APPS is a concept Next.js doesn't have. When you create a new app (startapp blog), you MUST add 'blog' to INSTALLED_APPS in settings.py. Otherwise Django doesn't know it exists. This is the #1 mistake beginners make — they create an app but forget to register it.

The Dev Server

Django's development workflow uses manage.py commands, similar to npm scripts.

Action Next.js Django
Start dev server npm run dev python manage.py runserver
Create migrations npx prisma migrate dev python manage.py makemigrations
Run migrations npx prisma migrate deploy python manage.py migrate
Interactive shell node (limited) python manage.py shell
Create admin user Manual python manage.py createsuperuser
Run tests npm test python manage.py test
Collect static files npm run build python manage.py collectstatic

manage.py is your Swiss Army knife. It's like if package.json scripts, npx prisma, and node REPL were all combined into one tool. The Django shell (python manage.py shell) is incredibly powerful — you can import your models and run database queries interactively, like having Prisma Studio in your terminal.

The Full Setup Flow

# 1. Create virtual environment (like npm init)
python -m venv venv
source venv/bin/activate

# 2. Install Django (like npm install next)
pip install django

# 3. Create project (like npx create-next-app)
django-admin startproject mysite
cd mysite

# 4. Create an app (like adding a route group)
python manage.py startapp blog

# 5. Add 'blog' to INSTALLED_APPS in settings.py

# 6. Run migrations (sets up the database)
python manage.py migrate

# 7. Create an admin user
python manage.py createsuperuser

# 8. Start the server!
python manage.py runserver

# Visit http://127.0.0.1:8000/ — your site
# Visit http://127.0.0.1:8000/admin/ — free admin panel!

The Request/Response Cycle

When someone visits a URL on your Django site, here's what happens — and how it compares to Next.js:

  1. Request comes in — GET /blog/my-article/
  2. URL routing — Django checks urls.py patterns (Next.js checks the app/ directory)
  3. Middleware runs — CSRF, sessions, auth (same concept in both)
  4. View executes — Your function runs, queries the database, prepares data (like a Server Component or API route)
  5. Template renders — Data gets inserted into HTML (like JSX rendering)
  6. Response returns — HTML or JSON sent back to the browser

The big picture: Django's flow is: URL → View → Template → Response. Next.js's flow is: URL → Route → Component → HTML. Same steps, same order, different names. Once you see this pattern, Django stops feeling foreign.

Check Your Understanding

Quick Quiz

1. What's the Django equivalent of npx create-next-app mysite?

django-admin startproject mysite — Creates the project structure with settings.py, urls.py, and manage.py.

2. In Next.js, routes are defined by file structure. How are routes defined in Django?

Explicitly in urls.py files using path() patterns. You map URL patterns to view functions. It's more explicit but gives you more control.

3. What's the Django equivalent of a Prisma schema?

models.py — You define your database tables as Python classes. Django's ORM handles migrations, queries, and relationships, just like Prisma.

4. What do you get for free in Django that you'd need NextAuth for in Next.js?

Authentication! django.contrib.auth gives you user registration, login, logout, password reset, permissions, and groups — all built-in. Plus a free admin panel via django.contrib.admin.

5. After creating a new app with python manage.py startapp blog, what's the crucial step most beginners forget?

Adding 'blog' to INSTALLED_APPS in settings.py. Without this, Django doesn't know the app exists and won't find its models, templates, or migrations.
