The same idea, in every language.

The fastest way to learn a new language is to see a concept you already know, written the other way. Here is the same small task across JavaScript, Python, SQL, R, and Pandas.

Variables & assignment

Binding a name to a value so it can be reused later.

JavaScript
const x = 5;
let y = x + 1;
Python
x = 5
y = x + 1
R
x <- 5
y <- x + 1
SQL
DECLARE @x INT = 5;
DECLARE @y INT = @x + 1;

Connection Python and R just bind a name to a value, while JavaScript forces you to pick const vs let and SQL has no general-purpose variable at all (the DECLARE form is a procedural extension, not standard SQL). The R gotcha is the <- arrow where every other language uses =.

Lists, arrays, dicts & objects

Ways to store an ordered sequence of values and to store key-to-value mappings.

JavaScript
const nums = [1, 2, 3];
const person = { name: "Ada", age: 36 };
console.log(nums[0], person.name);
Python
nums = [1, 2, 3]
person = {"name": "Ada", "age": 36}
print(nums[0], person["name"])
R
nums <- c(1, 2, 3)
person <- list(name = "Ada", age = 36)
print(c(nums[1], person$name))

Connection The sequence/map split is the same everywhere, but indexing differs: JavaScript and Python start at 0 while R starts at 1, and R's c() coerces to one type whereas a Python list or JS array can mix types.

Transform every item (map)

Apply the same function to each element of a collection, producing a new collection of the results.

JavaScript
const nums = [1, 2, 3];
const doubled = nums.map(n => n * 2);
Python
nums = [1, 2, 3]
doubled = [n * 2 for n in nums]
R
nums <- c(1, 2, 3)
doubled <- nums * 2
Pandas
doubled = df["nums"] * 2

Connection JavaScript and Python apply the function element-by-element, while R and Pandas vectorize: the operation broadcasts over the whole vector or column at once, so you usually skip the explicit loop entirely.

Keep some items (filter / WHERE)

Produce a subset of a collection by keeping only the items that satisfy a condition.

JavaScript
const adults = users.filter(u => u.age >= 18);
Python
adults = [u for u in users if u["age"] >= 18]
SQL
SELECT * FROM users
WHERE age >= 18;
Pandas
adults = df[df["age"] >= 18]

Connection JavaScript and Python evaluate a per-item predicate (a function or expression), while SQL's WHERE and Pandas' boolean indexing describe the whole condition at once. The classic gotcha is Pandas: you must use & / | with parentheses around each comparison, not Python's and / or, because it operates element-wise on the column.

Defining functions

Naming a reusable block of code that takes inputs and returns a result.

JavaScript
function square(x) {
  return x * x;
}
Python
def square(x):
    return x * x
R
square <- function(x) {
  x * x
}

Connection All three name a function and take a parameter, but R has no return keyword in idiomatic code (the last expression is returned automatically) and assigns the function to a name with <-, while JavaScript and Python use explicit return.

Looping & iteration

Repeatedly running a block of code, once for each value in a sequence.

JavaScript
for (let i = 1; i <= 3; i++) {
  console.log(i);
}
Python
for i in range(1, 4):
    print(i)
R
for (i in 1:3) {
  print(i)
}

Connection All three iterate over a range, but the bounds differ: Python's range(1, 4) and JavaScript's i <= 3 both stop before/at the limit you must reason about, while R's 1:3 is inclusive on both ends. The bigger gotcha in R is that explicit for loops are often discouraged in favor of vectorized operations.

Conditionals & truthiness

Branch the program down different paths depending on whether a condition evaluates to true or false.

JavaScript
if (x > 0) {
  console.log("positive");
} else {
  console.log("non-positive");
}
Python
if x > 0:
    print("positive")
else:
    print("non-positive")
R
if (x > 0) {
  print("positive")
} else {
  print("non-positive")
}

Connection All three branch the same way, but truthiness differs: JavaScript treats 0, "", null, undefined, and NaN as falsy, Python treats 0, "", and empty containers as falsy, while R's if() demands a single logical value and errors (or warns) on a multi-element vector or NA.

String formatting / interpolation

Building a string by embedding the values of variables directly inside it.

JavaScript
const name = "Ada";
const age = 36;
console.log(`${name} is ${age}`);
Python
name = "Ada"
age = 36
print(f"{name} is {age}")
R
name <- "Ada"
age <- 36
cat(sprintf("%s is %d", name, age))

Connection JavaScript and Python both inline the variable inside the literal (backtick `${}` vs the f-string `{}` prefix), so the names live in the string. R's idiomatic sprintf instead uses positional placeholders (%s, %d) with the values passed as separate arguments, which trips people up because order, not name, decides what goes where.

Grouping & aggregation

Split rows into groups by a key column, then compute one summary value per group.

SQL
SELECT dept, AVG(salary)
FROM employees
GROUP BY dept;
Pandas
df.groupby("dept")["salary"].mean()
R
aggregate(salary ~ dept, data = employees, FUN = mean)

Connection All three split-apply-combine the same way, but the result shape differs: SQL returns flat rows, Pandas returns a Series indexed by the group key (call .reset_index() to get a flat DataFrame), and R's aggregate already gives you a plain data frame.

Combining datasets (JOIN / merge)

Stitch two datasets together side by side by matching rows on a shared key.

SQL
SELECT *
FROM orders o
INNER JOIN customers c
  ON o.customer_id = c.id;
Pandas
merged = orders.merge(
    customers, left_on="customer_id", right_on="id", how="inner"
)
R
merged <- merge(orders, customers,
  by.x = "customer_id", by.y = "id")

Connection All three default differently: a bare SQL JOIN is an inner join, pandas merge also defaults to inner, but base R merge() defaults to an inner join only when all.x/all.y are FALSE (its default) yet silently does a full cross join if you forget the by argument. Always name your keys and join type explicitly.

Async & concurrency

Running work that waits on I/O without blocking, so other tasks can proceed while a result is pending.

JavaScript
async function main() {
  const res = await fetch("https://api.example.com/data");
  const data = await res.json();
}
Python
import aiohttp, asyncio
async def main():
    async with aiohttp.ClientSession() as s:
        async with s.get("https://api.example.com/data") as res:
            data = await res.json()

Connection Both use async/await and an event loop, but JavaScript runs its loop automatically while Python must explicitly start one with asyncio.run(main()) — forgetting that leaves your coroutine unawaited and never executed.

Missing & null values

A way to represent the absence of a value, and to substitute a fallback when one is missing.

JavaScript
const name = user.name ?? "Anonymous";
Python
name = user.get("name") or "Anonymous"
SQL
SELECT COALESCE(name, 'Anonymous') AS name
FROM users;
Pandas
df["name"] = df["name"].fillna("Anonymous")

Connection Each language has its own "missing" token (JS null/undefined, Python None, SQL NULL, Pandas NaN/NA), and the gotcha is that they don't compare or coerce the same way: SQL NULL is never equal to anything (use IS NULL), Pandas NaN != NaN, and JS ?? only catches null/undefined while Python's or also swallows falsy values like 0 and "".
