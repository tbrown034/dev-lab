--- LESSON CONTENT ---

The Problem

Bars without labels are meaningless. The viewer sees 253 pixels tall but doesn't know that means 20 million people. An axis renders tick marks and labels that translate pixel positions back to human-readable data values.

Axes Are Generated FROM Scales

This is the key insight: you create an axis FROM an existing scale. The axis reads the scale's domain and range to figure out where to put ticks and what to label them.

Step 1: Create a scale. Step 2: Create an axis from the scale (d3.axisLeft(yScale)). Step 3: Render it into a <g> group using .call().

Four axis orientations:
- d3.axisBottom(scale) -- ticks below the line (X-axis at bottom)
- d3.axisTop(scale) -- ticks above the line
- d3.axisLeft(scale) -- ticks to the left (Y-axis)
- d3.axisRight(scale) -- ticks to the right

What does .call() do? It passes the selection to a function. group.call(xAxis) is the same as xAxis(group). The axis function creates multiple child elements (line, ticks, text labels) inside the group. .call() keeps the chain going -- without it, you'd need to break the chain to pass the group.

Customizing Axes:
- .ticks(5) -- suggest approximately 5 tick marks
- .tickFormat(d => "$" + d/1000 + "k") -- custom label formatting
- .tickSize(-width) -- extend ticks to create grid lines
- .tickValues([0, 25, 50, 75, 100]) -- exact tick positions

Grid lines are just long ticks. Setting .tickSize(-width) extends each tick line across the full chart width. No separate grid system needed.

The Margins Convention (Full Implementation)

The standard pattern every professional D3 chart uses:

Step 1 - Define margins: const margin = { top: 20, right: 30, bottom: 40, left: 50 };

Step 2 - Calculate inner dimensions: width = 700 - margin.left - margin.right (= 620). height = 400 - margin.top - margin.bottom (= 340).

Step 3 - SVG at full size, group pushed inward:
const svg = d3.select("#chart")
  .append("svg")
  .attr("viewBox", "0 0 700 400")
  .append("g")
  .attr("transform", `translate(${margin.left}, ${margin.top})`);

Now (0,0) in the group = top-left of chart area. Scales use width and height (inner dimensions). Axes fit in the margin space.

The X-axis is positioned with translate(0, height) to push it to the bottom of the chart area. Without this translate, the X-axis would appear at the top.

Complete Pattern: margin -> dimensions -> SVG+g -> scales -> axes -> grid lines -> bars -> labels. This is the template for every bar chart.

--- KEY CONCEPTS TO EMPHASIZE ---

1. Axes are generated FROM scales -- the axis reads the scale's domain and range
2. .call(axisGenerator) renders the axis into a <g> group
3. Grid lines = ticks with .tickSize(-width) -- no separate system
4. Margins convention is boilerplate: margin -> dimensions -> SVG+g -> scales -> axes -> data
5. X-axis needs translate(0, height) to appear at the bottom

--- COMMON PITFALLS ---

- Forgetting to translate the X-axis to the bottom: translate(0, height)
- Using total SVG dimensions instead of inner dimensions for scales (should use width after margins)
- Not styling axis elements (D3's default axis styling is minimal -- you need to set text fill, stroke colors)
- Forgetting that .tickSize(-width) creates grid lines (many people create a separate grid system unnecessarily)
- Putting axes BEFORE grid lines in drawing order (grid lines should be drawn first so bars appear on top)

--- BROADCAST TV CONNECTION ---

- Broadcast charts ALWAYS have clean, formatted axis labels -- "$50k" not "50000"
- Grid lines help viewers read values even from a distance on a TV screen
- The margins convention ensures labels don't get clipped at screen edges
- Election result charts need formatted tick labels: "2.5M votes" not "2500000"
- Weather forecast charts need time-formatted X-axes: "Mon", "Tue", "Wed"
- The complete pattern (margins + scales + axes + data) is what you'll write every time you build a broadcast graphic
