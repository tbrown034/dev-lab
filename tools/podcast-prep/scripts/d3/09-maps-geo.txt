--- LESSON CONTENT ---

The Two-Step Process

Radial charts in D3 use two separate tools that work together:

Step 1 -- d3.pie() computes angles:

const pie = d3.pie()
  .value(d => d.amount)    // which field determines slice size
  .sort(null);             // keep original data order

const arcsData = pie(data);
// Each item now has: { startAngle, endAngle, data, value, index, padAngle }

pie() does NOT draw anything. It takes your data array and returns a new array where each item has startAngle and endAngle (in radians) that sum to 2*pi (a full circle). The original data is preserved in the .data property of each angle object.

Step 2 -- d3.arc() draws slices:

const arc = d3.arc()
  .innerRadius(0)          // 0 = pie chart
  .outerRadius(150);       // radius of the chart

arc() does NOT know about your data. It only reads angle objects (startAngle, endAngle) and produces SVG path strings. You pass pie-computed angle objects to the arc generator to get path strings.

Step 3 -- Combine with data binding:

svg.selectAll("path")
  .data(arcsData)          // bind the pie-computed angles
  .join("path")
  .attr("d", arc)          // arc generator reads startAngle/endAngle
  .attr("fill", (d, i) => colors[i]);

Note that .data() is used here, not .datum(). Each slice is a separate <path> element, and each gets one angle object bound to it. This is the opposite of line charts (Lesson 08), where one path element used .datum() for the entire array.

Donut vs Pie

The only difference between a pie chart and a donut chart is innerRadius:

Pie chart: d3.arc().innerRadius(0).outerRadius(150) -- solid slices from center to edge.

Donut chart: d3.arc().innerRadius(70).outerRadius(150) -- hollow center, ring of slices.

The donut's center space is valuable real estate for summary statistics. Broadcast graphics commonly put the total value, a percentage, or a label in the center of a donut chart.

.sort(null): Preserving Data Order

By default, d3.pie() sorts slices by descending value. This can be surprising. Calling .sort(null) tells it to keep the original data order. This matters when:
- Your data has a meaningful sequence (chronological, matching a legend)
- You want consistent slice positioning across updates
- You're animating between different data states and need slices to stay put

Centroid for Label Placement

arc.centroid(d) returns an [x, y] coordinate for the geometric center of an arc slice. This is the ideal position for labels:

const labelArc = d3.arc()
  .innerRadius(outerRadius * 0.7)
  .outerRadius(outerRadius * 0.7);

svg.selectAll("text")
  .data(arcsData)
  .join("text")
  .attr("transform", d => `translate(${labelArc.centroid(d)})`)
  .text(d => d.data.label);

Create a separate arc generator for label positioning. Setting both innerRadius and outerRadius to the same value places the centroid exactly at that radius -- useful for positioning labels at a consistent distance from the center.

Polish: padAngle and cornerRadius

.padAngle(0.02) adds a small gap between adjacent slices, making them visually distinct.

.cornerRadius(4) rounds the corners of each slice for a modern, polished look.

Together they give donut charts the clean, broadcast-quality appearance viewers expect.

Animating Arcs

Arcs animate using attrTween("d", ...) with a custom interpolator:

1. Start each arc with endAngle equal to startAngle (zero-size slice)
2. Interpolate from the starting angle to the final endAngle
3. At each frame, call the arc generator with the interpolated angle to produce an intermediate path string

This creates the effect of slices growing from nothing to their final size. Combined with staggered delays, each slice appears to grow one after another.

The lesson demonstrates:
- A campaign finance donut chart showing congressional funding sources, with percentage labels at centroids, a total in the center, and a legend
- An animated budget breakdown donut with arcs that grow from zero using attrTween

--- KEY CONCEPTS TO EMPHASIZE ---

1. Two tools, two jobs: d3.pie() computes angles, d3.arc() draws paths -- they always work together
2. pie() does NOT draw anything; arc() does NOT know about your data -- separation of computation and rendering
3. innerRadius = 0 is a pie chart, innerRadius > 0 is a donut chart
4. arc.centroid() gives the midpoint [x, y] of each slice -- use it for label positioning
5. .padAngle() and .cornerRadius() add broadcast-quality polish
6. attrTween enables smooth arc entrance and update animations

--- COMMON PITFALLS ---

- Forgetting .sort(null) and getting unexpectedly reordered slices
- Confusing pie() output with arc() output -- pie() returns angle objects, arc() returns path strings
- Trying to position labels manually instead of using arc.centroid()
- Not understanding that the data you pass to .data() is the pie-computed array, not the original data
- Accessing original data via d.data.fieldName (not d.fieldName) because pie() wraps the original in an angle object
- Using .datum() instead of .data() for arcs -- each slice is a separate path element, so use .data()

--- BROADCAST TV CONNECTION ---

- Election results: pie chart showing vote share by candidate with percentages
- Budget breakdowns: city/state/federal spending by department as a donut chart
- Demographic composition: population breakdown by age group, race, or education level
