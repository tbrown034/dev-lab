--- LESSON CONTENT ---

The Problem Data Binding Solves

Say you have population data for 5 states and want a bar for each. Without data binding, you'd use forEach to append rects manually. This works but has a problem: there is no link between the data and the elements. If the data changes, you have to manually find, update, or remove bars. D3's data binding creates that permanent link.

What .data() Actually Does

When you call .data(array), D3 takes each item in the array and physically attaches it to a DOM element using a hidden property called __data__. After data binding, each rect literally carries its data: rect.__data__ returns { state: "CA", pop: 39538 }. You can verify this in DevTools by selecting an element and typing $0.__data__.

This is the "sticky note." Each DOM element carries its data around. When you write .attr("width", d => d.pop), that d parameter is the sticky note glued to that specific element.

The Pattern: selectAll -> data -> join

svg.selectAll("rect")   // 1. Select all existing rects (probably 0)
  .data(data)            // 2. Bind the data array
  .join("rect")          // 3. Create/update/remove rects to match data
  .attr("x", (d, i) => i * 100)  // 4. Use bound data to set attributes

Step 1 - selectAll("rect"): This usually selects ZERO elements on first run. That is fine. You're telling D3: "I want rects to represent my data." It's a declaration of intent, not a selection of existing elements. This is like React's return <div>{items.map(...)}</div> -- declaring what the DOM SHOULD look like.

Step 2 - .data(data): D3 compares the data array to existing elements by index. 5 data items, 0 rects = 5 new rects needed.

Step 3 - .join("rect"): Creates missing elements, keeps existing ones, removes extras. After .join("rect"), you have exactly one rect per data item, each with __data__ attached.

Step 4 - Accessor functions: Any method that takes a value can also take a function. D3 calls it once per element: d = the bound data (the sticky note), i = the index.

Side by Side: forEach vs D3

forEach approach: data.forEach((d, i) => { svg.append("rect").attr(...)}) -- no data link, can't update.

D3 approach: svg.selectAll("rect").data(data).join("rect").attr(...) -- data glued to elements, re-run with new data = auto-update.

The superpower: If you call the D3 version again with different data, it automatically updates existing bars, creates new ones, and removes old ones. The forEach version would just keep appending duplicates.

--- KEY CONCEPTS TO EMPHASIZE ---

1. .data() attaches each data item to a DOM element via the hidden __data__ property
2. selectAll -> data -> join is the ONE pattern to memorize
3. Empty selectAll is a "declaration of intent" -- it's fine and expected to match 0 elements
4. Accessor functions (d, i) => ... make each element data-driven
5. Re-running with new data automatically updates the chart -- this is the superpower

--- COMMON PITFALLS ---

- Being confused by selectAll("rect") matching zero elements -- it's a declaration of intent, not a bug
- Confusing .data() (one element per array item) with .datum() (one element for the entire array) -- .datum() comes later for line/area charts
- Forgetting that accessor functions receive (d, i) where d is __data__ and i is the index
- Using forEach to create elements instead of the data binding pattern -- leads to duplicates on re-render
- Not understanding that .join() replaces the old .enter().append() pattern from older D3 tutorials

--- BROADCAST TV CONNECTION ---

- Election night: as results pour in, re-call the data binding pattern with updated vote counts and bars automatically resize
- Weather dashboards: temperature data updates every few minutes; data binding keeps the chart in sync without rebuilding from scratch
- Census visualizations: filter by state, region, or demographic and the chart automatically adjusts to show the filtered data
- Breaking-news tickers: new data items appear (enter), old items update, stale items disappear (exit) -- all handled by .join()
- The key insight for broadcast: you write the chart code ONCE, then feed it different data. The data binding pattern handles the rest.
