--- LESSON CONTENT ---

The Problem

In Lesson 01, we wrote things like .attr("width", d => d.pop / 100). That "/ 100" is embarrassing -- just eyeballing a number. If the data changes or the screen is a different size, it breaks. Every chart has this problem: your data lives in one world (millions of people), your screen lives in another (600 pixels wide). You need a translator. That's what a SCALE is.

A Scale is a Translator

Imagine converting Fahrenheit to Celsius for a journalism piece. The formula maps one range of numbers to another range. A D3 scale works the same way, except instead of temperature conversion, it's Data-to-Pixels:

- Input: 0 (population) maps to Output: 0px
- Input: 39,538,000 (California's population) maps to Output: 500px
- Input: 20,000,000 (New York's population) maps to Output: 253px (proportional!)

A scale is a function. You give it a data value, it gives back a pixel value. D3 gives you convenient ways to create these translator functions.

Creating a Scale in D3

Two things to configure: DOMAIN (the range of your data / input world) and RANGE (the range of your pixels / output world).

const scale = d3.scaleLinear()
  .domain([0, 39_538_000])   // data goes from 0 to 39.5M
  .range([0, 500]);           // pixels go from 0 to 500

scale(39_538_000)   // returns 500   (max data = max pixels)
scale(20_000_000)   // returns 253   (proportional)
scale(0)            // returns 0     (min data = min pixels)

Different Data, Different Scales

scaleLinear -- for continuous numbers: population, money, temperature, percentages. Any number in a continuous range. For Y scales, the range is FLIPPED: .range([height, 0]). This is because SVG's Y-axis goes top-down. 0% should be at the bottom (large Y pixel), 100% at the top (small Y pixel = 0). You will do this for EVERY Y-axis.

scaleBand -- for categories (bar charts): state names, district labels, candidate names. Each gets its own "band" of space. scaleBand divides space evenly among categories and provides .bandwidth() -- the width each bar should be. No more manually calculating bar widths. Includes .padding() for gaps between bars.

scaleTime -- for dates: maps Date objects to pixels. Used for the X-axis of time series charts like polling trends.

scaleSequential -- for color: maps data values to colors instead of pixels. Essential for choropleth maps. Uses interpolators like d3.interpolateBlues to go from light blue to dark blue.

Axes Are Just Labels for Your Scale

You have a scale translating data to pixels, but the viewer doesn't know that. An axis draws tick marks and labels showing what the scale is doing. Without axis: "That bar is 253 pixels tall. So what?" With axis: "That bar represents 20 million people."

An axis is created FROM a scale:
- d3.axisBottom(xScale) -- ticks below the line (X-axis at bottom)
- d3.axisLeft(yScale) -- ticks left of the line (Y-axis)

Render it by calling it on a <g> group:
svg.append("g")
  .attr("transform", `translate(0, ${height})`)
  .call(xAxis);

The .call() method passes the selection to the axis function. The axis function creates lines, ticks, and text labels inside the group.

The Margins Convention

The standard boilerplate every professional D3 chart starts with:
1. Define margins: { top: 20, right: 30, bottom: 40, left: 50 }
2. Calculate inner chart dimensions: width = 700 - margin.left - margin.right
3. Create SVG at full size, append a <g> group translated inward by the margins
4. All drawing happens inside the group. Axes fit in the margin space.

Copy-paste this pattern into every chart. It's boilerplate.

Complete Pattern: margins + scales + axes + bars. Replace magic numbers with scale calls:
- .attr("x", d => x(d.state)) instead of manual positioning
- .attr("y", d => y(d.pop)) instead of eyeballing
- .attr("width", x.bandwidth()) instead of hardcoded widths

Useful Extras:
- .nice() rounds domain to clean axis numbers (e.g., 0 to 40000 instead of 0 to 39538)
- .tickFormat() formats labels (e.g., 10000 becomes "10k")
- viewBox makes charts responsive -- define coordinate space, let browser scale

--- KEY CONCEPTS TO EMPHASIZE ---

1. A scale is a function that translates data values to pixel values (or color values)
2. Domain = input range (data world), Range = output range (pixel world)
3. Y range is ALWAYS flipped [height, 0] because SVG Y goes top-down
4. scaleBand for categories, scaleLinear for numbers, scaleTime for dates
5. The margins convention is copy-paste boilerplate for every chart

--- COMMON PITFALLS ---

- Forgetting to flip the Y range (bars grow down instead of up)
- Using scaleLinear when you need scaleBand for categories (or vice versa)
- Setting scale domain/range in wrong order (domain is data, range is pixels)
- Not calling .nice() and getting ugly axis numbers like 0, 7907, 15815
- Confusing width/height (total SVG size) with the inner chart dimensions (after margins)

--- BROADCAST TV CONNECTION ---

- Election dashboards need scaleLinear for vote counts and scaleBand for candidate names
- Weather maps use scaleSequential for temperature-to-color mapping (blue = cold, red = hot)
- Census choropleth maps use scaleSequential for population density colors
- Poll tracking line charts use scaleTime for the X-axis (dates) and scaleLinear for the Y-axis (approval %)
- Breaking-news graphics need responsive charts via viewBox to fit different broadcast screen resolutions
- The margins convention creates the clean, structured layouts that broadcast design standards require
