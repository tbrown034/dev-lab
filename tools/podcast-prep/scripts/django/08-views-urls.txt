LESSON CONTENT
------------------------------------------------------------

THE MENTAL MODEL — VIEWS ARE ROUTE HANDLERS

In Next.js, you write route.ts files for API endpoints and page.tsx files for pages. Django combines both into one concept: views. A Django view is a Python function (or class) that receives an HTTP request and returns an HTTP response. Request in, response out. That's it.

FUNCTION-BASED VIEWS (FBVs) — THE FOUNDATION

The three response types you'll use constantly:

1. HttpResponse — raw text or HTML string. Like: return new Response('Hello')
2. JsonResponse — JSON data for APIs. Like: NextResponse.json({...})
3. render() — HTML template with data. The most common. Like a Server Component that fetches data and returns JSX.

The render() function takes three arguments: request, template path, and a context dictionary. The context dict is like props — each key becomes a variable available in the template.

ACCESSING THE REQUEST OBJECT

Django separates query parameters (request.GET) from form body data (request.POST). These are both dict-like objects. For raw JSON bodies (like from a React frontend), use json.loads(request.body). This is different from Next.js where you call request.json() for everything.

Key properties: request.method ("GET"/"POST"), request.GET.get("q") for query params, request.POST.get("title") for form data, request.headers.get("Authorization"), request.user (current user), request.path (URL path).

HANDLING HTTP METHODS

In Next.js, you export separate functions (GET, POST). In Django FBVs, you check request.method inside one function. The @require_http_methods decorator restricts which methods are allowed.

URL CONFIGURATION — WIRING VIEWS TO URLS

Writing a view is only half the job. You also need to tell Django which URL triggers that view. You wire them up explicitly in urls.py using path().

Two-level routing: Root urls.py uses include() to delegate to each app's urls.py. When Django sees path('newsroom/', include('newsroom.urls')), it strips /newsroom/ from the URL and hands the rest to the app's router. Same idea as nested route groups.

path(route, view, name) — each URL pattern has three parts: the URL string, the view function, and a unique name for reverse URL generation.

Named URLs are a Django superpower. Instead of hardcoding "/newsroom/city-council-vote/", you reference URLs by name: reverse('newsroom:article-detail', kwargs={'slug': 'city-council-vote'}). In templates: {% url 'newsroom:article-detail' slug='city-council-vote' %}. If you ever change the URL structure, every reference updates automatically.

URL PARAMETERS & CONVERTERS

Dynamic routes work differently but solve the same problem. Next.js uses [slug] in file names. Django uses <slug:slug> in the URL pattern.

Built-in converters with automatic validation: <str:name>, <int:pk> (guarantees actual integer!), <slug:slug>, <uuid:id>, <path:rest>. In Next.js, params.id is always a string — you have to parseInt() yourself. In Django, <int:pk> guarantees pk is a real integer. If someone visits /article/abc/, Django returns a 404 automatically.

TEMPLATE RENDERING FROM VIEWS

The most common pattern: view queries data and hands it to a template. The third argument to render() is the context dictionary — each key becomes a template variable. Same mental model as passing props to a React component.

CLASS-BASED VIEWS (CBVs)

Django also has class-based views for common patterns. ListView auto-paginates, DetailView auto-fetches by pk/slug, CreateView auto-handles forms. They reduce boilerplate but have a steeper learning curve.

Translation: Next.js doesn't have an equivalent — you'd build these patterns manually. CBVs are like pre-built page templates that handle common CRUD patterns.

Start with function-based views (explicit, easy to understand), then graduate to class-based views when you see the patterns repeating.

REDIRECTS AND ERROR HANDLING

redirect('article-detail', slug=article.slug) sends a 302 redirect. raise Http404("Article not found") returns a 404 page. get_object_or_404(Article, slug=slug) combines both — get the object or 404.

------------------------------------------------------------
KEY CONCEPTS TO EMPHASIZE
------------------------------------------------------------

1. Views are just route handlers — request in, response out. The same mental model as Next.js API routes.
2. The context dict is template props — every key in the dict becomes a variable in the template.
3. Named URLs prevent broken links — {% url 'name' %} means you never hardcode paths.
4. URL converters provide free validation — <int:pk> guarantees an integer, returns 404 for anything else.
5. Start with FBVs, graduate to CBVs — function-based views are explicit and easy to debug. Class-based views reduce boilerplate once you know the patterns.

------------------------------------------------------------
COMMON PITFALLS FOR JS DEVELOPERS
------------------------------------------------------------

- Expecting file-based routing (Django uses explicit urls.py wiring)
- Forgetting to create the app's urls.py file (startapp doesn't create one)
- Mixing up request.GET (query params) and request.POST (form body)
- Forgetting to pass context to render() and wondering why template variables are empty
- Hardcoding URLs instead of using {% url 'name' %} or reverse()
- Not using get_object_or_404() (manually handling DoesNotExist is verbose)
- Returning a template response after a successful POST (use redirect for PRG pattern)
- Forgetting that Django views are synchronous by default (no async/await needed)

------------------------------------------------------------
REAL-WORLD CONNECTION
------------------------------------------------------------

At a broadcast TV network, views are where your business logic lives. A story list view queries published articles with select_related, paginates them, and hands the data to a template. A detail view fetches a single story by slug and renders it with related sources and datasets. An editor dashboard view checks permissions, queries stories in review, and displays them for editorial approval. An API view returns JSON for the mobile app. Every URL is explicitly named, so when the URL structure changes for SEO, no links break. Named URLs are especially valuable in a large newsroom app with dozens of interlinked pages.
