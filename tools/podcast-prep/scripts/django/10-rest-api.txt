LESSON CONTENT
------------------------------------------------------------

THE MENTAL MODEL — WHAT DRF ACTUALLY IS

Django by itself serves HTML pages. Django REST Framework (DRF) is a third-party package that adds a JSON API layer on top of Django. It gives you serializers (for shaping data), views (for handling requests), authentication, permissions, pagination — everything you need to build a production API.

If Django is like Next.js pages (server-rendered HTML), then DRF is like Next.js API routes (route.ts) — but with a whole framework around them. Imagine if every route.ts file came with automatic input validation, typed responses, auth middleware, and a browsable documentation page.

INSTALLATION & SETUP

pip install djangorestframework. Add "rest_framework" to INSTALLED_APPS. Configure global settings in the REST_FRAMEWORK dict — pagination, auth classes, permission classes. This is your global API config that every endpoint inherits.

SERIALIZERS — YOUR API'S ZOD SCHEMAS

A serializer is a Zod schema + a data transformer in one object. It validates incoming data (like z.object({...}).parse(body)) and it formats outgoing data (like picking which fields to include in your JSON response). One class, two jobs.

ModelSerializer reads your Django model and automatically generates fields that match. You just tell it which model and which fields to expose. It's like if Zod could read your Prisma schema and auto-generate the validation rules.

Three things a serializer does:
1. Serialize (Python object -> JSON): ArticleSerializer(article).data
2. Deserialize + Validate (JSON -> Python): ArticleSerializer(data=request.data).is_valid()
3. Save (validated data -> database): serializer.save(author=request.user)

Custom validation: validate_title(self, value) validates just the title field. validate(self, data) validates the whole object (cross-field). DRF discovers these methods automatically by name.

Nested serializers: Embed related objects in responses. author = AuthorSerializer(read_only=True) replaces the author ID with a full author object in the JSON response.

APIVIEW — THE BUILDING BLOCK

APIView is DRF's equivalent of a Next.js route.ts file. You write methods for each HTTP verb: get(), post(), put(), delete(). It's explicit, easy to understand, and gives you full control.

The pattern: get the object, run it through the serializer, return the response. get_object_or_404() handles the "not found" case automatically.

VIEWSETS & ROUTERS — CRUD ON AUTOPILOT

A ViewSet combines the list view and the detail view into one class. A Router automatically generates all the URL patterns. Together, they give you a complete CRUD API in about 10 lines of code.

In Next.js, you'd write 2 files and ~60 lines for full CRUD. With DRF ViewSet + Router: 1 file, ~10 lines. ModelViewSet gives you GET (list), POST (create), GET (retrieve), PUT (update), PATCH (partial update), DELETE (destroy) — all with validation, error handling, and proper status codes.

The Router auto-generates: GET /api/articles/ (list), POST /api/articles/ (create), GET /api/articles/{id}/ (retrieve), PUT /api/articles/{id}/ (update), PATCH /api/articles/{id}/ (partial), DELETE /api/articles/{id}/ (delete), GET /api/ (browsable API root).

FILTERING, SEARCH & ORDERING

DRF integrates with django-filter for powerful filtering. Add filter_backends, filterset_fields, search_fields, and ordering_fields to your ViewSet. Clients can then use query parameters: ?status=published&search=election&ordering=-created_at.

AUTHENTICATION & PERMISSIONS

DRF permission classes: IsAuthenticated (must be logged in), IsAuthenticatedOrReadOnly (logged in for writes, anyone can read), IsAdminUser, custom permissions. Set globally in REST_FRAMEWORK settings or per-view.

Token and JWT authentication: TokenAuthentication for simple token auth. djangorestframework-simplejwt for JWT. Configure in DEFAULT_AUTHENTICATION_CLASSES.

Custom permissions: Create a class that checks has_permission() and has_object_permission(). Example: IsAuthorOrReadOnly — only the article's author can edit/delete it.

PAGINATION

Built-in. Set DEFAULT_PAGINATION_CLASS and PAGE_SIZE in settings. Responses automatically include count, next, previous, and results. The client gets ?page=2 support for free.

THE BROWSABLE API

DRF includes a browsable HTML interface at your API endpoints. Visit /api/articles/ in a browser and you get an interactive page with forms for POST/PUT, clickable links, and formatted JSON. It's like free API documentation + testing tool.

------------------------------------------------------------
KEY CONCEPTS TO EMPHASIZE
------------------------------------------------------------

1. Serializers do two jobs — validate input AND shape output. Like Zod + data transformer in one.
2. ViewSets generate all CRUD — one class, six endpoints, zero boilerplate.
3. Routers auto-generate URLs — register a ViewSet, get RESTful routes for free.
4. The browsable API is free documentation — visit your API endpoint in a browser and get an interactive explorer.
5. Permissions are declarative — IsAuthenticatedOrReadOnly is one line, not a middleware file.

------------------------------------------------------------
COMMON PITFALLS FOR JS DEVELOPERS
------------------------------------------------------------

- Forgetting to add "rest_framework" to INSTALLED_APPS
- Writing manual CRUD views when a ViewSet would do it in 10 lines
- Forgetting serializer.is_valid(raise_exception=True) and wondering why validation doesn't work
- Not setting read_only_fields for auto-generated fields (id, created_at)
- Forgetting to pass many=True when serializing a list of objects
- Not using nested serializers and returning raw foreign key IDs in the API
- Trying to use Next.js-style middleware instead of DRF permission classes
- Not setting a timeout or pagination, leading to slow/large responses

------------------------------------------------------------
REAL-WORLD CONNECTION
------------------------------------------------------------

At a broadcast TV network, the REST API serves multiple clients: the public website (React/Next.js frontend), mobile apps, the on-air graphics system, and internal dashboards. DRF ViewSets give you full CRUD for stories, datasets, and sources with minimal code. Serializers validate incoming data from reporters in the field (mobile app submissions). Permission classes ensure only editors can publish stories. The browsable API lets the development team test endpoints without Postman. Filtering and search let the newsroom search across thousands of articles: /api/stories/?search=election&status=published&ordering=-published_at. Pagination keeps responses fast when the archive grows to tens of thousands of stories. This is production API development, not a tutorial pattern.
