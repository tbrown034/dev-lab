Key JSX-to-DTL translations:
- {variable} -> {{ variable }}
- {condition && <p>...</p>} -> {% if condition %}<p>...</p>{% endif %}
- {items.map(i => ...)} -> {% for i in items %}...{% endfor %}
- layout.tsx + {children} -> {% extends "base.html" %} + {% block content %}
- <Component /> -> {% include "component.html" %}
- props -> context dict (from view) or {% include ... with var=val %}
- {title.toUpperCase()} -> {{ title|upper }}
- {title || 'default'} -> {{ title|default:"default" }}
- dangerouslySetInnerHTML -> {{ content|safe }}
- <Link href="/blog"> -> <a href="{% url 'blog-index' %}">

Key React-Forms-to-Django translations:
- useState + onChange -> Django Form class handles state
- Zod schema -> Form field definitions (CharField, EmailField)
- Zod .refine() -> clean_fieldname() method
- Zod .superRefine() -> clean() method
- onSubmit handler -> view function with request.method == "POST"
- Manual error display -> {{ form.field.errors }} (automatic!)
- CSRF token (manual) -> {% csrf_token %} (one line)
- ~50-100 lines per React form -> ~10-20 lines per Django form

Mental models:
1. DTL is JSX for server-side rendering — {{ }} for values, {% %} for logic
2. Template inheritance = layouts — {% extends %} + {% block %} = layout.tsx + {children}

LESSON CONTENT

DJANGO TEMPLATE LANGUAGE (DTL) OVERVIEW

DTL is Django's built-in way to generate HTML dynamically. If JSX lets you write HTML inside JavaScript, DTL lets you write Python-like logic inside HTML. The big difference: DTL runs entirely on the server — no JavaScript ships to the browser.

Double curly braces {{ }} output values (like JSX's single curly braces). Curly-percent {% %} runs logic (like JSX's ternaries and .map calls). Everything renders on the server before the browser sees it.

VARIABLES & FILTERS

Variables come from the context dictionary your view passes to the template. Filters transform them — like piping through utility functions.

{{ title|upper }} — uppercase. {{ title|length }} — count. {{ created|date:"M d, Y" }} — format date. {{ description|default:"No description" }} — fallback. {{ body|safe }} — render raw HTML (like dangerouslySetInnerHTML). {{ list|join:", " }} — join. {{ body|truncatewords:30 }} — truncate. {{ title|slugify }} — URL-safe slug.

Security note: Django auto-escapes all {{ }} output to prevent XSS attacks. The |safe filter turns that off. Only use |safe on content you trust.

TEMPLATE TAGS — LOGIC IN TEMPLATES

Conditionals: {% if user.is_authenticated %}...{% else %}...{% endif %}
Loops: {% for post in posts %}...{% empty %}...{% endfor %}

forloop magic variables: Inside {% for %}, Django gives you forloop.counter (1-indexed), forloop.counter0 (0-indexed), forloop.first, forloop.last. The {% empty %} block handles empty lists — no need for a separate conditional.

TEMPLATE INHERITANCE — LAYOUTS IN DJANGO

{% extends "base.html" %} = "wrap me in the layout." {% block content %}...{% endblock %} = "this is where {children} goes." You can have multiple blocks (title, content, sidebar, extra_js) — like having multiple {children} slots, which React doesn't natively support.

{% include %} — REUSABLE COMPONENTS

{% include "components/post_card.html" with post=post %} — the "with" keyword passes variables (like props), and "only" restricts scope. This is Django's version of reusable React components.

THE {% url %} TAG — TYPE-SAFE LINKS

{% url 'blog-detail' post.slug %} generates URLs from route names. If you rename a URL pattern, all links update automatically. Next.js has no equivalent — you hardcode paths.

DJANGO FORMS — THE BIG PICTURE

In React, forms require: state management (useState), change handlers (onChange), validation (Zod/Yup), CSRF protection (custom), and rendering. Django handles all of this in one class. ~10-20 lines vs ~50-100 lines per form.

A Django Form class is like combining your Zod schema, your React component's form JSX, and your onChange handlers into ONE thing. Define the fields once; Django generates the HTML inputs, validates the data, and reports errors.

MODELFORM — FORMS FROM MODELS

ModelForm generates a form directly from your database model. It reads your model's field types and automatically creates the right HTML input. CharField becomes <input type="text">, BooleanField becomes a checkbox, ForeignKey becomes a <select> dropdown. All validation constraints from the model are inherited.

RENDERING FORMS IN TEMPLATES

{{ form.as_p }} renders the entire form automatically. {{ form.field_name }} renders a single field for custom layout. {{ form.title.errors }} displays field errors. Never forget {% csrf_token %}! Every POST form needs it — Django rejects submissions without it (403 error).

VALIDATION — clean() METHODS

Three layers of validation:
1. Field-type validation (automatic) — CharField checks it's a string, EmailField validates email format
2. clean_fieldname() methods — per-field custom logic (like Zod .refine())
3. clean() method — cross-field validation (like Zod .superRefine())

All errors auto-attach to fields and render in the template automatically.

THE VIEW PATTERN — HANDLING FORM SUBMISSIONS

One view handles both displaying the form (GET) and processing the submission (POST). On GET, create an empty form. On POST, bind the submitted data. If validation fails, the same template re-renders with errors pre-filled. After a successful save, always redirect() instead of rendering (PRG pattern — Post/Redirect/Get).

FILE UPLOADS

Django handles file uploads natively. Two things to remember: the template form must include enctype="multipart/form-data", and the view must pass request.FILES to the form constructor.

WIDGETS

Widgets control how a field renders as HTML. In React, you write <input type="date" className="form-control" />. In Django, you configure the same thing via the widget's attrs dictionary.

------------------------------------------------------------
KEY CONCEPTS TO EMPHASIZE
------------------------------------------------------------

1. {{ }} for values, {% %} for logic — this is the entire DTL syntax in two symbols.
2. Template inheritance is layouts — {% extends %} + {% block %} = layout.tsx + {children}, but with multiple named slots.
3. Django Forms = Zod + React Hook Form in one — define fields once, get validation + HTML + error handling.
4. ModelForm is the ultimate shortcut — auto-generates forms from your database model with inherited validation.
5. {% csrf_token %} is non-negotiable — every POST form needs it. Django's built-in security for free.

------------------------------------------------------------
COMMON PITFALLS FOR JS DEVELOPERS
------------------------------------------------------------

- Using single curly braces {var} instead of double {{ var }} in templates
- Forgetting {% endif %} or {% endfor %} (block tags must be closed)
- Forgetting {% csrf_token %} in POST forms (results in 403 error)
- Not passing request.FILES for file upload forms
- Forgetting enctype="multipart/form-data" on file upload form tags
- Trying to write JavaScript logic in templates (DTL is server-side Python-like, not JS)
- Not using {% url 'name' %} for links (hardcoding URLs breaks when routes change)
- Expecting React-style component state in templates (templates are stateless, server-rendered)

------------------------------------------------------------
REAL-WORLD CONNECTION
------------------------------------------------------------

At a broadcast TV network, templates display the public-facing pages: story listings, article detail pages, reporter profiles, dataset previews. Template inheritance keeps the site consistent — one base.html with nav, footer, and branding. Forms power the internal workflow: reporters submit stories through a StoryForm, editors approve them through a status change form, researchers upload datasets through a file upload form. The admin panel handles the rest. ModelForm generates these forms from the database models you already defined — no duplicate schema definitions. CSRF protection is automatic for every form. This is the "full stack" in action: models define data, views process logic, templates display results, forms collect input.
