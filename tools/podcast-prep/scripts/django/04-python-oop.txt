LESSON CONTENT
------------------------------------------------------------

CLASSES — THE BASICS

Three differences from JS:
1. constructor() becomes __init__()
2. this becomes self — but self MUST be the first parameter of every method
3. No "new" keyword — just call the class like a function: Article("Scoop", "Trevor")

Forgetting self is the #1 class bug. Every instance method must include self as its first parameter. Without it: TypeError: publish() takes 0 positional arguments but 1 was given.

SELF VS THIS — EXPLICIT VS IMPLICIT

In JavaScript, "this" is implicit and its value depends on HOW the function is called (causing endless binding bugs). In Python, self is explicit and always refers to the instance.

When you call a.publish(), Python translates that to Article.publish(a). No magic. No surprises.

This is why Python doesn't need arrow functions — there's no "this" binding to fix.

INSTANCE VS CLASS VARIABLES

Class variables (defined outside __init__) are shared by ALL instances. Instance variables (defined with self.x in __init__) are unique per instance.

Mutable class variable trap: If you put tricks = [] as a class variable, ALL instances share that same list. Always define mutable attributes inside __init__.

DUNDER METHODS — Python's Operator Overloading

"Dunder" means double underscore. These special methods are called automatically by operators, built-in functions, and language features.

__str__: For humans. What print() shows. Django admin uses this to display model records.
__repr__: For developers. What the REPL shows. Should look like valid Python code.
__len__: Makes len(obj) work.
__eq__: Makes == work between objects.
__lt__: Makes < work and enables sorted().
__contains__: Makes "x in obj" work.
__getitem__: Makes obj[key] work.
__iter__: Makes "for x in obj" work. Uses yield for generators.
__call__: Makes obj() work (callable objects).

The mental model: When you write len(obj), Python calls obj.__len__(). When you write obj[key], Python calls obj.__getitem__(key). Every operator has a corresponding dunder. This is how Django's ORM makes queryset[0:5] and "item in queryset" feel like native Python.

@PROPERTY — GETTERS AND SETTERS

Same concept as JS get/set, different syntax. @property makes a method behave like an attribute (no parentheses to access it).

JS: get area() { return ... } -> Python: @property def area(self): return ...
JS: set diameter(d) { ... } -> Python: @diameter.setter def diameter(self, d): ...

Django uses @property on models for computed fields like full_name or is_overdue.

INHERITANCE & SUPER()

extends becomes parentheses: class Circle(Shape):
instanceof becomes isinstance(c, Shape)
super().method() works the same way.

Multiple inheritance: JS can't do it. Python can. class Article(Timestamped, Slugified): combines both parents' capabilities. Django uses this heavily for class-based views: class MyView(LoginRequiredMixin, ListView): combines authentication with list rendering.

@STATICMETHOD AND @CLASSMETHOD

@staticmethod: utility function namespaced to the class, no access to class or instance. Like JS static.
@classmethod: receives the class as cls, enabling factory methods. Using cls() instead of ClassName() means subclasses automatically get the right type back. JS has no equivalent.

DATACLASSES — TypeScript Interfaces Meet Classes

@dataclass reads type-annotated fields and auto-generates __init__, __repr__, __eq__. Like TypeScript interfaces that actually DO things at runtime.

@dataclass class Article: title: str; author: str; views: int = 0; published: bool = False

Mutable defaults in dataclasses use field(default_factory=list) instead of [].
frozen=True makes instances immutable (like Object.freeze).
order=True enables sorting by fields.

Django models are similar but more powerful — they generate database tables AND admin interfaces from field declarations.

PRIVACY CONVENTIONS

Python has no true privacy. _single_underscore = "treat as private" (convention). __double_underscore = name mangling (harder to access but still not truly private). JS #private fields are genuinely inaccessible.

Python's philosophy: "We're all consenting adults here."

ABSTRACT CLASSES

Python's ABC (Abstract Base Class) enforces at runtime — trying to instantiate an abstract class raises TypeError. TypeScript's abstract is compile-time only.

Django uses abstract models (class Meta: abstract = True) for shared fields that don't create database tables.

------------------------------------------------------------
KEY CONCEPTS TO EMPHASIZE
------------------------------------------------------------

1. self is not magic — it's just the first parameter. a.method() is really Class.method(a). This eliminates all binding confusion.
2. __str__ on every model — Django admin uses it to display records. Always define it. Without it, you see "Article object (1)" everywhere.
3. Dunder methods connect your classes to Python's built-in ecosystem — they're the reason len(), ==, "in", for loops, and [] all work on custom objects.
4. Multiple inheritance enables Django's mixin pattern — understanding this unlocks class-based views.
5. @dataclass eliminates boilerplate — but Django models do even more (database tables + admin UI from class declarations).

------------------------------------------------------------
COMMON PITFALLS FOR JS DEVELOPERS
------------------------------------------------------------

- Forgetting self as the first parameter of every method
- Using "new" before class instantiation (Python doesn't use new)
- Putting mutable values (lists, dicts) as class variables instead of instance variables
- Expecting # syntax for private fields (Python uses _ convention)
- Not defining __str__ on classes (Django admin shows ugly "Object (1)" without it)
- Trying single inheritance workarounds when multiple inheritance would be cleaner
- Forgetting the @property decorator and calling computed attributes with parentheses

------------------------------------------------------------
REAL-WORLD CONNECTION
------------------------------------------------------------

Every Django model is a class. When you write class Article(models.Model) in lesson 7, that's Python inheritance — Article inherits from Django's Model base class, which provides all the database magic. The __str__ method you define shows up in Django admin. The @property decorator creates computed fields. Django class-based views use multiple inheritance for mixins: class StoryList(LoginRequiredMixin, ListView) combines authentication and list display. At a broadcast network, your data models for shows, episodes, ratings, and schedules are all Python classes with dunder methods making them sortable, comparable, and printable.
