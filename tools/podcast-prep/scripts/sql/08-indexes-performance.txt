MENTAL MODELS THAT MATTER MOST:
1. Book index analogy: A database index is like the index at the back of a textbook. Instead of reading every page (sequential scan), you look up the term and go directly to the right pages (index scan).
2. B-tree is like a phone book: Sorted data structure that finds any value in ~20 steps regardless of table size. Works for =, <, >, <=, >=, BETWEEN, IN, LIKE 'prefix%', ORDER BY.
3. The tradeoff: Indexes speed up reads but slow down writes. Every INSERT, UPDATE, DELETE must update all indexes on the table. Extra storage space.
4. Composite index column order matters: Like a phone book sorted by (last_name, first_name). You can look up by last_name alone, but not by first_name alone.

--------------------------------------------------------------------------------
KEY CONCEPTS TO EMPHASIZE
--------------------------------------------------------------------------------

1. HOW B-TREE INDEXES WORK: The default index type. Sorted data structure that finds values by cutting possibilities in half at each step. 1 million rows = ~20 steps. Without an index, 1 million checks. B-trees work for equality (=), comparisons (<, >, etc.), BETWEEN, IN, LIKE 'prefix%', and ORDER BY. They do NOT help with LIKE '%suffix'.

2. CREATING AND DROPPING INDEXES: CREATE INDEX idx_name ON table (column). Naming convention: idx_tablename_columnname. CREATE UNIQUE INDEX enforces uniqueness + speeds up lookups. DROP INDEX removes it. PostgreSQL auto-creates indexes on primary keys and unique constraints but NOT on foreign keys.

3. COMPOSITE INDEXES: Cover multiple columns. Column order is critical (leftmost prefix rule). An index on (A, B, C) helps queries on (A), (A, B), (A, B, C) but NOT (B), (C), or (B, C). Put the most frequently filtered column first. Sort order (ASC/DESC) can be specified.

4. EXPLAIN AND EXPLAIN ANALYZE: EXPLAIN shows the planned execution strategy without running the query. EXPLAIN ANALYZE actually runs it and shows real timing. Key scan types: Seq Scan (slow, reading every row), Index Scan (fast, using index), Index Only Scan (fastest, all data from index), Bitmap Index Scan (good for many matches).

5. INDEX-ONLY SCANS AND COVERING INDEXES: When all requested columns are in the index, the database never reads the table rows. INCLUDE clause adds extra columns to the index for index-only scans without using them for sorting.

6. WHEN NOT TO INDEX: Small tables, boolean columns (low cardinality), write-heavy tables with rare reads, columns never used in WHERE/ORDER BY, frequently updated columns. Start with no indexes beyond automatic ones, add when EXPLAIN shows problems.

7. PARTIAL INDEXES: Only index rows matching a condition. CREATE INDEX ON articles (published_date DESC) WHERE published = true. Smaller and faster because they contain fewer rows.

8. EXPRESSION INDEXES: Index the result of a function. CREATE INDEX ON users (LOWER(email)). The query must use the exact same expression.

9. PRISMA VS RAW SQL: Prisma's @@index handles basics. Raw SQL is needed for partial indexes, expression indexes, INCLUDE columns, and custom sort orders.

--------------------------------------------------------------------------------

Performance is not an abstract concern for data journalists. When you are exploring a FOIA dataset with 500,000 rows, every query should return in under a second. If it takes 10 seconds, you lose your train of thought. If it takes a minute, you stop exploring altogether.

LESSON CONTENT

THE MENTAL MODEL -- BOOK INDEX ANALOGY

Imagine a 500-page textbook. You need to find every mention of "PostgreSQL." Two options:
- Without an index: Read every page, start to finish. Sequential scan. Works, but slow.
- With an index: Flip to the index at the back, look up "PostgreSQL," go directly to pages 12, 47, 189, 301. Done in seconds.

A database index works exactly the same way. It is a separate data structure that maps values to row locations. The database jumps directly to what it needs instead of scanning every row.

The tradeoff is the same: the index takes up extra space (storage) and needs updating when the data changes (write performance). But for lookups, it is dramatically faster.

HOW B-TREE INDEXES WORK

The default index type in PostgreSQL is a B-tree (balanced tree). Think of it like a phone book organized alphabetically. To find "Martinez": open to the middle, you are at "K," too early. Open to the middle of the second half, "P," too late. Middle of K-P, "M," close. A few more steps and you are there.

Each step cuts remaining possibilities in half. For 1 million rows, about 20 steps. Without an index: 1 million checks.

  Table Size        Without Index        With B-tree
  1,000 rows        1,000 checks         ~10 steps
  100,000 rows      100,000 checks       ~17 steps
  1,000,000 rows    1,000,000 checks     ~20 steps
  10,000,000 rows   10,000,000 checks    ~23 steps

B-trees handle more than equality. Because values are sorted, they work for =, <, >, <=, >=, BETWEEN, IN, LIKE 'prefix%', and ORDER BY. They do NOT help with LIKE '%suffix%' (no fixed prefix).

CREATING AND DROPPING INDEXES

  CREATE INDEX idx_articles_author_id ON articles (author_id);
  CREATE UNIQUE INDEX idx_users_email ON users (email);
  DROP INDEX idx_articles_author_id;

Naming convention: idx_tablename_columnname.

PostgreSQL automatically creates indexes for primary keys and unique constraints. It does NOT auto-create indexes on foreign keys. This is a common performance trap -- JOINs on unindexed foreign keys are slow.

Prisma's @@index([authorId]) generates CREATE INDEX. @unique generates CREATE UNIQUE INDEX.

COMPOSITE INDEXES -- COLUMN ORDER MATTERS

A composite index covers multiple columns. Column order is critical because B-trees sort left to right.

Think of a phone book sorted by (last_name, first_name):
- "Everyone named Smith" -- works (first column)
- "Smith, John" -- works (both columns)
- "Everyone named John" -- does NOT work (skips first column)

  This index: (section, published_date)
  HELPS: WHERE section = 'news'
  HELPS: WHERE section = 'news' AND published_date > '2024-01-01'
  DOES NOT HELP: WHERE published_date > '2024-01-01'

The leftmost prefix rule: An index on (A, B, C) can be used for (A), (A, B), or (A, B, C). But NOT (B), (C), or (B, C) alone.

EXPLAIN AND EXPLAIN ANALYZE

  EXPLAIN SELECT * FROM articles WHERE author_id = 5;
  -- Shows the plan without running the query

  EXPLAIN ANALYZE SELECT * FROM articles WHERE author_id = 5;
  -- Runs the query and shows real timing

Key scan types:
- Seq Scan: Reading every row. Slow on large tables.
- Index Scan: Using an index. Fast.
- Index Only Scan: Getting all data from the index itself. Fastest.
- Bitmap Index Scan: Using index to build a map, then fetching. Good for many matches.

EXPLAIN ANALYZE actually executes the query. For DELETE or UPDATE, wrap in BEGIN/ROLLBACK.

PARTIAL AND EXPRESSION INDEXES

Partial indexes only index rows matching a condition:
  CREATE INDEX idx_articles_published_only ON articles (published_date DESC) WHERE published = true;

Smaller and faster because they contain fewer rows.

Expression indexes index the result of a function:
  CREATE INDEX idx_users_email_lower ON users (LOWER(email));
  -- Query must use LOWER(email) to match

PRACTICAL: OPTIMIZING A SLOW QUERY

From 1.8 seconds to 0.1 milliseconds on a 500,000-row table:
1. No indexes: Seq Scan, 1847 ms
2. Add index on WHERE columns: 245 ms (7x faster)
3. Add sort column to index: 0.89 ms (2000x faster)
4. Add foreign key index for JOIN: 0.34 ms
5. Use partial index: 0.12 ms (14,000x faster than original)
