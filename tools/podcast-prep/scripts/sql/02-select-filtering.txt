KEY CONCEPTS TO EMPHASIZE
--------------------------------------------------------------------------------

1. SELECT EXPRESSIONS AND ALIASES: SELECT doesn't just pick columns -- it computes new ones. salary / 12 AS monthly_salary creates a calculated column. This is like .map(item => ({ ...item, monthly: item.salary / 12 })). AS names the computed column so it shows up with a proper label.

2. PATTERN MATCHING (LIKE, ILIKE, REGEX): LIKE uses % (any characters) and _ (one character) as wildcards. LIKE is case-sensitive. ILIKE is case-insensitive (PostgreSQL-specific). For journalism, always use ILIKE because public records have inconsistent capitalization. PostgreSQL also supports regex with ~ (case-sensitive) and ~* (case-insensitive), using the same regex patterns Trevor knows from JavaScript.

3. NULL HANDLING: This is the biggest "gotcha" for JavaScript developers learning SQL. NULL means "unknown" -- not "empty" or "zero." NULL = NULL evaluates to NULL (not true). You must use IS NULL and IS NOT NULL. COALESCE(value, 'default') is the SQL version of value ?? 'default'. NULLIF(a, b) returns NULL if a equals b, useful for preventing division by zero.

4. CASE EXPRESSIONS: SQL's ternary/switch. CASE WHEN condition THEN result ELSE default END. Conditions evaluate top to bottom (first match wins). Can be used in SELECT, WHERE, ORDER BY, or inside aggregate functions. The CASE-inside-COUNT pattern is essential for data analysis: COUNT(CASE WHEN status = 'active' THEN 1 END).

5. DISTINCT: Removes duplicate rows. Like [...new Set(array)]. DISTINCT applies to the entire row, not individual columns. COUNT(DISTINCT column) counts unique values.

6. TYPE CASTING: PostgreSQL's :: operator (shorthand for CAST). '42'::INTEGER converts types. Important for mixed-type math and formatting.

7. BOOLEAN LOGIC PRECEDENCE: AND binds tighter than OR (just like && binds tighter than ||). Always use parentheses when mixing AND and OR to make intent clear.

DATA JOURNALISM CONNECTION

This lesson is the data journalist's Swiss Army knife. Every technique here maps directly to investigative work:

ILIKE for name searching: When investigating campaign donors, government contractors, or corporate officers, names in public records are spelled inconsistently. WHERE donor_name ILIKE '%smith%' catches "Smith," "SMITH," "smith," and "Blacksmith." Without ILIKE, you miss records and miss stories.

DISTINCT for data profiling: The first thing a data journalist does with a new dataset is ask "what are the unique values in each column?" SELECT DISTINCT column FROM table answers that instantly.

You need ILIKE to search station names, COALESCE to handle missing readings, CASE to flag extreme conditions, and DISTINCT to see what states are represented. SELECT EXPRESSIONS AND ALIASES

Lesson 01 taught SELECT to pick columns. But SELECT can also compute things. Think of it like .map(item => ({ ...item, fullName: item.first + ' ' + item.last })) -- you are transforming data on the way out.

Aliases with AS:
AS renames a column in the output. The data does not change -- just the label.

  SELECT
    first_name AS first,
    last_name AS last,
    salary / 12 AS monthly_salary
  FROM employees;

Without AS, a calculated column shows up as ?column? in the results. Aliases give computed columns a proper name.

JavaScript equivalent:
  employees.map(emp => ({
    first: emp.first_name,
    last: emp.last_name,
    monthly_salary: emp.salary / 12,
  }));

String Concatenation:
In PostgreSQL, use || to concatenate strings (like + in JavaScript):

  SELECT first_name || ' ' || last_name AS full_name
  FROM employees;

Useful String Functions:
UPPER(str) = str.toUpperCase()
LOWER(str) = str.toLowerCase()
LENGTH(str) = str.length
TRIM(str) = str.trim()
SUBSTRING(str, start, len) = str.substring(start, end)
REPLACE(str, from, to) = str.replace(from, to)

Important: SQL string positions start at 1, not 0. SUBSTRING('hello', 1, 3) gives 'hel'. This is different from JavaScript where indexes start at 0.

Math in SELECT:
Standard math operators work: +, -, *, /, % (modulo). ROUND(value, decimals) rounds to specified decimal places.

  SELECT
    name,
    salary,
    salary * 0.30 AS tax,
    salary * 0.70 AS take_home,
    ROUND(salary / 2080.0, 2) AS hourly_rate
  FROM employees;

COMPARISON OPERATORS

BETWEEN:
BETWEEN is inclusive on both ends. Cleaner than >= AND <=:

  WHERE salary BETWEEN 50000 AND 100000
  WHERE hire_date BETWEEN '2023-01-01' AND '2023-12-31'

IN:
IN checks if a value matches any item in a list. Like .includes() in JavaScript:

  JavaScript: ['NY', 'CA', 'TX'].includes(emp.state)
  SQL: WHERE state IN ('NY', 'CA', 'TX')

  NOT IN excludes specific values:
  WHERE department NOT IN ('HR', 'Legal')

PATTERN MATCHING -- LIKE, ILIKE, AND REGEX

Pattern matching is essential for investigative work. Searching names, addresses, document text -- you need to match partial strings.

LIKE and Wildcards:
LIKE uses two wildcards:
- % = any number of characters (like .* in regex)
- _ = exactly one character (like . in regex)

  WHERE name LIKE 'John%'     -- starts with 'John'
  WHERE name LIKE '%son'      -- ends with 'son'
  WHERE name LIKE '%smith%'   -- contains 'smith'
  WHERE code LIKE '_____'     -- exactly 5 characters
  WHERE code LIKE 'A_C%'      -- starts with A, any char, then C

ILIKE (PostgreSQL-Specific):
ILIKE is the case-insensitive version of LIKE. This is a PostgreSQL extension.

  WHERE name LIKE '%smith%'   -- case-sensitive (misses 'SMITH', 'Smith')
  WHERE name ILIKE '%smith%'  -- case-insensitive (catches all variations)

For journalism: always use ILIKE. Public records have inconsistent capitalization. ILIKE saves you from missing records because someone typed "JOHNSON" instead of "Johnson."

PostgreSQL Regex Matching:
For more complex patterns, PostgreSQL supports regex with ~ and ~*:

  ~ = regex match (case-sensitive)
  ~* = regex match (case-insensitive)
  !~ = does NOT match regex
  !~* = does NOT match regex (case-insensitive)

  WHERE name ~ '^John(son|ston)$'
  WHERE email ~* '^[a-z]+@gmail\.com$'
  WHERE phone !~ '^\d{3}-\d{3}-\d{4}$'

NULL HANDLING

This is the concept that trips up every developer. NULL in SQL is not null in JavaScript. It means "unknown" or "not applicable" -- not "empty" or "zero."

The critical rule: NULL is not equal to anything -- not even itself. NULL = NULL is NOT true. It evaluates to NULL (unknown). You MUST use IS NULL or IS NOT NULL instead of =.

  WRONG: SELECT * FROM contacts WHERE phone = NULL;
  CORRECT: SELECT * FROM contacts WHERE phone IS NULL;
  SELECT * FROM contacts WHERE phone IS NOT NULL;

COALESCE -- SQL's Nullish Coalescing:
COALESCE returns the first non-NULL value from a list. Exactly like ?? in JavaScript.

  JavaScript: const display = name ?? 'Anonymous';
  SQL: SELECT COALESCE(name, 'Anonymous') AS display;

  Chain multiple fallbacks (like a ?? b ?? c):
  SELECT COALESCE(mobile_phone, work_phone, home_phone, 'No phone') AS best_phone
  FROM contacts;

NULLIF:
NULLIF(a, b) returns NULL if a = b, otherwise returns a. Useful for preventing division by zero:

  SELECT votes_for / NULLIF(total_votes, 0) AS percentage FROM results;

BOOLEAN LOGIC -- AND, OR, NOT

Same as JavaScript's &&, ||, !, but operator precedence matters more.

AND binds tighter than OR (just like && binds tighter than ||). Use parentheses to be explicit:

  DANGEROUS:
  WHERE beat = 'politics' AND trust_level > 7 OR beat = 'crime'
  -- Means: (politics AND trust > 7) OR crime -- returns ALL crime sources

  SAFE:
  WHERE (beat = 'politics' OR beat = 'crime') AND trust_level > 7

Rule of thumb: If you have both AND and OR in a WHERE clause, always use parentheses.

DISTINCT -- Unique Values

DISTINCT removes duplicate rows. Like [...new Set(array)] in JavaScript.

  JavaScript: const departments = [...new Set(employees.map(e => e.department))];
  SQL: SELECT DISTINCT department FROM employees;

  Count unique values: SELECT COUNT(DISTINCT department) AS dept_count FROM employees;

  DISTINCT applies to the entire row. SELECT DISTINCT department, title returns unique (department, title) pairs, not unique departments and unique titles separately.

CASE EXPRESSIONS -- SQL's Ternary / Switch

CASE is SQL's version of if/else or switch. It creates computed columns with conditional logic.

  JavaScript ternary: const label = salary > 100000 ? 'Senior' : 'Standard';
  SQL: SELECT name, CASE WHEN salary > 100000 THEN 'Senior' ELSE 'Standard' END AS label;

Multi-condition CASE:
  SELECT name,
    CASE
      WHEN salary > 150000 THEN 'Executive'
      WHEN salary > 100000 THEN 'Senior'
      WHEN salary > 60000  THEN 'Mid'
      ELSE 'Entry'
    END AS tier
  FROM employees;

Structure: CASE WHEN condition THEN result ... ELSE default END. Conditions are evaluated top to bottom -- first match wins.

CASE can be used anywhere an expression is valid -- SELECT, WHERE, ORDER BY, even inside functions.

CASE for Conditional Counting (power pattern):
  SELECT
    COUNT(*) AS total_articles,
    COUNT(CASE WHEN published = true THEN 1 END) AS published_count,
    COUNT(CASE WHEN published = false THEN 1 END) AS draft_count,
    COUNT(CASE WHEN views > 1000 THEN 1 END) AS viral_count
  FROM articles;

This pattern is incredibly useful for journalism. Count subgroups in a single query: complaints per category, donors per bracket, violations per district.

CASTING AND TYPE CONVERSION

Sometimes you need to convert between types. PostgreSQL has two syntaxes:

CAST() -- Standard SQL:
  SELECT CAST('42' AS INTEGER);
  SELECT CAST(salary AS TEXT);

:: Operator -- PostgreSQL Shorthand:
  SELECT '42'::INTEGER;
  SELECT salary::TEXT;
  SELECT '2024-01-15'::DATE;

When do you need casting? Most often when: (1) comparing columns of different types, (2) formatting output, (3) doing math with mixed types (integer division gives integers -- use ::NUMERIC to get decimals).

PUTTING IT ALL TOGETHER

A realistic query combining everything from this lesson -- investigating campaign donations:

  SELECT
    donor_name,
    UPPER(donor_state) AS state,
    amount,
    CASE
      WHEN amount >= 10000 THEN 'Major Donor'
      WHEN amount >= 1000  THEN 'Mid-Level'
      ELSE 'Small Donor'
    END AS donor_tier,
    COALESCE(employer, 'Not disclosed') AS employer_display,
    donation_date::TEXT AS date_string
  FROM donations
  WHERE donor_state IN ('NY', 'CA', 'TX', 'FL')
    AND amount > 500
    AND donor_name ILIKE '%smith%'
    AND donation_date BETWEEN '2024-01-01' AND '2024-06-30'
  ORDER BY amount DESC
  LIMIT 50;

Read it out loud: "Select the donor name, state (uppercased), amount, a tier label based on the amount, the employer (or 'Not disclosed' if null), and the date as text -- from donations -- where the state is one of four states, amount is over 500, the name contains 'smith' (case-insensitive), and the date is in the first half of 2024 -- sorted by amount descending, top 50."
