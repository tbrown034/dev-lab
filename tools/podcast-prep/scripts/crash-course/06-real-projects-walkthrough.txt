THE TEAM'S OUTPUT — WHAT HIGH-OUTPUT DATA VISUALS TEAMS PRODUCE

A strong newsroom data visuals team contributes to roughly 100 stories per year. That's a story every 3-4 days. These are high-output, high-impact teams. Here are the types of standout projects that define what they do and how they think:

ELECTION RESULTS PAGES

This is the flagship product for most newsroom data teams.

- Features: county-level interactive maps, race-by-race results tables, personalized "find your races" search by address
- Tech stack in action: Django backend ingesting AP election data, Django REST Framework API serving JSON, D3 rendering maps and charts, React for interactive components
- Deployed as standalone apps, separate from the CMS
- Why it matters: the developer who owns this system is responsible for the most high-traffic, high-stakes product the team ships. Understanding the full pipeline — the Django management commands that poll AP data, the ORM models that store results, the DRF serializers that shape the JSON, and the D3 code that renders county-level maps with geoPath and choropleth color scales — is essential.

CREATIVE DATA VISUALIZATIONS

- Examples include "fish in a pond" visual metaphors showing how only a fraction of voters decide elections, or geographic trackers showing infrastructure progress
- These are non-standard visualizations — the kind of thing D3 excels at because you can't do this with Chart.js or a charting library with pre-built chart types
- Shows that the best teams value creative storytelling, not just standard bar charts
- D3's power is custom mark rendering — when the story needs a unique visual form, D3 can build it from primitives (circles, paths, force simulations, whatever the metaphor demands)

HOUSING AND ZONING ANALYSIS

- Gathering and analyzing zoning data from multiple major cities
- Showing how dominant single-family zoning is across metro areas
- These projects involve significant data gathering, analysis, AND visualization — the full data journalism pipeline
- The workflow: acquire data (likely through FOIA and direct requests to city planning departments) -> clean and analyze -> visualize -> tell the story
- This is the kind of project where investigative journalism skills (records requests, data validation, source verification) become a superpower

GEOGRAPHIC AND POLITICAL ANALYSIS

- Tracking political shifts in voting patterns over time
- Time-series data visualization — line charts or small multiples showing voting trends across multiple election cycles
- This is the "data reporter" role in action: finding the story in the numbers, not just visualizing someone else's data

REDISTRICTING TOOLS

- Interactive tools for exploring proposed district boundaries
- Uses D3 mapping extensively — drawing district boundaries with geoPath, overlaying demographic data as choropleths or proportional symbols
- Readers can explore how proposed maps would affect their community
- This is a tool, not just a graphic — it has user interaction, search, comparison features
- The engineering challenge: handling complex GeoJSON data efficiently, making comparisons between proposed and current districts intuitive, keeping the interface fast with large geographic datasets

LEGISLATIVE SESSION TRACKERS

- Tracking key proposals through a legislature
- Letting readers see how their representatives voted on consequential measures
- Combines database work (tracking hundreds of bills and votes over months) with visualization
- This is the kind of project that lives for an entire legislative session — it needs to be maintained, updated, and reliable over months
- The Django admin likely plays a key role here: editors and reporters can flag which bills to feature, add context, update statuses without needing a developer for every change

PERSONALIZED LOOKUP TOOLS

- Water crisis explainers where readers enter their location to see projected shortage severity
- School explorers where readers look up any school and see ratings, demographics, test scores
- Personalized data lookup — the same UX pattern as the election results "find your races" feature
- Shows the team's recurring pattern: take big statewide data and make it personally relevant to individual readers
- The technical pattern: user enters input (address, zip code, county) -> query a database or API -> return personalized results -> render with D3 or React

BREAKING NEWS VISUALIZATION

- Flood coverage, natural disaster mapping, outbreak trackers
- Shows the team can move fast on breaking news, not just planned features
- Breaking news visualization requires a different skill set: rapid data acquisition, fast but accurate rendering, clear communication under pressure
- Newsroom experience with deadline journalism is directly relevant here

PUBLIC HEALTH TRACKERS

- Analyzing vaccination trends in schools, tracking disease outbreaks
- Public health data + geographic mapping
- Another example of data reporting: acquiring the data, analyzing it, finding the story, then building the visualization
- This is the full pipeline from data acquisition to published graphic

SCHOOL DATA APPS — ONGOING PRODUCTS

- Long-running data apps that let readers look up any school and see ratings, demographics, test scores
- Built with Python/Django, with separate repos for the app and the datasets
- Uses education data from state agencies
- These are maintained products, not one-off graphics — they get updated every school year
- The engineering challenge is different from a one-off graphic: data migrations when agencies change formats, maintaining backward compatibility, handling edge cases in school data (closures, mergers, name changes)


PATTERNS IN THEIR WORK — WHAT TOP DATA TEAMS VALUE

Looking across all these projects, five patterns emerge that define how the best newsroom data visuals teams think:

1. PERSONALIZATION
Almost every major project lets readers enter their address or location to see personally relevant data. Election results pages have "find your races." Water crisis tools have "see your area." School explorers have "look up your school." Redistricting tools have "see your district." The best teams don't just publish statewide data — they make it about YOUR school, YOUR district, YOUR water supply, YOUR representative. This is a design philosophy, not just a feature. It's rooted in the journalistic principle that stories matter most when readers see themselves in them.

2. DATA ACQUISITION AS JOURNALISM
Many projects involve gathering data that isn't readily available — FOIA requests, scraping state agencies, combining multiple datasets from different sources. Housing zoning analysis requires collecting data from multiple city planning departments. Health trackers require getting records from schools. The best teams don't just visualize data that falls into their lap — they go get it, like investigative reporters working a story.

3. CREATIVE VISUALIZATION
Fish-in-a-pond voter metaphors, housing zoning maps, geographic trackers — the best teams don't default to bar charts. They find the visual form that tells the story best. When a bar chart is the right answer, they use a bar chart. But when the story calls for something novel — a metaphor, a geographic rendering, a custom interactive — they build it from scratch with D3. This is why teams use D3 instead of Chart.js or Highcharts: D3 gives them the freedom to invent.

4. TOOLS, NOT JUST GRAPHICS
School explorers, redistricting tools, legislative trackers, election results pages — these aren't single-view graphics that get published and forgotten. They're products that readers use over weeks and months. School explorers run for years. Election results systems get rebuilt every cycle. This requires a different engineering mindset: thinking about maintainability, data updates, edge cases, and long-term reliability.

5. SPEED ON BREAKING NEWS
Flood coverage, election night results — the team can ship quality work under extreme deadline pressure. This means having systems (like the election pipeline) that can be activated quickly, having team members who can make editorial and technical decisions fast, and having deployment workflows that don't bottleneck on approvals.

6. DATA ANALYSIS AS JOURNALISM
The "data reporter" role shows these teams do the analysis, not just the visualization. They find the story in the data. Analyzing shifts in voting patterns across election cycles isn't assigned by an editor who already knows the trend — the data reporter finds it by analyzing the data. This is investigative journalism applied to datasets instead of documents.


HOW A JOURNALIST-DEVELOPER CONTRIBUTES TO EACH TYPE OF PROJECT

ELECTION RESULTS:
- Maintain the Django backend: update models if AP changes their data format, optimize management commands for faster data ingestion, ensure the DRF API serves the right shape of JSON
- Update and improve D3 visualizations: county-level choropleth maps, results tables, the enter/update pattern for live election night updates
- A journalism background is an asset here that most developers don't have: understanding which races matter, what readers want to know on election night, how to handle race calls and corrections, when to show "too early to call" vs. projecting a winner
- Investigative skills add a dimension: finding stories IN the election data — unusual turnout patterns, demographic shifts, anomalies that deserve follow-up reporting

DATA ACQUISITION PROJECTS:
- Investigative journalists know FOIA inside and out, public records law, data scraping, how to verify data against official sources
- Building large-scale research databases means understanding data at scale
- Data cleaning, data validation, source verification — skills that most developers learn on the job but that journalists have been doing for years
- On projects like housing zoning analysis, a journalist-developer doesn't just build the visualization — they can help acquire the data, validate it, and spot the editorial angles

PERSONALIZED LOOKUP TOOLS:
- Full-stack experience with React/Next.js + PostgreSQL transfers directly
- The pattern — user enters input, query a database, display personalized results — is a standard full-stack workflow
- Experience with authentication and API development transfers to building tools where users can save preferences, create accounts, or bookmark results

CREATIVE VISUALIZATIONS:
- This is where the learning curve is steepest — the specific D3 API for custom marks, force simulations, novel visual metaphors
- But editorial judgment — knowing what to show, what to emphasize, what to leave out, what simplifies the story for readers — is a unique strength
- Most D3 developers can build a force-directed graph but can't tell you whether it's the right choice for the story. A journalist can make that editorial call.

AI EXPERIMENTATION:
- Many job listings now specifically mention interest in AI exploration
- Practical AI experience is valuable: using LLMs for data analysis and story discovery, making graphics more accessible with AI-generated alt text and summaries, exploring AI-assisted data cleaning and validation
- Most D3/Django developers don't have hands-on AI building experience — this is a strong differentiator


THE WORK RHYTHM — WHAT A TYPICAL WEEK LOOKS LIKE

Understanding the cadence helps you picture yourself in the role:

- Monday: Team standup. Review what's in progress, pick up tickets, discuss upcoming stories that need data support.
- Tuesday-Wednesday: Heads-down building time. Iterate on a visualization or data pipeline. Code review with teammates. Work through D3 rendering challenges or Django data model questions.
- Thursday: Polish and prepare for publishing. Help a reporter set up data in the Django admin. Test on mobile devices. Cross-browser QA.
- Friday: Ship, test, and deploy a graphic or update for the weekend edition. Update documentation. Prepare for next week.
- Election weeks: All hands on deck. Monitoring AP data pipelines, watching for ingestion errors, rapid iteration on results pages as data flows in, staying up late on election night.
- Legislative session: Ongoing updates to the session tracker, adding new bills, updating vote tallies, responding to reporter requests for quick data checks.
- Ongoing maintenance: School explorers need annual data updates when state agencies release new ratings. Election systems need pre-election prep months before election day. Long-running tools need bug fixes, performance improvements, and occasional redesigns.


THE BIG PICTURE — WHY THESE ROLES EXIST

Newsroom data visuals teams make information accessible to the public. They take complex datasets — election results, school ratings, legislative votes, water projections, zoning regulations — and turn them into tools that help people understand how policy affects their lives. A data visuals developer role isn't just a coding job. It's a journalism job that requires coding. The combination of investigative journalism expertise, full-stack development skills, and AI experience is what makes someone a uniquely strong fit. It's not just about knowing how to build the tools — it's about understanding why they matter.
