08 Tailwind v4 — CSS-First Architecture

← Back to lessons

You use Tailwind. Now understand it. Tailwind v4 is a complete rewrite. The JavaScript config file is gone. Everything is CSS-first. The Oxide engine is 10x faster. And if you've followed this course, the architecture will look familiar — it's built on the CSS features you just learned.

The Mental Model: Tailwind v4 Is a CSS Compiler

Tailwind v3 was a JavaScript plugin that happened to generate CSS. Tailwind v4 is a CSS tool that happens to use Rust for speed. The mental model shift:

Tailwind v3 (JS-first)
// tailwind.config.js
module.exports = {
  theme: {
    colors: {
      primary: '#a855f7',
      surface: '#0e0e12',
    },
    extend: {
      spacing: { '18': '4.5rem' },
    },
  },
  plugins: [
    require('@tailwindcss/forms'),
  ],
};

/* Config is JavaScript.
   Build reads JS, generates CSS.
   Customization = JS objects. */
Tailwind v4 (CSS-first)
/* app.css */
@import "tailwindcss";

@theme {
  --color-primary: #a855f7;
  --color-surface: #0e0e12;
  --spacing-18: 4.5rem;
}

@plugin "@tailwindcss/forms";

/* Config is CSS.
   Build reads CSS, generates CSS.
   Customization = CSS custom properties.
   No JS config file needed. */

Tailwind v4 tokens are CSS custom properties. When you write --color-primary: #a855f7 inside @theme, Tailwind generates the utility text-primary, bg-primary, border-primary, etc. AND it creates a CSS custom property --color-primary that you can use in your own CSS. The tokens live in CSS, not trapped in a JS config.

@theme: The New Config

@import "tailwindcss";

@theme {
  /* Colors — generates text-*, bg-*, border-*, etc. utilities */
  --color-primary: #a855f7;
  --color-primary-light: #c084fc;
  --color-surface: #0e0e12;
  --color-surface-2: #16161d;

  /* Spacing — generates p-*, m-*, gap-*, w-*, h-* utilities */
  --spacing-xs: 0.25rem;
  --spacing-sm: 0.5rem;
  --spacing-md: 1rem;
  --spacing-lg: 2rem;
  --spacing-xl: 4rem;

  /* Font families */
  --font-sans: 'DM Sans', system-ui, sans-serif;
  --font-mono: 'JetBrains Mono', monospace;
  --font-serif: 'Instrument Serif', Georgia, serif;

  /* Border radius */
  --radius-sm: 4px;
  --radius-md: 8px;
  --radius-lg: 12px;
  --radius-full: 9999px;

  /* Shadows */
  --shadow-card: 0 4px 12px rgba(0, 0, 0, 0.15);
  --shadow-elevated: 0 12px 40px rgba(0, 0, 0, 0.25);

  /* Animations */
  --ease-out-expo: cubic-bezier(0.16, 1, 0.3, 1);
}

/* Now you can use:
   text-primary, bg-surface, p-md, gap-lg,
   font-sans, rounded-md, shadow-card
   AND var(--color-primary), var(--spacing-md), etc. */

Connection to Lesson 05. The @theme directive is literally the "primitive tokens" layer from our design token architecture. Tailwind v4 takes those tokens and generates both utilities AND custom properties. This means your Tailwind config IS your design token system.

@utility: Custom Utilities

In v3, creating custom utilities required a JavaScript plugin. In v4, it's pure CSS:

/* Define a custom utility */
@utility text-balance {
  text-wrap: balance;
}

/* Now use it: <h1 class="text-balance"> */

/* Utility with modifiers (responsive, hover, etc.) */
@utility glass {
  background: rgba(255, 255, 255, 0.05);
  backdrop-filter: blur(20px) saturate(1.4);
  border: 1px solid rgba(255, 255, 255, 0.1);
}

/* Now use it: class="glass hover:glass md:glass" — modifiers work automatically */

/* Utility with a value (functional utility) */
@utility tab-size-* {
  tab-size: --value(--tab-size-*);
}

/* Now use: class="tab-size-4" */

@utility gets ALL Tailwind modifiers for free. When you define a custom utility with @utility, it automatically works with hover:, focus:, dark:, responsive prefixes, and every other Tailwind modifier. No extra configuration. This is the power of the CSS-first approach.

@variant: Custom Modifiers

/* Create a custom variant */
@variant pointer-coarse (@media (pointer: coarse));
@variant pointer-fine (@media (pointer: fine));

/* Use it: class="pointer-coarse:p-4 pointer-fine:p-2"
   Touch devices get more padding, mouse users get less */

/* Custom variant based on a data attribute */
@variant data-active (&[data-active="true"]);

/* Use it: class="data-active:bg-primary data-active:text-white" */

/* Custom variant for container queries */
@variant @md (@container (min-width: 448px));

/* Use it: class="@md:grid-cols-2 @md:gap-4" */

How Tailwind v4 Uses @layer Internally

/* Tailwind v4's internal layer structure */
@layer theme, base, components, utilities;

/* @layer theme — your @theme tokens + CSS custom properties */
/* @layer base — reset + default styles */
/* @layer components — @apply usage + component classes */
/* @layer utilities — all utility classes */

/* This is why utilities ALWAYS beat components:
   The utilities layer has higher cascade priority.
   Specificity doesn't matter.
   .card .title in components layer loses to
   .text-white in utilities layer. */

/* Your own layers slot in: */
@layer base {
  body { font-family: var(--font-sans); }
}

@layer components {
  .card {
    background: var(--color-surface);
    border-radius: var(--radius-md);
    padding: var(--spacing-md);
  }
}

Connection to Lesson 01. Remember how @layer order beats specificity? That's exactly what Tailwind v4 relies on. Utilities are in the highest layer, so text-white always beats .card .title { color: gray; } regardless of specificity. The cascade architecture you learned in Lesson 01 is the engine that makes Tailwind work.

The Oxide Engine: Why It's 10x Faster

Tailwind v4's build engine is rewritten in Rust (the "Oxide" engine). Key architectural differences:

  • Lightning CSS replaces PostCSS for CSS parsing and transformation
  • Content detection is built-in — no content config needed, it auto-detects your templates
  • Zero-config — install and import, it just works
  • Incremental builds — only reprocesses changed files
  • CSS-native — no JavaScript AST parsing needed for the config
/* v4 setup — literally this simple */

/* 1. Install */
/* npm install tailwindcss@next @tailwindcss/vite */

/* 2. Add to your CSS entry file */
@import "tailwindcss";

/* 3. Done. No config file. No content paths. No PostCSS config.
   It scans your project, finds all class names, generates CSS. */

Dark Mode and Multi-Theme in v4

/* Dark mode uses CSS custom properties under the hood */
@theme {
  --color-bg: #f5f4f1;
  --color-text: #1a1918;
}

@theme dark {
  --color-bg: #060608;
  --color-text: #e8e6f0;
}

/* This generates:
   :root { --color-bg: #f5f4f1; --color-text: #1a1918; }
   @media (prefers-color-scheme: dark) {
     :root { --color-bg: #060608; --color-text: #e8e6f0; }
   }
*/

/* Or use class-based dark mode for manual toggle: */
@variant dark (&:where(.dark, .dark *));

/* Multi-brand theming — same pattern as Lesson 05's token layers */
@layer base {
  [data-brand="news"] {
    --color-primary: #ef4444;
    --color-accent: #f97316;
  }
  [data-brand="sports"] {
    --color-primary: #22c55e;
    --color-accent: #10b981;
  }
}

Check Your Understanding

1. What's the fundamental architecture change from Tailwind v3 to v4?

v3 was JS-first (JavaScript config file, PostCSS plugin). v4 is CSS-first (CSS @theme directive, no JS config needed, uses Lightning CSS and the Rust-based Oxide engine).

2. Where do design tokens live in Tailwind v4?

In the @theme { } block in your CSS file. They become both Tailwind utilities AND CSS custom properties. You can access them via utility classes (text-primary) or CSS (var(--color-primary)).

3. Why does text-white always beat .card .title { color: gray; } in Tailwind v4?

Tailwind v4 uses @layer internally. Utilities are in the utilities layer, which has higher cascade priority than the components layer. Layer order beats specificity, so the utility always wins regardless of selector complexity.

4. How do you create a custom utility in v4 vs v3?

v3: write a JavaScript plugin using the addUtilities API. v4: write @utility name { property: value; } in CSS. The v4 approach is simpler and automatically gets all modifier support (hover, responsive, etc.).

5. What makes the Oxide engine faster than v3's build system?

Oxide is written in Rust (not JavaScript), uses Lightning CSS instead of PostCSS, auto-detects content files (no manual paths), does incremental builds, and doesn't need to parse JavaScript config. Combined, this yields roughly 10x faster build times.

Recap

  1. Tailwind v4 is CSS-first — @theme replaces tailwind.config.js
  2. @theme tokens become CSS custom properties — usable everywhere, not locked in JS
  3. @utility creates custom utilities with all modifiers for free
  4. @variant creates custom modifiers from CSS
  5. @layer powers the cascade — utilities always beat components
  6. Oxide engine (Rust + Lightning CSS) is 10x faster than v3
  7. Zero config — install, import, go. No config file, no PostCSS, no content paths
  8. Tokens are the bridge between utility classes and raw CSS
