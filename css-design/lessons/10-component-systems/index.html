A component is a contract. It defines what can vary, what's fixed, and how it interacts with other components. The CSS methodology you choose determines how that contract is expressed and enforced.

The Mental Model: CSS as an API

Think of each component's CSS as an API. It has:

  • Public interface — custom properties that consumers can override (--btn-bg, --btn-padding)
  • Internal implementation — layout, positioning, structure that shouldn't be changed
  • Variants — predefined states that modify the public interface (.btn-lg, .btn-outline)
  • Slots — areas where consumer content goes (children, named regions)

Good component CSS is like good function design. Clear inputs (custom properties, variant classes). Predictable outputs (visual appearance). No side effects (doesn't leak styles to siblings or parents). Encapsulated implementation (internal selectors don't matter to consumers).

The Three Approaches

1. BEM (Block Element Modifier)

/* Naming convention: block__element--modifier */
.card { }
.card__title { }
.card__body { }
.card__footer { }
.card--featured { }
.card--compact { }

/* Strengths:
   - Clear naming tells you what belongs to what
   - Flat specificity (everything is a single class)
   - Works without build tools
   - Self-documenting: .card__title clearly belongs to .card

   Weaknesses:
   - Verbose class names
   - Convention-based (nothing prevents .card__title outside a .card)
   - No actual encapsulation (styles CAN leak) */

2. Utility-First (Tailwind)

/* Component = composition of utilities */
<div class="flex flex-col gap-4 p-6 bg-surface rounded-lg border border-border">
  <h3 class="text-lg font-semibold text-text">...</h3>
  <p class="text-sm text-text-secondary">...</p>
</div>

/* Strengths:
   - No CSS file to maintain
   - No naming decisions
   - Co-located styles (see what it looks like in the template)
   - No unused CSS in production

   Weaknesses:
   - Long class strings
   - Repetition across instances (solved with components/partials)
   - Harder to see the "design intent"
   - Requires build tool */

3. CSS Modules / Scoped CSS

/* card.module.css */
.card { background: var(--color-surface); }
.title { font-weight: 600; }
.body { color: var(--color-text-secondary); }

/* Build tool transforms to unique class names:
   .card_a1b2c3 { background: var(--color-surface); }
   .title_d4e5f6 { font-weight: 600; }
   Guaranteed no conflicts. */

/* Strengths:
   - True encapsulation (unique class names prevent leaking)
   - Use any class names you want (no naming convention)
   - TypeScript integration (import styles and get autocomplete)

   Weaknesses:
   - Requires build tool
   - Dynamic styling requires extra work
   - Harder to override from outside the component */

There is no "best" approach. BEM works great for simpler projects and content sites. Tailwind excels for application UIs with lots of one-off variations. CSS Modules are ideal for component libraries with strict encapsulation needs. Many production codebases use a hybrid: Tailwind for layout utilities + component classes for complex components.

The Headless Component Pattern

The most influential modern pattern: separate behavior from appearance. Libraries like Radix UI and Headless UI provide accessible behavior; you provide all the styling.

/* Radix UI gives you unstyled, accessible primitives */
<Dialog.Root>
  <Dialog.Trigger className="...">Open</Dialog.Trigger>
  <Dialog.Portal>
    <Dialog.Overlay className="..." />
    <Dialog.Content className="...">
      <Dialog.Title className="...">Title</Dialog.Title>
      <Dialog.Description className="...">...</Dialog.Description>
    </Dialog.Content>
  </Dialog.Portal>
</Dialog.Root>

/* You style with whatever approach you prefer:
   - Tailwind classes directly
   - CSS custom properties
   - CSS Modules
   - data-state attributes for animations */

/* Radix uses data-state attributes for interactive states */
[data-state="open"] { animation: fade-in 200ms ease-out; }
[data-state="closed"] { animation: fade-out 150ms ease-in; }

The shadcn/ui breakthrough. shadcn/ui took the headless pattern further: instead of installing a component library (dependency), you copy component files into your project (ownership). The components use Radix for behavior + Tailwind for styling + class-variance-authority for variants. You own the code. You can modify anything. This is the dominant pattern in the React ecosystem right now.

Class Variance Authority (CVA): Typed Variants

/* CVA creates a typed variant system for Tailwind classes */
import { cva } from "class-variance-authority";

const button = cva(
  /* Base styles (always applied) */
  "inline-flex items-center justify-center rounded-md font-medium transition-colors focus-visible:outline-none focus-visible:ring-2",
  {
    variants: {
      variant: {
        default: "bg-primary text-white hover:bg-primary/90",
        outline: "border border-input bg-transparent hover:bg-accent",
        ghost: "hover:bg-accent hover:text-accent-foreground",
        destructive: "bg-destructive text-white hover:bg-destructive/90",
      },
      size: {
        sm: "h-8 px-3 text-xs",
        md: "h-10 px-4 text-sm",
        lg: "h-12 px-6 text-base",
      },
    },
    defaultVariants: {
      variant: "default",
      size: "md",
    },
  }
);

/* Usage in a React component: */
<button className={button({ variant: "outline", size: "lg" })}>
  Click me
</button>

/* CVA gives you:
   - TypeScript autocomplete for variants
   - Compound variants (variant + size combinations)
   - Default variants
   - The Tailwind approach but with structure */

Pure CSS Component Pattern (No JS Framework)

For design systems that need to work everywhere (not just React), here's the pure CSS approach using modern features:

/* button.css — framework-agnostic component */
@layer components {
  .btn {
    /* Component token API (public) */
    --btn-bg: var(--color-action);
    --btn-text: var(--color-action-text);
    --btn-padding-x: var(--spacing-element);
    --btn-padding-y: var(--spacing-tight);
    --btn-radius: var(--radius-md);
    --btn-font-size: var(--text-sm);

    /* Implementation (internal) */
    display: inline-flex;
    align-items: center;
    justify-content: center;
    gap: var(--spacing-tight);
    padding: var(--btn-padding-y) var(--btn-padding-x);
    background: var(--btn-bg);
    color: var(--btn-text);
    border: 1px solid transparent;
    border-radius: var(--btn-radius);
    font-size: var(--btn-font-size);
    font-weight: 500;
    cursor: pointer;
    transition: background 200ms ease-out, border-color 200ms ease-out;

    &:hover {
      filter: brightness(1.1);
    }

    &:focus-visible {
      outline: 2px solid var(--color-action);
      outline-offset: 2px;
    }

    &:disabled {
      opacity: 0.5;
      cursor: not-allowed;
    }
  }

  /* Variants override token API */
  .btn-outline {
    --btn-bg: transparent;
    --btn-text: var(--color-action);
    border-color: var(--color-action);
  }

  .btn-ghost {
    --btn-bg: transparent;
    --btn-text: var(--color-text);
  }

  .btn-lg {
    --btn-padding-x: var(--spacing-section);
    --btn-padding-y: var(--spacing-element);
    --btn-font-size: var(--text-base);
  }

  .btn-sm {
    --btn-padding-x: var(--spacing-tight);
    --btn-padding-y: var(--space-1);
    --btn-font-size: var(--text-xs);
  }
}

This pattern works in React, Vue, Svelte, plain HTML, or web components. Because it's just CSS classes, any framework can use it. The component token API (--btn-bg, etc.) lets consumers customize without modifying the source CSS. This is the approach used by many enterprise design systems (IBM Carbon, Salesforce Lightning).

Composition Patterns

The compound component pattern

/* A card with structured slots */
.card { }
.card-header { }
.card-body { }
.card-footer { }

/* Usage: */
<div class="card">
  <div class="card-header">...</div>
  <div class="card-body">...</div>
  <div class="card-footer">...</div>
</div>

/* Style relationships between parts using :has() */
.card:has(.card-footer) .card-body {
  padding-bottom: 0;  /* remove bottom padding when footer exists */
}
.card:has(img:first-child) {
  padding-top: 0;     /* remove top padding when image is first */
}

The data-attribute variant pattern

/* Variants via data attributes instead of classes */
.btn[data-variant="outline"] { --btn-bg: transparent; }
.btn[data-variant="ghost"] { --btn-bg: transparent; --btn-text: var(--color-text); }
.btn[data-size="sm"] { --btn-padding: 0.25rem 0.5rem; }
.btn[data-size="lg"] { --btn-padding: 0.75rem 1.5rem; }

/* Usage: */
<button class="btn" data-variant="outline" data-size="lg">
  Click me
</button>

/* Benefits:
   - Class list stays clean (just .btn)
   - Variants are explicit attributes
   - Works great with Radix/headless component data-state */

Check Your Understanding

1. What does "headless component" mean?

A headless component provides behavior and accessibility (keyboard handling, ARIA attributes, focus management) without any visual styling. The consumer provides all CSS. Libraries like Radix UI and Headless UI are headless — they handle the hard parts (accessibility, state) and let you style however you want.

2. What problem does BEM solve, and what's its main weakness?

BEM solves naming collisions and makes component ownership clear through naming conventions (.block__element--modifier). Its main weakness: it's convention-based, not enforced. Nothing prevents someone from using .card__title outside a .card, and styles can still leak.

3. What's the difference between installing a component library and the shadcn/ui approach?

Traditional libraries (Material UI, Chakra) are npm dependencies — you import and use them, but they own the code. shadcn/ui copies component files into YOUR project — you own the code and can modify anything. It's the difference between renting (dependency) and owning (source code).

4. Why are component tokens (like --btn-bg) useful?

Component tokens create an explicit API. They tell consumers "you can customize these properties" without exposing the internal CSS implementation. Variants override tokens, not CSS rules directly. This makes components predictable and safe to customize.

5. When would you use data attributes instead of classes for component variants?

Data attributes keep the class list clean (just .btn) and make variants explicit. They're especially useful with headless component libraries that already use data-state attributes. They also work well with JavaScript frameworks that bind dynamic attributes.

Recap

  1. Components are contracts — public API (tokens), internal implementation, variants, slots
  2. BEM = naming convention, Tailwind = utility composition, CSS Modules = build-time encapsulation
  3. Headless components separate behavior from styling — the dominant modern pattern
  4. shadcn/ui = ownership over dependency — copy code, don't install packages
  5. CVA provides typed variants for Tailwind-based component systems
  6. Pure CSS components with @layer work in any framework
  7. Component tokens define the customization API
  8. Data attributes are a clean alternative to modifier classes
