09 Design Tokens — The Single Source of Truth

← Back to lessons

A design token is a named value that represents a design decision. Not a color. Not a number. A decision. "The primary action color is this." "The spacing between cards is this." Tokens are the contract between design and engineering.

The Mental Model: Tokens Are Decisions, Not Values

When a designer says "use blue for buttons," they're making a decision. When that decision changes to "use purple for buttons," you don't want to find-and-replace #3b82f6 across 200 files. You want to change ONE token.

/* Without tokens: the value IS the decision */
.btn { background: #3b82f6; }
.link { color: #3b82f6; }
.badge { border-color: #3b82f6; }
/* Change requires finding every instance of #3b82f6.
   What if some #3b82f6 values are intentional but unrelated?
   You're guessing. */

/* With tokens: the decision is named */
:root { --color-action: #3b82f6; }
.btn { background: var(--color-action); }
.link { color: var(--color-action); }
.badge { border-color: var(--color-action); }
/* Change once. Everything updates. The NAME tells you WHY. */

The name IS the value. A well-named token like --color-action-primary tells you the intent. If the brand changes from blue to purple, you change the token's value, not its name. Components don't care about the actual color — they care about "the primary action color, whatever it happens to be."

The Three-Layer Token Architecture

Production design systems organize tokens into three layers. This isn't optional complexity — it's what makes themes, brands, and dark mode possible without rewriting components.

Layer 1: Primitive Tokens (the palette)

/* Raw values. No semantic meaning. Like a paint store inventory. */
:root {
  /* Colors — the full palette */
  --purple-50: #faf5ff;
  --purple-100: #f3e8ff;
  --purple-200: #e9d5ff;
  --purple-300: #d8b4fe;
  --purple-400: #c084fc;
  --purple-500: #a855f7;
  --purple-600: #9333ea;
  --purple-700: #7e22ce;
  --purple-800: #6b21a8;
  --purple-900: #581c87;
  --purple-950: #3b0764;

  --gray-50: #f9fafb;
  --gray-100: #f3f4f6;
  --gray-900: #111827;
  --gray-950: #030712;

  /* Spacing scale */
  --space-0: 0;
  --space-1: 0.25rem;
  --space-2: 0.5rem;
  --space-3: 0.75rem;
  --space-4: 1rem;
  --space-6: 1.5rem;
  --space-8: 2rem;
  --space-12: 3rem;
  --space-16: 4rem;

  /* Font sizes */
  --text-xs: 0.75rem;
  --text-sm: 0.875rem;
  --text-base: 1rem;
  --text-lg: 1.125rem;
  --text-xl: 1.25rem;
  --text-2xl: 1.5rem;
  --text-3xl: 1.875rem;
}

/* These tokens answer: "what values exist?"
   They do NOT answer: "what is this value used for?" */

Layer 2: Semantic Tokens (the decisions)

/* Map primitives to purposes. This is where design decisions live. */
:root {
  /* Surfaces */
  --color-bg: var(--gray-950);
  --color-surface: var(--gray-900);
  --color-surface-elevated: var(--gray-800);

  /* Text */
  --color-text: var(--gray-50);
  --color-text-secondary: var(--gray-400);
  --color-text-muted: var(--gray-500);

  /* Actions */
  --color-action: var(--purple-500);
  --color-action-hover: var(--purple-400);
  --color-action-text: white;

  /* Feedback */
  --color-success: var(--green-500);
  --color-error: var(--red-500);
  --color-warning: var(--amber-400);

  /* Borders */
  --color-border: var(--gray-800);
  --color-border-hover: var(--gray-700);

  /* Spacing decisions */
  --spacing-page: var(--space-8);
  --spacing-section: var(--space-12);
  --spacing-element: var(--space-4);
  --spacing-tight: var(--space-2);
}

/* Theme override — ONLY changes semantic tokens */
[data-theme="light"] {
  --color-bg: var(--gray-50);
  --color-surface: white;
  --color-text: var(--gray-900);
  --color-text-secondary: var(--gray-600);
  --color-border: var(--gray-200);
  --color-action: var(--purple-600);
  /* Notice: primitive palette doesn't change.
     Only the semantic mapping changes. */
}

Layer 3: Component Tokens (the interface)

/* Each component exposes its own token API */
.card {
  --card-bg: var(--color-surface);
  --card-border: var(--color-border);
  --card-padding: var(--spacing-element);
  --card-radius: 8px;
  --card-title-color: var(--color-text);
  --card-body-color: var(--color-text-secondary);

  background: var(--card-bg);
  border: 1px solid var(--card-border);
  border-radius: var(--card-radius);
  padding: var(--card-padding);
}

.card h3 { color: var(--card-title-color); }
.card p { color: var(--card-body-color); }

/* Variants override component tokens */
.card-featured {
  --card-bg: var(--color-action);
  --card-title-color: white;
  --card-body-color: rgba(255, 255, 255, 0.8);
  --card-border: transparent;
}

/* Components don't know about themes.
   They reference semantic tokens.
   Semantic tokens change per theme.
   Components update automatically. */

The three layers create indirection. Components reference semantic tokens. Semantic tokens reference primitives. When you add a "dark mode," you only change the semantic layer. When you rebrand, you might change some primitives. Components never need to change. This indirection is what makes a design system maintainable at scale.

Multi-Brand Token Systems

The real power of tokens shows when one codebase serves multiple brands:

/* Brand A: News Network */
[data-brand="news"] {
  --color-action: var(--red-600);
  --color-action-hover: var(--red-500);
  --color-bg: var(--gray-950);
  --color-accent: var(--orange-500);
}

/* Brand B: Sports Network */
[data-brand="sports"] {
  --color-action: var(--green-600);
  --color-action-hover: var(--green-500);
  --color-bg: var(--gray-950);
  --color-accent: var(--emerald-400);
}

/* Brand C: Entertainment */
[data-brand="entertainment"] {
  --color-action: var(--purple-600);
  --color-action-hover: var(--purple-500);
  --color-bg: var(--gray-950);
  --color-accent: var(--pink-400);
}

/* Every component uses the SAME token names.
   Switch brands by changing one data attribute on <body>.
   <body data-brand="news"> → red buttons, orange accents
   <body data-brand="sports"> → green buttons, emerald accents
   ZERO component code changes. */

Broadcast connection. This is exactly how TV networks handle election night. The same lower-third component displays in the network's brand colors. Switch from the primary channel to the sports channel, and the lower third turns green. Same HTML, same CSS, different tokens. You'll build this in Lesson 11.

The W3C Design Tokens Format

The W3C is standardizing a JSON format for design tokens, enabling tokens to flow from design tools (Figma) to code (CSS, iOS, Android):

/* tokens.json — W3C Design Tokens Community Group format */
{
  "color": {
    "action": {
      "$value": "{color.purple.500}",
      "$type": "color",
      "$description": "Primary action color for buttons and links"
    },
    "bg": {
      "$value": "{color.gray.950}",
      "$type": "color"
    }
  },
  "spacing": {
    "element": {
      "$value": "1rem",
      "$type": "dimension"
    }
  }
}

/* Tools like Style Dictionary or Tokens Studio
   transform this JSON into:
   - CSS custom properties
   - Tailwind @theme values
   - iOS Swift constants
   - Android XML resources
   - Design tool (Figma) variables
   One source → all platforms */

Token Naming Conventions

/* Common naming pattern: category-property-variant-state */

/* Category: what kind of token */
--color-*
--spacing-*
--font-*
--radius-*
--shadow-*

/* Property: what role within the category */
--color-bg-*
--color-text-*
--color-border-*
--color-action-*

/* Variant: which version */
--color-bg-primary
--color-bg-secondary
--color-bg-elevated

/* State: interactive state */
--color-action-hover
--color-action-active
--color-action-disabled

/* Full examples */
--color-bg-primary
--color-text-secondary
--color-action-hover
--color-border-focus
--spacing-section-x
--spacing-element-y
--font-size-heading-lg
--radius-card
--shadow-elevated

Check Your Understanding

1. What's the difference between a primitive token and a semantic token?

Primitive tokens are raw values with no semantic meaning (--purple-500: #a855f7). Semantic tokens map primitives to purposes (--color-action: var(--purple-500)). Semantic tokens answer "what is this for?" while primitives answer "what values exist?"

2. When you implement dark mode, which token layer changes?

Only the semantic layer. You remap semantic tokens to different primitives. For example, --color-bg might point to --gray-950 in dark mode and --gray-50 in light mode. Primitives and component tokens stay the same.

3. What problem do component tokens solve?

Component tokens create an explicit API for each component. They document what can be customized (e.g., --card-bg, --card-padding) and what can't. Variants override component tokens rather than overriding CSS properties directly, making the component's interface predictable.

4. How does a multi-brand token system work?

Each brand defines its own set of semantic tokens (usually via a data attribute or class on the root element). Components reference the same token names. Switching brands changes which semantic tokens are active, and all components update automatically. Zero component code changes.

5. Why use three layers instead of just one?

Indirection. With one layer, changing from blue to purple means changing every component. With three layers, you change ONE semantic token and everything updates. Adding dark mode requires only a new semantic layer mapping. Adding a brand requires only new semantic overrides. Components never change.

Exercise: Build a Multi-Theme Token System

  1. Define primitive tokens: a full color palette (3 hues, 10 shades each), spacing scale, font sizes
  2. Create semantic tokens for dark and light modes
  3. Create component tokens for a card, button, and input
  4. Build those three components using only token references
  5. Add a brand override that changes the accent color and test switching
  6. Bonus: Export your tokens as a JSON file following the W3C format

Recap

  1. Tokens are named decisions, not just variables with values
  2. Three layers: primitive, semantic, component — each serves a different purpose
  3. Themes change the semantic layer — components don't need to change
  4. Multi-brand = multiple semantic overrides sharing the same components
  5. Component tokens define the customization API for each component
  6. W3C Design Tokens format enables token flow from design tools to all platforms
  7. Naming convention: category-property-variant-state
