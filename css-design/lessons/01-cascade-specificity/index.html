01 Cascade & Specificity — How CSS Actually Decides

← Back to lessons

Every CSS bug you've ever had was a cascade bug. The cascade is the algorithm CSS uses to decide which styles win when multiple rules target the same element. Master it, and CSS stops feeling random.

The Mental Model: CSS Is a Courtroom

Think of CSS like a courtroom where every property declaration is a lawyer arguing for their value. When two lawyers both argue that color should be different values, the judge (the cascade algorithm) uses a strict hierarchy to decide who wins:

  1. Origin & Importance — Where did the rule come from? (User agent, author, user? Does it have !important?)
  2. Inline styles — Styles in the style="" attribute beat external stylesheets
  3. Layers — @layer rules have explicit ordering (new in modern CSS)
  4. Specificity — More specific selectors win over less specific ones
  5. Source order — When everything else is equal, the last rule wins

The cascade is a tiebreaker chain. It only moves to the next step when the current step is a tie. Most of your daily CSS decisions are resolved at steps 4 and 5 — specificity and source order. But understanding the full chain prevents entire categories of bugs.

Specificity: The Scoring System

Specificity isn't magic. It's a three-number score: (A, B, C).

CategoryScore SlotExamples
IDsA (leftmost, heaviest)#header, #main
Classes, attributes, pseudo-classesB (middle).nav, [type="text"], :hover
Elements, pseudo-elementsC (rightmost, lightest)div, p, ::before

The comparison works like version numbers. (1, 0, 0) beats (0, 99, 99) because the leftmost slot wins first. A single ID outweighs any number of classes.

Scoring Examples

Lower specificity
/* (0, 0, 1) — one element */
p { color: gray; }

/* (0, 1, 0) — one class */
.intro { color: blue; }

/* (0, 1, 1) — one class + one element */
p.intro { color: green; }
Higher specificity
/* (0, 2, 0) — two classes */
.card .title { color: purple; }

/* (1, 0, 0) — one ID */
#hero { color: red; }

/* (1, 1, 1) — ID + class + element */
#hero .title h2 { color: orange; }

Tailwind connection. Tailwind utilities are all single classes, so they all have specificity (0, 1, 0). That's why Tailwind works — every utility has the same weight, and the last one in the CSS file wins. No specificity wars. When you write raw CSS with nested selectors like .sidebar .nav .link.active, you're at (0, 3, 0) and now need even more specific selectors to override it. That's the specificity trap Tailwind avoids entirely.

The Specificity Trap

Here's the pattern that ruins codebases:

/* Developer A writes: */
.card .title { color: white; }    /* (0, 2, 0) */

/* Developer B needs to override: */
.title { color: gray; }           /* (0, 1, 0) — LOSES */

/* Developer B escalates: */
.card .title.muted { color: gray; }  /* (0, 3, 0) — wins... for now */

/* Developer C needs to override B: */
.card .title.muted.special { ... }   /* (0, 4, 0) — the arms race begins */

/* Developer D gives up: */
.title { color: gray !important; }   /* nuclear option */

This is why Tailwind exists. By keeping every utility at (0, 1, 0) and relying on source order, Tailwind sidesteps the entire specificity arms race. But when you write custom CSS — which you'll need to for design systems and broadcast graphics — you need to manage specificity yourself.

Inheritance: The Silent Style Propagator

Some CSS properties flow from parent to child automatically. Others don't. The split is intuitive once you see the pattern:

Inherited (text-related)
color
font-family
font-size
font-weight
line-height
letter-spacing
text-align
visibility
cursor
list-style
NOT inherited (box-related)
background
border
margin
padding
width / height
display
position
overflow

The rule of thumb: If it's about how text looks, it inherits. If it's about the box a thing sits in, it doesn't. You wouldn't want every child <div> to inherit its parent's border or padding.

Controlling Inheritance

/* Force inheritance */
.child { color: inherit; }

/* Reset to initial browser value */
.child { color: initial; }

/* Act like the property was never set
   (inherits if inheritable, initial if not) */
.child { color: unset; }

/* Reset ALL properties on an element */
.clean-slate { all: unset; }

The all: unset trick. When building components for a design system, all: unset strips away all inherited and default styles. It's a clean-room reset. Use it on buttons, links, and interactive elements where browser defaults fight your design.

@layer: The Modern Solution to Cascade Management

CSS Cascade Layers (@layer) let you explicitly control which groups of styles beat other groups, regardless of specificity. This is the modern solution to the specificity trap.

/* Declare layer order — first declared = lowest priority */
@layer reset, base, components, utilities;

@layer reset {
  *, *::before, *::after { box-sizing: border-box; margin: 0; }
}

@layer base {
  h1 { font-size: 2rem; }
  a { color: var(--accent); }
}

@layer components {
  .card { background: var(--surface); border: 1px solid var(--border); }
  .card .title { font-weight: 600; }  /* (0, 2, 0) — but layer order beats specificity */
}

@layer utilities {
  .text-muted { color: var(--text-muted); }  /* (0, 1, 0) — but in highest layer, so wins */
}

Layers beat specificity. A utility with specificity (0, 1, 0) in the utilities layer beats a component rule with specificity (0, 5, 0) in the components layer. This is exactly how Tailwind v4 works internally — it puts its utilities in the highest layer so they always win.

The Complete Decision Tree

When the browser encounters conflicting declarations, it walks this tree top to bottom. The first step that produces a winner is the answer:

1. Is one !important and the other isn't?
   → !important wins (but !important has its own reversed layer order)

2. Are they in different layers?
   → Higher-priority layer wins

3. Is one an inline style (style="") and the other isn't?
   → Inline wins

4. Do they have different specificity?
   → Higher specificity wins

5. Everything else is equal?
   → Last one in source order wins

In practice, 90% of CSS conflicts are resolved by specificity or source order. But when your styles "don't work" and you can't figure out why, walk through this tree step by step. The answer is always in here.

Practical Patterns for Clean Specificity

Pattern 1: Keep specificity flat

/* BAD — specificity escalation */
.sidebar .nav .link { color: gray; }
.sidebar .nav .link.active { color: white; }
.sidebar .nav .link.active:hover { color: purple; }

/* GOOD — flat specificity */
.nav-link { color: gray; }
.nav-link--active { color: white; }
.nav-link--active:hover { color: purple; }

Pattern 2: Use :where() to zero out specificity

/* :where() has 0 specificity — great for defaults */
:where(h1, h2, h3) { margin-top: 1.5em; }

/* Any class can override this easily */
.compact-title { margin-top: 0.5em; }  /* wins because (0,1,0) > (0,0,0) */

Pattern 3: Use :is() for grouping (takes highest specificity)

/* :is() takes the specificity of its most specific argument */
:is(.card, #hero) .title { color: white; }
/* This has specificity (1, 0, 0) because #hero is in the :is() */
/* Use :is() for convenience, but be aware of this gotcha */

The :is() vs :where() trap. :is(.foo, #bar) takes the specificity of #bar — (1, 0, 0) — even when matching .foo. Use :where() when you want zero-specificity grouping (like resets and defaults). Use :is() when you want the grouping convenience and are okay with the specificity.

Check Your Understanding

1. What is the specificity of .card .title:hover?

(0, 3, 0) — three entries in the B slot: two classes (.card, .title) and one pseudo-class (:hover).

2. Which wins: #header a or .nav .link.active:hover?

#header a wins. Its specificity is (1, 0, 1). The other selector is (0, 3, 0). The ID in slot A beats any number of classes in slot B.

3. What is the specificity of :where(.card, #hero) .title?

(0, 1, 0). The :where() pseudo-class has zero specificity — it's as if the contents aren't there. Only .title contributes.

4. Both rules target the same element with the same specificity. Which color wins?

.title { color: red; }
.heading { color: blue; }
blue wins. Same specificity = last rule in source order wins. This is step 5 in the cascade.

5. A style in the components layer has specificity (0, 3, 0). A style in the utilities layer has specificity (0, 1, 0). Layers are declared as @layer components, utilities;. Which wins?

The utilities layer wins. Layer order beats specificity. Since utilities is declared after components, it has higher priority regardless of the specificity scores.

6. Does background-color inherit from parent to child?

No. Box-related properties like background, border, margin, and padding do not inherit. Only text-related properties (color, font-family, etc.) inherit by default.

Exercise: Debug the Cascade

Open your browser DevTools on any page of this platform and try this:

  1. Find an element where multiple CSS rules compete for the same property
  2. In the Styles panel, look for struck-through declarations — those are the rules that lost
  3. For each losing rule, identify WHY it lost: specificity, source order, or layer?
  4. Try adding :where() around a selector in the DevTools to see its specificity drop to zero
  5. Try adding !important to a losing rule and watch it jump to the top

The goal: build muscle memory for reading the cascade in DevTools. This is how you debug CSS in production.

Recap

  1. The cascade is a tiebreaker chain — importance, layers, inline, specificity, source order
  2. Specificity is (A, B, C) — IDs, classes/pseudo-classes, elements. Left slot beats right.
  3. Keep specificity flat — use single classes, avoid deep nesting
  4. @layer controls cascade order — layer order beats specificity
  5. :where() zeroes specificity — use it for defaults and resets
  6. Inheritance flows down — text properties inherit, box properties don't
  7. !important is a code smell — it usually means your specificity architecture is broken
