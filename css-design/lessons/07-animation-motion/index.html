07 Animation & Motion — Making Interfaces Feel Alive

← Back to lessons

Animation is communication. Every transition, every keyframe, every scroll-driven effect should communicate something — a state change, a spatial relationship, a response to user input. Motion without meaning is just distraction.

The Mental Model: Motion As Language

Good animation follows the same principles as good writing: clarity, purpose, and economy. Every animation answers one of these questions:

  • What changed? — Transition on a button's background tells users "you're hovering"
  • Where did it come from? — A slide-in tells users "this came from the right"
  • What's the relationship? — A shared-element transition shows "this card expanded into this page"
  • What should I pay attention to? — A pulse animation says "look here"

The 200ms rule. Most UI transitions should be 150-300ms. Under 100ms feels instant (no animation needed). Over 400ms feels sluggish. The sweet spot for most hover states and small transitions is 200ms. Larger elements (modals, page transitions) can be 300-500ms because the eye needs more time to track larger movements.

Transitions: Animate Property Changes

Transitions animate from one state to another when a property changes. They're the most common animation in UI development.

/* Basic transition */
.btn {
  background: var(--accent);
  transition: background 200ms ease-out;
}
.btn:hover {
  background: var(--accent-hover);
}

/* Multiple properties */
.card {
  transition: transform 200ms ease-out,
              box-shadow 200ms ease-out,
              border-color 300ms ease-out;
}
.card:hover {
  transform: translateY(-2px);
  box-shadow: 0 8px 25px rgba(0, 0, 0, 0.15);
  border-color: var(--accent);
}

/* Transition shorthand: property duration timing-function delay */
transition: all 200ms ease-out 0ms;

/* Tailwind: transition-all duration-200 ease-out */

The performance rule. Only animate transform and opacity for smooth 60fps animations. Everything else triggers layout or paint, which can cause frame drops. If you need to "animate" width, use transform: scaleX() instead. If you need to "move" an element, use transform: translateX() instead of left.

Easing: The Personality of Motion

Easing controls the acceleration curve of an animation. It's what makes motion feel natural vs robotic.

/* Built-in easing functions */
transition-timing-function: linear;       /* constant speed (robotic) */
transition-timing-function: ease;         /* slow start, fast middle, slow end (default) */
transition-timing-function: ease-in;      /* slow start (things entering) */
transition-timing-function: ease-out;     /* slow end (things arriving — MOST USED) */
transition-timing-function: ease-in-out;  /* slow both ends */

/* Custom cubic-bezier (from cubic-bezier.com) */
transition-timing-function: cubic-bezier(0.16, 1, 0.3, 1);
/* This is an "expo out" curve — fast start, very gentle landing.
   It's the easing this platform uses (var(--ease-out-expo)). */

/* Spring-like easing with linear() — new in CSS */
transition-timing-function: linear(
  0, 0.006, 0.025, 0.058, 0.104, 0.163, 0.234, 0.316,
  0.408, 0.509, 0.615, 0.726, 0.838, 0.946, 1.043,
  1.120, 1.175, 1.205, 1.212, 1.199, 1.171, 1.133,
  1.091, 1.048, 1.010, 0.979, 0.956, 0.942, 0.935,
  0.937, 0.944, 0.955, 0.968, 0.981, 0.992, 1.000
);

ease-out for entering, ease-in for leaving. When something appears (dropdown opening, modal entering), use ease-out — it decelerates into its final position, feeling like it's arriving. When something disappears, use ease-in — it accelerates away. This matches real-world physics where objects slow down as they stop.

Keyframe Animations: Multi-Step Motion

Transitions go from A to B. Keyframes go from A to B to C to D... with full control at each step.

/* Define the animation */
@keyframes slide-in {
  from {
    opacity: 0;
    transform: translateY(20px);
  }
  to {
    opacity: 1;
    transform: translateY(0);
  }
}

/* Apply it */
.card {
  animation: slide-in 500ms ease-out forwards;
  /* name | duration | easing | fill-mode */
}

/* Multi-step animation */
@keyframes pulse {
  0%   { transform: scale(1); opacity: 1; }
  50%  { transform: scale(1.05); opacity: 0.8; }
  100% { transform: scale(1); opacity: 1; }
}

.live-indicator {
  animation: pulse 2s ease-in-out infinite;
}

/* Staggered entrance (combine with custom properties) */
.card {
  animation: slide-in 400ms ease-out forwards;
  animation-delay: calc(var(--index) * 80ms);
  opacity: 0; /* start hidden before animation runs */
}

/* Set --index per card: style="--index: 0", style="--index: 1", etc.
   Each card enters 80ms after the previous one. */

The forwards fill mode. By default, after a keyframe animation ends, the element snaps back to its pre-animation state. animation-fill-mode: forwards keeps it in the final keyframe state. Almost always what you want for entrance animations.

Scroll-Driven Animations: No JavaScript Needed

CSS can now tie animations to scroll position. Previously this required JavaScript scroll listeners.

/* Progress bar that fills as you scroll the page */
.progress-bar {
  position: fixed;
  top: 0;
  left: 0;
  height: 3px;
  background: var(--accent);
  transform-origin: left;
  animation: grow-bar linear;
  animation-timeline: scroll();
}

@keyframes grow-bar {
  from { transform: scaleX(0); }
  to   { transform: scaleX(1); }
}

/* Reveal elements as they enter the viewport */
.reveal {
  animation: fade-in linear;
  animation-timeline: view();
  animation-range: entry 0% entry 100%;
}

@keyframes fade-in {
  from { opacity: 0; transform: translateY(30px); }
  to   { opacity: 1; transform: translateY(0); }
}

/* Parallax effect */
.hero-bg {
  animation: parallax linear;
  animation-timeline: scroll();
}

@keyframes parallax {
  to { transform: translateY(-30%); }
}

Scroll-driven animations run on the compositor thread. Unlike JavaScript scroll listeners (which run on the main thread and can jank), CSS scroll-driven animations are handled by the browser's compositor. They're guaranteed smooth 60fps, even during heavy JavaScript execution. This is a game-changer for performance-sensitive scroll effects.

View Transitions API: Page-Level Animation

The View Transitions API animates between DOM states — including full page navigations. It creates smooth transitions between "views" without SPA frameworks.

/* Enable View Transitions for same-document navigations */
@view-transition {
  navigation: auto;
}

/* Name elements that should animate between states */
.card-image {
  view-transition-name: card-hero;
}

/* Customize the transition animation */
::view-transition-old(card-hero) {
  animation: fade-out 300ms ease-in;
}
::view-transition-new(card-hero) {
  animation: fade-in 300ms ease-out;
}

/* Shared element transition: an element "morphs" from one page to another */
/* Card thumbnail on list page: */
.card img { view-transition-name: hero-image; }
/* Hero image on detail page: */
.detail-hero { view-transition-name: hero-image; }
/* The browser automatically animates between them
   — position, size, and content. Like magic. */

View Transitions replace what React Router + Framer Motion do. Shared element transitions, page crossfades, and morphing animations that previously required complex JavaScript libraries are now possible with a few CSS rules. This is still a newer API, but browser support is growing rapidly.

Motion and Accessibility

/* ALWAYS respect prefers-reduced-motion */
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}

/* Or selectively reduce (preferred approach) */
@media (prefers-reduced-motion: reduce) {
  .hero-animation { animation: none; }
  .card { transition-duration: 0ms; }
  /* Keep functional transitions (like focus indicators) */
  :focus-visible { transition: outline-offset 100ms; }
}

Motion sensitivity is a medical condition. Vestibular disorders affect 35% of adults over 40. Parallax scrolling, spinning animations, and auto-playing video can trigger nausea, vertigo, and migraines. prefers-reduced-motion is not optional — it's an accessibility requirement. Always include it.

Check Your Understanding

1. Why should you animate transform and opacity instead of width and top?

transform and opacity are handled by the GPU compositor thread and don't trigger layout recalculation. width, height, top, left etc. trigger layout for every frame, causing performance jank.

2. When should you use ease-out vs ease-in?

ease-out for elements entering/arriving (decelerates to rest). ease-in for elements leaving/disappearing (accelerates away). This matches natural physics.

3. What does animation-fill-mode: forwards do?

It keeps the element in its final keyframe state after the animation ends. Without it, the element snaps back to its pre-animation state when the animation completes.

4. How are scroll-driven animations more performant than JS scroll listeners?

Scroll-driven animations run on the browser's compositor thread, not the main JavaScript thread. They're processed at the GPU level and guaranteed to hit 60fps even if JavaScript is busy with other tasks.

5. What's the purpose of the view-transition-name property?

It identifies elements that should animate between DOM states. When two elements on different pages share the same view-transition-name, the browser creates a smooth morphing animation between their positions, sizes, and appearances.

Recap

  1. Motion is communication — every animation should answer "what changed?"
  2. 150-300ms for UI transitions — the sweet spot for interactive feedback
  3. Only animate transform and opacity for smooth 60fps performance
  4. ease-out for entering, ease-in for leaving
  5. Keyframes for multi-step, transitions for A-to-B state changes
  6. Scroll-driven animations replace JavaScript scroll listeners with GPU-powered CSS
  7. View Transitions API enables page-level animations natively
  8. prefers-reduced-motion is mandatory — motion sensitivity is a medical condition
